
## SSE가 실제로 필요한 MCP 서버 종류

Jira는 맞아, 전형적인 **요청-응답형**이라 SSE의 실시간 푸시가 크게 의미없어.

SSE가 진짜 필요한 건 이런 서버들이야:

**① 파일시스템 감시**

```
로컬 파일 MCP Server
→ 파일이 변경되면 Claude에게 즉시 알림
"README.md가 수정됐어, 다시 읽어"
```

**② 데이터베이스 변경 감지**

```
DB MCP Server (with CDC*)
→ 테이블에 새 row 들어오면 즉시 푸시
"orders 테이블에 새 주문 들어왔어"

*CDC: Change Data Capture
```

**③ 로그 스트리밍**

```
서버 로그 MCP Server
→ 에러 로그 실시간 스트리밍
"지금 이런 에러 계속 찍히고 있어"
```

**④ 메시지 큐 / 이벤트 브로커**
```
Kafka / RabbitMQ MCP Server
→ 새 메시지 도착하면 즉시 Claude에게 전달
```

**⑤ 협업 도구 실시간 알림**
```
Slack MCP Server
→ 특정 채널에 메시지 오면 Claude에게 푸시
(Jira도 webhook 기반으로 만들면 가능은 해)
```

---

## 정리하면

| 구분          | 방식                   | 예시                 |
| ----------- | -------------------- | ------------------ |
| **요청-응답형**  | stdio / HTTP 단순 POST | Jira, Gmail, DB 조회 |
| **실시간 푸시형** | SSE 필요               | 파일 감시, 로그, 메시지 큐   |

Jira MCP가 SSE를 쓰더라도 실제로는 푸시 기능을 거의 안 쓰고, **전송 방식으로 HTTP+SSE 스펙을 채택했을 뿐**인 경우가 대부분이야. 스펙이 SSE를 포함하고 있어도 실제 푸시 이벤트를 안 보내면 그냥 요청-응답이랑 동일하게 동작하거든.
## 먼저 배경부터: 왜 변경이 필요했나

HTTP + SSE 방식의 구조를 다시 보면:

```
Claude                    MCP Server
  │                           │
  ├─── GET /sse ─────────────►│  (1) 먼저 SSE 채널 오픈
  │◄── 연결 유지 ──────────────│
  │                           │
  ├─── POST /messages ───────►│  (2) 요청 전송
  │◄── SSE 채널로 응답 ────────│  (3) 응답은 SSE로 옴
```

**문제 1: 무조건 SSE 연결을 먼저 열어야 함**

```
POST /messages를 보내기 전에
반드시 GET /sse가 살아있어야 함
→ 순서 의존성 발생
→ SSE 끊기면 POST도 못 보냄
```

**문제 2: 서버리스 환경에서 불가**

```
AWS Lambda / Vercel 같은 서버리스는
요청 하나 처리하고 프로세스가 죽음

GET /sse로 연결 유지?
→ 불가능. 연결 유지할 프로세스가 없음
```

**문제 3: 단순한 요청도 SSE 채널 필요**

```
"현재 시각 알려줘" 같은 단순 툴도
SSE 연결 없으면 동작 안 함
→ 오버헤드
```

## Streamable HTTP 핵심 아이디어

**"POST 하나로 다 해결하자. 응답 방식은 서버가 상황에 따라 결정"**

```
엔드포인트: POST /mcp 하나만 존재

서버가 응답할 때:
- 단순하면  → 그냥 JSON으로 응답
- 복잡하면  → SSE 스트림으로 전환해서 응답
```

---

## 케이스별 동작 방식

### Case 1: 단순 요청 (JSON 응답)

```
Claude ──► POST /mcp
           Content-Type: application/json
           {
             "method": "tools/list"
           }

Server ◄── 200 OK
           Content-Type: application/json
           {
             "result": { "tools": [...] }
           }
```

그냥 일반 REST API랑 완전히 동일해.

---

### Case 2: 스트리밍 필요한 요청 (SSE 응답)

```
Claude ──► POST /mcp
           Content-Type: application/json
           Accept: text/event-stream        ← "스트림도 받을 수 있어"
           {
             "method": "tools/call",
             "params": { "name": "analyze_large_log" }
           }

Server ◄── 200 OK
           Content-Type: text/event-stream  ← 서버가 스트림 선택
           
           data: {"type": "progress", "value": "파일 읽는 중..."}
           
           data: {"type": "progress", "value": "분석 중... 30%"}
           
           data: {"type": "progress", "value": "분석 중... 80%"}
           
           data: {"type": "result", "content": [{"type":"text","text":"완료"}]}
           
           data: [DONE]
```

**같은 POST /mcp 엔드포인트**인데, 서버가 응답 Content-Type만 바꾼 거야.

---

### Case 3: 서버가 먼저 보내야 할 때 (서버 → 클라이언트 푸시)

SSE 방식에서 서버 푸시가 필요한 경우 어떻게 해결하냐:

```
Claude ──► POST /mcp
           {
             "method": "initialize"
           }

Server ◄── 200 OK
           Content-Type: text/event-stream
           
           // 초기화 응답
           data: {"id":1, "result": {"capabilities": {...}}}
           
           // 연결 유지하면서 서버가 나중에 푸시
           data: {"method": "notifications/tools/list_changed"}
           
           data: {"method": "notifications/resources/updated", 
                  "params": {"uri": "file:///log/app.log"}}
```

initialize 요청의 응답을 SSE 스트림으로 열어놓으면, 그 스트림이 서버 푸시 채널이 되는 거야.

## HTTP + SSE vs Streamable HTTP 구조 비교

```
[HTTP + SSE]

연결 1: GET /sse ──────────────────────────────► (항상 열려있어야 함)
                 ◄── 서버 푸시 ─────────────────
연결 2: POST /messages ──► (요청)
                       ◄── (응답은 연결 1로 옴)  ← 요청이랑 응답이 다른 채널


[Streamable HTTP]

연결 1: POST /mcp ──► (요청)
                 ◄── JSON or SSE stream          ← 요청이랑 응답이 같은 채널
(필요하면 새 연결)
연결 2: POST /mcp ──► (다른 요청)
                 ◄── JSON or SSE stream
```

---

## 클라이언트 입장에서 Accept 헤더가 핵심

```http
POST /mcp HTTP/1.1
Content-Type: application/json
Accept: application/json, text/event-stream
```

`Accept`에 둘 다 적는 게 포인트야.

```
서버 응답 시나리오:

Content-Type: application/json     → 클라이언트: "아 단순 응답이네" → JSON 파싱
Content-Type: text/event-stream    → 클라이언트: "아 스트림이네"    → SSE 파싱
```

클라이언트가 **응답 Content-Type 보고 파싱 방식을 동적으로 결정**하는 구조야.

## 서버리스에서 왜 되는가

```
[Lambda 실행 흐름]

요청 도착 → Lambda 프로세스 시작
         → POST /mcp 처리
         → 단순하면: JSON 반환 후 프로세스 종료  ✅
         → 스트리밍: SSE 스트림 다 보내고 종료  ✅

GET /sse처럼 연결을 "유지"할 필요가 없음
각 요청이 독립적으로 완결됨
```

## 세 방식 최종 비교

|구분|stdio|HTTP + SSE|Streamable HTTP|
|---|---|---|---|
|**엔드포인트**|없음|GET /sse + POST /messages|POST /mcp 하나|
|**응답 채널**|stdout|SSE 채널 (별도)|요청과 동일 채널|
|**서버 푸시**|가능|SSE 채널|SSE 스트림 유지|
|**서버리스**|불가|불가|가능|
|**단순 요청**|가능|SSE 필요|JSON만으로 가능|
|**권장 상황**|로컬 툴|구버전 호환|신규 원격 서버|

---

## 한 줄 요약

```
stdio          → 프로세스 파이프
HTTP + SSE     → 채널 2개 (요청/응답이 다른 선)
Streamable HTTP → 채널 1개 (요청/응답이 같은 선, 응답 형식만 유동적)
```

## 성능 관점 비교 

**HTTP + SSE가 유리한 이유:**

```
SSE 채널이 항상 열려있음
→ 서버 푸시할 때 새 연결 수립 비용 없음
→ TCP 핸드셰이크, TLS 협상 없음
→ 지연시간(latency) 최소화
```

**Streamable HTTP의 비용:**

```
서버가 클라이언트에게 먼저 보내야 할 때:

방법 1: 기존 POST 응답 스트림을 오래 유지
→ 연결 유지 비용 발생

방법 2: 클라이언트가 polling
→ 불필요한 요청 발생
```

---

## 실제로 차이가 나는 케이스

```
실시간성이 중요한 상황:
(로그 스트리밍, 실시간 알림, 주식 데이터)

HTTP + SSE:
서버 이벤트 발생 → 즉시 푸시  ✅

Streamable HTTP:
서버 이벤트 발생 → 열린 스트림 있으면 푸시
                → 없으면? 다음 요청 올 때까지 대기  ❌
```

---

## 결국 설계 목적이 달라

|구분|HTTP + SSE|Streamable HTTP|
|---|---|---|
|**최적화 대상**|실시간 푸시 성능|배포 환경 유연성|
|**적합한 서버**|상시 실행 서버|서버리스, 엣지|
|**적합한 용도**|실시간 스트리밍|일반적인 툴 호출|

---

## MCP 공식 입장도 이걸 인정해

Streamable HTTP가 "권장"인 이유는 성능이 아니라:

```
대부분의 MCP 서버는 Jira, Gmail 같은
요청-응답형 툴이 주력

→ 실시간 푸시가 거의 없음
→ SSE 상시 연결 유지가 오히려 낭비
→ 서버리스 배포 가능한 게 더 실용적
```

실시간성이 핵심인 서버라면 오히려 HTTP + SSE가 더 나은 선택이야.
