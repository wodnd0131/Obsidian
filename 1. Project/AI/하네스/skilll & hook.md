## 정의

### Skill

리포지토리에 내장된 **실행 가능한 스크립트 단위**야. 에이전트가 "도구"처럼 호출할 수 있는 캡슐화된 능력.

원문에서 Codex가 `gh`, 로컬 스크립트, "repository-native skills"를 직접 사용한다고 했는데 그게 이거야. 사람이 CLI에 복붙하지 않아도 에이전트가 스스로 호출함.

```
Skill의 특징:
- 리포지토리 안에 버전 관리됨
- 입력/출력이 명확히 정의됨
- 에이전트가 이름으로 호출 가능
- 실행 결과가 에이전트 컨텍스트로 돌아옴
```

### MCP Tool (Hook)

에이전트가 **외부 시스템과 상호작용하는 표준 인터페이스**야. Anthropic이 정의한 프로토콜로, 에이전트가 파일시스템/DB/API/브라우저 등을 일관된 방식으로 다룰 수 있게 함.

```
MCP의 특징:
- 에이전트 런타임에 연결되는 서버 형태
- Tool 호출 → 결과 반환 구조
- 외부 시스템을 에이전트 컨텍스트 안으로 끌어옴
- "에이전트가 볼 수 없는 건 존재하지 않는다"를 해결하는 수단
```

## 하네스에서의 역할

원문의 핵심 문제의식이 이거잖아.

> **"에이전트가 컨텍스트에서 접근할 수 없는 것은 존재하지 않는 것"**

Skill과 MCP는 이 문제를 서로 다른 방향에서 해결해.

```
┌─────────────────────────────────────────────────────────┐
│  Skill    → 에이전트의 "행동 능력" 확장                  │
│             "무엇을 할 수 있는가"                        │
│                                                         │
│  MCP Tool → 에이전트의 "인식 범위" 확장                  │
│             "무엇을 볼 수 있는가"                        │
└─────────────────────────────────────────────────────────┘
```

구체적으로:

**Skill이 없으면** — 에이전트가 린트 실행, 테스트 실행, PR 생성을 하려면 매번 사람이 명령어를 복붙해줘야 함. 자율성 0.

**MCP가 없으면** — 에이전트가 빌드 실패 로그, 테스트 결과, DB 상태를 볼 수 없음. 피드백 루프가 끊김.

## 현재 계획에서 담당할 수 있는 위치

```
Phase 1 (기반): 패키지 구조 + AGENTS.md + docs/
Phase 2 (울타리): Pre-commit Hook + 커스텀 린터
Phase 3 (검증): TDD 구조 + ArchUnit + CI Gate
Phase 4 (자동화): Garbage Collection + doc-gardening
```

각 Phase에 매핑하면 이래.

### Phase 2에 Skill 진입

```
scripts/skills/
  ├── lint.sh          # 린트 실행 + 결과를 에이전트 readable 포맷으로 출력
  ├── test.sh          # 테스트 실행 + 실패 위치/원인 구조화 출력
  ├── build-check.sh   # 빌드 검증
  └── arch-check.sh    # 아키텍처 경계 검사
```

Pre-commit Hook이 이 Skill들을 호출하는 구조가 되는 거야. Hook은 트리거, Skill은 실제 실행 단위로 분리됨.

```
커밋 시도
  → Pre-commit Hook 발동
    → skills/lint.sh 호출
      → 실패 시 에러 메시지 + 수정 지침 출력 (에이전트가 읽는 포맷)
      → 커밋 차단
```

### Phase 3에 MCP 진입

CI가 실패했을 때 에이전트가 그 결과를 **직접 읽고 수정**할 수 있으려면 MCP가 필요해.

```
MCP Server 구성:
  ├── filesystem MCP   → 빌드 로그, 테스트 결과 파일 읽기
  ├── github MCP       → PR 상태, CI 결과, 리뷰 코멘트 읽기/쓰기
  └── (선택) database MCP → 로컬 테스트 DB 상태 확인
```

```
CI Gate 실패
  → MCP(github)로 실패 로그 수집
  → MCP(filesystem)로 관련 파일 읽기
  → 에이전트가 원인 파악 + Skill(test.sh) 로컬 재실행
  → 수정 후 PR 업데이트
```

### Phase 4에 Skill + MCP 조합

Garbage Collection이 온전히 자율화되는 지점이야.

```
doc-gardening 에이전트:
  MCP(filesystem)로 코드 변경 감지
    → Skill(validate-docs.sh)로 문서 freshness 검사
      → 불일치 발견 시 Skill(create-pr.sh)로 수정 PR 자동 생성

pattern-sweep 에이전트:
  MCP(filesystem)로 코드 패턴 스캔
    → Skill(arch-check.sh)로 황금 원칙 위반 감지
      → Skill(create-pr.sh)로 리팩터링 PR 자동 생성
```

## 전체 그림

```
┌──────────────────────────────────────────────────────────────┐
│                        에이전트                               │
│                                                              │
│         "무엇을 볼 것인가"      "무엇을 할 것인가"            │
│         MCP (인식)         →    Skill (행동)                  │
└──────────────────────────────────────────────────────────────┘
         │                              │
         ▼                              ▼
  빌드 로그 읽기               린트 실행
  테스트 결과 읽기             테스트 실행
  PR 상태 읽기                 PR 생성
  코드 변경 감지               문서 검사
```

Skill이 **행동의 원자 단위**고, MCP가 **인식의 채널**이야. 이 둘이 붙어야 원문에서 말하는 "에이전트가 6시간 동안 자율 실행"이 가능해지는 구조가 됨.
