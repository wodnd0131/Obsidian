# SVN
**Apache Subversion**의 약자로, 2000년대 초에 CVS(더 오래된 VCS)의 단점을 개선하려고 만든 버전 관리 시스템이야.

### 작동 방식

```
[중앙 서버 저장소]
       ↑↓ commit/update
  [개발자 A]  [개발자 B]  [개발자 C]
```

모든 히스토리가 **서버에만** 있어. 개발자는 필요한 파일만 로컬에 받아서 작업함.

반면 Git은:

```
[원격 저장소 (GitHub)]
       ↑↓ push/pull
[개발자 A]  [개발자 B]  [개발자 C]
(각자 전체 히스토리 보유)
```

---

## SVN vs Git 상세 비교

|항목|SVN|Git|
|---|---|---|
|**저장소 구조**|중앙 집중식|분산형|
|**브랜치**|디렉토리 복사 방식 (무거움)|포인터 방식 (가벼움)|
|**커밋**|서버 연결 필수|로컬에서 가능|
|**히스토리**|서버에만 존재|모든 로컬에 존재|
|**속도**|네트워크 의존|로컬 작업이라 빠름|
|**학습 난이도**|상대적으로 낮음|높음 (브랜치/머지 복잡)|
|**권한 관리**|디렉토리/파일 단위 세밀하게 가능|저장소 단위|

---
### SVN이 상대적으로 유리한 상황
- **바이너리 파일이 많은 프로젝트** (게임 에셋, 디자인 파일 등) → SVN은 파일 단위 잠금(lock) 기능이 있어서 충돌 방지에 유리
- **디렉토리 일부만 체크아웃** 하고 싶을 때 → 대형 모노레포에서 필요한 폴더만 받기 가능
- **세밀한 접근 권한** 이 필요할 때 → 특정 폴더는 A만, 다른 폴더는 B만 접근하게 설정 가능

### Git이 유리한 상황 (거의 대부분)
- 오픈소스 / 협업 프로젝트
- 브랜치 전략이 필요한 경우 (feature 브랜치, PR 리뷰 등)
- 오프라인 작업이 필요한 경우
- CI/CD 파이프라인 연동

---
## 결론

> SVN은 언어가 아니라 **팀 규모, 파일 종류, 조직 구조**에 따라 선택하는 거야.

요즘 신규 프로젝트는 거의 100% Git 써. SVN은 오래된 기업 레거시 환경에서만 남아있는 추세고, pickeat처럼 GitHub Actions 쓰는 프로젝트라면 Git이 당연한 선택이지.
