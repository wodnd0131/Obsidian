```
객체 생성 문제
    ├── 로직을 객체 밖으로 뺐다          → Utility Class
    ├── 의존성을 전역에서 꺼냈다          → Service Locator
    └── 선택적 파라미터를 오버로딩으로    → Telescoping Constructor
                                            └── 해결: Builder / 정적 팩토리
```

# 1. Utility Class
## 1. Problem First

```java
public class LottoUtils {
    public static boolean isValidNumber(int number) {
        return number >= 1 && number <= 45;
    }

    public static int calculateMatchCount(List<Integer> ticket, List<Integer> winning) {
        return (int) ticket.stream().filter(winning::contains).count();
    }

    public static double calculateProfitRate(int prize, int totalPrice) {
        return (double) prize / totalPrice;
    }
}
```

언뜻 보면 깔끔해 보인다. 근데 이 코드가 커지면:

```java
public class LottoUtils {
    public static boolean isValidNumber(int number) { ... }
    public static boolean isValidTicket(List<Integer> numbers) { ... }
    public static int calculateMatchCount(...) { ... }
    public static double calculateProfitRate(...) { ... }
    public static List<Integer> generateAutoNumbers() { ... }
    public static String formatTicket(List<Integer> numbers) { ... }
    // 계속 늘어난다
}
```

로또와 관련된 모든 로직이 여기 모인다. 이게 **Utility Class 안티패턴**이다.

---

## 2. Mechanics

Utility Class의 구조적 특징:

```java
public class LottoUtils {
    private LottoUtils() {} // 인스턴스화 방지 — 혹은 이것도 없음

    public static ... // 모든 메서드가 static
}
```

상태가 없다. 인스턴스를 만들 이유가 없다. 순수하게 입력 → 출력만 한다.

문제는 **책임의 위치**다. `isValidNumber`는 `LottoNumber`가 스스로 알아야 할 것이고, `calculateMatchCount`는 `Lotto`나 `WinningLotto`가 알아야 할 것이다. 이 로직들이 객체 밖으로 빠져나와 유틸에 모이면:

```java
// Utility Class 방식 — 절차지향
boolean valid = LottoUtils.isValidNumber(7);
int matchCount = LottoUtils.calculateMatchCount(ticket, winning);

// OOP 방식 — 객체가 스스로 판단
boolean valid = lottoNumber.isValid();
int matchCount = lotto.countMatch(winningLotto);
```

전자는 데이터와 로직이 분리된다. 객체는 데이터만 들고 있고, 로직은 유틸에 있다. 이게 Fowler가 말한 **Anemic Domain Model**의 전형적인 증상이다.

---

## 3. 공식 근거

**Martin Fowler, "AnemicDomainModel" (bliki, 2003):**

> "The fundamental horror of this anti-pattern is that it's so contrary to the basic idea of object-oriented design; which is to combine data and process together."

**Effective Java 3rd, Item 4:** Bloch은 Utility Class를 안티패턴으로 보지 않는다. 단, **인스턴스화 불가능하게 만들어야 한다**고 명시한다(`private` 생성자). 그리고 사용 맥락을 제한한다 — 진짜 유틸성 연산(`Math`, `Arrays`)에만.

두 입장이 다르다. Fowler는 도메인 로직이 유틸로 가는 것 자체를 문제로 보고, Bloch은 유틸 클래스 자체를 부정하지 않되 남용을 경계한다.

---

## 4. 트레이드오프

| |설명|
|---|---|
|테스트 용이성|static이라 mock 불가. 의존하는 쪽 테스트가 어려워짐|
|재사용성|단순해 보이지만 로직이 분산되어 파악이 어려움|
|확장성|상태가 없으니 다형성 적용 불가|
|절차지향|OOP의 캡슐화, Tell Don't Ask 원칙 위반|

---

## 5. 안티패턴 검증

**개념 자체의 오남용:** `Math.abs()`, `Collections.sort()` 같은 진짜 도메인 무관 연산은 Utility Class가 맞다. 문제는 **도메인 로직**이 유틸로 가는 것.

```java
// 정당한 유틸 — 도메인과 무관한 범용 연산
public class StringUtils {
    public static boolean isBlank(String str) { ... }
}

// 안티패턴 — 도메인 로직이 유틸로 탈출
public class LottoUtils {
    public static int calculateMatchCount(List<Integer> ticket, List<Integer> winning) { ... }
}
```

구분 기준: **이 로직이 특정 도메인 객체의 책임인가?** 그렇다면 유틸이 아니라 그 객체 안으로.

---

## 6. 실전 함정

1학년 코드에서 가장 자주 보이는 패턴:

```java
// LottoService.java
public class LottoService {
    public static List<Integer> generateNumbers() { ... }
    public static boolean isWinning(...) { ... }
    public static double calculateProfit(...) { ... }
}
```

이름만 Service인데 실질적으로 Utility Class다. static 메서드만 있고 상태가 없다. 던질 질문:

> "`isWinning` 로직이 `LottoService`에 있어야 하는 이유가 뭔가요? `Lotto` 객체가 당첨 여부를 스스로 알면 안 되나요?"

# 2. Service Locator

---

## 1. Problem First

```java
// 의존성을 어떻게 전달할 것인가
public class LottoMachine {
    public List<Lotto> generate(int count) {
        LottoNumberGenerator generator = ???; // 어디서 가져오지?
        ...
    }
}
```

이 문제를 "전역에서 꺼내면 되지"로 해결한 게 Service Locator다:

```java
public class LottoMachine {
    public List<Lotto> generate(int count) {
        LottoNumberGenerator generator =
            ServiceLocator.get(LottoNumberGenerator.class); // 전역에서 꺼냄
        ...
    }
}
```

---

## 2. Mechanics

```java
public class ServiceLocator {
    private static final Map<Class<?>, Object> registry = new HashMap<>();

    public static <T> void register(Class<T> type, T instance) {
        registry.put(type, instance);
    }

    public static <T> T get(Class<T> type) {
        return (T) registry.get(type);
    }
}

// 앱 시작 시
ServiceLocator.register(LottoNumberGenerator.class, new RandomLottoNumberGenerator());

// 사용처
LottoNumberGenerator gen = ServiceLocator.get(LottoNumberGenerator.class);
```

겉으로는 의존성 주입처럼 보인다. 구현체가 분리되어 있고, 인터페이스를 쓴다. 근데 의존성이 **숨겨져** 있다.

```java
// 이 클래스의 시그니처만 보면 의존성을 알 수 없다
public class LottoMachine {
    public List<Lotto> generate(int count) { // 파라미터에 의존성 없음
        LottoNumberGenerator generator = ServiceLocator.get(...); // 내부에서 꺼냄
    }
}
```

`LottoMachine`을 테스트하려면 `ServiceLocator`에 무언가 등록되어 있어야 한다. 클래스 시그니처만 보고 이걸 알 수 없다.

---

## 3. 공식 근거

**Martin Fowler, "Inversion of Control Containers and the Dependency Injection pattern" (2004):**

> "The key difference is that with a Service Locator, every user of a service has a dependency on the locator. The locator can hide dependencies to other implementations, but you do need to see the locator."

Fowler는 Service Locator를 금지하지 않는다. 단, **의존성 주입이 가능한 상황에서는 DI가 낫다**고 명시한다. 숨겨진 의존성이 핵심 문제.

**Mark Seemann, "Service Locator is an Anti-Pattern" (2010) — 커뮤니티 영향력 높은 글:**

> "The problem with Service Locator is that it hides a class's dependencies, causing run-time errors instead of compile-time errors."

이쪽은 더 강경하다. 컴파일 타임에 잡힐 오류가 런타임으로 밀린다는 게 핵심 주장.

---

## 4. 트레이드오프

| |Service Locator|의존성 주입|
|---|---|---|
|의존성 가시성|숨겨짐|시그니처에 노출|
|테스트|전역 상태 오염|Mock 주입 가능|
|컴파일 타임 안전성|없음 (런타임 에러)|있음|
|설정 편의성|간단|설정 필요|

---

## 5. 안티패턴 검증

**논쟁이 있는 지점:**

Fowler는 안티패턴으로 단정하지 않는다. 레거시 환경이나 DI 프레임워크를 쓸 수 없는 상황에서는 Service Locator가 현실적 선택일 수 있다고 본다.

Seemann은 안티패턴으로 단정한다. 의존성이 숨겨지는 순간 설계가 거짓말을 하는 것이라고.

1학년 코드 리뷰 맥락에서는 Seemann 쪽이 더 유용하다. "이 클래스가 뭘 필요로 하는지 시그니처만 보고 알 수 있어야 한다"는 원칙으로 가르치면 된다.

---


# 3. Telescoping Constructor

---

## 1. Problem First

```java
public class LottoTicketOrder {
    public LottoTicketOrder(int totalCount) { ... }
    public LottoTicketOrder(int totalCount, int manualCount) { ... }
    public LottoTicketOrder(int totalCount, int manualCount, boolean hasSuperLotto) { ... }
    public LottoTicketOrder(int totalCount, int manualCount, boolean hasSuperLotto, boolean isPremium) { ... }
}

// 호출하는 쪽
new LottoTicketOrder(10, 3, false, true); // 이게 뭔 소리야?
```

파라미터가 늘어날수록 어떤 생성자를 써야 하는지, 각 인자가 뭘 의미하는지 알 수 없다. 망원경처럼 생성자가 겹겹이 쌓인다고 해서 Telescoping Constructor.

---

## 2. Mechanics

선택적 파라미터 문제를 생성자 오버로딩으로 해결하려는 시도가 근본 원인이다.

Bloch이 제시한 두 가지 해결책:

**해결책 1 — Builder 패턴:**

```java
LottoTicketOrder order = new LottoTicketOrder.Builder(10)
    .manualCount(3)
    .superLotto(false)
    .premium(true)
    .build();
```

각 파라미터가 이름을 가진다. 순서를 외울 필요가 없다.

**해결책 2 — 정적 팩토리 메서드 (파라미터 적을 때):**

```java
LottoTicketOrder.onlyAuto(10);
LottoTicketOrder.withManual(10, 3);
```

파라미터가 2~3개 이하면 팩토리 메서드 이름으로 의도를 표현하는 게 더 간결하다.

---

## 3. 공식 근거

**Effective Java 3rd, Item 2:**

> "The telescoping constructor pattern works, but it is hard to write client code when there are many parameters, and harder still to read it."

> "A second alternative when you are faced with many optional constructor parameters is the JavaBeans pattern... Unfortunately, the JavaBeans pattern has serious disadvantages of its own."

JavaBeans 패턴(setter 기반)도 대안으로 보지 않는다 — 불변성이 깨지기 때문. Builder가 Bloch의 최종 권장안.

---

## 4. 트레이드오프

Builder 패턴 적용 시 잃는 것:

| |설명|
|---|---|
|코드량 증가|Builder 내부 클래스 작성 비용|
|파라미터 2~3개|오버엔지니어링이 될 수 있음|
|불변성|Builder 자체는 가변 — `build()` 호출 전까지|

Bloch 기준: **파라미터 4개 이상이면 Builder를 고려하라.**

---

## 5. 안티패턴 검증

**잘못된 적용 방식 — Builder인데 검증이 없다:**

```java
// build()에서 검증 안 함
public LottoTicketOrder build() {
    return new LottoTicketOrder(this); // 유효하지 않은 상태로 생성 가능
}

// 올바른 방식
public LottoTicketOrder build() {
    validate(); // 여기서 검증
    return new LottoTicketOrder(this);
}
```

Builder를 쓰면서 검증을 `build()` 안에 넣지 않으면 불완전한 객체가 만들어진다.
