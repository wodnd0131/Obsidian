# 지속 가능하고 효율적인 소프트웨어 개발을 위한 ATDD 및 테스트 전략

## 핵심 요약 (Executive Summary)

본 문서는 우아한형제들의 기술 세미나 및 컨퍼런스 자료를 바탕으로, 현대적 소프트웨어 개발의 핵심 방법론인 **인수 테스트 주도 개발(ATDD, Acceptance Test Driven Development)**과 효율적인 테스트 작성 전략을 종합적으로 분석한다.

가장 핵심적인 통찰은 다음과 같다:

- **빠른 피드백의 가치:** 테스트는 단순한 검증 도구를 넘어 개발자에게 즉각적인 피드백을 제공하고 심리적 안정감을 부여하는 설계 활동이다.
- **협업의 도구로서의 ATDD:** ATDD는 개발, 기획, QA 등 다양한 직군이 공통의 이해를 도모하기 위한 협업 프로세스이며, 사용자 스토리와 인수 조건을 통해 요구사항의 모호성을 제거한다.
- **지속 가능한 실천:** 완벽한 규칙으로 시작하기보다 팀의 상황에 맞춰 점진적으로 개선해 나가는 '지속 가능한' 접근 방식이 필수적이다.
- **기술적 최적화:** 코틀린(Kotlin)의 언어적 특성을 활용한 DSL 구축, 프론트엔드 통합 테스트에서의 MSW 활용 등 기술적 도구를 통해 테스트의 가독성과 신뢰성을 높일 수 있다.

## 1. 인수 테스트와 ATDD의 개념 및 가치

### 1.1 인수 테스트(Acceptance Test)의 정의

- **인수의 의미:** 소프트웨어 개발을 의뢰한 측에서 결과물이 요구사항을 충족하는지 확인하고 수락하는 과정이다.
- **역할:** 시스템이 인수 기준을 만족하는지 검증하며, 주로 사용자 스토리와 시나리오를 기반으로 작성된다.
- **특징:** 내부 구현에 관심이 없는 '블랙박스 테스트'의 성격을 가지며, 사용자 관점에서 기능이 올바르게 동작하는지 확인한다.

### 1.2 ATDD(인수 테스트 주도 개발)의 핵심

- **프로세스:** 인수 조건을 정의하고 이를 만족하는 인수 테스트를 먼저 작성한 후 기능을 구현하는 방식이다.
- **장점:**
    - **배포 전 기능 검증:** 로컬 환경에서 배포 없이 기능을 바로 확인하여 피드백 주기를 단축한다.
    - **도메인 지식의 축적:** 테스트 코드가 비즈니스 흐름을 설명하는 문서 역할을 하여 도메인 정보 공유에 기여한다.
    - **요구사항 명확화:** 기획자와 개발자, QA 간의 오해를 방지하고 공통된 이해를 구축한다.

## 2. ATDD의 단계별 실천 프로세스

ATDD는 단순히 테스트 코드를 짜는 행위를 넘어, 기획부터 개발까지의 전 과정을 포괄한다.

|   |   |   |
|---|---|---|
|단계|활동 내용|비고|
|**요구사항 분석**|사용자 스토리 도출 (Who, Why, What 형식)|사용자 스토리 매핑 활용|
|**인수 조건 정의**|시나리오 기반의 인수 조건 작성 (Given/When/Then 형식)|모든 이해관계자 참여 권장|
|**인수 테스트 작성**|실제 요청과 응답을 주고받는 API 레벨 테스트 구현|RestAssured 등 도구 활용|
|**기능 구현**|인수 테스트를 성공시키기 위한 코드 작성|내부적으로 TDD 사이클 병행 가능|

### 주요 실천 전략

- **사용자 스토리 매핑:** 전체 시스템의 기능을 시각화하고 히스토리를 공유하여 개선 사항을 도출하는 활동이다. 결과물 자체보다 논의 과정에서의 '생각의 싱크'가 더 중요하다.
- **Given/When/Then 형식:**
    - **Given:** 테스트 전제 조건 (데이터 준비)
    - **When:** 검증하고자 하는 행위 (API 호출)
    - **Then:** 행위에 따른 결과 검증 (응답 확인)

## 3. 백엔드 및 코틀린(Kotlin) 테스트 최적화

코틀린의 특징을 활용하면 더 간결하고 읽기 쉬운 테스트 코드를 작성할 수 있다.

### 3.1 가독성 향상 기법

- **백틱(** **) 활용:** 테스트 함수명에 한글과 공백을 사용하여 테스트 의도를 명확히 드러낸다.
- **팩토리 함수와 기본 인자:** 테스트 픽스처(Fixture) 생성 시 팩토리 함수와 코틀린의 기본 인자를 활용하여 불필요한 코드 중복을 줄이고 관심사에 집중한다.
- **확장 함수 활용:** 도메인 특화된 검증 로직을 확장 함수로 만들어 자연어에 가까운 검증 코드를 작성한다.

### 3.2 테스트 라이브러리 및 도구

- **Kotest & MockK:** 자바 기반의 JUnit/Mockito보다 코틀린 친화적인 DSL을 제공하는 라이브러리 활용을 권장한다.
- **RestAssured:** 실제 웹 서버를 띄워(SpringBootTest의 RandomPort 설정 등) HTTP 요청을 보내는 방식으로 인수 테스트를 수행한다.

### 3.3 테스트 격리와 데이터 초기화

- **문제점:** 테스트 간 데이터 공유로 인해 특정 테스트의 결과가 다른 테스트에 영향을 주는 현상이 발생한다.
- **해결책:**
    - `@DirtiesContext`는 완전한 격리를 보장하지만 속도가 매우 느리다.
    - **데이터베이스 트렁케이트(Truncate):** 각 테스트 실행 전 테이블을 초기화하는 로직을 엔티티 매니저를 통해 구현하여 빠른 속도와 격리를 동시에 달성한다.

## 4. 프론트엔드 통합 테스트 전략

웹 프론트엔드에서도 단위 테스트만으로는 부족한 '모듈 간 상호작용'을 검증하기 위해 통합 테스트가 필수적이다.

### 4.1 통합 테스트의 필요성

- 단위 테스트는 개별 컴포넌트의 논리 오류는 잡을 수 있으나, 컴포넌트 간 연동 과정의 결함이나 비동기 타이밍 이슈를 발견하기 어렵다.
- 통합 테스트는 실제 유저의 경험과 유사한 시나리오를 검증하여 배포에 대한 자신감을 부여한다.

### 4.2 핵심 기술 및 접근법

- **MSW(Mock Service Worker):** 실제 서버에 요청을 보내는 대신 네트워크 레벨에서 API를 모킹하여 다양한 서버 응답 시나리오(성공, 에러, 로딩 등)를 검증한다.
- **Testing Library 활용 지침:**
    - `findByRole`: 사용자의 인지와 가장 유사한 역할(Role) 기반 검색을 최우선으로 사용한다.
    - `test-id`는 최후의 수단으로 사용하며, 실제 사용자의 행동에 기반한 테스트 작성을 지향한다.

## 5. 지속 가능한 테스트 문화를 위한 제언

테스트 문화의 정착은 기술적인 문제보다 조직적인 노력이 수반되어야 한다.

- **점진적 도입:** 처음부터 모든 프로세스를 강제하기보다, 팀원들이 테스트의 필요성을 느낄 수 있도록 작은 부분부터 적용하고 성과를 공유해야 한다.
- **테스트 코드 리팩터링:** 테스트 코드 역시 제품 코드만큼 중요하다. 중복을 제거하고 가독성을 높여 "관리하기 힘든 테스트"가 되어 버려지는 것을 방지해야 한다.
- **리더십과 성장:** 새로운 개발 문화를 도입하는 과정은 개발자 개인의 성장에 큰 도움이 되며, 이는 향후 더 좋은 리더로 성장하기 위한 필수적인 경험이다.
