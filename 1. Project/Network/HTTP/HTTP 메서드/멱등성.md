동일한 요청을 한 번 보내는 것과 여러 번 연속으로 보내는 것이 같은 효과를 지니고, 서버의 상태도 동일하게 남을 때, 해당 HTTP 메서드가 **멱등성**을 가졌다고 말합니다.

다른 말로는, 멱등성 메서드에는 통계기록 등을 제외하면 어떠한 부수효과(side effect)도 존재해서는 안됩니다. 예를 들어, 서버는 모든 요청을 받은 시간과 함께 기록할 수 있습니다. 멱등성(Idempotency)는 클라이언트가 의도한 효과에만 적용됩니다. 예를 들어, POST 요청은 데이터를 서버로 보내려고 하거나, DELETE 요청은 서버에서 리소스를 삭제하려고 합니다.

[safe](https://developer.mozilla.org/ko/docs/Glossary/Safe/HTTP) 메서드는 [`PUT`](https://developer.mozilla.org/ko/docs/Web/HTTP/Reference/Methods/PUT) and [`DELETE`](https://developer.mozilla.org/ko/docs/Web/HTTP/Reference/Methods/DELETE)와 마찬가지로 멱등적입니다. [`POST`](https://developer.mozilla.org/ko/docs/Web/HTTP/Reference/Methods/POST) 메서드는 멱등성이 아닙니다.

멱등성을 따질 땐 실제 서버의 백엔드 상태만 보면 되며, 각 요청에서 반환하는 응답 코드는 다를 수 있습니다. 첫 번째 [`DELETE`](https://developer.mozilla.org/ko/docs/Web/HTTP/Reference/Methods/DELETE) 요청이 [`200`](https://developer.mozilla.org/ko/docs/Web/HTTP/Reference/Status/200)을 반환한다면, 그 이후는 아마 [`404`](https://developer.mozilla.org/ko/docs/Web/HTTP/Reference/Status/404)를 반환할 것입니다. `DELETE`가 멱등성을 가진다는 것은, REST API에서 개발자는 `DELETE` 메서드를 사용해 "목록의 마지막 항목 제거" 기능을 구현해서는 안 된다는 것입니다.

다만, 서버는 메서드의 멱등성을 보장하지 않으며, 일부 애플리케이션은 잘못된 구현으로 멱등성 제약을 어길 수도 있습니다.

`GET /pageX HTTP/1.1`는 안전한(읽기 전용) 메서드이기 때문에 멱등성을 가집니다. 그 동안 서버의 데이터가 업데이트된 경우, 다음에 하는 호출은 클라이언트에 다른 데이터를 반환할 수 있습니다.

`POST /add_row HTTP/1.1`은 멱등성을 가지지 않습니다. 여러 번 호출되면 여러 행을 추가합니다.

```
POST /add_row HTTP/1.1
POST /add_row HTTP/1.1   -> Adds a 2nd row
POST /add_row HTTP/1.1   -> Adds a 3rd row
```
`DELETE /idX/delete HTTP/1.1`의 상태 코드는 응답마다 달라질 수 있지만, 그럼에도 멱등성을 가집니다.

```
DELETE /idX/delete HTTP/1.1   -> Returns 200 if idX exists
DELETE /idX/delete HTTP/1.1   -> Returns 404 as it just got deleted
DELETE /idX/delete HTTP/1.1   -> Returns 404
```

# 멱등성이 왜 중요한가

---

## 1. Problem First — 멱등성이 없을 때 어떤 일이 생기나

### 시나리오: 결제 요청 중 네트워크가 끊겼다

```
클라이언트                                                서버
    |                                                   |
    |--- POST /payment ------->|
    |                                                      | (처리 중... 결제 완료)
    |                         X (네트워크 단절)  |
    |<--- 응답이 안 옴                           |
    |                                                      |
    | "타임아웃. 실패한 건가?"               |
    |--- POST /payment ------->  | (재시도)
    |                                                      | (또 결제 완료)
    |<--- 200 OK                                |
```

서버는 결제를 **두 번** 처리했다.  
클라이언트는 응답을 못 받았으니 한 번 시도한 것으로 알고 있다.

사용자는 이중 결제를 당한다.

**이게 근본적으로 문제인 이유:**  
네트워크는 구조적으로 신뢰할 수 없다.  
패킷 손실, 타임아웃, 서버 재시작 — 이 중 어느 것도 "요청이 처리됐는지 안 됐는지"를 클라이언트에게 보장해주지 않는다.

> RFC 9110 §9.2.2: "Idempotent methods are distinguished because the request can be repeated automatically if a communication failure occurs before the client is able to read the server's response."
> 
> _→ 멱등성 메서드는 클라이언트가 응답을 읽기 전에 통신 장애가 발생했을 때 자동으로 재시도할 수 있다는 점에서 구별된다._

멱등성은 이 구조적 불확실성을 다루기 위한 설계 원칙이다.

---

## 2. Mechanics — 멱등성이 실제로 무엇을 보장하는가

### 핵심 정의 재정리

멱등성은 **서버의 최종 상태**에 관한 약속이다. 응답 코드가 아니다.

```
f(f(x)) = f(x)
```

```
DELETE /user/123  →  user 123 삭제됨 (200)
DELETE /user/123  →  이미 없음    (404)
DELETE /user/123  →  이미 없음    (404)
```

세 번 요청했지만 서버 상태는 "user 123이 없다" 로 동일하다.  
→ 멱등하다.

```
POST /orders  →  주문 생성됨 (id=1)
POST /orders  →  주문 생성됨 (id=2)   ← 다른 리소스가 생김
POST /orders  →  주문 생성됨 (id=3)
```

세 번 요청하면 세 개의 주문이 생긴다.  
→ 멱등하지 않다.

---

## 3. 왜 중요한가 — 강제되지 않는데도

### 핵심: 재시도 로직을 안전하게 짤 수 있다

멱등성이 보장되면, 클라이언트/인프라가 **응답을 못 받았을 때 그냥 다시 보내면 된다.**

```java
// 멱등성이 보장된 경우 — 재시도 로직이 단순하다
public void updateUserName(String userId, String name) {
    int retries = 3;
    while (retries-- > 0) {
        try {
            httpClient.put("/users/" + userId, Map.of("name", name));
            return;
        } catch (TimeoutException e) {
            // 타임아웃 → 그냥 재시도
            // PUT은 멱등하므로 서버에 같은 요청이 두 번 가도 결과 동일
        }
    }
}
```

```java
// 멱등성이 보장되지 않은 경우 — 재시도하면 큰일난다
public void createOrder(OrderRequest request) {
    try {
        httpClient.post("/orders", request);
    } catch (TimeoutException e) {
        // 재시도하면 주문이 두 번 생길 수 있음
        // → 재시도 못 함, 사용자에게 "다시 시도하시겠습니까?" 물어봐야 함
        // → 아니면 별도 중복 방지 로직 필요
        throw new OrderUncertainException("요청 결과를 알 수 없습니다");
    }
}
```

### 이걸 믿는 주체들

|주체|멱등성을 어떻게 활용하나|
|---|---|
|HTTP 클라이언트 라이브러리|GET/PUT은 자동 재시도, POST는 안 함|
|로드밸런서|연결 끊기면 GET은 다른 서버로 재전송 가능|
|브라우저|"뒤로가기" 시 GET은 재요청, POST는 "다시 제출하시겠습니까?"|
|CDN/프록시|GET 응답은 캐시하고 재활용|
|AWS ALB/API Gateway|재시도 정책을 메서드 기준으로 다르게 적용|

브라우저가 POST에서 뒤로가기할 때 경고창을 띄우는 것,  
이게 바로 멱등성 개념을 브라우저가 구현한 것이다.

---

## 4. 강제가 아닌데 왜 지켜야 하나

RFC가 강제하지 못한다. 서버가 `PUT`으로 매번 다른 결과를 만들어도 HTTP 자체는 막지 않는다.

그런데 **인프라 전체가 이 약속을 믿고 동작**한다.

### 약속을 어겼을 때 생기는 실제 장애

**사례 1: AWS ALB 재시도**

```
클라이언트 → ALB → 서버 A (처리 완료, 응답 전 재시작)
                 ↘ 서버 B (재시도 자동 전달)
```

ALB는 연결이 끊기면 GET/HEAD에 대해 자동으로 다른 인스턴스에 재시도한다.  
만약 GET 핸들러가 내부에서 insert를 한다면?  
→ ALB가 자동으로 두 번 insert를 만들어준다.  
→ 개발자는 ALB 로그 보기 전까지 이유를 모른다.

**사례 2: Spring RestTemplate / WebClient 재시도 설정**

```java
// 이런 재시도 설정은 흔하다
@Bean
public RestTemplate restTemplate() {
    HttpComponentsClientHttpRequestFactory factory = 
        new HttpComponentsClientHttpRequestFactory();
    factory.setConnectTimeout(3000);
    // Apache HttpClient는 기본적으로 GET/HEAD/DELETE/PUT을 재시도한다
    // POST는 재시도 안 함
    return new RestTemplate(factory);
}
```

Apache HttpClient는 멱등성 있는 메서드를 자동 재시도한다.  
`GET`으로 삭제 로직을 만든 핸들러가 있으면, 네트워크 불안정 시 자동으로 두 번 삭제 시도가 간다.

**사례 3: 브라우저 prefetch (앞에서 다룬 Google Web Accelerator 사건)**  
GET 링크를 미리 긁었더니 서버가 그 GET에서 데이터 삭제 → 데이터 손실.

## 5. 멱등성이 없는 POST를 멱등하게 만드는 실무 패턴

POST 자체는 멱등하지 않다. 하지만 **멱등성을 애플리케이션 레벨에서 구현**할 수 있다.

### 패턴: Idempotency Key

```
클라이언트가 요청마다 고유한 키를 헤더에 붙인다
```

```java
// 클라이언트
String idempotencyKey = UUID.randomUUID().toString(); // 요청 생성 시 한 번만 만든다

httpClient.post("/orders")
    .header("Idempotency-Key", idempotencyKey)
    .body(orderRequest)
    .execute();

// 타임아웃 → 같은 idempotencyKey로 재시도
httpClient.post("/orders")
    .header("Idempotency-Key", idempotencyKey) // 동일한 키
    .body(orderRequest)
    .execute();
```

```java
// 서버
@PostMapping("/orders")
public ResponseEntity<Order> createOrder(
    @RequestHeader("Idempotency-Key") String idempotencyKey,
    @RequestBody OrderRequest request
) {
    // 이미 처리한 키인지 확인
    Optional<Order> existing = idempotencyStore.find(idempotencyKey);
    if (existing.isPresent()) {
        return ResponseEntity.ok(existing.get()); // 그냥 이전 결과 반환
    }

    // 실제 처리
    Order order = orderService.create(request);
    idempotencyStore.save(idempotencyKey, order);
    return ResponseEntity.ok(order);
}
```

```
1차 요청 (key=abc123) → 주문 생성 → 저장 → 응답 못 받음
2차 요청 (key=abc123) → 이미 처리됨 → 저장된 결과 반환
결과: 주문 1개
```

Stripe, Kakao Pay 등 결제 API가 이 패턴을 공식으로 사용한다.  
Stripe API 문서는 이를 명시적으로 요구한다.

---

## 6. 설계 트레이드오프

| |멱등성 보장|멱등성 포기|
|---|---|---|
|재시도|인프라가 자동으로 해줌|직접 중복 방지 로직 필요|
|구현 복잡도|낮음|높음 (idempotency key 등)|
|잃는 것|"매번 다른 결과"가 필요한 경우 표현 어려움|인프라 자동화 혜택 없음|

"목록의 마지막 항목 제거"를 DELETE로 구현하면 안 된다는 예시가 바로 이 트레이드오프다.  
`DELETE /items/last` — 이걸 두 번 보내면 두 개가 지워진다.  
인프라는 DELETE가 멱등하다고 믿고 재시도할 수 있는데, 의도치 않게 두 개를 지우게 된다.

구현하고 싶다면 `POST /items/pop` 처럼 명시적으로 비멱등 메서드를 써서  
"이건 재시도하면 안 된다"는 신호를 인프라에 주는 게 맞다.

## 7. 이어지는 개념

1. **HTTP 캐시 (Cache-Control, ETag)** — 멱등성의 직접 활용. 안전+멱등 메서드만 캐시 가능하다. 캐시 없이는 멱등성의 실무 가치 절반이 사라진다.
    
2. **분산 시스템의 재시도와 At-least-once / Exactly-once 전달** — 멱등성은 네트워크 레이어 문제의 HTTP 버전이다. Kafka 등에서 같은 문제가 더 복잡하게 나온다.
    
3. **낙관적 잠금 (Optimistic Locking) / ETag 조건부 요청** — PUT이 멱등하려면 "내가 본 버전을 기준으로 업데이트"해야 한다. 그렇지 않으면 동시 수정 시 마지막 쓴 사람이 이기는 문제가 생긴다. `If-Match` 헤더가 이걸 해결한다.

# PUT 멱등성 / DELETE 404·200 / ETag + 낙관적 잠금

---

## 1. PUT이 멱등적이라는 게 뭔 뜻인가

### "상태를 바꾸는데 멱등하다" — 모순처럼 보이는 이유

"멱등 = 상태를 안 바꾼다"로 오해하는 경우가 많다.  
앞에서 정리했듯이 멱등성은 **"몇 번 보내도 최종 상태가 같다"** 는 것이지, 상태 변경 자체를 안 한다는 게 아니다.

```
PUT /users/123
{ "name": "김철수" }
```

```
1회 요청 → users/123의 name = "김철수"
2회 요청 → users/123의 name = "김철수"  (똑같음)
3회 요청 → users/123의 name = "김철수"  (똑같음)
```

상태는 바꿨다. 근데 **최종 상태는 항상 동일**하다.  
이게 멱등하다는 것이다.

### POST와 비교하면 명확해진다

```
POST /users
{ "name": "김철수" }

1회 요청 → id=1, name="김철수" 생성
2회 요청 → id=2, name="김철수" 생성  ← 다른 리소스가 생김
3회 요청 → id=3, name="김철수" 생성
```

POST는 호출할 때마다 **새로운 리소스를 만든다.**  
최종 상태가 달라지므로 멱등하지 않다.

### PUT이 멱등할 수 있는 이유 — URI가 리소스를 특정한다

PUT의 핵심은 **"어디에 무엇을 넣을지"를 클라이언트가 완전히 지정**한다는 것이다.

```
PUT /users/123      ← "123번 자리에"
{ "name": "김철수" } ← "이 상태로 덮어써라"
```

같은 내용으로 100번 보내도 `/users/123`은 항상 `name="김철수"`다.  
새로운 리소스가 생기지 않는다.

반면 POST는 "서버야, 너가 알아서 어딘가에 만들어줘" 구조다.  
서버가 매번 새 id를 발급해서 새 리소스를 만든다.

> RFC 9110 §9.3.4: "A successful PUT of a given representation would suggest that a subsequent GET on that same target resource will result in an equivalent representation being sent"
> 
> _→ PUT이 성공하면, 이후 같은 리소스에 GET을 했을 때 동등한 표현이 반환될 것임을 의미한다._

---

## 2. DELETE 후 404 vs 200 — 뭘 보내야 하나

### RFC가 말하는 것

> RFC 9110 §9.3.5: "If the DELETE method is successfully applied, the origin server SHOULD send ... 202 (Accepted) if the action will likely succeed but has not yet been enacted, 204 (No Content) if the action has been enacted and no further information is to be supplied, or 200 (OK) if the action has been enacted and the response message includes a representation describing the status."
> 
> _→ DELETE 성공 시 200, 202, 204를 권장한다. 이미 없는 리소스에 대한 재시도 응답은 별도로 명시하지 않는다._

RFC는 "이미 삭제된 것을 다시 DELETE하면 뭘 보내라"를 명시하지 않는다.  
실무에서 두 가지 입장이 있다.

### 입장 A: 404를 보내야 한다

```
DELETE /users/123  →  200 (삭제 성공)
DELETE /users/123  →  404 (없는 리소스)
```

"없는 리소스를 요청했으니 404가 맞다"는 논리.  
응답 코드가 서버의 현재 상태를 정확히 반영한다.

### 입장 B: 200/204를 보내도 된다

```
DELETE /users/123  →  200 (삭제 성공)
DELETE /users/123  →  200 (이미 없음, 그래도 '삭제된 상태'는 맞음)
```

"클라이언트의 의도는 '123이 없는 상태로 만들어라'였고, 실제로 없으니 의도 달성"이라는 논리.  
재시도 로직 구현이 단순해진다.

### 실무에서 어떻게 결정하나

**재시도 로직에서 오류 처리 방식**이 기준이 된다.

```java
// 클라이언트가 404를 에러로 처리하면 → 문제 발생
public void deleteUser(String userId) {
    try {
        httpClient.delete("/users/" + userId); // 타임아웃
        // 재시도
        httpClient.delete("/users/" + userId); // 404 반환
        // 404를 예외로 던지면 → "삭제 실패"로 처리됨
        // 실제로는 삭제가 됐는데
    } catch (NotFoundException e) {
        log.error("삭제 실패"); // 잘못된 알람
    }
}
```

팀 내 클라이언트 코드가 **404를 "처리 실패"로 간주**하는 구조라면,  
재시도 시나리오에서 오탐(false negative)이 발생한다.

이 경우 서버가 200/204를 반환하는 게 더 안전하다.

**결론:** RFC는 둘 다 틀리다고 하지 않는다. 팀 내 클라이언트-서버 계약으로 정하면 된다.  
단, **한 시스템 안에서 일관되게** 정해야 한다.
