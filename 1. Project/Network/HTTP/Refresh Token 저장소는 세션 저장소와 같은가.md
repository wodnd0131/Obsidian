#Network/HTTP/Security #Network/HTTP #Network/JWT
타협은 맞다.

JWT의 본래 가치는 **"저장소 없이 서명만으로 인증"** 인데, Refresh Token을 서버에 저장하는 순간 저장소 의존이 다시 생긴다. 완전한 무상태는 아니게 된다.

이 긴장 관계를 업계가 어떻게 바라보는지, 그리고 다른 방향의 접근들을 정리한다.

---

## 1. 긴장 관계의 본질

```
JWT 본래 목표:
"저장소 없이 → 서명 검증만으로 → 어느 서버든 독립적으로 인증"

현실:
Access Token  → 무상태 ✓ (서명 검증만)
Refresh Token → 서버 저장소 필요 ✗ (무효화를 위해)

결국:
"완전한 무상태"는 이론적 이상
"Refresh Token 저장소"는 현실적 타협
```

그렇다면 왜 이 타협을 받아들이는가. 저장소 조회 빈도가 다르기 때문이다.

```
세션 방식:     모든 API 요청마다 저장소 조회
               초당 10만 요청 = 초당 10만 Redis 조회

JWT + Refresh: Access Token 유효한 동안은 저장소 조회 없음
               Refresh Token 조회는 15분에 한 번 (Access Token 만료 시에만)
               초당 10만 요청 ≠ 초당 10만 Redis 조회
```

완전한 무상태는 아니지만, **저장소 부하를 극적으로 줄이는 타협**이다.

---

## 2. 다른 방향의 접근들

### 2-1. 짧은 Access Token 단독 사용 — Refresh Token 자체를 없앤다

```
Access Token 만료를 매우 짧게 (5분) → 탈취 피해 시간 최소화
Refresh Token 없음 → 만료 시 재로그인

단점: UX가 나빠진다
      5분마다 로그인 = 실사용 불가
      
현실적 사용처:
- 금융/의료 등 보안이 극단적으로 중요한 영역
- 일회성 작업 토큰 (비밀번호 변경, 결제 확인 등)
```

---

### 2-2. 토큰 내부에 무효화 버전을 박는다 — 저장소 조회를 최소화

서버 저장소에 토큰 전체를 저장하는 대신, **사용자별 버전 숫자 하나만** 저장한다.

```json
// JWT Payload에 버전 포함
{
  "userId": 1,
  "tokenVersion": 7,
  "exp": ...
}
```

```
서버 저장소: {userId: 1, currentVersion: 7}  ← 숫자 하나만

검증 시:
1. 서명 검증 (저장소 조회 없음)
2. JWT의 tokenVersion(7) == DB의 currentVersion(7) ? ✓

강제 로그아웃:
DB에서 currentVersion을 8로 변경
→ 기존 발급된 모든 토큰의 tokenVersion(7) != currentVersion(8)
→ 전체 무효화
```

```
세션 방식:     요청마다 세션 전체 조회
Refresh 방식:  재발급 시마다 토큰 전체 조회
버전 방식:     요청마다 숫자 하나만 조회 (훨씬 가볍다)
               + 캐싱 적용 시 조회 빈도 더 감소
```

단, 이 방식도 저장소 조회가 모든 요청에 발생한다. 세션과 구조적으로 유사해진다. **JWT를 굳이 쓰는 이유가 줄어든다.**

---

### 2-3. Sender-Constrained Token — 토큰을 특정 클라이언트에 묶는다

탈취 자체의 **효용을 없애는** 방향이다.

현재 Bearer Token 방식의 문제:

```
Bearer Token:
"이 토큰을 가진 자(bearer)는 누구든 인증된다"
→ 탈취하면 그냥 쓸 수 있다
```

이를 해결하는 것이 **DPoP (Demonstrating Proof of Possession)** 이다.

```
클라이언트: 자신만의 공개키/개인키 쌍 생성
            토큰 발급 시 공개키를 서버에 등록
            → 서버는 이 공개키를 JWT에 바인딩해서 발급

API 요청 시:
Authorization: DPoP <JWT>
DPoP: <현재 요청에 대해 클라이언트 개인키로 서명한 proof>

서버 검증:
1. JWT 서명 검증
2. DPoP proof가 JWT에 바인딩된 공개키로 유효한가
3. 이 proof가 현재 요청(URL, 시각)에 대한 것인가

공격자가 JWT를 탈취해도:
→ 클라이언트의 개인키가 없으면 DPoP proof 생성 불가
→ 토큰 단독으로는 쓸 수 없음
```

> RFC 9449 (DPoP), Section 1: _"DPoP... can be used to prevent unauthorized or illegitimate parties from using leaked or stolen access tokens."_ (DPoP는 유출되거나 탈취된 액세스 토큰을 비인가된 또는 불법적인 당사자가 사용하는 것을 방지하는 데 사용될 수 있다.)

현재 OAuth 2.0 생태계에서 채택이 늘고 있는 방식이다. 단, 클라이언트가 키 쌍을 관리해야 하므로 구현 복잡도가 높다.

---

### 2-4. 세션으로 돌아간다 — JWT가 항상 답이 아니다

업계 일각에서는 아예 이렇게 말한다.

> _"Stop using JWT for sessions"_ — Joséph Heck, 2014 (보안 커뮤니티에서 널리 인용되는 글)

JWT가 유리한 상황과 세션이 유리한 상황이 명확히 다르다.

```
JWT가 유리한 상황:
- 여러 도메인/서비스가 토큰을 검증해야 하는 경우 (MSA, 서드파티 API)
- 인증 서버와 리소스 서버가 분리된 경우 (OAuth 2.0)
- 서버리스 환경 (Lambda 등 — 공유 세션 저장소 구성이 어려움)

세션이 유리한 상황:
- 단일 서비스, 단일 도메인
- 즉각적인 권한 취소가 중요한 경우 (금융, 의료)
- 팀이 JWT의 보안 복잡성을 감당하기 어려운 경우
```

## 3. 정리 — 무엇을 선택할 것인가

```
완전한 무상태 추구
        │
        ▼
Access Token 단독 (Refresh 없음)
→ 저장소 없음, 무효화 지연 감수, UX 희생

        │ UX 포기 못 하면
        ▼
JWT + Refresh Token (서버 저장)
→ 저장소 조회 빈도 낮춤, 완전한 무상태는 아님

        │ 탈취 위협 줄이고 싶으면
        ▼
DPoP 바인딩
→ 탈취해도 사용 불가, 구현 복잡도 높음

        │ 즉각 무효화가 핵심이면
        ▼
세션으로 회귀
→ 단순하고 명확, 저장소 부하는 감수
```

결국 이것이다.

> **JWT는 "저장소 조회를 줄이는 도구"지, "저장소를 완전히 없애는 도구"가 아니다.** 완전한 무상태와 즉각적 무효화는 근본적으로 양립하기 어렵다. 시스템의 요구사항(보안 수준, 규모, 팀 역량)에 따라 어느 지점에서 타협할지 결정하는 것이 설계다.
