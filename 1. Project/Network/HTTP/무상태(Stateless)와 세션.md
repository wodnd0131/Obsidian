#Network/HTTP/Security #Network/HTTP #Network/JWT
## 1. Problem First

### 무상태성이란 정확히 무엇인가

HTTP의 무상태성을 RFC는 이렇게 정의한다.

> RFC 9110, Section 3.7: _"HTTP is a stateless protocol, meaning that each request message's semantics can be understood in isolation."_ 
> (HTTP는 무상태 프로토콜이다. 즉, 각 요청 메시지의 의미는 독립적으로 이해될 수 있다.)

"독립적으로 이해될 수 있다" — 서버가 이 요청을 처리하는 데 **이전 요청의 기억이 전혀 필요 없다**는 뜻이다.

```
요청 1: GET /main       → 서버 처리 → 응답 → 서버는 이 요청을 완전히 잊음
요청 2: GET /products   → 서버 처리 → 응답 → 서버는 이 요청을 완전히 잊음
요청 3: POST /cart      → 서버는 요청 1, 2를 전혀 모름
```

각 요청은 서버 입장에서 **처음 만나는 낯선 사람**이다.

---

### 무상태성이 없었다면 — 설계 근거

먼저 "왜 상태를 저장하지 않도록 설계했는가"를 이해해야 한계도 보인다.

**서버가 상태를 기억하는 구조였다면:**

```
[사용자 1000만 명] ──▶ [서버]
                          │
                    사용자 1: {현재 페이지: /main, 마지막 클릭: ...}
                    사용자 2: {현재 페이지: /cart, 장바구니: [...]}
                    사용자 3: {...}
                    ...
                    사용자 1000만: {...}
```

- 서버 메모리가 접속자 수에 비례해 증가
- 서버를 여러 대로 늘리면(Scale Out) **어느 서버가 어느 사용자 상태를 갖고 있는가** 문제 발생
- 서버 한 대가 죽으면 그 서버가 기억하던 모든 사용자 상태가 사라짐

무상태 설계는 이 복잡성을 근본적으로 제거한다. 서버는 요청 하나를 처리하고 끝이다.

---

### 그런데 현실 서비스는 상태가 필요하다

```
// 쇼핑몰 시나리오
요청 1: POST /login      { id: "kim", pw: "1234" }  → 로그인 성공
요청 2: POST /cart/add   { productId: 42 }           → 누가 추가하는 건가?
요청 3: POST /order      { }                         → 누구의 주문인가?
```

서버가 요청 1을 잊어버리면 요청 2에서 "이 사람이 로그인한 사람인지" 알 수 없다. **인증 상태를 어딘가에 저장해야 한다.**

이것이 무상태성의 한계다. 그런데 이 한계를 해결하면서도 **HTTP 자체의 무상태성은 그대로 유지**된다. 이게 어떻게 가능한가.

---

## 2. Mechanics

### 2-1. 핵심 아이디어 — 상태를 서버가 아니라 요청에 담는다

무상태성의 정의를 다시 보자.

> "각 요청 메시지의 의미는 **독립적으로** 이해될 수 있다"

독립적으로 이해되려면, **필요한 정보가 요청 안에 다 들어있으면 된다.**

```
// 상태를 서버가 기억하는 방식 (Stateful)
요청 2: POST /cart/add { productId: 42 }
서버: "이전에 kim이 로그인했으니까 kim의 장바구니에 추가"  ← 서버가 기억

// 상태를 요청에 담는 방식 (Stateless 유지)
요청 2: POST /cart/add { productId: 42 }
        Cookie: session_id=abc123          ← 요청 안에 신원 정보 포함
서버: "abc123이 kim이라는 걸 조회해서 kim의 장바구니에 추가"
```

서버는 여전히 이전 요청을 **기억하지 않는다.** 클라이언트가 매 요청마다 필요한 정보를 **직접 들고 온다.** 이것이 무상태성을 잃지 않는 이유다.

---

### 2-2. 쿠키 — 클라이언트가 상태를 운반하는 메커니즘

쿠키는 **서버가 클라이언트에게 값을 저장시키고, 클라이언트가 이후 요청마다 자동으로 그 값을 돌려보내는** 메커니즘이다.

```
1단계: 서버 → 클라이언트 (쿠키 설정)

HTTP/1.1 200 OK
Set-Cookie: session_id=abc123; HttpOnly; Secure; SameSite=Strict; Max-Age=3600
            │             │         │          │              │
            저장할 값      JS접근불가  HTTPS만    CSRF방어      1시간 후 만료


2단계: 클라이언트 → 서버 (이후 모든 요청에 자동 포함)

GET /cart HTTP/1.1
Cookie: session_id=abc123    ← 브라우저가 자동으로 붙임
```

> RFC 6265, Section 1: _"This document defines the HTTP Cookie and Set-Cookie header fields. These header fields can be used by HTTP servers to store state (called cookies) at HTTP user agents, letting the servers maintain a stateful session over the mostly stateless HTTP protocol."_ (이 문서는 HTTP Cookie와 Set-Cookie 헤더 필드를 정의한다. 이 헤더 필드들은 HTTP 서버가 HTTP 사용자 에이전트에 상태(쿠키)를 저장하는 데 사용될 수 있으며, 대부분 무상태인 HTTP 프로토콜 위에서 서버가 상태 있는 세션을 유지하도록 한다.)

---

### 2-3. 세션 — 실제 상태는 어디에 저장되는가

쿠키만으로 모든 상태를 저장할 수는 없다. 쿠키는 클라이언트에 저장되므로:

- 사용자가 직접 열어볼 수 있다
- 조작 가능하다
- 용량 제한이 있다 (4KB)

그래서 실제 구현은 이렇게 나뉜다.

```
쿠키에는 "열쇠(식별자)"만 저장
실제 "금고(상태)"는 서버 측에 저장

[브라우저]                    [서버]                  [세션 저장소]
   │                           │                          │
   │──GET /cart ──────────────▶│                          │
   │  Cookie: session_id=abc123│                          │
   │                           │──조회(abc123)───────────▶│
   │                           │◀─{userId:1, cart:[...]}──│
   │                           │                          │
   │◀──200 OK + 장바구니 내용───│
```

**세션 저장소의 종류와 트레이드오프:**

|저장소|특징|문제점|
|---|---|---|
|서버 메모리 (기본값)|빠름|서버 재시작 시 세션 소멸, 서버 여러 대면 문제|
|Redis|빠르고 서버 독립적|별도 인프라 필요|
|DB|영속성 보장|느림|
|Sticky Session|같은 사용자는 항상 같은 서버로|특정 서버에 부하 집중|

---

### 2-4. 서버 여러 대일 때 세션의 문제

무상태성이 Scale Out에 유리하다고 했는데, 세션 저장소가 서버 메모리에 있으면 이 장점이 사라진다.

```
[사용자] ──로그인──▶ [서버 1] (세션 abc123 저장)
[사용자] ──장바구니──▶ [서버 2] (abc123? 모름 → 로그인 필요)
```

이를 해결하는 방법이 **세션 저장소 외부화**다.

```java
// Spring Session + Redis로 세션을 서버 외부에 저장
// build.gradle
// implementation 'org.springframework.session:spring-session-data-redis'

// application.yml
// spring:
//   session:
//     store-type: redis

// 이것만으로 세션이 Redis에 저장됨
// 어느 서버로 요청이 가도 같은 세션 조회 가능
```

```
[사용자] ──로그인──▶ [서버 1] ──세션 저장──▶ [Redis]
[사용자] ──장바구니──▶ [서버 2] ──세션 조회──▶ [Redis] → abc123 찾음
```

이 구조에서 HTTP는 여전히 무상태다. 서버가 이전 요청을 기억하는 게 아니라, 매 요청마다 Redis를 조회하는 것이다.

#### 세션 저장소는 무상태성을 깨는가

엄밀히 말하면 **그렇다.** 세션 방식은 HTTP 무상태성의 정신에서 벗어난다.

```
순수 무상태:
요청 자체에 모든 정보 포함 → 서버는 요청만 보고 처리 완료

세션 방식:
요청에 session_id 포함 → 서버가 저장소 조회 → 이전 상태 참조 → 처리
                                    ↑
                           이전 요청의 흔적에 의존
```

서버가 이전 요청이 남긴 상태(세션)를 참조해야 현재 요청을 처리할 수 있다. 이건 분명히 **상태 의존**이다.

---

#### 그러면 왜 "무상태 위에서 세션을 구현한다"고 말하는가

표현이 정확하지 않아서 생기는 혼란이다. 더 정확히는 이렇다.

```
HTTP 프로토콜 자체는 무상태를 유지한다
            +
애플리케이션 레이어에서 상태를 별도로 관리한다
```

HTTP는 session_id를 그냥 문자열 헤더로 전달할 뿐이다. 그 문자열이 무슨 의미인지, 어디서 조회하는지는 HTTP가 모른다. HTTP 입장에서는 그냥 쿠키 헤더 하나를 전달한 것이다.

```
HTTP가 보는 것:   Cookie: session_id=abc123  (그냥 헤더)
애플리케이션이 하는 것: abc123으로 Redis 조회 → 상태 참조
```

레이어를 나누면 이렇게 된다.

```
HTTP 레이어:        무상태 ✓ (각 요청은 독립적인 메시지)
애플리케이션 레이어: 유상태 (세션 저장소에 상태 보관)
```

세션 방식은 시스템 전체로 보면 상태가 있는 구조다. "무상태 HTTP 위에 세션을 구현한다"는 말은 엄밀하게는 **"HTTP 프로토콜 규격을 어기지 않으면서, 애플리케이션 레이어에서 상태를 관리한다"** 는 뜻이다.

JWT가 진짜로 노리는 것은 이 애플리케이션 레이어의 상태도 없애는 것이다. 서버 어디에도 상태를 저장하지 않고, 토큰 자체가 필요한 모든 정보를 담는다. 그래서 JWT를 쓸 때 비로소 **서버 전체가 진정한 무상태**에 가까워진다. 단, 앞서 다룬 것처럼 토큰 무효화 문제라는 대가를 치른다.

---

### 2-5. [[JWT]] — 세션 저장소 자체를 없애는 접근

세션은 서버 측에 저장소가 필요하다는 문제가 있다. 
**JWT(JSON Web Token)** 는 상태를 토큰 자체에 담아 저장소를 없애는 방식이다.

```
// JWT 구조: Header.Payload.Signature
eyJhbGciOiJIUzI1NiJ9.eyJ1c2VySWQiOjF9.abc123signature

// Payload 디코딩하면:
{
  "userId": 1,
  "role": "USER",
  "exp": 1714176000    ← 만료시간
}
```

```
[클라이언트]                         [서버]
     │                                 │
     │──POST /login───────────────────▶│
     │◀──JWT 토큰 발급─────────────────│  세션 저장소에 아무것도 저장 안 함
     │                                 │
     │──GET /cart─────────────────────▶│
     │  Authorization: Bearer [JWT]     │
     │                                 │ JWT 서명 검증만 수행
     │                                 │ (저장소 조회 없음)
     │◀──200 OK───────────────────────│
```

**JWT가 무상태성을 완전히 유지하는 이유:**

서버는 토큰의 서명만 검증한다. 어떤 서버에서 검증해도 결과가 같다. 저장소가 필요 없다.

**JWT의 트레이드오프:**

|        | 세션               | JWT       |
| ------ | ---------------- | --------- |
| 서버 저장소 | 필요               | 불필요       |
| 토큰 무효화 | 즉시 가능 (저장소에서 삭제) | **어렵다**   |
| 토큰 크기  | 작음 (ID만)         | 큼 (정보 포함) |
| 서버 확장성 | Redis 필요         | 자유로움      |

JWT의 핵심 단점은 **발급한 토큰을 중간에 무효화하기 어렵다**는 것이다. 사용자가 로그아웃해도 토큰 만료 전까지는 기술적으로 유효하다. 
이를 해결하려면 블랙리스트를 Redis에 유지해야 하는데, 그러면 결국 저장소가 다시 필요해진다.

## 4. 이 설계를 이렇게 한 이유

### 무상태 + 쿠키/세션 조합이 얻는 것
- HTTP 인프라(서버, 프록시, 캐시)는 무상태 설계 그대로 유지
- 상태 관리 책임을 애플리케이션 레이어로 분리
- 서버 확장이 자유로움 (상태를 외부 저장소로 분리하면)

### 잃는 것

| 손실                      | 구체적 상황                                                                          |
| ----------------------- | ------------------------------------------------------------------------------- |
| **쿠키 보안 취약점**           | XSS 공격으로 쿠키 탈취 가능 → `HttpOnly`로 완화. CSRF 공격으로 쿠키 도용 가능 → `SameSite=Strict`으로 완화 |
| **세션 저장소가 단일 장애점**      | Redis가 죽으면 전체 사용자 로그아웃. HA 구성 필수                                                |
| **JWT 무효화 불가**          | 탈취된 JWT를 만료 전에 막으려면 블랙리스트 필요 → 무상태성 일부 포기                                       |
| **모바일/API 환경에서 쿠키 부적합** | 쿠키는 브라우저 중심 메커니즘. 모바일 앱은 `Authorization: Bearer` 헤더로 JWT를 직접 관리하는 방식을 선호        |

|순서|개념|이유|
|---|---|---|
|**1**|**쿠키 보안 속성 (`HttpOnly`, `Secure`, `SameSite`) + XSS/CSRF**|세션/쿠키를 도입하는 순간 이 공격 벡터가 생긴다. 구현 전에 반드시 알아야 할 보안 지식|

|**3**|**Spring Security 세션/JWT 처리 흐름**|위 개념들이 Spring에서 어떻게 구현되는지. `SecurityContextHolder`, `OncePerRequestFilter` 등 실무 코드와 직결|
