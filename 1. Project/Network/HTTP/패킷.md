# 패킷 (Packet)

패킷 또는 네트워크 패킷은 네트워크를 통해 전송되는 형식화된 데이터 덩어리입니다. 네트워크 패킷의 주요 컴포넌트는 사용자 데이터와 제어 정보입니다. 사용자 데이터는 '페이로드(payload)'로 알려져 있습니다. 제어정보는 페이로드를 전달하기 위한 정보입니다. 소스와 목적지에 대한 네트워크 주소, 순서 정보, 오류 감지 코드로 구성되며 일반적으로 패킷 헤더와 푸터에서 발견됩니다.

## 패킷에 포함된 내용

### 홉 제한

홉은 패킷이 한 네트워크에서 다음 네트워크로 전달될 때 발생합니다. 패킷이 통과할 때마다 1씩 감소하는 필드입니다. 홉 제한이 0에 도달하면 전송 작업이 실패하고 패킷이 삭제됩니다.

시간이 지남에 따라 패킷 수로 인해 폐쇄 회로 내에서 횡단이 발생할 수 있고, 순환하는 패킷 수가 증가하여 궁극적으로 네트워킹 실패로 이어질 수 있습니다.

### 오류 감지 및 수정

오류 감지 및 수정은 데이터가 수신기로 전송될 때 발생하는 오류를 감지하고 수정하는 데 사용되는 코드입니다. 오류 수정에는 역방향 오류 수정과 순방향 오류 수정의 두 가지 유형이 있습니다. 역뱡향 오류 수정은 수신자가 발신자에게 전체 데이터 단위를 재전송하도록 요청하는 경우입니다. 순방향 오류 수정은 수신자가 자동으로 오류를 수정하는 오류 수정 코드를 사용하는 것입니다.

송신기에서는, 패킷이 전송되기 전에 계산이 수행됩니다. 대상에서 수신되면 체크섬이 다시 계산되어 패킷의 체크섬과 비교됩니다.

### 우선 순위

이 필드는 어떤 패킷이 다른 패킷보다 더 높은 우선순위를 가져야 하는지를 나타냅니다. 높은 우선순위 대기열은 네트워크가 혼잡할 때 낮은 우선순위 대기열보다 더 빨리 비워지게 됩니다.

### 주소

네트워크 패킷을 라우팅할 때 송신 **호스트의 소스 주소**와 **수신 호스트의 대상 주소**라는 두 개의 네트워크 주소가 필요합니다.

### 사용자 데이터 - 페이로드

페이로드는 애플리케이션을 대신하여 전달되는 데이터입니다. 일반적으로 네트워크 프로토콜과 때로는 경로의 장비에 의해 설정된 최대 길이까지 가변 길이입니다.

# 패킷과 홉(Hop) — 네트워크에서 데이터가 실제로 이동하는 방식

## 1. Problem First

### "인터넷에서 데이터를 보낸다" 는 게 실제로 무슨 뜻인가

서울 서버에서 뉴욕 서버로 데이터를 보낼 때, 두 서버가 직접 연결된 전선 하나로 이어져 있다고 생각하기 쉽다. 하지만 실제로는 그렇지 않다.

인터넷은 수많은 **라우터(Router)** 들이 연결된 그물망이다. 데이터는 이 라우터들을 **하나씩 거쳐가며** 목적지로 향한다.

### 홉 제한이 없다면 생기는 문제

```
[서울] → [도쿄] → [LA] → [뉴욕]
                ↑
         라우팅 테이블 오류 발생
         
[뉴욕] → [LA] → [도쿄] → [뉴욕] → [LA] → [도쿄] → ...
```

라우팅 설정 오류나 네트워크 장애로 패킷이 **루프(loop)** 에 빠지면, 홉 제한이 없을 경우 해당 패킷은 **영원히 네트워크를 떠돌게 된다.**

라우터 하나가 초당 수백만 개의 패킷을 처리하는 상황에서, 좀비 패킷들이 누적되면 네트워크 전체가 마비된다. 이를 막기 위해 설계된 것이 **홉 제한(Hop Limit)** 이다.

---

## 2. Mechanics

### 2-1. 패킷이란 무엇인가 — 구조부터

TCP는 앞서 다뤘으니, 그 아래 레이어인 **IP 패킷**에 집중한다.

네트워크 계층 모델로 보면 위치는 이렇다.

```
[ 애플리케이션 ]  HTTP 메시지
      ↓
[  전송 계층  ]  TCP 세그먼트  ← 앞서 다룬 영역
      ↓
[  네트워크 계층 ]  IP 패킷     ← 지금 다루는 영역
      ↓
[  데이터링크 계층 ]  이더넷 프레임
      ↓
[  물리 계층  ]  실제 전기/광신호
```

TCP 세그먼트가 IP 계층으로 내려오면, **IP 헤더가 앞에 붙어** IP 패킷이 된다.

**IPv4 패킷 헤더 구조** (RFC 791)

![[../../../Repository/패킷-IPv4 패킷 헤더 구조 (RFC 791).png]]

여기서 핵심 필드는 **TTL (Time to Live)** — IPv4에서 홉 제한을 구현하는 필드다.

IPv6에서는 이름이 명확하게 **Hop Limit** 으로 바뀐다. (RFC 8200)

---

### 2-2. 홉(Hop)이란 정확히 무엇인가

**홉 = 라우터 한 개를 통과하는 것**

```
[내 서버]──────[라우터 A]──────[라우터 B]──────[라우터 C]──────[목적지]
         홉 1           홉 2           홉 3           홉 4
```

"라우터를 거친다" 는 게 실제로 어떤 일인가.

```
라우터가 패킷을 받았을 때 하는 일:

1. 패킷의 IP 헤더에서 Destination Address 읽기
2. 자신의 라우팅 테이블 조회
   ("192.0.2.x 로 가려면 다음 라우터(next-hop)는 저쪽이군")
3. TTL 값을 1 감소
4. TTL이 0이 됐는가?
   - YES → 패킷 폐기 + 출발지에 ICMP "Time Exceeded" 메시지 전송
   - NO  → 다음 라우터로 포워딩
```

핵심은 **라우터가 패킷을 저장하지 않는다** 는 것이다. 읽고, TTL 줄이고, 던진다. 이게 인터넷이 빠른 이유다.

---

### 2-3. TTL 값의 생애 주기

패킷이 출발할 때 OS가 TTL 초기값을 설정한다.

|OS|기본 TTL 초기값|
|---|---|
|Linux|64|
|Windows|128|
|macOS|64|
|오래된 Unix|255|

> 이 기본값들은 OS 커널 소스에서 확인 가능하다. Linux의 경우 `net/ipv4/ip_output.c` 에서 `IPDEFTTL` 상수로 정의되며, 이는 `/proc/sys/net/ipv4/ip_default_ttl` 로 런타임 조회 가능하다.

```
출발 (TTL=64)
    │
    ▼
[라우터 A]  TTL: 64 → 63  포워딩
    │
    ▼
[라우터 B]  TTL: 63 → 62  포워딩
    │
    ▼
    ...
    │
    ▼
[라우터 N]  TTL: 1  → 0   패킷 폐기
                           ICMP Type 11 "Time Exceeded" → 출발지 전송
```

---

### 2-4. ICMP — 라우터가 보내는 사망 통보

TTL이 0이 되면 라우터는 조용히 버리지 않는다. 출발지 IP로 **ICMP(Internet Control Message Protocol) Type 11, Code 0 — "Time Exceeded"** 메시지를 보낸다.

> RFC 792: _"If the gateway processing a datagram finds the time to live field is zero it must discard the datagram. The gateway may also notify the source host via the time exceeded message."_ (게이트웨이가 데이터그램을 처리하는 중 TTL 필드가 0임을 발견하면 반드시 해당 데이터그램을 폐기해야 한다. 게이트웨이는 또한 Time Exceeded 메시지를 통해 출발지 호스트에 알릴 수 있다.)

이 ICMP 응답에는 **어느 라우터에서 폐기됐는지(= 해당 라우터의 IP 주소)** 가 담겨 있다.

---

### 2-5. traceroute — TTL을 무기로 경로를 추적하는 도구

`traceroute` (Windows: `tracert`) 는 이 TTL 메커니즘을 **역이용** 해서 경로상의 모든 라우터를 알아낸다.

```
원리:
TTL=1 패킷 전송 → 첫 번째 라우터에서 폐기 → ICMP로 해당 라우터 IP 획득
TTL=2 패킷 전송 → 두 번째 라우터에서 폐기 → ICMP로 해당 라우터 IP 획득
TTL=3 패킷 전송 → 세 번째 라우터에서 폐기 → ICMP로 해당 라우터 IP 획득
...
TTL=N 패킷 전송 → 목적지 도달 → 완료
```

실제 출력 예시 (`traceroute google.com` 을 서울에서 실행):

```
traceroute to google.com (142.250.196.142)
 1  192.168.1.1       1.2 ms   ← 내 공유기 (홉 1)
 2  10.0.0.1          3.4 ms   ← ISP 첫 번째 라우터 (홉 2)
 3  112.188.x.x       5.1 ms   ← KT 백본 (홉 3)
 4  * * *                      ← ICMP 차단된 라우터 (홉 4, IP 모름)
 5  72.14.x.x        31.2 ms   ← Google 피어링 포인트 (홉 5)
 6  142.250.196.142  33.8 ms   ← 목적지 도달
```

`* * *` 는 해당 라우터가 보안상 ICMP 응답을 차단한 것이다. 패킷은 통과하지만 자신의 존재를 알리지 않는다.

---

### 2-6. IPv6에서의 변화 — TTL → Hop Limit

IPv6(RFC 8200)에서는 필드 이름이 **Hop Limit** 으로 바뀐다.

> RFC 8200, Section 3: _"Hop Limit: 8-bit unsigned integer. Decremented by 1 by each node that forwards the packet. The packet is discarded if Hop Limit is decremented to zero."_ (Hop Limit: 8비트 부호 없는 정수. 패킷을 포워딩하는 각 노드에 의해 1씩 감소된다. Hop Limit이 0으로 감소되면 패킷은 폐기된다.)

기능은 동일하다. 이름이 바뀐 이유는 원래 TTL이 **시간(초) 단위** 로 설계됐기 때문이다.

RFC 791 원본 설계에서는 라우터가 패킷을 1초 이상 처리하면 TTL을 추가로 감소시켜야 했다. 하지만 실제 구현에서 라우터 처리 시간은 마이크로초 단위라 모든 구현이 그냥 "1씩 감소" 로 통일됐다. IPv6는 이 혼란을 정리하고 실제 의미인 "홉 횟수 제한" 을 이름에 반영했다.

## 3. 공식 근거

|주장|출처|
|---|---|
|IPv4 TTL 필드 정의 및 동작|RFC 791, Section 3.1 — _"Time to Live: This field indicates the maximum time the datagram is allowed to remain in the internet system."_|
|TTL=0 시 폐기 및 ICMP 통보|RFC 792 (ICMP), Type 11 정의|
|IPv6 Hop Limit 정의|RFC 8200, Section 3|
|Linux 기본 TTL 값|Linux 커널 소스 `include/net/ip.h`, `IPDEFTTL = 64`|
|traceroute 동작 원리|RFC 792 + 각 OS 구현체. 공식 알고리즘 문서는 없으며, Van Jacobson이 1988년 최초 구현|

## 4. 이 설계를 이렇게 한 이유

### 얻는 것

- 라우팅 루프로 인한 좀비 패킷 누적 방지
- 네트워크 자원(라우터 CPU, 회선 대역폭) 보호
- ICMP와 결합해 경로 디버깅 도구(traceroute) 구현 가능

### 잃는 것 / 한계

|한계|구체적 상황|
|---|---|
|**TTL 값이 작으면 정상 패킷도 폐기**|기본값 64는 대부분 충분하지만, 위성 통신이나 복잡한 VPN 터널 환경에서는 홉이 많아 도달 실패 가능|
|**루프 감지가 아니라 루프 제한**|TTL은 루프를 **탐지해 고치는** 게 아니라 **일정 횟수 후 포기** 하는 것이다. 근본 원인(잘못된 라우팅 테이블)은 BGP/OSPF 같은 라우팅 프로토콜이 해결해야 한다|
|**ICMP 차단 환경에서 디버깅 불가**|보안 정책으로 ICMP를 막으면 traceroute의 `* * *` 처럼 경로 파악이 불가능해진다|
|**TTL은 보안 도구가 아니다**|TTL 값을 보고 OS를 추정하는 **OS Fingerprinting** 공격에 활용될 수 있다. 공격자가 목적지까지의 홉 수를 알면 초기 TTL을 역산해 OS를 추정한다|

## 5. 이어지는 개념

|순서|개념|이유|
|---|---|---|
|**1**|**라우팅 테이블과 라우터의 포워딩 결정**|홉을 이해했으면 "라우터가 다음 홉을 어떻게 결정하는가" 가 자연스러운 다음 질문이다. 이걸 알아야 패킷의 경로 전체가 보인다|
|**2**|**ICMP 전체**|TTL 만료 외에 `ping`, 경로 MTU 탐색 등 네트워크 진단의 핵심. traceroute를 이해했으면 ICMP의 다른 역할도 같이 정리할 필요가 있다|
|**3**|**MTU와 패킷 단편화(Fragmentation)**|패킷이 너무 크면 라우터가 쪼개야 한다. 이것도 IP 헤더(Flags, Fragment Offset)와 직접 연관되며, HTTP 성능 문제와도 이어진다|
|**4**|**BGP / OSPF — 라우팅 프로토콜**|라우팅 테이블이 어떻게 만들어지는가. AWS VPC, 온프레미스 연동을 다루면 반드시 마주치는 개념|
