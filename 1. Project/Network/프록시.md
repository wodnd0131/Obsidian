# 프록시 (Proxy)

## 1. Problem First

### 클라이언트와 서버가 직접 연결되면 생기는 문제들

```
[브라우저] ──────────────────────────────▶ [서버]
```

이 단순한 구조에서 발생하는 현실적인 문제들:

```
문제 1. 서버가 하나면 트래픽이 몰릴 때 죽는다
문제 2. 서버 IP가 직접 노출되면 공격 대상이 된다
문제 3. 사내에서 특정 사이트 접근을 막을 수단이 없다
문제 4. 누가 어떤 요청을 했는지 기록이 없다
문제 5. 같은 응답을 매번 서버에서 만들어야 한다 (캐싱 편 참고)
```

이 문제들을 클라이언트나 서버 코드를 건드리지 않고, **중간에서** 해결하는 것이 프록시다.

---

## 2. Mechanics

### 2-1. 프록시의 핵심 특성 — 투명성

인용문에서 말하는 "눈에 보이거나 그렇지 않을 수도 있다"는 것이 바로 **투명성(Transparency)** 이다.

```
투명 프록시 (Transparent Proxy):
[클라이언트] ──요청 그대로 전달──▶ [프록시] ──요청 그대로──▶ [서버]
 클라이언트는 프록시 존재를 모름. 요청/응답이 변경되지 않음

비투명 프록시 (Non-transparent Proxy):
[클라이언트] ──요청──▶ [프록시] ──변형된 요청──▶ [서버]
 헤더 추가/제거, 내용 검사, 응답 수정 등이 일어남
```

ISP나 회사 네트워크에서 트래픽을 검사하는 프록시는 대부분 **투명 프록시**다. 사용자가 별도 설정을 하지 않아도 모든 트래픽이 자동으로 프록시를 거친다.

---

### 2-2. 위치에 따른 분류 — 누구 앞에 서있는가

이건 앞서 캐싱 편에서 다뤘지만, 기능 설명의 전제가 되므로 요약한다.

```
포워드 프록시:          클라이언트 집단 앞
[사내 PC] ──▶ [프록시] ──▶ 외부 인터넷

리버스 프록시:          서버 집단 앞
외부 인터넷 ──▶ [프록시] ──▶ [서버들]
```

**어떤 기능이 어느 위치에서 주로 쓰이는가:**

|기능|포워드 프록시|리버스 프록시|
|---|---|---|
|캐싱|✓ (사내 트래픽 절감)|✓ (CDN, Nginx)|
|필터링|✓ (유해 사이트 차단)|✓ (WAF, 바이러스 검사)|
|로드 밸런싱|✗|✓ (ALB, Nginx)|
|인증|△ (사용자 인증)|✓ (리소스 접근 제어)|
|로깅|✓ (사용자 행동 기록)|✓ (API 호출 기록)|

---

### 2-3. 기능별 내부 동작

#### ① 캐싱 — 이전 편에서 상세히 다룸

```
[클라이언트] ──GET /image.png──▶ [프록시]
                                    │ 저장된 응답 있음
[클라이언트] ◀──200 OK + 이미지───── 원본 서버 요청 없음
```

`Cache-Control: public`인 응답만 프록시가 저장 가능. `private`이면 브라우저만 저장.

---

#### ② 필터링 — 요청/응답 내용 검사

```
[사내 PC] ──GET http://malware-site.com──▶ [포워드 프록시]
                                                │
                                    URL 블랙리스트 확인
                                    바이러스 시그니처 검사
                                                │ 차단
[사내 PC] ◀──403 Forbidden / 차단 페이지──────────
                                    원본 서버에 요청 도달 안 함
```

리버스 프록시에서는 **WAF(Web Application Firewall)** 형태로 동작한다.

```
외부 공격자 ──POST /api (SQL Injection 패턴)──▶ [WAF/리버스 프록시]
                                                      │
                                          요청 페이로드 검사
                                          악성 패턴 탐지
                                                      │ 차단
[Spring 서버]에는 요청이 도달하지 않음
```

---

#### ③ 로드 밸런싱 — 트래픽 분산

리버스 프록시만의 기능이다. 단일 서버의 한계를 극복한다.

```
                              ┌──▶ [서버 1] (정상)
외부 요청 ──▶ [리버스 프록시] ──▶ [서버 2] (정상)
                              └──▶ [서버 3] (과부하 → 요청 줄임)
```

분산 알고리즘:

|알고리즘|동작|적합한 상황|
|---|---|---|
|Round Robin|순서대로 돌아가며 배분|요청 처리 시간이 균일할 때|
|Least Connections|현재 연결이 가장 적은 서버로|처리 시간이 들쭉날쭉할 때|
|IP Hash|클라이언트 IP 기반 고정 배분|세션 유지가 필요할 때|

AWS ALB는 기본적으로 Round Robin을 사용하고, 느린 서버를 자동으로 감지해 배분을 줄이는 **Slow Start** 기능을 제공한다.

---

#### ④ 인증 — 리소스 접근 제어

Spring 서버까지 요청이 도달하기 전에 프록시 레이어에서 인증을 처리할 수 있다.

```
[클라이언트] ──GET /admin (토큰 없음)──▶ [리버스 프록시]
                                              │
                                      Authorization 헤더 확인
                                              │ 없음
[클라이언트] ◀──401 Unauthorized──────────────
                                    Spring 서버에 요청 도달 안 함
```

AWS API Gateway + Lambda Authorizer 구조가 이 패턴이다. Spring 서버는 이미 인증된 요청만 받는다.

---

#### ⑤ 로깅 — 모든 요청의 기록

프록시는 모든 트래픽이 통과하는 지점이므로, **중앙 집중식 로깅**에 이상적이다.

```
[클라이언트 A] ──▶ ┐
[클라이언트 B] ──▶ ├──▶ [프록시/로드밸런서] ──▶ [서버들]
[클라이언트 C] ──▶ ┘         │
                         모든 요청 기록
                         - 시각, IP, URL, 응답코드, 응답시간
                              │
                         [중앙 로그 저장소]
```

Spring 서버가 여러 대일 때 각 서버 로그를 따로 보는 것보다, **프록시 로그 하나**로 전체 트래픽을 파악할 수 있다. AWS ALB Access Log가 이 역할을 한다.

---

### 2-4. 실무에서 마주치는 프록시들

```
[브라우저]
    │
    ▼
[포워드 프록시]  ← 사내 네트워크, VPN
    │
    ▼ (인터넷)
    │
    ▼
[CDN 엣지 서버]  ← CloudFront (캐싱 + 필터링)
    │
    ▼
[로드밸런서]     ← AWS ALB (로드밸런싱 + 인증 + 로깅)
    │
    ▼
[API Gateway]   ← 인증 + 필터링 + 로깅
    │
    ▼
[Spring 서버]
```

Spring 서버에 요청이 도달할 때까지 이미 여러 프록시를 거친다. 이것이 Spring에서 `request.getRemoteAddr()`가 **클라이언트 실제 IP 대신 프록시 IP를 반환하는 이유**다.

실제 클라이언트 IP는 프록시가 추가한 헤더에서 꺼내야 한다.

```java
// 프록시를 거친 경우 실제 IP 추출
String clientIp = request.getHeader("X-Forwarded-For");
// ALB가 "실제IP, 프록시IP" 형태로 추가하는 헤더
```

> RFC 7239: _"In addition to the issues with using X-Forwarded-For, the Forwarded header field allows for more complete information about the chain of communication."_ (X-Forwarded-For 사용의 문제에 더해, Forwarded 헤더 필드는 통신 체인에 대해 더 완전한 정보를 제공한다.)

## 4. 이 설계를 이렇게 한 이유

### 얻는 것

- 클라이언트/서버 코드 변경 없이 **횡단 관심사(cross-cutting concerns)** 처리
- 보안, 성능, 가용성을 인프라 레이어에서 통합 관리

### 잃는 것 / 트레이드오프

| 손실                  | 구체적 상황                                                                                 |
| ------------------- | -------------------------------------------------------------------------------------- |
| **단일 장애점**          | 프록시 하나가 죽으면 전체 트래픽이 막힌다. 고가용성(HA) 구성 필수                                                |
| **레이턴시 추가**         | 중간 홉이 하나 늘어난다. 일반적으로 수 ms지만 체인이 길면 누적                                                  |
| **클라이언트 IP 소실**     | `X-Forwarded-For` 헤더를 신뢰해야 하는데, 이 헤더는 클라이언트가 위조할 수 있다. IP 기반 인증에 취약점이 생긴다              |
| **HTTPS 트래픽 검사 불가** | 암호화된 트래픽은 프록시가 내용을 볼 수 없다. 필터링/로깅이 제한된다. 기업 환경에서 SSL Inspection을 강제하면 TLS 신뢰 체계가 복잡해진다 |
| **설정 복잡도**          | 프록시 체인이 길어질수록 디버깅이 어렵다. 어느 레이어에서 요청이 변형됐는지 추적하기 힘들다                                    |

# 캐시 역할을 하는 프록시 (Caching Proxy)

## 1. Problem First

### 같은 데이터를 매번 원본 서버에서 가져와야 하는가

뉴스 사이트 메인 페이지를 생각해보자. 1초에 1000명이 같은 페이지를 요청한다.

```
[사용자 1000명] ──── 동일한 GET /index.html ────▶ [뉴스 서버]
                                                       │
                                              매번 DB 조회
                                              매번 HTML 렌더링
                                              매번 응답 전송
                                                  (1000회)
```

내용이 바뀌지도 않았는데 서버는 동일한 작업을 1000번 반복한다. 이 구조에서 발생하는 문제는 두 가지다.

**서버 입장**: 불필요한 연산과 대역폭 낭비 
**사용자 입장**: 서버가 멀수록 응답이 느리다 (한국 사용자가 미국 서버에 매번 요청)

---

## 2. Mechanics

### 2-1. 캐싱 프록시의 기본 동작

```
첫 번째 요청:
[클라이언트] ──GET /index.html──▶ [캐싱 프록시]
                                       │ 캐시 없음
                                       │──GET /index.html──▶ [원본 서버]
                                       │◀──200 OK + HTML────
                                       │ 캐시에 저장
[클라이언트] ◀──200 OK + HTML──────────

두 번째 요청 (캐시 유효 기간 내):
[클라이언트] ──GET /index.html──▶ [캐싱 프록시]
                                       │ 캐시 있음 ✓
[클라이언트] ◀──200 OK + HTML──────────
                                  원본 서버에 요청 없음
```

핵심은 **원본 서버까지 요청이 도달하지 않는다**는 것이다. 프록시가 저장해둔 응답을 바로 돌려준다.

---

### 2-2. 어떤 응답을 캐싱할 수 있는가 — HTTP 캐시 헤더

캐싱 프록시는 **HTTP 헤더**를 보고 이 응답을 저장해도 되는지, 얼마나 저장해도 되는지 판단한다.

**Cache-Control 헤더** — 캐싱 동작의 핵심 지시어

```http
HTTP/1.1 200 OK
Cache-Control: public, max-age=3600
Content-Type: text/html

<html>...</html>
```

|지시어|의미|
|---|---|
|`public`|프록시 같은 중간 서버도 캐싱 가능|
|`private`|브라우저(최종 사용자)만 캐싱 가능, **프록시는 캐싱 불가**|
|`max-age=3600`|3600초(1시간) 동안 유효|
|`no-cache`|캐시 저장은 하되, 쓰기 전에 서버에 유효성 검증 필요|
|`no-store`|캐시 저장 자체를 금지|

> RFC 9111 (HTTP Caching), Section 5.2: _"The 'public' response directive indicates that any cache MAY store a response, even if the response would normally be non-cacheable."_ ('public' 응답 지시어는 일반적으로 캐시 불가능한 응답이더라도 어떤 캐시든 응답을 저장할 수 있음을 나타낸다.)

---

### 2-3. 캐시가 만료됐을 때 — 검증(Validation)

`max-age`가 지나면 프록시는 캐시가 여전히 유효한지 서버에 확인한다. 이때 **전체 응답을 다시 받는 게 아니라** 변경 여부만 확인한다.

**방법 1: ETag (Entity Tag)**

```http
# 서버 첫 응답
HTTP/1.1 200 OK
ETag: "abc123"         ← 이 응답의 고유 식별자 (내용의 해시값 등)
Cache-Control: max-age=3600

# 캐시 만료 후 프록시가 검증 요청
GET /index.html HTTP/1.1
If-None-Match: "abc123"  ← "이 ETag랑 같으면 안 보내도 돼"

# 서버 응답 - 변경 없음
HTTP/1.1 304 Not Modified   ← 본문 없이 헤더만 전송
                             프록시는 기존 캐시 계속 사용

# 서버 응답 - 변경됨
HTTP/1.1 200 OK
ETag: "xyz789"          ← 새로운 ETag
[새로운 본문]
```

**방법 2: Last-Modified**

```http
# 서버 첫 응답
Last-Modified: Mon, 27 Apr 2026 09:00:00 GMT

# 검증 요청
If-Modified-Since: Mon, 27 Apr 2026 09:00:00 GMT

# 변경 없으면 304, 변경됐으면 200 + 새 내용
```

이 덕분에 내용이 바뀌지 않은 경우 **네트워크 대역폭을 거의 쓰지 않고** 캐시 유효성을 확인할 수 있다.

---

### 2-4. 포워드 프록시 vs 리버스 프록시 — 누구 앞에 있는가

캐싱 프록시는 위치에 따라 두 종류로 나뉜다.

**포워드 프록시 (Forward Proxy)** — 클라이언트 앞에 위치

```
[사내 PC들]
[사내 PC들] ──▶ [포워드 프록시] ──▶ 인터넷 ──▶ [외부 서버]
[사내 PC들]

- 클라이언트들이 프록시를 통해 외부에 접근
- 외부 서버 입장에서 클라이언트 IP가 숨겨짐
- 사용처: 사내 인터넷 필터링, 트래픽 캐싱으로 외부 대역폭 절감
```

**리버스 프록시 (Reverse Proxy)** — 서버 앞에 위치

```
인터넷 ──▶ [리버스 프록시] ──▶ [서버 1]
                          ──▶ [서버 2]
                          ──▶ [서버 3]

- 클라이언트 입장에서는 프록시가 서버처럼 보임
- 실제 서버 IP가 숨겨짐
- 사용처: CDN, 로드밸런서, API Gateway, Nginx
```

Spring 애플리케이션 앞에 놓이는 **Nginx, AWS ALB, CloudFront** 가 모두 리버스 프록시다.

### 2-5. 실제로 어디서 마주치는가

**CDN (Content Delivery Network)** — 지리적으로 분산된 캐싱 프록시

```
[한국 사용자] ──▶ [AWS CloudFront 서울 엣지] ──(캐시 없을 때만)──▶ [미국 오리진 서버]
                         │
                  캐시 있으면 여기서 바로 응답
                  RTT: 서울→서울 ~5ms
                  (vs 서울→미국 ~150ms)
```

**Nginx 리버스 프록시 캐싱**

```nginx
# Spring 앞에 Nginx를 두고 정적 응답 캐싱
proxy_cache_path /var/cache/nginx levels=1:2 keys_zone=my_cache:10m;

server {
    location /api/products {
        proxy_cache my_cache;
        proxy_cache_valid 200 1h;    # 200 응답은 1시간 캐싱
        proxy_pass http://spring-app:8080;
    }
}
```

**Spring 애플리케이션에서 캐시 헤더 직접 제어**

```java
@GetMapping("/products/{id}")
public ResponseEntity<Product> getProduct(@PathVariable Long id) {
    Product product = productService.findById(id);
    
    return ResponseEntity.ok()
        .cacheControl(CacheControl.maxAge(1, TimeUnit.HOURS).cachePublic())
        // "public, max-age=3600" 헤더 추가
        // → 이 응답을 받은 CDN/프록시가 1시간 캐싱
        .eTag(String.valueOf(product.getVersion()))
        .body(product);
}
```

## 4. 이 설계를 이렇게 한 이유

### 얻는 것

- 원본 서버 부하 감소
- 사용자 응답 속도 향상 (지리적으로 가까운 캐시)
- 네트워크 대역폭 비용 절감

### 잃는 것 / 트레이드오프

|손실|구체적 상황|
|---|---|
|**캐시 무효화의 어려움**|서버에서 데이터가 바뀌었는데 프록시가 여전히 이전 응답을 줄 수 있다. `max-age` 이내에는 강제로 비우기 어렵다. Phil Karlton의 유명한 말 — _"캐시 무효화와 이름 짓기는 컴퓨터 과학의 두 가지 어려운 문제다"_|
|**`private` 데이터 캐싱 오류**|로그인한 사용자 A의 개인정보 응답에 `Cache-Control: public`을 잘못 설정하면 프록시가 이를 캐싱해 사용자 B에게 줄 수 있다. 실제 발생한 보안 사고 유형|
|**캐시 일관성 문제**|여러 프록시가 각자 다른 버전을 캐싱하면 사용자마다 다른 응답을 받을 수 있다|
|**동적 콘텐츠 캐싱 불가**|사용자마다 다른 응답(장바구니, 알림 등)은 캐싱 자체가 불가능. 캐싱 가능한 API와 불가능한 API를 명확히 설계해야 한다|
