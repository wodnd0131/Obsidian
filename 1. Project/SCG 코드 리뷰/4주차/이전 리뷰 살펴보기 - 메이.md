## 접근 제어자 (Access Modifier)

### 종류와 범위

|제어자|클래스 내부|같은 패키지|자식 클래스|외부 전체|
|---|:-:|:-:|:-:|:-:|
|`private`|✅|❌|❌|❌|
|`(default)`|✅|✅|❌|❌|
|`protected`|✅|✅|✅|❌|
|`public`|✅|✅|✅|✅|

명시 안 하면 **default(package-private)** 이에요. 같은 패키지 안에서만 접근 가능해요.

# ArithmeticException
정수를 0으로 나눌 때 발생하는 런타임 예외예요.

```java
int result = 10 / 0; // ArithmeticException: / by zero
```

---
### 상속 구조

```
Throwable
└── Exception
    └── RuntimeException
        └── ArithmeticException
```

`RuntimeException` 계열이라 **unchecked exception**이에요. 즉 try-catch를 강제하지 않아요.

## Static 리뷰의 의도 분석

이 리뷰는 사실 **두 가지 별개의 학습 포인트**를 동시에 던지고 있어요.

---

### 1. Static 메서드의 장단점 (원래 의도)

리뷰어가 원래 묻고 싶었던 건 이거예요.

```java
// 현재 — 인스턴스 메서드
Calculator calculator = new Calculator();
calculator.add(1, 2);

// Static 적용 시
Calculator.add(1, 2);  // 객체 생성 불필요
```

**장점**

- 객체 생성 비용 없음
- 상태가 없는 유틸성 클래스에 적합 (`Math.max()` 같은 경우)

**단점**

- 인터페이스로 추상화 불가 → 다형성 활용 못함
- 테스트에서 mock으로 대체 불가
- 전역 상태를 가지면 테스트 간 오염 위험

스터디원이 "메모리 이점"을 언급한 건 방향은 맞지만, **더 중요한 단점인 테스트 가능성과 OOP 설계 측면을 놓쳤어요.**

---

### 2. 리뷰어가 추가로 던진 질문 (Given-When-Then)

스터디원이 "Given 절이 비는데 어떻게 하나요?"라고 물었을 때, 리뷰어가 방향을 틀어서 더 본질적인 질문을 던졌어요.

```java
// Given이 있는 경우
@Test
void 덧셈_테스트() {
    // Given
    int a = 1;
    int b = 2;
    
    // When
    int result = Calculator.add(a, b);
    
    // Then
    assertThat(result).isEqualTo(3);
}

// Given 없이 When-Then만
@Test
void 덧셈_테스트() {
    assertThat(Calculator.add(1, 2)).isEqualTo(3);
}
```

> "Given이 없으면 When-Then으로 충분하지 않을까?"

이건 **형식보다 의도가 중요하다**는 걸 가르치려는 거예요. 단순한 케이스에서 Given-When-Then 주석을 억지로 채우는 건 오히려 노이즈가 될 수 있어요.

---

### 리뷰 전체가 남기고자 한 것

```
Static 장단점 탐구
        ↓
테스트에서 객체 생성이 왜 필요한가?
        ↓
Given-When-Then은 규칙이 아니라 도구다
        ↓
형식에 매몰되지 말고 의도를 전달하는 게 핵심
```

리뷰어가 "정답은 없어요"라고 한 게 핵심이에요. Static의 기술적 차이를 외우게 하는 게 목적이 아니라, **스스로 트레이드오프를 고민하는 습관**을 기르게 하려는 거예요. 오브젝트에서 조영호가 강조한 "설계는 항상 트레이드오프"라는 관점과 정확히 맞닿아 있어요.
