## 접근 제어자 (Access Modifier)

### 종류와 범위

| 제어자         | 클래스 내부 | 같은 패키지 | 자식 클래스 | 외부 전체 |
| ----------- | :----: | :----: | :----: | :---: |
| `private`   |   ✅    |   ❌    |   ❌    |   ❌   |
| `(default)` |   ✅    |   ✅    |   ❌    |   ❌   |
| `protected` |   ✅    |   ✅    |   ✅    |   ❌   |
| `public`    |   ✅    |   ✅    |   ✅    |   ✅   |

명시 안 하면 **default(package-private)** 이에요. 같은 패키지 안에서만 접근 가능해요.

# ArithmeticException
정수를 0으로 나눌 때 발생하는 런타임 예외예요.

```java
int result = 10 / 0; // ArithmeticException: / by zero
```

---
### 상속 구조

```
Throwable
└── Exception
    └── RuntimeException
        └── ArithmeticException
```

`RuntimeException` 계열이라 **unchecked exception**이에요. 즉 try-catch를 강제하지 않아요.

## Static 리뷰의 의도 분석

이 리뷰는 사실 **두 가지 별개의 학습 포인트**를 동시에 던지고 있어요.

---

### 1. Static 메서드의 장단점 (원래 의도)

리뷰어가 원래 묻고 싶었던 건 이거예요.

```java
// 현재 — 인스턴스 메서드
Calculator calculator = new Calculator();
calculator.add(1, 2);

// Static 적용 시
Calculator.add(1, 2);  // 객체 생성 불필요
```

**장점**

- 객체 생성 비용 없음
- 상태가 없는 유틸성 클래스에 적합 (`Math.max()` 같은 경우)

**단점**

- 인터페이스로 추상화 불가 → 다형성 활용 못함
- 테스트에서 mock으로 대체 불가
- 전역 상태를 가지면 테스트 간 오염 위험

스터디원이 "메모리 이점"을 언급한 건 방향은 맞지만, **더 중요한 단점인 테스트 가능성과 OOP 설계 측면을 놓쳤어요.**

---

### 2. 리뷰어가 추가로 던진 질문 (Given-When-Then)

스터디원이 "Given 절이 비는데 어떻게 하나요?"라고 물었을 때, 리뷰어가 방향을 틀어서 더 본질적인 질문을 던졌어요.

```java
// Given이 있는 경우
@Test
void 덧셈_테스트() {
    // Given
    int a = 1;
    int b = 2;
    
    // When
    int result = Calculator.add(a, b);
    
    // Then
    assertThat(result).isEqualTo(3);
}

// Given 없이 When-Then만
@Test
void 덧셈_테스트() {
    assertThat(Calculator.add(1, 2)).isEqualTo(3);
}
```

> "Given이 없으면 When-Then으로 충분하지 않을까?"

이건 **형식보다 의도가 중요하다**는 걸 가르치려는 거예요. 단순한 케이스에서 Given-When-Then 주석을 억지로 채우는 건 오히려 노이즈가 될 수 있어요.

---

### 리뷰 전체가 남기고자 한 것

```
Static 장단점 탐구
        ↓
테스트에서 객체 생성이 왜 필요한가?
        ↓
Given-When-Then은 규칙이 아니라 도구다
        ↓
형식에 매몰되지 말고 의도를 전달하는 게 핵심
```

리뷰어가 "정답은 없어요"라고 한 게 핵심이에요. Static의 기술적 차이를 외우게 하는 게 목적이 아니라, **스스로 트레이드오프를 고민하는 습관**을 기르게 하려는 거예요. 오브젝트에서 조영호가 강조한 "설계는 항상 트레이드오프"라는 관점과 정확히 맞닿아 있어요.

## @BeforeAll

### 기본 개념

```java
// @BeforeEach — 각 테스트마다 실행
@BeforeEach
void setUp() {
    calculator = new Calculator(); // 매 테스트마다 새로 생성
}

// @BeforeAll — 전체 테스트 클래스에서 딱 한 번 실행
@BeforeAll
static void setUpAll() {
    calculator = new Calculator(); // 한 번만 생성
}
```

`@BeforeAll`은 **static 메서드**여야 해요. 그래서 리뷰어가 static이랑 함께 언급한 거예요.

---

### JUnit5 생명주기

```
테스트 클래스 로드
        ↓
@BeforeAll    ← 딱 한 번
        ↓
@BeforeEach   ← 테스트마다
테스트 실행
@AfterEach    ← 테스트마다
        ↓
@AfterAll     ← 딱 한 번
```

---

### 언제 쓰나요?

```java
// ✅ @BeforeAll이 적합한 케이스
// 생성 비용이 크고, 상태가 변하지 않는 객체
@BeforeAll
static void setUp() {
    // DB 연결, 파일 로드 등 무거운 작업
    dbConnection = Database.connect();
    
    // Calculator처럼 상태 없는 유틸성 객체
    calculator = new Calculator();
}
```

```java
// ✅ @BeforeEach가 적합한 케이스
// 테스트마다 새로운 상태가 필요한 객체
@BeforeEach
void setUp() {
    // 매번 새로운 로또 생성
    lotto = new Lotto(List.of(1, 2, 3, 4, 5, 6));
}
```

---

### ⚠️ 조심할 점

**가장 큰 위험 — 테스트 간 상태 공유**

```java
@BeforeAll
static void setUp() {
    lottoMachine = new LottoMachine(); // 하나를 공유
}

@Test
void 테스트1() {
    lottoMachine.addTicket(ticket1); // 상태 변경
}

@Test
void 테스트2() {
    // 테스트1이 변경한 상태가 그대로 남아있음
    // 테스트 순서에 따라 결과가 달라짐 💥
}
```

테스트는 **독립적**이어야 해요. 순서가 바뀌어도 결과가 같아야 하는데, `@BeforeAll`로 상태 있는 객체를 공유하면 이게 깨져요.

**Calculator는 괜찮은 이유**

```java
@BeforeAll
static void setUp() {
    calculator = new Calculator(); // 상태가 없음
}

@Test
void 덧셈() {
    assertThat(calculator.add(1, 2)).isEqualTo(3);
    // calculator 내부 상태가 변하지 않음 → 공유해도 안전
}
```

상태가 없는 객체는 공유해도 테스트 독립성이 깨지지 않아요.

---

### 핵심 판단 기준

```
이 객체는 테스트 실행 후 상태가 변하는가?
        ↓
YES → @BeforeEach (매번 새로 생성)
NO  → @BeforeAll 고려 가능
```

리뷰어가 static과 `@BeforeAll`을 함께 언급한 건, **"왜 static이어야 하는지"** 를 `@BeforeAll` 제약 조건을 통해 자연스럽게 깨닫게 하려는 의도예요.

## 스프링 부트 환경에서 @BeforeAll 주의점

### 스프링 테스트의 기본 동작 방식

```java
@SpringBootTest
class LottoServiceTest {
    
    @Autowired
    private LottoService lottoService; // 스프링이 주입
    
    @BeforeAll
    static void setUp() {
        // static이라 스프링 컨텍스트 접근 불가 💥
        // lottoService가 null
    }
}
```

스프링 빈은 스프링 컨텍스트가 관리해요. `@BeforeAll`은 static이라 컨텍스트 로드 전에 실행될 수 있어서 **주입된 빈에 접근이 안 돼요.**

---

### 주의점 1 — ApplicationContext 공유 문제

```java
// 이렇게 하면 테스트마다 컨텍스트를 새로 띄움 💥
@SpringBootTest
class TestA {
    @MockBean
    LottoRepository lottoRepository; // 컨텍스트 오염
}

@SpringBootTest  
class TestB {
    // TestA가 MockBean으로 오염시킨 컨텍스트와
    // 다른 컨텍스트가 필요해서 새로 로드
}
```

스프링은 기본적으로 컨텍스트를 **캐싱해서 재사용**해요. 그런데 `@MockBean`, `@SpyBean`을 쓰거나 설정이 달라지면 캐시가 깨지고 컨텍스트를 새로 띄워요. 컨텍스트 로딩은 매우 무거운 작업이라 테스트가 느려져요.

```java
// ✅ 컨텍스트 재사용을 위해 설정을 통일
@SpringBootTest
@Transactional
class LottoServiceTest {
    // MockBean 최소화
    // 설정 통일
}
```

---

### 주의점 2 — @BeforeAll과 트랜잭션

```java
@SpringBootTest
class LottoRepositoryTest {

    @Autowired
    LottoRepository lottoRepository;

    @BeforeAll
    static void setUp(@Autowired LottoRepository repo) {
        // JUnit5 5.9부터 @Autowired 파라미터 주입 가능
        repo.save(new Lotto(List.of(1, 2, 3, 4, 5, 6)));
    }

    @Test
    @Transactional
    void 조회_테스트() {
        // @BeforeAll은 트랜잭션 밖에서 실행됨
        // setUp에서 저장한 데이터가 다른 트랜잭션에 있어서
        // 안 보일 수 있음 💥
    }
}
```

`@BeforeAll`은 `@Transactional` 범위 밖이에요. DB 데이터를 세팅하면 트랜잭션 경계가 꼬일 수 있어요.

```java
// ✅ DB 초기 데이터는 @BeforeEach + @Transactional 조합
@BeforeEach
void setUp() {
    lottoRepository.save(new Lotto(List.of(1, 2, 3, 4, 5, 6)));
}

@AfterEach
void tearDown() {
    lottoRepository.deleteAll(); // 테스트 후 정리
}
```

---

### 주의점 3 — @TestInstance 활용

```java
// @BeforeAll을 non-static으로 쓰고 싶다면
@SpringBootTest
@TestInstance(TestInstance.Lifecycle.PER_CLASS)
class LottoServiceTest {

    @Autowired
    private LottoService lottoService; // 주입 가능

    @BeforeAll
    void setUp() { // static 아니어도 됨
        // lottoService 접근 가능 ✅
    }
}
```

기본값은 `PER_METHOD`라 매 테스트마다 인스턴스를 새로 만들어요. `PER_CLASS`로 바꾸면 인스턴스를 공유하고 `@BeforeAll`이 non-static이어도 돼요.

**단 조심할 점**

```java
@TestInstance(TestInstance.Lifecycle.PER_CLASS)
class LottoServiceTest {

    private int count = 0; // 인스턴스 공유 → 테스트 간 상태 공유 위험

    @Test
    void 테스트1() { count++; }

    @Test
    void 테스트2() { 
        // count가 0인지 1인지 모름 💥
    }
}
```

---

### 정리

```
순수 단위 테스트
→ @BeforeAll 비교적 안전
→ 상태 없는 객체만 공유할 것

스프링 통합 테스트
→ 컨텍스트 캐싱 오염 주의
→ 트랜잭션 경계 주의  
→ @TestInstance(PER_CLASS) 활용 시 상태 공유 주의
→ 가능하면 @BeforeEach + @AfterEach로 격리
```

스프링으로 넘어가면 **테스트 속도 vs 테스트 격리** 사이의 트레이드오프가 핵심이에요. `@BeforeAll`로 무거운 세팅을 한 번만 하고 싶은 욕구와, 테스트 독립성을 지켜야 한다는 원칙이 충돌하는 지점이거든요.

## No newline at end of file

### 뭔가요?

파일의 마지막 줄에 개행문자(`\n`)가 없는 상태예요.

```
// ❌ 개행 없음
public class Lotto {
    ...
}   ← 여기서 파일 끝 (줄바꿈 없음)

// ✅ 개행 있음
public class Lotto {
    ...
}
↵  ← 마지막에 \n 존재
```

---

### 왜 문제가 되나요?

**POSIX 표준** 때문이에요. POSIX에서 텍스트 파일의 줄(line)은 `\n`으로 끝나야 한다고 정의해요. 개행이 없으면 엄밀히 말해 마지막 줄이 "줄"로 인식되지 않아요.

**Git에서 diff가 지저분해져요**

```bash
# 개행 없는 파일을 수정하면
-}
\ No newline at end of file
+}
\ No newline at end of file

# 내용은 안 바뀌었는데 diff에 노이즈가 생김
```

**협업 시 불필요한 변경이 생겨요**

```bash
# A가 개행 추가하면
# 코드 변경 없이 파일이 수정된 것으로 잡힘
# → 코드 리뷰에서 불필요한 diff 발생
```

---

### 해결 방법

**IntelliJ 설정**

```
Settings
→ Editor
→ General
→ "Ensure every saved file ends with a line break" 체크
```

**EditorConfig로 팀 전체 통일**

```ini
# .editorconfig
root = true

[*]
insert_final_newline = true  # 모든 파일에 개행 강제
```

프로젝트 루트에 `.editorconfig` 파일 하나로 팀원 전체 에디터 설정을 통일할 수 있어요.

---

### 사소해 보이지만 중요한 이유

```
작은 컨벤션을 지키는 습관
        ↓
코드 리뷰에서 본질적인 내용에 집중 가능
        ↓
불필요한 diff 노이즈 제거
        ↓
협업 품질 향상
```

스터디에서 이런 작은 것들을 짚어주는 게 중요한 이유는, 실무에서 PR 올렸을 때 **개행 하나 때문에 리뷰어의 집중력이 흐트러지는 걸 방지**하기 위해서예요.
