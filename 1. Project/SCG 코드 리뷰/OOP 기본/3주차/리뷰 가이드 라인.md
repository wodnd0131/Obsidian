## 리뷰 우선 순위 기준 잡기

> **"이게 없으면 다음 개념을 배워도 의미없다"** 순서로 쌓아야 해. 기초가 흔들리면 위에 올린 개념도 다 무너지거든.

---

## Priority 1 — 반드시 이번 리뷰에서

**도메인이 데이터 덩어리가 되지 않는 것** — 이게 핵심이야.

| 개념         | 이유                                   |
| ---------- | ------------------------------------ |
| **원시값 포장** | 검증 로직이 여기저기 복붙되는 걸 막는 첫 번째 관문        |
| **일급 컬렉션** | `Lotto` 클래스 자체가 존재해야 나머지 개념을 붙일 수 있음 |
| **불변 객체**  | 포장했는데 외부에서 바꿀 수 있으면 포장한 의미가 없음       |

이 세 개는 **하나의 덩어리**야. 원시값 포장 → 일급 컬렉션 → 불변 순서로 연결해서 짚어줘. 따로따로 피드백하면 연결이 안 보여.

---

## Priority 2 — 1을 고치고 나서

**객체가 스스로 일하는 것**

| 개념                   | 이유                                                |
| -------------------- | ------------------------------------------------- |
| **Tell, Don't Ask**  | P1을 잘 했으면 자연스럽게 "그럼 이 로직 어디 둬요?" 질문이 나옴           |
| **도메인 책임 위치 + Enum** | Tell Don't Ask의 실제 적용처. Rank 없이는 당첨 계산이 서비스에 흘러넘침 |

P1 피드백 받고 고치는 과정에서 **스스로 이 질문이 생기는 타이밍**이 있어. 그때 던져주는 게 제일 흡수가 잘 돼.

---

## Priority 3 — 여유가 있을 때

**설계 구조 레벨**

| 개념                | 이유                                            |
| ----------------- | --------------------------------------------- |
| **SRP**           | P1, P2가 잡히면 자연스럽게 클래스 분리 욕구가 생김               |
| **의존성 방향**        | 분리하고 나서 "어 근데 domain이 view를 알아도 돼요?" 질문이 나올 때 |
| **정적 팩토리 + 값 객체** | 상대적으로 체감 임팩트가 낮음. 나중에 캐싱이나 가독성 얘기할 때 자연스럽게    |

이건 솔직히 **이번 미션에서 안 나와도 괜찮아.** 다음 미션에서 이 개념이 필요한 순간이 오면 그때 짚어줘도 늦지 않아.

---

## 한 줄 요약

```
P1 (원시값 포장 → 일급 컬렉션 → 불변)
  "Lotto라는 객체가 제대로 존재하는가?"

P2 (Tell Don't Ask → Enum)
  "그 객체가 스스로 일하고 있는가?"

P3 (SRP → 의존성 → 팩토리)
  "클래스들이 올바른 관계를 맺고 있는가?"
```

P1 없이 P2 피드백하면 붕 떠. P2 없이 P3 피드백하면 더 붕 떠. **상대방 코드가 어느 단계에 있는지 먼저 보고** 거기서 한 단계만 올려주는 게 좋은 리뷰야.

# 전체 가이드라인

설명 방식은 이렇게 갈게.

1. **왜 이게 문제인지** 먼저 — 절차적 코드에서 어떤 불편함이 생기는지
2. **핵심 개념** 한 줄 정의
3. **Before / After 코드** — 로또 미션 맥락 그대로

총 9개 (도메인 모델링 6 + 설계 원칙 3), 3개씩 나눠서 진행할게.

---

## 1. 원시값 포장 (Primitive Obsession)

**왜 문제가 생기나**

```java
// 여기서 price는 뭘 의미하는 int야?
int price = 14000;
int count = 14;
int number = 7;

void buy(int price) { ... }
```

`int`만 봐선 이게 "금액"인지 "개수"인지 "로또 번호"인지 알 수 없어. 더 큰 문제는, 유효성 검증 로직이 여기저기 흩어지게 된다는 거야.

```java
// 로또 번호 쓰는 곳마다 이 검증이 복붙됨
if (number < 1 || number > 45) {
    throw new IllegalArgumentException("번호는 1~45 사이여야 합니다.");
}
```

**핵심 개념**

> 의미 있는 값은 그 값만을 위한 클래스로 감싼다. 검증과 책임이 그 안에 모인다.

**After**

```java
public class LottoNumber {
    private static final int MIN = 1;
    private static final int MAX = 45;

    private final int value;

    public LottoNumber(int value) {
        validate(value);
        this.value = value;
    }

    private void validate(int value) {
        if (value < MIN || value > MAX) {
            throw new IllegalArgumentException("번호는 1~45 사이여야 합니다.");
        }
    }
}
```

이제 `LottoNumber`를 만드는 순간 검증이 보장돼. 쓰는 쪽은 믿고 쓰면 돼.

## 2. 일급 컬렉션 (First-Class Collection)

**왜 문제가 생기나**

```java
// 이 List가 뭔지, 몇 개여야 하는지, 중복은 안 되는지...
// 쓰는 쪽에서 전부 신경 써야 함
List<LottoNumber> numbers = new ArrayList<>();
```

컬렉션을 그냥 들고 다니면, 그 컬렉션에 대한 규칙(6개여야 한다, 중복 없어야 한다)이 **사용하는 곳마다** 생겨나.

**핵심 개념**

> 컬렉션 하나만 멤버변수로 갖는 클래스를 만든다. 그 컬렉션의 규칙과 행동이 한 곳에 모인다.

---

핵심은 **컴파일러가 잡아주지 못하는 실수**야.

```java
void buy(int price, int count) { ... }

// 부르는 쪽에서 순서 바꿔도 컴파일 에러 없음
buy(14, 14000); // 💥 근데 아무도 모름
```

`int`는 그냥 숫자야. `price` 자리에 `count`를 넣어도 자바는 모르거든.

포장하면 **타입 자체가 의미를 강제**해.

```java
void buy(LottoPrice price, LottoCount count) { ... }

buy(new LottoCount(14), new LottoPrice(14000)); // 💥 컴파일 에러
```

`LottoPrice` 자리에 `LottoCount`를 넣는 순간 컴파일이 안 돼. 실수를 **런타임이 아니라 컴파일 타임에** 잡아주는 거야.

---

**Before**

```java
// 여기저기서 직접 꺼내서 검사
List<LottoNumber> lotto = new ArrayList<>();
if (lotto.size() != 6) { ... }
if (new HashSet<>(lotto).size() != lotto.size()) { ... }
```

**After**

```java
public class Lotto {
    private final List<LottoNumber> numbers;

    public Lotto(List<LottoNumber> numbers) {
        validate(numbers);
        this.numbers = Collections.unmodifiableList(new ArrayList<>(numbers));
    }

    private void validate(List<LottoNumber> numbers) {
        validateSize(numbers);
        validateNoDuplicate(numbers);
    }

    private void validateSize(List<LottoNumber> numbers) {
        if (numbers.size() != 6) {
            throw new IllegalArgumentException("로또 번호는 6개여야 합니다.");
        }
    }

    private void validateNoDuplicate(List<LottoNumber> numbers) {
        if (new HashSet<>(numbers).size() != numbers.size()) {
            throw new IllegalArgumentException("로또 번호는 중복될 수 없습니다.");
        }
    }
}
```

## 3. 불변 객체 (Immutable)

**왜 문제가 생기나**

원시값 포장이랑 일급 컬렉션을 잘 만들어도, 외부에서 바꿀 수 있으면 의미가 없어.

```java
Lotto lotto = new Lotto(numbers);

// 이게 가능하면?
lotto.getNumbers().add(new LottoNumber(46)); // 💥 6개짜리 로또가 7개가 됨
```

생성자에서 검증을 열심히 해도, 만들고 나서 바꿔버릴 수 있으면 **신뢰할 수 없는 객체**가 돼.

**핵심 개념**

> 한 번 만들어진 객체는 내부 상태가 바뀌지 않아야 한다. `final` + 방어적 복사가 기본 도구다.

**After**

```java
public class Lotto {
    private final List<LottoNumber> numbers; // final로 재할당 차단

    public Lotto(List<LottoNumber> numbers) {
        // 방어적 복사 — 외부 리스트가 변해도 영향 없음
        this.numbers = Collections.unmodifiableList(new ArrayList<>(numbers));
    }

    public List<LottoNumber> getNumbers() {
        // unmodifiableList라 add/remove 호출하면 예외 발생
        return numbers;
    }
}
```

포인트 두 가지야.

- `new ArrayList<>(numbers)` — 외부에서 넘긴 리스트가 나중에 바뀌어도 내 것은 안 바뀜
- `Collections.unmodifiableList(...)` — 꺼내서 직접 수정하는 것도 차단

## 4. 값 객체 (Value Object)

**왜 문제가 생기나**

```java
LottoNumber a = new LottoNumber(7);
LottoNumber b = new LottoNumber(7);

System.out.println(a == b);      // false
System.out.println(a.equals(b)); // false (재정의 안 했으면)
```

자바에서 `==`는 주소 비교야. `equals`도 재정의 안 하면 똑같이 주소 비교를 해. 그런데 `LottoNumber(7)`은 어디서 만들든 **"7번"이라는 동일한 의미**를 가져야 하잖아.

당첨 번호 비교할 때 이게 터져.

```java
// 이러면 항상 false — 당첨자가 없는 로또가 됨
winningNumbers.contains(lottoNumber);
```

**핵심 개념**

> 식별자(id)가 아니라 **값 자체로 동등성**을 판단하는 객체. `equals`와 `hashCode`를 반드시 재정의한다.

**After**

```java
public class LottoNumber {
    private final int value;

    // ...생성자, 검증 생략

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof LottoNumber)) return false;
        LottoNumber that = (LottoNumber) o;
        return value == that.value;
    }

    @Override
    public int hashCode() {
        return Objects.hash(value);
    }
}
```

`equals`만 재정의하고 `hashCode`를 빠뜨리는 실수가 많아. `HashSet`, `HashMap`은 `hashCode` 먼저 보거든. 둘은 항상 같이 재정의해야 해.

## 5. Tell, Don't Ask

**왜 문제가 생기나**

```java
// 외부(서비스, 컨트롤러)에서 꺼내서 직접 판단
int matchCount = 0;
for (LottoNumber number : lotto.getNumbers()) {
    if (winningNumbers.contains(number)) {
        matchCount++;
    }
}
```

이 코드의 문제는 `Lotto` 안을 **꺼내서** 판단한다는 거야. `Lotto`가 자기 번호를 알고 있는데, 왜 외부에서 하나씩 꺼내서 세고 있어?

이런 패턴이 쌓이면 도메인 로직이 서비스 레이어에 녹아들어가고, `Lotto` 클래스는 그냥 데이터 덩어리(getter 모음)가 돼.

**핵심 개념**

> 객체에게 데이터를 달라고 하지 말고, **일을 시켜라.** 판단은 그 데이터를 가진 객체가 한다.

**Before**

```java
// 서비스에서 꺼내서 판단
List<LottoNumber> numbers = lotto.getNumbers();
long count = numbers.stream()
        .filter(winningNumbers::contains)
        .count();
```

**After**

```java
// Lotto 스스로 몇 개 일치하는지 계산
public class Lotto {
    private final List<LottoNumber> numbers;

    public int countMatch(List<LottoNumber> winningNumbers) {
        return (int) numbers.stream()
                .filter(winningNumbers::contains)
                .count();
    }
}

// 부르는 쪽은 그냥 시키면 돼
int matchCount = lotto.countMatch(winningNumbers);
```

## 6. 도메인 책임 위치 + Enum 활용

이 두 개는 묶어서 보는 게 자연스러워. 책임 위치를 잡다 보면 자연스럽게 Enum이 필요해지거든.

**왜 문제가 생기나**

```java
// 서비스나 컨트롤러에 이런 코드가 생김
if (matchCount == 3) {
    prize = 5000;
} else if (matchCount == 4) {
    prize = 50000;
} else if (matchCount == 5) {
    prize = 1500000;
} else if (matchCount == 6) {
    prize = 2000000000;
}
```

등수가 추가되거나 상금이 바뀌면 **이 분기문을 찾아서** 고쳐야 해. 찾지 못하면 버그야. 그리고 이 로직은 명백히 **당첨 규칙**이라는 도메인 지식인데, 서비스 코드 한가운데 박혀있어.

**핵심 개념**

> 등수와 상금처럼 **연관된 상수 묶음 + 그에 딸린 로직**은 Enum 안으로 응집한다.

**After**

```java
public enum Rank {
    FIRST(6, 2_000_000_000),
    SECOND(5, 1_500_000),  // 보너스 번호 있는 경우는 다음 단계에서
    THIRD(4, 50_000),
    FOURTH(3, 5_000),
    MISS(0, 0);

    private final int matchCount;
    private final int prize;

    Rank(int matchCount, int prize) {
        this.matchCount = matchCount;
        this.prize = prize;
    }

    public static Rank of(int matchCount) {
        return Arrays.stream(values())
                .filter(rank -> rank.matchCount == matchCount)
                .findFirst()
                .orElse(MISS);
    }

    public int getPrize() {
        return prize;
    }
}
```

이렇게 하면 서비스는 이렇게 단순해져.

```java
Rank rank = Rank.of(lotto.countMatch(winningNumbers));
int prize = rank.getPrize();
```

등수가 추가되면 Enum에 항목 하나만 추가하면 돼. 서비스 코드는 안 건드려도 돼.

# 7~9번 (설계 원칙)

## 7. SRP (단일 책임 원칙)

**왜 문제가 생기나**

미션 초반에 절차적으로 짜면 자연스럽게 이렇게 돼.

```java
public class LottoMachine {

    public void run() {
        // 입력
        Scanner scanner = new Scanner(System.in);
        System.out.println("구입금액을 입력해 주세요.");
        int price = scanner.nextInt();

        // 검증
        if (price % 1000 != 0) {
            throw new IllegalArgumentException("1000원 단위로 입력해주세요.");
        }

        // 로또 생성
        int count = price / 1000;
        List<List<LottoNumber>> tickets = new ArrayList<>();
        for (int i = 0; i < count; i++) {
            List<LottoNumber> numbers = createLotto();
            tickets.add(numbers);
        }

        // 출력
        System.out.println(count + "개를 구매했습니다.");
        for (List<LottoNumber> ticket : tickets) {
            System.out.println(ticket);
        }

        // 당첨 계산
        // ...
    }
}
```

`run()` 하나가 입력 / 검증 / 생성 / 출력 / 계산을 다 해. 한 곳을 고치면 다른 곳이 깨질 가능성이 항상 있어.

**핵심 개념**

> 하나의 클래스는 하나의 이유로만 바뀌어야 한다. "이 클래스는 ___을 담당한다" 한 문장으로 설명 안 되면 SRP 위반 신호야.

**After — 책임을 쪼개면**

```java
// 입출력만
public class InputView {
    public int readPurchaseAmount() { ... }
    public List<Integer> readWinningNumbers() { ... }
}

public class OutputView {
    public void printTickets(LottoTickets tickets) { ... }
    public void printResult(LottoResult result) { ... }
}

// 로또 생성만
public class LottoMachine {
    public LottoTickets purchase(LottoPrice price) { ... }
}

// 당첨 계산만
public class LottoResult {
    public Map<Rank, Integer> calculate(LottoTickets tickets, WinningLotto winningLotto) { ... }
}

// 흐름 조립만
public class LottoController {
    public void run() {
        LottoPrice price = new LottoPrice(inputView.readPurchaseAmount());
        LottoTickets tickets = lottoMachine.purchase(price);
        outputView.printTickets(tickets);
        // ...
    }
}
```

각 클래스가 바뀌는 이유가 딱 하나씩이야. 출력 형식이 바뀌면 `OutputView`만, 생성 규칙이 바뀌면 `LottoMachine`만 건드리면 돼.

## 8. 의존성 방향

**왜 문제가 생기나**

SRP로 클래스를 쪼개도, 의존 방향이 잘못되면 또 문제가 생겨.

```java
// domain 패키지 안에 있는 Lotto가
public class Lotto {

    public String toOutputFormat() {
        // view의 출력 형식을 도메인이 알고 있음 💥
        return numbers.stream()
                .map(String::valueOf)
                .collect(Collectors.joining(", ", "[", "]"));
    }
}
```

출력 형식이 `[1, 2, 3]`에서 `1-2-3`으로 바뀌면 **도메인 코드를 수정**해야 해. 도메인이 view에 오염된 거야.

**핵심 개념**

> 의존성은 항상 **바깥 → 안쪽** 방향이어야 한다. `domain`은 아무것도 몰라야 한다. `view`와 `controller`가 `domain`을 알면 돼.

```
controller
    ↓
  domain   ← view도 여기에 의존
```

**After**

```java
// domain — 출력 형식 모름, 자기 데이터만 제공
public class Lotto {
    public List<LottoNumber> getNumbers() {
        return numbers;
    }
}

// view — domain을 알고, 출력 형식은 여기서 결정
public class OutputView {
    public void printTicket(Lotto lotto) {
        String formatted = lotto.getNumbers().stream()
                .map(n -> String.valueOf(n.getValue()))
                .collect(Collectors.joining(", ", "[", "]"));
        System.out.println(formatted);
    }
}
```

형식이 바뀌어도 `OutputView`만 고치면 돼. `Lotto`는 그대로야.

실제로 패키지 구조로 강제하면 더 명확해.


## 9. 정적 팩토리 메서드

**왜 문제가 생기나**

생성자를 그냥 쓰면 두 가지 문제가 생겨.

```java
// 이게 뭘 만드는 건지 이름만으로 알 수 없음
Lotto lotto = new Lotto(numbers);

// 자동 생성이야? 수동 생성이야?
// 생성자 시그니처가 같으면 구분 자체가 불가능
```

그리고 생성자는 **캐싱, 하위 타입 반환** 같은 걸 할 수 없어. `LottoNumber(7)`을 만들 때마다 새 객체를 만드는 건 낭비야. 1~45는 고정이거든.

**핵심 개념**

> 생성자 대신 **이름 있는 정적 메서드**로 객체를 만든다. 의도를 이름으로 표현하고, 내부 구현(캐싱 등)을 숨길 수 있다.

**After**

```java
public class LottoNumber {
    private static final Map<Integer, LottoNumber> CACHE = new HashMap<>();

    static {
        for (int i = 1; i <= 45; i++) {
            CACHE.put(i, new LottoNumber(i));
        }
    }

    private final int value;

    private LottoNumber(int value) { // 생성자를 private으로
        this.value = value;
    }

    public static LottoNumber from(int value) { // 이름으로 의도 표현
        if (!CACHE.containsKey(value)) {
            throw new IllegalArgumentException("번호는 1~45 사이여야 합니다.");
        }
        return CACHE.get(value);
    }
}
```

```java
// 쓰는 쪽 — 의미가 명확하고 같은 번호는 같은 객체
LottoNumber number = LottoNumber.from(7);
```

`from(7)`을 100번 호출해도 객체는 하나야. 그리고 4번에서 다룬 `equals`/`hashCode`도, 같은 객체면 `==` 비교도 통과돼서 더 안전해져.

## 전체 흐름 정리

```
원시값 포장          →  의미 있는 값에 이름과 검증을 부여
일급 컬렉션          →  컬렉션의 규칙을 한 곳에 응집
불변 객체            →  만들어진 후 신뢰 보장
값 객체              →  값으로 동등성 판단 (equals/hashCode)
Tell Don't Ask      →  판단은 데이터를 가진 객체가
도메인 책임 + Enum   →  변하는 규칙을 한 곳에
SRP                 →  하나의 클래스, 하나의 이유
의존성 방향          →  domain은 아무것도 모른다
정적 팩토리          →  생성 의도를 이름으로
```

앞 개념들이 뒤 개념의 기반이 돼. 원시값 포장 없이 값 객체가 없고, 일급 컬렉션 없이 Tell Don't Ask가 어색해지는 식으로.
