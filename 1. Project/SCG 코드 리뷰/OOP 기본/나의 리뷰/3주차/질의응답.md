값을 전달할 때마다 검증하는 문제에 대해서는 조금 찾아보니 인스턴스 캐싱이라는 것이 있어 적용해 보았습니다. 티켓을 8개 이상 구매하면 메모리적으로 1~45 캐싱이 더 효율적이고, 8개 미만 구매보다 8개 이상 구매가 더 흔한 케이스일 것 같아 적용했습니다!

## 한 함수에만 쓰이는 검증 로직을 위해 하나의 클래스를 분리하는 것은 낭비이지 않을까 싶어요.
`그럼 모든 값을 포장해야 할까?` 에 대한 고민을 해주신  것 같아요😊
- LottoNumber나 Price의 경우, 효율적이고 보기 좋다.
- 반면, 일회용 검증을 위한, 활용도가 낮은 경우는 포장하지 않는 게 좋은 것 같다.

저도 모든 값을 포장할 필요는 없다고 생각합니다😀
이 질문을 남긴 이유는 `포장해야 하는 기준을 세웠는가?` 를 묻기 위함입니다.
그런 측면에서, 좋은 경우와 그렇지 않은 경우를 같이 이야기해주신 점이 마음에 쏙 드네요😊😊

제가 해석하기로는, 연우님께서 `클래스를 나누어 포장할 이유` 를 `코드의 반복` 과 `직관성` 에서 찾으신 것 같아요.
그리고 `직관성`이란 장점은, `코드 반복` 이 없는 경우에서는 포장할 이유까진 되지 않는다 보신 것이 두 번째 이유이구요. 제가 예시로 들었던 검증의 반복과 같이, 개발자를 가장 불편하게 만드는 점이 코드 반복이라 생각합니다.🤔
`직관성` 또한, 결국 클래스가 늘어나면서 복잡함도 늘어나니, 마냥 장점이라 보기 어려운 점이구요.

때문에, `그럼 모든 값을 포장해야 할까?` 에 대한 타당한 기준을 말씀해주신 것 같아요.😄

다만 한 가지 질문을 드리고 싶어요!
만약 `UserId`라는 값이 있고, 검증 로직이 `User` 생성 로직, 딱 한 곳에서만 쓰인다면 
연우님 기준대로라면 포장하지 않는 게 맞을까요?

저는 ``UserId`` 클래스로 감싸 포장할 수 있다 생각해요!🤔
앞선 두 기준에선 부족할 수 있지만, `UserId`라는 개념을 `도메인`으로 볼 수 있기 때문입니다. 
`UserId`은 항상 `UserId`로 그 자체여야 하고, 다른 값과 혼동되면 안되니까요!

포장을 하면 그 값이 존재하는 한 **항상 유효함이 보장한다**는 의미이기 때문입니다.
- `User`를 생성할때 `UserId`가 `Long`으로 선언되든, `UUID`로 선언되었든 신경쓰지 않겠다. 
- `UserId` 내부에 무엇이 일어나든 `User`의 생성은 알 필요없이 `User` 생성을 완수해야 한다.
- 이 값은 오롯이 `UserId`라는 개념 자체로만 보겠다.

유효함을 보장하면, `User`와 `UserId` 같이 유연한 관계를 만들 수 있습니다.
자연스럽게 `UserId`로 인한 코드 중복이나 직관성 문제는 예방될 수 있구요.
이를 위해서라면, 저는 `UserId`를 분리할 수 있을 것 같아요.

Count의 역할에 대해 잘 고민해주셨어요!😊

다만 getter를 쓰는 이유가 "역산 가능하니 숨겨도 무의미하다"라면,
조금 다르게 생각해볼 수 있어요.

getter를 쓰는 방향은 맞아요. 
다만, 그 이유가 "역산 가능하니 숨겨도 무의미"라면 조금 다르게 생각해볼 수 있어요. 
OOP의 관점에서는 "Count의 책임 범위 밖이니 숨기면 안 된다"가 더 정확한 이유예요.
Count에 곱셈 책임을 넣으면 Count가 상금 개념을 알아야 하고, 
그건 단순히 `개수`라는 개념만 알고 있던 Count의 책임에서 벗어나니, 단일 책임 원칙([SRP](https://tecoble.techcourse.co.kr/post/2020-07-31-solid-1/))에 위반하거든요!😮

그리고 한 가지 더 고민해볼 부분이 있어요.
Count가 제공하는 가치가 "음수 예외 처리" 뿐이라면,
오히려 Count 자체가 필요한가를 먼저 고민해봐야 해요.🤔

앞선 리뷰처럼 값을 객체로 포장하는 것은 도메인 개념을 표현할 때 의미있는데,
Count는 "0 이상의 정수"라는 기술적 제약만 표현하거든요.
(특히, 로또 개수는 양수끼리 연산되는 맥락이라,음수가 들어올 경로 자체가 없어요. 
그럼 이 예외를 위해 Count로 포장하는 것은 유의미한가?도 고민해봐야 할 것 같구요.)

"타입 안전성을 위한 래퍼"로서의 가치는 있지만, 
`Price`나 `LottoRank`처럼 도메인 개념을 포장하는 값 객체와 비교했을 때 큰 역할을 찾기 어려워요.

현재 Count가 쓰이는 맥락들은 다음과 같아요.
- 구매 가능한 로또 수 (Price.getBuyableLottoCount())
- 번호 일치 개수 (LottoTicket.getMatchCount())
- 랭크별 당첨 횟수 (LottoResult의 Map<LottoRank, Count>)
- 랭크 기준 일치 수 (LottoRank enum 상수들)

맥락적으로 보았을 때, Count는  int로 대체되어도 표현에서 크게 차이점이 없다는 점에서 눈에 들어왔어요. 그로 인해, 무엇을 위한 Count인지, 예상할 수 없었어요.🤔
어떠한 개념을 표현하기 위한 Count가 아니라 단순히 숫자이기에, 오히려 헷갈린 것 같아요.

`Count` 클래스는 첫번째 리뷰 코멘트와와 함께 생각해보면 좋을 것 같아요!🙌

### Q2. Constants 클래스 — 안티패턴

public class RandomLottoTicketGenerator {

    // LottoTicket.TICKET_LENGTH 참조
}

### Q3. priceOfOneLotto — public으로 열기

**Q3. priceOfOneLotto — public으로 열기**

테스트 클래스에 똑같이 선언하는 건 값이 바뀌면 두 곳 다 고쳐야 하는데
컴파일러가 잡아주지 못해요.

public static final int PRICE_OF_ONE_LOTTO = 1000;

new Price(Price.PRICE_OF_ONE_LOTTO * 3)  // 의도도 명확하고 변경에도 안전

로또 한 장 가격이 1000원이라는 건 비즈니스 규칙이라
외부에 공개해도 자연스러운 값이에요.😄

---





---



>또한, TICKET_LENGTH와 같은 여러 클래스에 걸쳐 사용되는 값들을 Constants라던가 하는 클래스에 담는 아이디어를 생각했는데, (공개하면 안되는 값이 아니라면) 괜찮을 지 궁금합니다!
>
>또한, priceOfOneLotto처럼 한 클래스에서 사용되는 상수도, 변경하게 되면 Test코드에는 반영이 안되는 문제점에 대해, 그 상수를 public으로 만들고 Test에서 사용하는 것은 어떤지 궁금합니다. 아니면 test 클래스에서 똑같은 상수를 선언해 두는 것은 어떤지 궁금합니다!


Constants는 응집도가 떨어져서 안티패턴으로 취급되기도 합니다!
```
public class Constants {
    public static final int TICKET_LENGTH = 6;
    public static final int PRICE_OF_ONE_LOTTO = 1000;
    public static final String ERROR_MESSAGE = "잘못된 입력";
    public static final int MAX_RETRY = 3;
}
```
이 상수들은 서로 연관성이 없습니다. 
단지 `static final` 이라는 이유만으로 한 곳에 모이게 됩니다. 애매하면 결국 여기에 몰아넣는 경우도 생기구요..!😮
이러면, 티켓 규칙이 바뀌어도, 가격이 바뀌어도, 에러 메시지가 바뀌어도 Constants를 함께 수정해야 합니다.
이는 SRP 위반이구요.🤔
또한, TICKET_LENGTH가 누구 개념인지 Constants만 보면 알 수 없습니다. LottoTicket에 있어야 해당 상수가 누구 소유인지 명확해집니다.
이런 측면에서는 Constants가 필요해 보이는 순간, 사실 그 상수의 주인을 정의하지 못했을 가능성도 있구요.🤔

따라서 `public static final int ARRAY_INIT_SIZE = 10;` 와 같이 구현 세부 사항을 나타내는 경우가 아닌,
`public static final int TICKET_LENGTH = 6;`와 같이 비즈니스 로직을 나타내는 경우라면 public으로 두어도 좋다고 생각합니다.😄 `LottoTicket.TICKET_LENGTH` 이 직관성도  좋으니까요!😊

### LottoTicketTest ⛔ 절반만 쓸모 있음

**생성자 검증 3종 (길이, 중복, 정상)** → 쓸모 있어요.

**근데 `getMatchCount()`가 없어요. 이게 심각한 문제예요.**

`LottoTicket`의 생성자는 "올바른 티켓을 만든다"는 책임이고, `getMatchCount()`는 "당첨 번호와 비교한다"는 책임이에요. 지금 테스트는 전자만 있어요.

실제 소스를 보면:

```java
// LottoTicket.java 32~36줄
public Count getMatchCount(LottoTicket winnerTicket) {
    int matchCount = 0;
    for (LottoNumber number : winnerTicket.lottoNumbers) {
        matchCount += Boolean.compare(lottoNumbers.contains(number), false);
    }
    return new Count(matchCount);
}
```

`Boolean.compare(contains, false)` 이 표현이 꽤 비직관적이에요. `contains(number) ? 1 : 0` 을 우회해서 쓴 건데, 이런 코드일수록 테스트가 반드시 있어야 해요. 지금은 이 로직이 틀려도 테스트가 안 잡아요.

**추가해야 할 것:**

```java
// 3개 일치
@Test
void testGetMatchCount_partial() {
    LottoTicket ticket = new LottoTicket(Arrays.asList(
        LottoNumber.valueOf(1), LottoNumber.valueOf(2), LottoNumber.valueOf(3),
        LottoNumber.valueOf(4), LottoNumber.valueOf(5), LottoNumber.valueOf(6)
    ));
    LottoTicket winner = new LottoTicket(Arrays.asList(
        LottoNumber.valueOf(1), LottoNumber.valueOf(2), LottoNumber.valueOf(3),
        LottoNumber.valueOf(14), LottoNumber.valueOf(15), LottoNumber.valueOf(16)
    ));
    assertThat(ticket.getMatchCount(winner)).isEqualTo(new Count(3));
}

// 0개 일치
// 6개 일치 (완전 일치)
```
