##  `lottoRankOfEachTicket` 만 넘기고 `LottoResult`가 스스로 집계하는 것의 차이가 뭔가요?
스스로 집계하는 게 좋을 것 같네요! 반영했습니다!

이렇게 변경하고 나니, LottoResult 생성자에서 잘 집계하는지 테스트하면 좋을 것 같다고 생각했습니다! 

그런데 임의의 리스트로 LottoResult를 만드는 방법이 없어 결과물과 비교할 expected를 만드려면 
결국 테스트하는 함수인 생성자를 사용해야 하는 상황이 생깁니다. 

이러한 테스트는 안만들어도 괜찮은 것일까요, 
아니면 테스트를 위한 생성자(이전에 있던 생성자 같은 것)를 만들어서 테스트하는 것이 좋을까요?

먼저, 테스트를 위한 생성자는 만들어도 괜찮다고 생각합니다.
물론, 테스트 작성을 위한 코드를  만든다는 것은, 그만큼 기존 로직이 테스트하기 어렵다는 신호이긴 합니다.
하지만 `LottoResult(Map<LottoRank, Integer> ticketCountOfEachRank) {}`와 같은 기본 생성자의 경우, 예외라고 생각해요. 말 그대로 `기본 생성자`니까요. 
테스트를 위해 만들었다는 전제 이전에, 기본적으로 다른 것으로 대신할 수 없는 기본적인 생성자입니다.
equals을 사용하지 않는다고, 강제로 지워버릴 필요가 없는 것과 비슷하게 생각해요.

이게 로직을 작성하고, 테스트를 작성하는 흐름에서 저도 연우님처럼 `테스트를 위한 코드` 로 보였어요. 하지만 TDD와 같이 테스트를 먼저 작성하고, 로직을 작성해보니 `테스트를 위해서` 라는 맥락 이전에, 객체가 당연히 가져야 할 코드 라는게 조금 더 다가왔던 것 같아요!

그래서 여기에 getter 같은 것도 포함될 수 있습니다. 
물론 이전에 언급했던 불변성 등이 보장된다면 말이죠.🫠

또한, 로직도 for문을 하나로 줄여볼 수 있을 것 같아요.

```
for (LottoRank rank : LottoRank.values()) { 
ticketCountOfEachRank.put(rank, Collections.frequency(lottoRankOfEachTicket, rank)); 
}
```

# LottoTest

> 하지만 티켓 리스트가 잘못된 건지, 당첨 티켓과 비교가 실패한건지, 등수별로 티켓 개수 리스트 반환하는 것이 실패한 것인지 또 몇 등에서 틀린 건지 알기 어렵다 느꼈거든요. 그래서 테스트 목적이 모호하다 느꼈습니다.

티켓 리스트가 잘못된 건지 -> LottoTicket의 책임  
당첨 티켓 비교가 실패한건지 -> LottoTicket의 책임  
몇등에서 틀린건지 -> 등수에 따라 실패한다면, LottoRank의 책임

이라고 생각하여, 총 티켓 개수와 수동 티켓 개수 (자동은 모두 1등, 수동은 모두 실패로 설정하여)를 파라미터로 테스트를 개편해 보았습니다! 등수별 티켓 개수 리스트 반환에 초점을 맞춰서요.  
이런 방향성은 어떤지 피드백 부탁드립니다!

Fixture가 선언적이기에, 새 코드는 `totalTicketCount`와 `manualTicketCount`라는 변수명만으로 시나리오가 읽혀지네요.

방향성 완전히 맞아요. 😄

지금은 `totalTicketCount`, `manualTicketCount` 두 변수만으로 "총 10장 중 5장은 수동(꽝), 5장은 자동(1등)" 이라는 시나리오 바로 읽히네요.😄

# View 와 도메인의 분리

"결국 누군가는 해야 한다"는 건 맞아요. 핵심은 **누가 하냐**가 아니라 **변경 이유가 같은 코드끼리 모여있냐**예요.

---

## View가 바뀌는 이유 vs 도메인이 바뀌는 이유

```
View가 바뀌는 이유    → UI가 바뀔 때
                       Scanner → Web 요청으로 바뀔 때
                       콘솔 → GUI로 바뀔 때

도메인이 바뀌는 이유  → 비즈니스 규칙이 바뀔 때
                        LottoTicket 구조가 바뀔 때
```

**변경 이유가 완전히 달라요.**

---

## 지금 코드의 진짜 문제

```java
// InputView가 바뀌어야 하는 상황이 두 가지예요
public LottoTicket inputWinnerTicket() {  
    return new LottoTicket(...)  // 도메인이 바뀔 때
    System.out.println(...)      // UI가 바뀔 때
}
```

View 안에 **두 가지 변경 이유**가 공존해요. 이게 SRP 위반이에요.

Controller는 원래부터 도메인 변경에 반응하는 레이어예요. 도메인이 바뀌면 Controller가 바뀌는 건 자연스러운 거고, View가 바뀌는 건 아닌 거예요.
