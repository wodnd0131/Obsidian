**A.**
포인터의 개념을 생각하는 방향성을 생각하고 질문하신 의도가 맞는진 모르겠지만, 제가 생각하기에 두 차이는 원본의 유지에 있는 것 같습니다,

첫 번째 class의 멤버 함수로서의 add는 일반적으로 return 값을 가지고 활용하기 때문에 a, b를 건드리지 않고 별도의 메모리에 저장하기 때문에, 메모리 면에서는 비효율적일 수 있겠지만 값을 온전히 보존할 수 있다는 장점은 있을 것 같습니다. 이는 class에서 private으로 선언된 변수일 경우 더욱 명확하게 확인될 것 같습니다. private 변수는 주로 외부의 메서드나 로직으로 인해 바뀌면 안 되는 값으로 선언되어 있는 경우가 다수니까요.

Q.
답변 정말 인상 깊게 읽었습니다.
제가 처음에 던진 질문의 의도와 정확히 같지는 않았지만, 질문을 본인만의 기준으로 해석해서 끝까지 고민해보신 점이 아주 좋았습니다!
특히 원본 보존과 상태 변경 가능성이라는 축으로 정리해주신 부분에서, 설계의 차이를 스스로 이해하려는 태도가 느껴졌어요. 이런 고민이 쌓이면 앞으로도 정말 많이 성장하실 것 같습니다!

제가 처음에 이야기하고 싶었던 핵심은, `calculator.add(a, b)` 와 `a.add(b)` 의 차이를 메모리나 호출 방식보다는 책임을 어디에 두는가의 관점에서 바라보는 것이었습니다.

예를 들면,
- calculator.add(a, b) 는 계산 책임을 외부 객체인 Calculator 에 두는 방식
- a.add(b) 는 계산 책임을 a 자신에게 두는 방식 (`a: Number(int value)`)
이라고 볼 수 있습니다.

즉, 이 연산을 가장 자연스럽게 알고 수행해야 하는 주체가 누구인가? 를 생각해보면 좋겠다는 의도였습니다.
현실의 비유로 보면 계산기가 계산을 수행하는 모습이 익숙하지만, 객체지향에서는 Number 역시 자신의 연산을 스스로 수행하는 주체로 볼 수 있어 이것도 충분히 자연스러운 설계가 될 수 있습니다. 다만 주의하실 점은, 어느 한 쪽이 정답인 것은 절대 아니라는 점입니다.

또 한 가지 덧붙이자면, `a.add(b)` 형태라고 해서 반드시 a의 상태를 직접 변경하는 방식으로만 구현되는 것은 아닙니다.
예를 들어 불변 객체처럼 설계하면, `a.add(b)` 가 내부적으로는 a를 바꾸는 대신 새로운 결과 객체 c를 만들어 반환하는 방식으로도 충분히 구현할 수 있습니다. (`return new Number(...)`)
그래서 `a.add(b)` 를 곧바로 원본을 변경하는 방식이라고 단정하기는 어렵고, 핵심은 호출 문법 자체보다 그 객체를 가변으로 설계했는지, 불변으로 설계했는지, 그리고 그 책임을 그 객체가 가지는 것이 자연스러운지에 더 가깝습니다.

---

질문 마지막에 말씀해주신 low-level 성능 차이에 대한 고민도 좋았습니다.
실제로는 메서드 호출 형태가 calculator.add(a, b) 인지 a.add(b) 인지 자체보다,
- 객체가 새로 생성되는지
- JVM이 인라이닝 같은 최적화를 하는지
- 불변 객체를 얼마나 자주 만들어내는지
- primitive 중심인지 wrapper/object 중심인지
같은 요소가 성능에 더 큰 영향을 주는 경우가 많습니다.

즉, 저수준에서 차이가 전혀 없다고 단정할 수는 없지만, 일반적인 애플리케이션 레벨에서는 메서드 호출 형태 자체보다 설계의 자연스러움과 책임 배치가 훨씬 더 중요한 경우가 많다고 볼 수 있습니다. 또한 이런 성능 특성은 언어와 런타임, JVM 버전이나 최적화 상황에 따라 달라질 수 있습니다.

현재 상황에서는 Calculator 가 primitive 타입을 파라미터로 받아 연산하므로 더 효율적으로 작동한다고 생각할 수 있습니다. 다만 값이 음수가 될 수 없다는 제약이 있다면, 이런 규칙을 여러 곳에 흩어진 로직으로 검사하기보다 Number 라는 객체가 생성될 때부터 일관되게 검증하도록 만드는 설계도 충분히 의미 있다고 생각합니다.

2. **StringCalculator도 1번의 호출로 결과를 받아볼 수 있을까요?**

**A.**
빼먹었네요..! 죄송합니다 ㅠㅠ add 함수를 split 안에 집어넣으면, 애초에 split 함수 자체의 로직을 분할 후 합까지 도출하는 것으로 만들면 추가적인 변수 할당이나 함수 실행을 줄일 수 있기 때문에 메모리나, 실행 측면에서 더욱 효율적일 수 있을 것 같습니다.

처음 저 코드를 구성할 시에 addAll 함수를 작성하고, Custom 구분자와 basic 구분자의 경우에 모두 사용하려고 따로 분할해 두긴 했는데, 병합하는 게 더 나을까요? 호출 면에서는 당연히 변수를 그대로 사용하는 방향이 좋겠지만, 기능별로 메서드를 분할해 두는 것도 충분히 효율적이라 생각해서요.

**Q.**
기능별로 메서드를 분리한 판단 자체는 아주 좋습니다! 
지금 구조의 아쉬운 점은, **호출하는 쪽이 계산 과정의 절차까지 알고 있어야 한다는 점**입니다.

```java
String[] result = calculator.basicSplit("1,2,4");
int fin = calculator.addAll(result);
```

이 구조는 먼저 split하고, 그 결과를 addAll에 넣어야 한다는 흐름을 사용하는 쪽에서 직접 알아야 한다는 뜻이거든요.
즉, StringCalculator 가 어떻게 동작하는지를 외부에서 순서대로 맞춰 호출하고 있는 셈입니다.

비유하자면, 커피 자판기에 커피 버튼 하나만 있으면 되는데, 사용자가 직접 물 끓이기 -> 원두 갈기 -> 섞기 버튼을 순서대로 눌러야 하는 구조와 비슷합니다.

그래서 답은 분리할까, 병합할까 둘 중 하나만 고르는 것이 아니라, 내부 구현은 분리하되 외부에는 하나의 완성된 기능으로 제공하는 것에 가깝습니다.

```java
public int calculate(String input) {
    String[] tokens = basicSplit(input);
    return addAll(tokens);
}
```

이렇게 하면 basicSplit, addAll 은 내부에서 역할별로 나뉘어 있어 읽기 쉽고, 외부에서는 calculate("1,2,4") 하나만 호출하면 되니 구현 상세를 몰라도 됩니다.

이게 바로 캡슐화의 핵심이라고 볼 수 있어요.
내부는 나누어도 되고, 외부에는 필요한 기능만 단순하게 보여줄 수 있습니다.
즉, 기능 분리 자체는 좋은 방향이었고, 여기에 외부에 어떤 기능을 제공할지를 한 번 더 고민해보면 더 좋아질 것 같습니다.

그리고 말씀해주신 성능 관점도 좋은 고민입니다.  
다만 이 경우에는 `split()` 과 `addAll()` 을 외부에서 각각 호출하느냐, 내부에서 한 번에 묶어 호출하느냐에 따른 오버헤드는 일반적으로 크지 않은 편입니다.  
오히려 더 중요한 부분은, 사용하는 쪽에서 계산 절차를 알고 있어야 하느냐, 아니면 `calculate()` 같은 하나의 메시지만 보내면 되느냐에 있습니다.  

즉, 여기서의 핵심은 미세한 성능 차이보다는 **역할은 내부에서 나누되, 외부에는 단순한 인터페이스(사용법)를 제공하는 구조**를 익히는 것에 더 가깝다고 생각합니다.

---

더 나아가서 고민해보면, `StringCalculator` 안에 문자열 해석과 계산 책임이 함께 들어 있는지도 살펴볼 수 있을 것 같아요.  
예를 들어 문자열을 숫자 목록으로 변환하는 `StringParser` 와, 변환된 값을 계산하는 `Calculator` 로 역할을 나누면 각자의 책임이 더 또렷해질 수 있습니다.

즉, 지금 단계에서는 먼저 외부에서 사용하는 방법을 단순하게 감추는 것을 고민해보고, 그 다음 단계로는 문자열을 해석하는 책임과 계산하는 책임을 분리할 수 있을지도 생각해보면 좋겠습니다.

**A.**

말씀해 주신 피드백, 깊게 고민해 볼 만 요소네요. 각각의 객체가 어디까지 알고 있어야 하는가? 에 대한 질문을 계속 던지면서, 메소드와 변수가 어디까지 수행 능력을 갖추어야 하는지에 대한 확실한 이해를 바탕으로 코드를 구성해야겠다는 것을 배웠습니다.
