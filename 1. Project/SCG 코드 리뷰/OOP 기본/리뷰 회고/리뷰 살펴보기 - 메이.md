### ❓ 이전에 구현 및 테스트한 기능을 여러 번 수행하는 함수는 테스트가 필요한가요?

만약 요구사항 변경에 의해 자동차 이름이 4글자까지만 가능하다면 테스트코드를 수정해야 할 것이고, Car, Name에서 Car name 관련 테스트를 해줬다면 테스트 수정 지점이 늘어날 것 같아요.

지금은 자동차 경주에 대해서만 다루는 애플리케이션을 구현하기에 Race에서만 name을 이용하지만, 만약 자동차를 이용한 다양한 기능이 추가됨에 따라 Car을 사용하는 객체가 추가된다면(ex. 자동차 수리, ...) 전부 테스트하기는 어려울 것 같다는 입장입니다!

### ❓ 랜덤 값 테스트에 관하여

Random 모듈 자체는 신뢰하시는데, "내가 Random 모듈을 제대로 사용하고 있는가"에 대해서 의심이 든 적이 있으신가요? (예를 들어 random.nextInt(10);가 `0 ~ 10`인지 `1 ~ 10`인지 헷갈릴 수도 있구요.)

### ❓ 에러 관련

이 부분은 각각의 장단점이 있어서 취향이라고 생각해요. 예를 들어 class로 정의하면 예외 상황에 따라 클래스가 무한정 늘어날 수 있다는 단점이 있고, 메시지로 알려주면 메시지 변경에 따라 테스트 코드도 같이 변경해야 한다는 단점이 있을 것 같아요.

### ❓ README.md 관련

요구사항 상세하게 잘 적어주셔서 저는 보기 편했습니다 ㅎㅎ

연우님이 README를 작성하는 목적이 무엇인가요? 예를 들어 스터디 요구사항을 충족했는지 확인하거나, 리뷰어에게 구현 내용과 요구사항에 대한 이해를 공유하는 용도라면 현재 형태도 충분히 괜찮아보여요.

그런데 "내가 만든 자동차 경주 애플리케이션은 이렇게 동작해요"를 표현하는 것이 목적이라면 이렇게 생각할 수도 있을 것 같아요. (이 목적이 정답이라는 것이 아닙니다!)

- 자동차 이름에 대한 제약 조건이 명시되어 있다면 좋을 것 같아요. 특히나 이름에 대한 조건은 LMS 요구사항에는 명시되어 있지 않았던 것 같아서, 연우님의 생각으로 판단한 내용(ex. 자동차 이름은 5글자 이하)은 미래의 연우님과 팀원들을 위해 문서화로 남겨두는 것도 좋을 것 같습니다!
- "자동차"에 대한 기능, "경주"에 대한 기능 이렇게 나눠서 기능 목록을 작성할 수도 있을 것 같아요. 혹은 연우님이 구현하신 객체별로 작성할 수도 있고요!
