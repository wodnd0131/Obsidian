2. **StringCalculator도 1번의 호출로 결과를 받아볼 수 있을까요?**
```
String[] result = calculator.basicSplit("1,2,4");
int fin = calculator.addAll(result);
```

학생이 `basicSplit()` → `addAll()` 을 외부에서 순서대로 호출하는 구조를 짰고, 리뷰어가 **"호출하는 쪽이 절차를 알고 있다"는 캡슐화 위반**을 지적했다. `calculate()` 하나로 감싸되 내부는 분리하라는 피드백.

### 대화에서 더 끌어낼 수 있었던 것들

#### 1. [[../../../JAVA/OOP/캡슐화|캡슐화]]의 정확한 정의

리뷰에서 캡슐화를 **"내부를 감춘다"** 수준으로 설명했는데, 조금 더 정밀하게 짚을 필요가 있다.

캡슐화는 두 가지를 함께 의미한다:

- **정보 은닉(Information Hiding)**: 내부 구현을 외부에 노출하지 않는다
- **불변식 보호(Invariant Protection)**: 객체가 항상 유효한 상태를 유지하도록 외부의 개입을 막는다

이 대화에서 핵심은 **정보 은닉** 쪽이다. `basicSplit → addAll`을 외부에서 순서대로 호출해야 한다는 건, 구현 상세(split 후 add한다는 사실)가 외부로 새어나온 것이다. 내부 구현이 바뀌면(예: delimiter를 다르게 처리하기로 변경) 호출하는 쪽 코드도 바꿔야 한다. **이게 결합도(coupling)가 높다는 의미다.**

```java
// 결합도 높음: 외부가 내부 절차에 의존
String[] tokens = calculator.basicSplit("1,2,4");
int result = calculator.addAll(tokens);

// 결합도 낮음: 외부는 의도(what)만 전달
int result = calculator.calculate("1,2,4");
```

#### 2. Tell, Don't Ask와의 연결

리뷰에서 캡슐화로 설명했지만, 사실 이 문제는 **Tell, Don't Ask** 원칙으로도 정확하게 설명된다.

> 객체에게 상태를 물어본 뒤 외부에서 판단하지 말고, 하라고 시켜라.

```java
// Ask: split 결과를 꺼내서 외부에서 판단
String[] tokens = calculator.basicSplit(input);
int result = calculator.addAll(tokens); // 외부가 절차를 조립

// Tell: 그냥 시킨다
int result = calculator.calculate(input);
```

`basicSplit()`의 반환값을 꺼내서 `addAll()`에 넣는 행위 자체가 **"내부 중간 상태를 외부에서 조립"** 하는 것이다. `String[]` 이라는 중간 산출물이 외부로 노출된 게 문제의 본질이다.

#### 3. 리뷰에서 언급된 [[../../../JAVA/OOP/SRP (단일 책임 원칙)|SRP (단일 책임 원칙)]] 힌트 — 더 밀어붙일 수 있다

리뷰 마지막에 `StringParser` + `Calculator` 분리를 제안했다. 이게 사실 이 대화에서 가장 중요한 다음 단계인데, 

```java
// 현재: StringCalculator이 파싱 + 계산을 모두 담당
class StringCalculator {
    public int calculate(String input) {
        String[] tokens = basicSplit(input);  // 파싱 책임
        return addAll(tokens);                // 계산 책임
    }
}

// 분리 후
class StringParser {
    public List<Integer> parse(String input) { ... }
}

class Calculator {
    public int sum(List<Integer> numbers) { ... }
}

class StringCalculator {
    private final StringParser parser;
    private final Calculator calculator;

    public int calculate(String input) {
        return calculator.sum(parser.parse(input));
    }
}
```

**왜 분리해야 하는가?** 단순히 "역할이 둘이니까"가 아니라 **변경 이유가 다르기 때문**이다.

- `StringParser`는 구분자 규칙이 바뀔 때 변경된다
- `Calculator`는 연산 종류가 바뀔 때 변경된다

변경 이유가 다른 것들이 같은 클래스에 있으면, 한쪽을 바꿀 때 다른 쪽이 영향을 받을 위험이 생긴다. **이게 SRP의 핵심이다. "하나의 책임"이란 "하나의 변경 이유"다.**


다음 질문으로 밀어붙이면 좋다:

> "그럼 `basicSplit()`을 `public`으로 두면 어떤 문제가 생길 수 있을까요?"

`public`으로 열려 있으면 다른 누군가가 언제든 `split → addAll` 순서로 외부에서 조립하는 코드를 짤 수 있다. 캡슐화는 **접근 제한자로 강제해야** 비로소 완성된다.

```java
class StringCalculator {
    public int calculate(String input) { ... }     // 외부에 공개
    private String[] basicSplit(String input) { ... }  // 내부 구현 은닉
    private int addAll(String[] tokens) { ... }        // 내부 구현 은닉
}
```

---

#### 한 줄 요약

이 대화의 본질은 **"외부 인터페이스(what)와 내부 구현(how)을 분리하라"** 이고, 그것을 강제하는 도구가 **접근 제한자**이며, 그 이유가 **결합도를 낮추고 변경을 격리하기 위해서**다. SRP, 캡슐화, Tell Don't Ask가 사실 같은 문제를 다른 각도에서 바라보는 것임을 연결해서 가르치면 학생 이해가 깊어질 것이다.

### `String[] tokens`가 외부로 노출된 것이 핵심

`basicSplit()`의 반환 타입을 보면 된다.

```java
String[] result = calculator.basicSplit("1,2,4");
int fin = calculator.addAll(result);
```

`basicSplit()`이 `String[]`을 반환한다는 건, **계산의 중간 상태**가 외부로 나왔다는 뜻이다. `String[]`은 `calculate`의 최종 결과물이 아니라, `addAll`을 호출하기 위한 **중간 산출물**이다.

외부 코드가 `String[]`을 들고 있다는 것 자체가 "다음에 `addAll`에 넣어야 한다"는 **절차를 알고 있다는 증거**다.

---

#### 일반화하면

```
중간 산출물이 외부로 나온다
    → 외부가 그걸 다음 단계에 넘겨야 한다
    → 외부가 단계의 순서를 알아야 한다
    → 절차가 외부에 노출된 것
```

반환 타입이 **최종 결과**냐 **중간 상태**냐를 보면 된다. 중간 상태가 반환되는 순간, 호출하는 쪽이 그 상태를 어딘가에 넘기는 코드를 추가로 써야 한다는 뜻이고, 그게 절차 노출이다.

> 추가적인 변수 할당이나 함수 실행을 줄일 수 있기 때문에 메모리나, 실행 측면에서 더욱 효율적일 수 있을 것 같습니다.
> 
> 처음 저 코드를 구성할 시에 addAll 함수를 작성하고, 
> Custom 구분자와 basic 구분자의 경우에 모두 사용하려고 따로 분할해 두긴 했는데, 
> 병합하는 게 더 나을까요?|
>  
> 호출 면에서는 당연히 변수를 그대로 사용하는 방향이 좋겠지만, 
> 기능별로 메서드를 분할해 두는 것도 충분히 효율적이라 생각해서요.
> 
> **Q.**
> 기능별로 메서드를 분리한 판단 자체는 아주 좋습니다! 
> 지금 구조의 아쉬운 점은, **호출하는 쪽이 계산 과정의 절차까지 알고 있어야 한다는 점**입니다.
> 
> ```java
> String[] result = calculator.basicSplit("1,2,4");
> int fin = calculator.addAll(result);
> ```
> 
> 이 구조는 먼저 split하고, 그 결과를 addAll에 넣어야 한다는 흐름을 
> 사용하는 쪽에서 직접 알아야 한다는 뜻이거든요.
> 즉, StringCalculator 가 어떻게 동작하는지를 **외부에서 순서대로 맞춰 호출하고 있는 셈입니다.**
> 
> 비유하자면, 커피 자판기에 커피 버튼 하나만 있으면 되는데, 
> 사용자가 직접 물 끓이기 -> 원두 갈기 -> 섞기 버튼을 순서대로 눌러야 하는 구조와 비슷합니다.
> 
> 그래서 답은 분리할까, 병합할까 둘 중 하나만 고르는 것이 아니라, 내부 구현은 분리하되 외부에는 하나의 완성된 기능으로 제공하는 것에 가깝습니다.
> 
> ```java
> public int calculate(String input) {
>     String[] tokens = basicSplit(input);
>     return addAll(tokens);
> }
> ```
> 
> 이렇게 하면 basicSplit, addAll 은 내부에서 역할별로 나뉘어 있어 읽기 쉽고, 
> 외부에서는 calculate("1,2,4") 하나만 호출하면 되니 구현 상세를 몰라도 됩니다.
> 
> 이게 바로 캡슐화의 핵심이라고 볼 수 있어요.
> 내부는 나누어도 되고, 외부에는 필요한 기능만 단순하게 보여줄 수 있습니다.
> 
> 즉, 기능 분리 자체는 좋은 방향이었고, 
> 여기에 외부에 어떤 기능을 제공할지를 한 번 더 고민해보면 더 좋아질 것 같습니다.
> 
> 그리고 말씀해주신 성능 관점도 좋은 고민입니다.  
> 다만 이 경우에는 `split()` 과 `addAll()` 을 외부에서 각각 호출하느냐, 
> 내부에서 한 번에 묶어 호출하느냐에 따른 오버헤드는 일반적으로 크지 않은 편입니다.  
> 
> 오히려 더 중요한 부분은, 사용하는 쪽에서 계산 절차를 알고 있어야 하느냐, 
> 아니면 `calculate()` 같은 하나의 메시지만 보내면 되느냐에 있습니다.  
> 
> 즉, 여기서의 핵심은 미세한 성능 차이보다는 **역할은 내부에서 나누되, 
> 외부에는 단순한 인터페이스(사용법)를 제공하는 구조**를 익히는 것에 더 가깝다고 생각합니다.
> 
> ---
> 
> 더 나아가서 고민해보면, `StringCalculator` 안에 문자열 해석과 계산 책임이 함께 들어 있는지도 살펴볼 수 있을 것 같아요.  
> 예를 들어 문자열을 숫자 목록으로 변환하는 `StringParser` 와, 변환된 값을 계산하는 `Calculator` 로 역할을 나누면 각자의 책임이 더 또렷해질 수 있습니다.
> 
> 즉, 지금 단계에서는 먼저 외부에서 사용하는 방법을 단순하게 감추는 것을 고민해보고, 
> 그 다음 단계로는 문자열을 해석하는 책임과 계산하는 책임을 분리할 수 있을지도 생각해보면 좋겠습니다.
> 
> **A.**
> 
> 말씀해 주신 피드백, 깊게 고민해 볼 만 요소네요. 각각의 객체가 어디까지 알고 있어야 하는가? 에 대한 질문을 계속 던지면서, 메소드와 변수가 어디까지 수행 능력을 갖추어야 하는지에 대한 확실한 이해를 바탕으로 코드를 구성해야겠다는 것을 배웠습니다.

### 요구사항을 검증하기 위한 테스트

```
구현을 본다
→ 해당 구현이 통과하는 테스트를 작성한다
→ 통과한다
```

이건 **테스트가 아니라 그냥 코드 복사**입니다.

테스트의 역할은 **요구사항을 검증**하는 것인데, 구현을 먼저 보고 테스트를 맞춰버리면 → 잘못된 구현도 통과하게 됩니다.

```
올바른 순서: 요구사항 → 테스트 → 구현
잘못된 순서: 구현 → 구현에 맞는 테스트 → 통과 (의미없음)
```

---

> "구현에 매몰되다 보니 시야가 좁아졌다"

이게 TDD가 권장되는 이유 중 하나입니다. 구현 전에 테스트를 먼저 짜면 **구현에 끌려가지 않고** 요구사항 관점을 유지할 수 있어요.
