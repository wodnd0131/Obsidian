## 코드 리뷰

### 잘한 점

**1. DTO 어노테이션 선택이 정확해**

[ReservationRequest.java](https://claude.ai/epitaxy/src/main/java/roomescape/reservations/dto/request/ReservationRequest.java)에서 `LocalTime`, `LocalDate`에 `@NotBlank`가 아닌 `@NotNull`을 쓴 것 — 당연해 보이지만 놓치기 쉬운 부분이야. `@NotBlank`는 `String` 전용이라 다른 타입에 붙이면 조용히 무시돼. 타입을 보고 어노테이션을 고른 게 맞아.

**2. Domain 불변화 방향이 맞아**

`final` 필드 + setter 제거. "도메인 객체는 생성 시점에 완전해야 한다"는 원칙을 코드로 표현한 거야. 의도가 코드에 잘 드러나.

**3. Repository에서 `orElse(null)` 제거**

null을 반환하는 건 API 계약이 불명확하다는 신호야. `orElseThrow`로 바꿔서 "이 메서드는 항상 값을 반환하거나 예외를 던진다"는 계약을 명확히 한 것 좋아.

### 완전히 틀린 점

#### 1. HTTP 상태 코드 의미를 잘못 쓰고 있어

[GlobalExceptionHandler.java:16-18](https://claude.ai/epitaxy/src/main/java/roomescape/global/error/GlobalExceptionHandler.java)

```java
@ExceptionHandler(ReservationNotFoundException.class)
public ResponseEntity<String> handleReservationNotFoundException(ReservationNotFoundException e) {
    return ResponseEntity.badRequest().body(e.getMessage()); // 400???
}
```

`ReservationNotFoundException`은 400이 아니라 **404**야. HTTP 상태 코드는 서버가 클라이언트에게 보내는 언어야.

|상황|맞는 코드|이유|
|---|---|---|
|`ReservationNotFoundException`|**404** Not Found|리소스가 존재하지 않음|
|`DuplicateReservationException`|**409** Conflict|이미 존재하는 리소스와 충돌|
|`ReservationFullException`|**409** Conflict 또는 **422** Unprocessable|규칙 위반|
|`InvalidReservationTimeException`|**400** Bad Request|잘못된 입력값|

400은 "클라이언트가 잘못된 데이터를 보냈다"는 의미야. 없는 예약을 삭제하려는 건 입력값이 잘못된 게 아니라 대상 리소스가 없는 거야.

#### 2. Service에서 검증 전에 ID를 먼저 채번하고 있어

[ReservationService.java:45-56](https://claude.ai/epitaxy/src/main/java/roomescape/reservations/service/ReservationService.java)

```java
public ReservationResponse createReservation(ReservationRequest newReservation) {
    Reservation reservation = new Reservation(
            idGenerator.getAndIncrement(), // ← ID가 여기서 이미 증가
            newReservation.name(),
            newReservation.date(),
            newReservation.time()
    );

    validateDuplicateReservation(reservation);    // 여기서 실패하면?
    validateCapacityPerTime(reservation.getDate(), reservation.getTime());
    // ID 1 소비 → 실패 → 다음 성공 예약은 ID 2
```

중복 예약이나 정원 초과로 예외가 터지면 ID 카운터는 이미 올라간 뒤야. 재현 시나리오:

```
POST /reservations (중복) → 예외 발생, idGenerator는 1→2로 증가
POST /reservations (정상) → id=2로 저장 (id=1 영원히 결번)
```

검증을 먼저 하거나, `Reservation` 생성 전에 비즈니스 규칙을 검사해야 해.

### 개선하면 좋을 점

#### 1. `MethodArgumentNotValidException` 처리가 클라이언트에 쓸모없는 정보를 내려보내

[GlobalExceptionHandler.java:21-24](https://claude.ai/epitaxy/src/main/java/roomescape/global/error/GlobalExceptionHandler.java)

```java
@ExceptionHandler(MethodArgumentNotValidException.class)
public ResponseEntity<String> handleReservationRequestWithMissingArgumentsException(MethodArgumentNotValidException e) {
    return ResponseEntity.badRequest().body(e.getMessage()); // 수백 자짜리 프레임워크 내부 문자열
}
```

`e.getMessage()`를 그대로 내려보내면 클라이언트는 이런 걸 받아:

```
Validation failed for argument [0] in public org.springframework.http.ResponseEntity
roomescape.reservations.controller.ReservationController.createReservation(
roomescape.reservations.dto.request.ReservationRequest): [Field error in object ...
```

실제로 필요한 건 "어떤 필드가 왜 틀렸는가"야:

```java
@ExceptionHandler(MethodArgumentNotValidException.class)
public ResponseEntity<Map<String, String>> handleValidation(MethodArgumentNotValidException e) {
    Map<String, String> errors = new LinkedHashMap<>();
    e.getBindingResult().getFieldErrors()
        .forEach(err -> errors.put(err.getField(), err.getDefaultMessage()));
    return ResponseEntity.badRequest().body(errors);
}
```

클라이언트가 받는 응답:

```json
{
  "name": "에약하실 분의 성함을 입력해주세요!",
  "date": "예약 날짜는 필수로 입력해주셔야 해요!"
}
```

#### 2. `ReservationResponse`의 필드명이 Java 컨벤션 위반이야

[ReservationResponse.java:7](https://claude.ai/epitaxy/src/main/java/roomescape/reservations/dto/response/ReservationResponse.java)

```java
public record ReservationResponse(
        Long Id,   // ← 대문자 I
        ...
)
```

Java 필드명은 camelCase고, 첫 글자는 소문자야. 이 상태로 JSON 직렬화하면 응답 키가 `"Id"`로 나와. API 스펙이 `"id"`를 기대하는 클라이언트가 있다면 조용히 깨져. `id`로 바꿔야 해. Controller에서도 `newReservation.Id()`로 접근하고 있어서 같이 수정해야 함.

#### 3. `protected Reservation()`이 도메인 불변식을 구멍 뚫고 있어

[Reservation.java:18-23](https://claude.ai/epitaxy/src/main/java/roomescape/reservations/model/Reservation.java)

```java
protected Reservation() {
    this.id = null;
    this.name = null; // 검증 없이 null 상태로 객체 생성 가능
    this.date = null;
    this.time = null;
}
```

"도메인 객체는 항상 유효한 상태여야 한다"고 했는데, 이 생성자는 그 원칙을 깨. 같은 패키지에서 `new Reservation()`을 호출하면 null 투성이인 `Reservation`이 만들어져. `protected`가 JPA 요구사항 때문에 어쩔 수 없이 있는 건지, 단순히 `final` 필드 때문에 추가한 건지가 불명확해. 현재 이 코드는 JPA도 아니고 인메모리 저장소인데 왜 있는지 설명이 필요해.

### 확장해서 생각해볼 질문들

**검증 레이어 관련**

> `Reservation` 도메인은 영업시간 검증을 직접 하고 있어. 그런데 Service도 중복/정원 검증을 해. 둘 다 "도메인 규칙"처럼 보이는데, 왜 어떤 건 Domain에, 어떤 건 Service에 있을까? 그 기준이 뭘까?

힌트: DB 조회가 필요한가, 아닌가.

**HTTP 상태 코드 관련**

> 현재 `deleteReservationById`는 존재하지 않는 ID를 삭제하려 할 때 400을 돌려줘. 실제 REST API를 설계할 때 없는 리소스를 DELETE하면 404를 내야 할까, 아니면 "어차피 없으니까 성공(204)"으로 봐야 할까?

정답이 하나가 아닌 질문이야. 멱등성(Idempotency) 개념과 연결돼 있어.

**예외 설계 관련**

> 현재 `ReservationNotFoundException`은 메시지가 다른 생성자 두 개를 가지고 있어. 왜 두 개가 필요할까? 그리고 Service에서 던지는 메시지(`"해당 id로는 예약건이 존재하지 않아요!"`)와 Repository에서 던지는 기본 메시지(`"일치하는 예약건이 없어요!"`)가 다른데 — 같은 예외가 다른 메시지로 두 군데서 던져지는 게 좋은 설계일까?
