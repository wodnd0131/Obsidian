파일들 이미 읽었으니 바로 정리할게.

## 질문별 코멘트

---

### Q1. DTO → Entity 변환 로직은 Service에서?

**판단 자체는 맞아.** 다만 "왜 Service인가"의 근거를 좀 더 정확히 잡는 게 좋아.

Service가 맞는 이유는 Controller가 도메인 모델을 몰라야 하기 때문이야. Controller는 HTTP 계층이고, `Reservation`이 어떤 필드를 갖는지, 어떻게 생성하는지 알면 안 돼. 지금 구조는 방향은 맞아.

그런데 현재 코드에서 변환 방식이 두 군데 다르게 구현돼 있어:

```java
// Service.createReservation — 직접 생성자 호출
Reservation reservation = new Reservation(
        idGenerator.getAndIncrement(),
        newReservation.name(),  // Request에서 꺼내서
        newReservation.date(),
        newReservation.time()
);

// Service.convertIntoResponseDTO — 변환 메서드 분리
private ReservationResponse convertIntoResponseDTO(Reservation reservation) {
    return new ReservationResponse(reservation.getId(), ...);
}
```

`Request → Domain`은 인라인, `Domain → Response`는 메서드 분리. 일관성이 없어. 실무에서 자주 쓰는 패턴 두 가지를 소개할게:

**패턴 A: DTO에 팩토리 메서드**

```java
// Response DTO가 자기 자신을 어떻게 만들지 알고 있음
public record ReservationResponse(...) {
    public static ReservationResponse from(Reservation reservation) {
        return new ReservationResponse(reservation.getId(), ...);
    }
}
```

**패턴 B: Domain에 팩토리 메서드**

```java
// Domain이 자기 자신을 어떻게 만들지 알고 있음
public class Reservation {
    public static Reservation from(Long id, ReservationRequest request) {
        return new Reservation(id, request.name(), request.date(), request.time());
    }
}
```

어떤 패턴이든 "변환 책임이 어디에 있는가"를 명시적으로 드러낼 수 있어. 현재 Service 인라인 코드는 책임이 숨어있는 형태야.

### Q2. 어디까지 커스텀 예외를 만들 것인가?

**방향은 맞는데, 적용이 일관되지 않아.**

"비즈니스 로직과 관련된 것들은 전부 커스텀 예외"라고 했는데, [`Reservation.java`](https://claude.ai/epitaxy/src/main/java/roomescape/reservations/model/Reservation.java)를 보면:

```java
private void validateNameNotEmpty(String name) {
    throw new IllegalArgumentException("이름을 필수로 입력해주셔야 해요!"); // 커스텀 아님
}

private void validateReservationTimeIsNotPast(LocalDate date, LocalTime time) {
    throw new IllegalArgumentException("지난 날짜로는 예약할 수 없어요!"); // 커스텀 아님
}

private void validateReservationInBusinessHour(LocalTime reservationTime) {
    throw new InvalidReservationTimeException(...); // 커스텀
}
```

같은 도메인 클래스 안에서 어떤 건 커스텀, 어떤 건 `IllegalArgumentException`. 원칙을 세웠으면 일관되게 적용해야 해.

실무에서 커스텀 예외를 나누는 더 실용적인 기준은 이거야:

> **"이 예외를 잡아서 다른 처리를 해야 할 일이 생기는가?"**

`IllegalArgumentException`으로 묶으면 나중에 "이름 누락"과 "과거 날짜 예약"을 서로 다른 상태 코드로 내려보내고 싶을 때 `GlobalExceptionHandler`에서 구분할 방법이 없어.

### Q3. 검증 로직의 위치

각 케이스에 대해 하나씩 코멘트할게.

**i) 스키마 필드 검증 → Domain (tell, don't ask)**

방향 맞아. 다만 지금 코드에서 **이중 검증**이 발생하고 있어:

```
@NotBlank on ReservationRequest.name  (Controller 레이어에서 막음)
    ↓ 통과
Reservation 생성자의 validateNameNotEmpty  (Domain에서 또 막음)
```

`name`이 blank면 Controller에서 이미 400 반환하니까, Domain의 `validateNameNotEmpty`는 Controller를 통하는 한 절대 실행 안 돼. 이건 "Defense in Depth(심층 방어)"로 의도적일 수 있어. 중요한 건 **의도한 이중 검증인지, 아닌지**를 인지하고 있어야 한다는 거야.

실무에서는 "DTO 검증은 형식, Domain 검증은 의미"로 분리하면 자연스럽게 이중 검증이 생겨. 이름 비어있는 것 자체를 두 번 검증하는 건 과잉이라는 시각도 있어.

**ii) 비즈니스 로직 → Service**

맞아. 중복 예약, 정원 초과는 DB(또는 저장소) 조회가 필요하니까 Domain이 알 수 없고, Service가 맞아.

**iii) 단순 누락 → Controller (@Valid)**

맞아. 이게 Controller 레이어 검증의 전형적인 역할이야.

### 실무에서 빠진 예외 처리

현재 `GlobalExceptionHandler`가 잡지 못하는 케이스들이 있어:

**케이스 1. JSON 파싱 실패 (`HttpMessageNotReadableException`)**

```http
POST /reservations
{"name": "홍길동", "date": "2025-13-45", "time": "25:99"}
```

`LocalDate`, `LocalTime`으로 역직렬화할 수 없는 값이 들어오면 `@NotNull` 검증 전에 파싱 자체가 실패해. Spring이 `HttpMessageNotReadableException`을 던지는데, 현재 핸들러에 없으면 500으로 내려가.

**케이스 2. Path Variable 타입 불일치**

```http
GET /reservations/abc
```

`@PathVariable Long id`에 문자열이 오면 `MethodArgumentTypeMismatchException`. 역시 핸들러 없으면 500.

**케이스 3. 지원하지 않는 HTTP 메서드 (`MethodNotAllowedException`)**

```http
PATCH /reservations/1
```

405로 내려가야 하는데 기본 Spring 에러 처리에 맡기면 응답 형식이 다른 에러들과 달라져.

### DTO 사용 적절성

한 가지 이슈가 있어. [ReservationController.java:12](https://claude.ai/epitaxy/src/main/java/roomescape/reservations/controller/ReservationController.java):

import roomescape.reservations.model.Reservation; // 사용하지 않는 import

`Reservation` 도메인 모델이 Controller에 import돼 있어. 실제로 사용하지 않는 죽은 import인데, Controller가 Domain을 참조한다는 잘못된 신호를 줘. 정리 필요해.

### Spring 컨벤션 위반

**1. `ReservationResponse`의 필드명**

[ReservationResponse.java:7](https://claude.ai/epitaxy/src/main/java/roomescape/reservations/dto/response/ReservationResponse.java)

```java
public record ReservationResponse(
        Long Id, // ← Java 명명 규칙 위반, JSON 직렬화 시 "Id"로 나옴
```

Spring이 Jackson으로 직렬화할 때 record의 accessor 메서드 이름을 그대로 써. `Id()`라는 메서드가 생기고 JSON 키는 `"Id"`가 돼. 클라이언트가 `"id"` 키를 기대하면 조용히 깨지는 버그야. Controller에서 `newReservation.Id()`로 호출하는 것도 컨벤션 위반이고.

**2. 와일드카드 import**

[ReservationController.java:8](https://claude.ai/epitaxy/src/main/java/roomescape/reservations/controller/ReservationController.java)

```java
import org.springframework.web.bind.annotation.*;
```

IDE에서는 관행적으로 쓰기도 하지만, 실무 코드 리뷰에서 자주 지적받아. 어떤 어노테이션을 쓰는지 명시적으로 보이지 않고, 이름 충돌 위험도 있어.

**3. `protected Reservation()` 생성자**

[Reservation.java:18](https://claude.ai/epitaxy/src/main/java/roomescape/reservations/model/Reservation.java)

JPA 엔터티에서는 프록시 생성을 위해 no-arg 생성자가 강제돼. 그래서 `protected`로 만드는 패턴이 있어. 그런데 이 코드는 JPA를 쓰지 않는 인메모리 저장소야. 이유 없이 도메인 불변식을 우회하는 생성자가 열려있는 거야. 왜 남겼는지 설명이 필요한 코드야.

### 연관된 내용 정리

위 내용들이 서로 연결되는 구조를 정리하면:

```
[Q1 변환 위치] → convertIntoResponseDTO 방식 불일치
    ↕ 연결
[Spring 컨벤션] → ReservationResponse.Id 대문자 문제
                    (변환 메서드가 잘못된 필드명을 그대로 씀)

[Q2 커스텀 예외] → Domain에서 IllegalArgumentException 혼용
    ↕ 연결
[Q3 검증 위치] → Domain 검증이 커스텀 예외를 일관성 없이 사용
    ↕ 연결
[빠진 예외 처리] → GlobalExceptionHandler가 HttpMessageNotReadableException 미처리
                    (JSON 파싱 실패 → 500)
```

핵심 공통 원칙으로 모아보면: **"일관성"**이야. 방향은 전부 맞는데 끝까지 밀지 않은 지점들이 있어. 커스텀 예외를 쓰기로 했으면 Domain에서도, Service에서도 동일하게. DTO를 쓰기로 했으면 import까지 정리. 검증 레이어를 나눴으면 각 레이어에서 같은 방식으로 처리. 방향을 결정했다면 그 결정의 끝까지 가는 게 실무 코드야.
