
> "디비에 데이터를 추가하면서 1씩 증가하는 id를 생성하고 그 추가한 데이터를 식별자 id를 포함하여 객체로 변환해서 Reservation 객체에 저장하는 걸로 해석하면 될까요?"

---
먼저 jdbcTemplate을 사용하여, 객체를 DB에 저장하는 플로우에 대한 질문으로 보겠습니다.

1. 디비에 데이터를 추가하면서 1씩 증가하는 id를 생성하고
2. 그 추가한 데이터를 식별자 id를 포함하여 객체로 변환해서
3. Reservation 객체에 저장하는 걸로 해석하면 될까요?

살짝 다듬으면 더 명확해질 것 같아요.
플로우를 코드랑 대입하면 다음과 같습니다.

1. name, date, time을 담아 INSERT 요청을 보냄 
2. DB가 id를 자동 생성해서 row를 만듦
3. DB가 만든 id를 `KeyHolder`를 통해 Java로 돌려받음
4. 그 id로 응답용 `Reservation`을 새로 만들어 반환

먼저 흐름은 잘 이해하신 것 같아요.  😀
말씀처럼 DB가 Id 할당의 주체이고, id 값만 돌려 받아 서버 측에서 새로 객체를 만듭니다.

>아 그리고 KeyHolder가 들어가면 코드 길이가 엄청 길어지는 것 같더라고요.  
아직 keyholder는 생소한 편이라 여기 위주로 설명주셔도 좋습니다.

이렇게 정리하고 보니까, 다음 질문과 함께 좀더 구체적인 과정에 대해 궁금해하셨을까 싶네요.🤔

1. [PreparedStatement](https://docs.oracle.com/en/java/javase/17/docs/api/java.sql/java/sql/PreparedStatement.html) 생성 
`PreparedStatement`는  DB 서버에 미리 준비해둔 SQL 실행 계획을 가리키는 핸들이에요. `connection.prepareStatement(sql, ...)`를 호출하면 드라이버가 DB와 통신해서 SQL을 파싱시키고 실행 계획을 보관시킨 뒤, 그 자원을 가리키는 객체를 Java로 돌려줍니다.
그래서 `?` 자리에 값을 바꿔 끼우며 여러 번 실행해도 DB는 같은 실행 계획을 재사용할 수 있습니다.

```
PreparedStatement ps = connection.prepareStatement(  
        "INSERT INTO reservation (name, date, time) VALUES (?, ?, ?)",  
        new String[]{"id"});
```
여기서 `new String[]{"id"}`가 `INSERT` 끝나면 `id` 컬럼의 생성된 값을 회수할 수 있게 준비하라는 요청입니다. 
`new String[]{"id"}` 같은 generated key 회수 신호는 `prepareStatement()`를 호출하는 그 순간에만 줄 수 있습니다.

PreparedStatement는 DB 서버에 이미 준비된 statement의 핸들이기에, generated key 옵션 여부는 그 준비 단계에서 SQL 실행 방식 자체를 바꿉니다. 그래서 사후에 옵션만 바꾸는 건 불가능하고, 바꾸려면 DB 측에서 새로 prepare하는 수밖에 없습니다.

그래서 PreparedStatement 생성 자체를 사용자가 제어해야 하도록 넘겨, `PreparedStatementCreator` 를 파라미터로 받게 된 거예요.

*jdbcTemplate.update() 명세
`public int update(final PreparedStatementCreator psc, final KeyHolder generatedKeyHolder)`
    
2. `executeUpdate()` 실행
여긴 희창님의 말씀처럼, DB가 INSERT를 수행하면서 AUTO_INCREMENT id를 채워넣습니다.👍
    
3. `getGeneratedKeys()`로 회수
1번에서 신호를 줬기 때문에, DB가 생성한 id가 `ResultSet` 형태로 회수 가능한 상태가 됩니다.

[ResultSet](https://docs.oracle.com/en/java/javase/17/docs/api/java.sql/java/sql/ResultSet.html)은 DB가 돌려준 결과 row들을 한 줄씩 순회할 수 있게 해주는 커서(cursor) 같은 객체입니다.
`rs.next()`로 다음 row로 이동하고 `rs.getObject(컬럼)`으로 값을 꺼내는 식입니다. 그래서 결과를 다 쓰기 전까지는 DB 측 자원을 계속 점유하게 됩니다. 여기서 사용자는 이 ResultSet을 직접 열고, 순회하고, 닫아야 합니다. 매번 반복되는 자원 관리·순회·예외 처리가 보일러플레이트로 쌓이고, 자원 해제 누락으로 인한 connection leak도 흔한 문제였어요. JdbcTemplate은 이 ResultSet 자체를 사용자에게 노출하지 않고 내부에서 다 처리하는 방향으로 풀었습니다.

JdbcTemplate은 이 `ResultSet`을 사용자에게 그대로 노출하지 않고, 내부에서 끝까지 순회하면서 각 row를 `Map<컬럼명, 값>` 형태로 변환해 `KeyHolder`에 채워넣은 뒤 ResultSet을 닫습니다. 
[KeyHolder](https://docs.spring.io/spring-framework/docs/current/javadoc-api/org/springframework/jdbc/support/KeyHolder.html)는 `prepareStatement`에서 회수 신호를 준 컬럼 값들이 담기는, Spring이 만든 결과 그릇입니다. (JDBC 표준 x ) 그래서 사용자 코드에서는 ResultSet을 다룰 일이 없고, `keyHolder.getKey()` 호출로 그 안에 담긴 첫 값을 꺼내기만 하면 되도록 보다 간단히 만든 것입니다.

KeyHolder가 길어 보이는 건, generated key를 받으려면 JDBC 레벨에서 이만큼은 직접 다뤄야 하기 때문입니다. 반대로 JdbcTemplate가 대신 다루어주는 부분도 비교해보면 좋을 것 같아요😄
JdbcTamplete은 또 순수 JDBC의 무엇을 대신하고 있을까요?

## 1. "Reservation은 Entity인가요?"

이 질문은 **단어의 모호성** 때문에 답이 갈려요. "Entity"라는 단어가 두 가지 의미로 쓰이거든요:

**의미 A: JPA `@Entity`**  
`@Entity` 어노테이션이 붙은, JPA가 관리하는 영속 객체. DB 테이블과 매핑된 클래스.

**의미 B: DDD의 Entity**  
도메인 주도 설계에서 말하는 "고유 식별자를 가진 도메인 객체". 영속성과 무관하게, **식별자로 구분되고 생명주기를 가지는 도메인 개념**.

지금 코드에서 `Reservation`이 어느 쪽인지는 클래스 정의를 봐야 알지만, **메서드 이름이 `toEntity()`이고 `id`를 받는다는 점**에서 짐작이 가요:

```java
public static Reservation toEntity(Reservation reservation, Long id) {
    return new Reservation(id, ...);
}
```

이건 **"식별자(id) 없는 상태 → 식별자 있는 상태로 전환"**하는 메서드예요. 즉 의미 B(DDD의 Entity)에 가까워요. DB에 저장돼서 PK를 부여받은 "정식 Entity"로 승격시키는 작업.

그래서 "Reservation은 Entity인가?" → **결과물(id를 가진 Reservation)은 Entity가 맞아요.** 입력으로 들어온 Reservation(id 없음)은 엄밀히 말하면 아직 Entity가 아니라 **transient 상태**거나 **도메인 모델의 미저장 형태**고요.

## 2. 코드의 진짜 이상한 점

근데 사실 wddy가 짚어야 할 더 큰 문제는 **이 메서드의 위치와 형태**예요.

```java
public static Reservation toEntity(Reservation reservation, Long id) {
    return new Reservation(id, reservation.getName(), reservation.getDate(), reservation.getTime());
}
```

이 메서드, **자기 자신을 받아서 자기 자신을 반환**하고 있어요. 이게 좀 이상한 신호예요. 보통 `toEntity()`는:

```java
// 일반적인 패턴 1: DTO가 가짐
public class ReservationRequest {
    public Reservation toEntity() {
        return new Reservation(this.name, this.date, this.time);
    }
}

// 일반적인 패턴 2: 별도 매퍼
public class ReservationMapper {
    public static Reservation toEntity(ReservationRequest dto) { ... }
}
```

이런 식으로 **"다른 타입(DTO) → Entity"** 변환에 쓰여요. 그런데 지금 코드는 `Reservation → Reservation`이에요. 변환이 아니라 **복제 + id 주입**이에요.

## 3. "기본 생성자가 아닌 정적 팩토리 메서드를 둔 이유"

질문 자체는 정적 팩토리 메서드의 일반적 장점을 묻는 거니까 거기 답해드릴게요.

**기본 생성자(new) 대비 정적 팩토리 메서드의 장점** (Effective Java Item 1):

**1) 이름을 가질 수 있다**  
`new Reservation(...)`보다 `Reservation.toEntity(...)`, `Reservation.withId(...)`, `Reservation.of(...)`가 의도를 명확하게 표현해요. 생성자는 클래스명과 같아서 이름으로 의도를 못 드러내요.

**2) 호출할 때마다 새 객체를 만들 필요가 없다**  
캐싱이나 싱글톤이 가능. `Boolean.valueOf(true)`가 매번 새 객체 안 만드는 것처럼.

**3) 반환 타입의 하위 타입을 반환할 수 있다**  
인터페이스를 반환하면서 실제로는 다양한 구현체를 줄 수 있음. `List.of(...)`가 상황에 따라 다른 구현 반환하는 것처럼.

**4) 입력 파라미터에 따라 다른 클래스의 객체를 반환할 수 있다**  
다형성 활용.

**5) 클래스가 작성된 시점에 반환 객체 클래스가 존재하지 않아도 된다**  
서비스 프로바이더 프레임워크(JDBC 같은)의 기반.

## 4. 그런데 이 코드의 경우에 진짜 정당화되는가?

솔직히 말하면, **지금 코드에서 정적 팩토리 메서드의 장점들 중 어느 것도 강하게 작동하지 않아요.**

- 이름? `toEntity`라는 이름이 적절한가? → **부적절해요.** 입력도 Entity, 출력도 Entity니까 변환이 아니라 "withId" 같은 게 더 정확.
- 캐싱? → 매번 `new Reservation` 호출하니까 X
- 하위 타입 반환? → 그냥 Reservation 반환 X
- 다른 클래스 반환? → X

**즉 이 정적 팩토리 메서드의 진짜 용도는 "id가 없는 Reservation에 id를 주입한 새 Reservation을 만들기"인데, 그렇다면 이름과 시그니처를 다르게 짜는 게 맞아요.**

## 5. 더 자연스러운 설계 대안들

**대안 A: 인스턴스 메서드로 (불변 객체 패턴)**

```java
public class Reservation {
    public Reservation withId(Long id) {
        return new Reservation(id, this.name, this.date, this.time);
    }
}

// 사용
Reservation saved = reservation.withId(generatedId);
```

이게 가장 자연스러워요. "기존 Reservation을 기반으로 id만 다른 새 인스턴스 생성"이라는 의도가 메서드 이름에 그대로 드러나요.

**대안 B: 정적 팩토리지만 의도 명확하게**

```java
public static Reservation of(Long id, ReservationRequest request) {
    return new Reservation(id, request.getName(), request.getDate(), request.getTime());
}
```

입력이 DTO라면 "DTO → Entity" 변환으로서의 `toEntity`가 의미 있어요. 단, 이러면 `Reservation`이 `ReservationRequest`를 알아야 해서 도메인이 DTO에 의존하는 게 좀 찝찝해요.

**대안 C: 매퍼 분리**

```java
public class ReservationMapper {
    public static Reservation toEntity(ReservationRequest request, Long id) {
        return new Reservation(id, request.getName(), ...);
    }
}
```

Entity와 DTO 변환 로직을 별도 클래스로 분리. 의존성 방향이 깔끔.

## 6. 그래서 질문에 정리해서 답하면

> **Reservation는 Entity인가요?**

→ 네, id를 가진 도메인 객체이므로 Entity 맞아요. `@Entity` 어노테이션 여부와는 별개로 DDD 관점에서 Entity 역할이에요.

> **본인을 생성하는 작업인데, 기본 생성자가 아닌 정적 팩토리 메서드를 둔 이유가 무엇인가요?**

→ 일반적인 정적 팩토리 메서드의 장점은 "이름을 가질 수 있다, 캐싱 가능, 하위 타입 반환 가능" 등이 있어요. **하지만 지금 이 코드에서는 그 장점들이 거의 활용되지 않아요.**

이 코드의 진짜 의도는 **"id가 없는 Reservation을 받아서 id가 채워진 새 Reservation을 만드는 것"**인데, 그렇다면 `toEntity()`라는 이름은 어색하고 `withId()` 같은 인스턴스 메서드가 의도를 더 잘 표현해요.

질문을 던지신 분이 의도한 답이 "정적 팩토리 메서드의 일반론적 장점"인지, "이 코드가 왜 이상한지 짚어보라"인지에 따라 답 방향이 달라질 것 같은데, **둘 다 답할 만한 가치가 있는 좋은 포인트**예요.
