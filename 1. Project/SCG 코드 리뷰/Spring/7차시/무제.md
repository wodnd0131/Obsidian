
> "디비에 데이터를 추가하면서 1씩 증가하는 id를 생성하고 그 추가한 데이터를 식별자 id를 포함하여 객체로 변환해서 Reservation 객체에 저장하는 걸로 해석하면 될까요?"

---
먼저 jdbcTemplate을 사용하여, 객체를 DB에 저장하는 플로우에 대한 질문으로 보겠습니다.

1. 디비에 데이터를 추가하면서 1씩 증가하는 id를 생성하고
2. 그 추가한 데이터를 식별자 id를 포함하여 객체로 변환해서
3. Reservation 객체에 저장하는 걸로 해석하면 될까요?

살짝 다듬으면 더 명확해질 것 같아요.
플로우를 코드랑 대입하면 다음과 같습니다.

1. name, date, time을 담아 INSERT 요청을 보냄 
2. DB가 id를 자동 생성해서 row를 만듦
3. DB가 만든 id를 `KeyHolder`를 통해 Java로 돌려받음
4. 그 id로 응답용 `Reservation`을 새로 만들어 반환

먼저 흐름은 잘 이해하신 것 같아요.  😀
말씀처럼 DB가 Id 할당의 주체이고, id 값만 돌려 받아 서버 측에서 새로 객체를 만듭니다.

>아 그리고 KeyHolder가 들어가면 코드 길이가 엄청 길어지는 것 같더라고요.  
아직 keyholder는 생소한 편이라 여기 위주로 설명주셔도 좋습니다.

이렇게 정리하고 보니까, 다음 질문과 함께 좀더 구체적인 과정에 대해 궁금해하셨을까 싶네요.🤔

1. [PreparedStatement](https://docs.oracle.com/en/java/javase/17/docs/api/java.sql/java/sql/PreparedStatement.html) 생성 
`PreparedStatement`는  DB 서버에 미리 준비해둔 SQL 실행 계획을 가리키는 핸들이에요. `connection.prepareStatement(sql, ...)`를 호출하면 드라이버가 DB와 통신해서 SQL을 파싱시키고 실행 계획을 보관시킨 뒤, 그 자원을 가리키는 객체를 Java로 돌려줍니다.
그래서 `?` 자리에 값을 바꿔 끼우며 여러 번 실행해도 DB는 같은 실행 계획을 재사용할 수 있습니다.

```
PreparedStatement ps = connection.prepareStatement(  
        "INSERT INTO reservation (name, date, time) VALUES (?, ?, ?)",  
        new String[]{"id"});
```
여기서 `new String[]{"id"}`가 `INSERT` 끝나면 `id` 컬럼의 생성된 값을 회수할 수 있게 준비하라는 요청입니다. 
`new String[]{"id"}` 같은 generated key 회수 신호는 `prepareStatement()`를 호출하는 그 순간에만 줄 수 있습니다.

PreparedStatement는 DB 서버에 이미 준비된 statement의 핸들이기에, generated key 옵션 여부는 그 준비 단계에서 SQL 실행 방식 자체를 바꿉니다. 그래서 사후에 옵션만 바꾸는 건 불가능하고, 바꾸려면 DB 측에서 새로 prepare하는 수밖에 없습니다.

그래서 PreparedStatement 생성 자체를 사용자가 제어해야 하도록 넘겨, `PreparedStatementCreator` 를 파라미터로 받게 된 거예요.

*jdbcTemplate.update() 명세
`public int update(final PreparedStatementCreator psc, final KeyHolder generatedKeyHolder)`
    
2. `executeUpdate()` 실행
여긴 희창님의 말씀처럼, DB가 INSERT를 수행하면서 AUTO_INCREMENT id를 채워넣습니다.👍
    
3. `getGeneratedKeys()`로 회수
1번에서 신호를 줬기 때문에, DB가 생성한 id가 `ResultSet` 형태로 회수 가능한 상태가 됩니다.

[ResultSet](https://docs.oracle.com/en/java/javase/17/docs/api/java.sql/java/sql/ResultSet.html)은 DB가 돌려준 결과 row들을 한 줄씩 순회할 수 있게 해주는 커서(cursor) 같은 객체입니다.
`rs.next()`로 다음 row로 이동하고 `rs.getObject(컬럼)`으로 값을 꺼내는 식입니다. 그래서 결과를 다 쓰기 전까지는 DB 측 자원을 계속 점유하게 됩니다. 여기서 사용자는 이 ResultSet을 직접 열고, 순회하고, 닫아야 합니다. 매번 반복되는 자원 관리·순회·예외 처리가 보일러플레이트로 쌓이고, 자원 해제 누락으로 인한 connection leak도 흔한 문제였어요. JdbcTemplate은 이 ResultSet 자체를 사용자에게 노출하지 않고 내부에서 다 처리하는 방향으로 풀었습니다.

JdbcTemplate은 이 `ResultSet`을 사용자에게 그대로 노출하지 않고, 내부에서 끝까지 순회하면서 각 row를 `Map<컬럼명, 값>` 형태로 변환해 `KeyHolder`에 채워넣은 뒤 ResultSet을 닫습니다. 
[KeyHolder](https://docs.spring.io/spring-framework/docs/current/javadoc-api/org/springframework/jdbc/support/KeyHolder.html)는 `prepareStatement`에서 회수 신호를 준 컬럼 값들이 담기는, Spring이 만든 결과 그릇입니다. (JDBC 표준 x ) 그래서 사용자 코드에서는 ResultSet을 다룰 일이 없고, `keyHolder.getKey()` 호출로 그 안에 담긴 첫 값을 꺼내기만 하면 되도록 보다 간단히 만든 것입니다.

KeyHolder가 길어 보이는 건, generated key를 받으려면 JDBC 레벨에서 이만큼은 직접 다뤄야 하기 때문입니다. 반대로 JdbcTemplate가 대신 다루어주는 부분도 비교해보면 좋을 것 같아요😄
JdbcTamplete은 또 순수 JDBC의 무엇을 대신하고 있을까요?
