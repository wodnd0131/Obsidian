
## Q1 답변 초안

> "디비에 데이터를 추가하면서 1씩 증가하는 id를 생성하고 그 추가한 데이터를 식별자 id를 포함하여 객체로 변환해서 Reservation 객체에 저장하는 걸로 해석하면 될까요?"

---
먼저 jdbcTemplate을 사용하여, 객체를 DB에 저장하는 플로우에 대한 질문으로 보겠습니다.

1. 디비에 데이터를 추가하면서 1씩 증가하는 id를 생성하고
2. 그 추가한 데이터를 식별자 id를 포함하여 객체로 변환해서
3. Reservation 객체에 저장하는 걸로 해석하면 될까요?

살짝 다듬으면 더 명확해질 것 같아요.
플로우를 코드랑 대입하면 다음과 같습니다.

1. name, date, time을 담아 INSERT 요청을 보냄 
2. DB가 id를 자동 생성해서 row를 만듦
3. DB가 만든 id를 `KeyHolder`를 통해 Java로 돌려받음
4. 그 id로 응답용 `Reservation`을 새로 만들어 반환

흐름은 잘 이해하신 것 같아요. DB가 Id 할당의 주체이고, id 값만 돌려 받아 서버 측에서 새로 객체를 만듭니다.
이렇게 정리하고 보니까 `KeyHolder `로 id를 돌려받는 과정에 대해 궁금해하셨다는 생각이 드네요.🤔

 `jdbcTemplate.update(...)`를 호출하셨을 때, 그 안에서 일어나는 일을 큰 그림으로 보면 사실 JDBC 표준이 정해둔 3단계가 전부입니다.

1. PreparedStatement 생성 
```
PreparedStatement ps = connection.prepareStatement(  
        "INSERT INTO reservation (name, date, time) VALUES (?, ?, ?)",  
        new String[]{"id"});
```
여기서 `new String[]{"id"}`가 `INSERT` 끝나면 `id` 컬럼의 생성된 값을 회수할 수 있게 준비하라는 요청입니다. 
`new String[]{"id"}` 같은 generated key 회수 신호는 `prepareStatement()`를 호출하는 그 순간에만 줄 수 있습니다. 만들어진 뒤에는 추가할 수 없어요. 

PreparedStatement는 DB 서버에 이미 준비된 statement의 핸들이에요. generated key 옵션은 그 준비 단계에서 SQL 실행 방식 자체를 바꿔요. 그래서 사후에 옵션만 바꾸는 건 불가능하고, 바꾸려면 DB 측에서 새로 prepare하는 수밖에 없어요. JDBC가 옵션 setter를 안 만든 이유고요.


그래서 PreparedStatement 생성 자체를 사용자가 제어해야 하고, 그 결과 `PreparedStatementCreator` 람다를 파라미터로 받게 된 거예요. 

Connection 설정과 달리 사용자 제어가 필요해 탬플릿으로 내부에서 알아서 설정하지 않고, 외부에서 전달받기로한겁니다.

*jdbcTemplate.update() 명세
`public int update(final PreparedStatementCreator psc, final KeyHolder generatedKeyHolder)`
    
2. `executeUpdate()` 실행
여기서는 희창님의 말씀처럼, DB가 INSERT를 수행하면서 AUTO_INCREMENT id를 채워넣습니다.
    
3. `getGeneratedKeys()`로 회수
1번에서 신호를 줬기 때문에, DB가 생성한 id가 ResultSet 형태로 회수 가능한 상태가 됩니다.
받은 ResultSet을 순회하면서 `KeyHolder`에 `Map<컬럼명, 값>` 형태로 채워 넣습니다.
그래서 `keyHolder.getKey()`만 호출하면 그 안에 담긴 첫 값(id)을 꺼낼 수 있게 됩니다.


- [`Connection.prepareStatement(String, String[])` Javadoc](https://docs.oracle.com/en/java/javase/17/docs/api/java.sql/java/sql/Connection.html#prepareStatement\(java.lang.String,java.lang.String%5B%5D\)) — 그 배열의 정확한 역할 명세
- [`Statement.getGeneratedKeys()` Javadoc](https://docs.oracle.com/en/java/javase/17/docs/api/java.sql/java/sql/Statement.html#getGeneratedKeys\(\)) — 회수 메서드 명세
- [Spring 공식 레퍼런스 — Retrieving Auto-generated Keys](https://docs.spring.io/spring-framework/reference/data-access/jdbc/core.html#jdbc-auto-generated-keys) — JdbcTemplate이 이걸 어떻게 감싸는지

---

**그리고 이 질문에는 사실 더 큰 맥락이 깔려 있어요.**

진우님이 지금 다루고 있는 건 "**객체와 DB row 사이를 어떻게 오가게 할 것인가**"라는 주제예요. 지금은 그 작업을 컨트롤러 안에서 직접 손으로 하고 계신 거고요. PreparedStatement 만들고, 파라미터 바인딩하고, KeyHolder로 키 회수하고, 그 결과로 새 객체를 조립하고.

이 과정을 **반복하다 보면 자연스럽게 떠오르는 질문**이 있어요. "이 객체↔row 변환 책임을 한 군데로 모을 수 없을까?" 그 답이 DAO(Data Access Object)고, JdbcTemplate은 그중 보일러플레이트를 줄여주는 도구예요. 더 나아가서 "Java 객체를 다루는 것처럼 쓰면 ORM이 알아서 row로 바꿔주면 안 되나?"가 JPA의 출발점이고요.

지금 단계에서는 이걸 다 알 필요는 없어요. 다만 지금 손으로 짜고 있는 이 코드가 **그 큰 흐름의 가장 첫 자리**에 있다는 것만 알아두시면, 앞으로 새로운 도구를 만날 때마다 "아, 이건 지금 내가 손으로 하던 그 부분을 줄여주는 거구나"가 보이실 거예요 😊

---

코드 시각화는 빼고 흐름 위주로만 정리했어. 톤 맞는지 봐줘.

## Q2 답변 초안

> "KeyHolder가 들어가면 코드 길이가 엄청 길어지는 것 같더라고요."

---

이 체감, 정확히 짚으셨어요. 실제로 코드가 길어집니다. 그리고 그게 왜 그런지 이해하시면 KeyHolder가 훨씬 덜 생소해지실 거예요 😊

**길이의 책임은 Spring이 아니라 JDBC API에 있어요**

6단계에서 쓰셨을 짧은 형태부터 비교해볼게요.

```java
jdbcTemplate.update(
    "INSERT INTO reservation (name, date, time) VALUES (?, ?, ?)",
    name, date, time
);
```

이렇게 쓰면 깔끔하지만, **DB가 만든 id를 돌려받을 방법이 없어요.** id를 받으려면 JDBC 표준이 제공하는 `Statement.getGeneratedKeys()`를 사용해야 하는데, 이걸 쓰려면 PreparedStatement를 만들 때 "생성된 키를 돌려받을게요"라고 미리 알려줘야 합니다.

```java
connection.prepareStatement(sql, new String[]{"id"});
```

여기서 핵심은 이 플래그를 **`prepareStatement()`를 호출하는 그 순간에 넘겨야 한다**는 점이에요. PreparedStatement가 만들어진 뒤에는 못 바꿔요. JDBC API가 그렇게 생겼습니다.

그래서 짧은 `jdbcTemplate.update(String, Object...)` 오버로드는 generated key를 받지 못해요. 내부에서 PreparedStatement를 만들 때 그 플래그를 안 넘기거든요. 만약 generated key를 받고 싶다면, **PreparedStatement 생성 자체를 사용자가 제어**해야 합니다. 그게 진우님 코드의 `PreparedStatementCreator` 람다 부분이에요.

```java
jdbcTemplate.update(connection -> {
    PreparedStatement ps = connection.prepareStatement(
            "INSERT INTO reservation (name, date, time) VALUES (?, ?, ?)",
            new String[]{"id"});   // ← 이 플래그가 핵심
    ps.setString(1, reservation.getName());
    ps.setString(2, reservation.getDate());
    ps.setString(3, reservation.getTime());
    return ps;
}, keyHolder);
```

즉 람다 안의 6~7줄은 Spring이 강요한 게 아니라, **JDBC 표준이 "generated key를 받으려면 이만큼은 직접 다뤄야 한다"고 정한 결과**예요. Spring은 그 위에 KeyHolder라는 받는 그릇만 얹어준 거고요.

**JdbcTemplate이 KeyHolder를 채우는 흐름**

내부 동작을 간단히 따라가보면 이해가 더 단단해질 것 같아요.

1. 진우님이 넘긴 람다를 호출해서 PreparedStatement를 받아옴 (이때 generated key 플래그가 켜져 있음)
2. `ps.executeUpdate()`로 INSERT 실행
3. `ps.getGeneratedKeys()`로 DB가 만든 키 ResultSet을 회수
4. ResultSet의 각 row를 Map 형태로 KeyHolder에 채워넣음
5. 진우님이 `keyHolder.getKey()`로 그 값을 꺼냄

여기서 짚어두면 좋은 건, INSERT와 키 회수가 **같은 statement 한 번의 실행 안에서** 일어난다는 점이에요. 그래서 "INSERT하고 따로 SELECT MAX(id) 하면 안 되나?" 같은 우회는 안전하지 않아요. 동시에 다른 요청이 INSERT를 끼워넣으면 엉뚱한 id를 받게 될 수 있거든요. KeyHolder 패턴이 굳이 이렇게 생긴 이유가 여기 있습니다.

**더 짧은 대안도 있어요**

7단계 힌트에 살짝 언급된 `SimpleJdbcInsert`도 같은 일을 해주는 도구예요.

```java
SimpleJdbcInsert insert = new SimpleJdbcInsert(jdbcTemplate)
    .withTableName("reservation")
    .usingGeneratedKeyColumns("id");

Map<String, Object> params = Map.of(
    "name", reservation.getName(),
    "date", reservation.getDate(),
    "time", reservation.getTime()
);
Long id = insert.executeAndReturnKey(params).longValue();
```

내부 동작은 똑같이 `getGeneratedKeys()`를 쓰지만, PreparedStatement 만드는 보일러플레이트를 라이브러리가 감춰줘요. 강의 힌트가 "SimpleJdbcInsert와 같은 새로운 객체를 활용할 수도 있습니다"라고 두 선택지를 다 열어둔 게 이런 이유 같아요.

다만 지금 단계에서 굳이 바꾸시라는 건 아니에요. KeyHolder 방식이 길어 보이긴 해도, **"generated key를 받으려면 JDBC 레벨에서 이만큼이 필요하다"는 걸 한 번 직접 다뤄본 경험**이 SimpleJdbcInsert를 쓸 때도 "아, 라이브러리가 이걸 대신 해주는 거구나"라는 이해로 이어지거든요. 지금 코드 그대로 두시고, 강의 진행하면서 한 번쯤 SimpleJdbcInsert 버전으로도 짜보시면 비교가 더 선명해질 것 같아요 😊
