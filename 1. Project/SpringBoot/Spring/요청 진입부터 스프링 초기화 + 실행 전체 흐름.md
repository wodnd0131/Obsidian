## 전제 — 두 가지 타임라인을 구분한다

```
① 서버 시작 타임라인  →  스프링이 준비되는 과정
② 요청 처리 타임라인  →  준비된 스프링이 요청을 처리하는 과정
```

이 둘을 섞으면 흐름이 안 보인다. 순서대로 간다.

---

## 타임라인 ① — 서버 시작 (스프링 초기화)

### SpringApplication.run() 진입

```
java -jar app.jar
        │
        ▼
SpringApplication.run()
        │
        ├─ 1. 환경 구성
        │       Environment 생성
        │       application.properties 로딩
        │       OS 환경변수, JVM 시스템 프로퍼티 등록
        │       PropertySource 우선순위 목록 구성
        │
        ├─ 2. ApplicationContext 생성 (인스턴스만)
        │       AnnotationConfigServletWebServerApplicationContext
        │       → 아직 빈 없음, 껍데기만
        │
        └─ 3. context.refresh() 호출  ← 여기서 모든 것이 시작
```

---

### context.refresh() 내부 — 단계별

```
context.refresh()
        │
        ▼
① BeanDefinition 수집
        │
        │  ClassPathScanningCandidateComponentProvider
        │  → @ComponentScan 경로 탐색
        │  → @Component, @Service, @Repository, @Controller 탐색
        │  → @Configuration 클래스의 @Bean 메서드 탐색
        │
        │  결과: BeanDefinitionRegistry에 메타데이터 저장
        │  {
        │    "orderService"  → BeanDefinition(class=OrderService, scope=singleton)
        │    "orderRepository" → BeanDefinition(class=OrderRepository, scope=singleton)
        │    "dataSource"    → BeanDefinition(class=HikariDataSource, ...)
        │    ...수백 개
        │  }
        │  (아직 인스턴스 없음)
        │
        ▼
② BeanFactoryPostProcessor 실행
        │
        │  ConfigurationClassPostProcessor
        │  → @Configuration 클래스 파싱
        │  → @Import, @PropertySource 처리
        │  → 자동 구성 클래스 로딩 (AutoConfiguration)
        │       @ConditionalOnClass 등 조건 평가
        │       → 조건 충족한 자동 구성만 BeanDefinition 추가 등록
        │
        │  PropertySourcesPlaceholderConfigurer
        │  → BeanDefinition 안의 "${db.url}" 리터럴
        │  → Environment에서 실제 값 조회
        │  → BeanDefinition 값 치환
        │
        ▼
③ BeanPostProcessor 등록
        │
        │  아직 실행 아님 — 등록만
        │  AutowiredAnnotationBeanPostProcessor   (@Autowired 처리 담당)
        │  CommonAnnotationBeanPostProcessor      (@PostConstruct 처리 담당)
        │  AbstractAutoProxyCreator               (AOP 프록시 생성 담당)
        │
        ▼
④ onRefresh()  ← Spring Boot 확장 지점
        │
        │  ServletWebServerApplicationContext.onRefresh()
        │  → 내장 Tomcat 인스턴스 생성
        │  → 포트, 커넥터 설정
        │  → DispatcherServlet을 Tomcat에 등록
        │  → Tomcat 시작 (소켓 열림 — 요청 수신 가능)
        │    그러나 스프링 빈이 아직 없으므로
        │    요청이 와도 처리 불가 상태
        │
        ▼
⑤ finishBeanFactoryInitialization()  ← 싱글톤 빈 전부 생성
        │
        │  BeanDefinition 목록 순회
        │  → 싱글톤 + NonLazy인 것들 선별
        │  → 의존관계 위상 정렬
        │       DataSource → OrderRepository → OrderService 순서 결정
        │
        │  각 빈마다:
        │  ┌─────────────────────────────────────┐
        │  │ a. 생성자 호출 (인스턴스 생성)        │
        │  │ b. 의존성 주입                        │
        │  │ c. BeanPostProcessor Before 실행     │
        │  │      → @PostConstruct 실행            │
        │  │ d. BeanPostProcessor After 실행      │
        │  │      → AOP 프록시 생성 (해당 시)      │
        │  │ e. singletonObjects Map에 등록        │
        │  └─────────────────────────────────────┘
        │
        │  결과: singletonObjects Map
        │  {
        │    "orderService"    → OrderService$$CGLIB 프록시
        │    "orderRepository" → SimpleJpaRepository 인스턴스
        │    "dataSource"      → HikariDataSource 인스턴스
        │    ...
        │  }
        │
        ▼
⑥ finishRefresh()
        │
        │  LifecycleProcessor.start()
        │  → SmartLifecycle 구현 빈들 시작
        │  ContextRefreshedEvent 발행
        │  → ApplicationListener들에게 통보
        │
        ▼
refresh() 완료
→ 서버 준비 완료 로그 출력
→ 요청 처리 가능 상태
```

## 타임라인 ② — 요청 처리

### Tomcat → DispatcherServlet 진입

```
클라이언트: POST /orders HTTP/1.1

        │
        ▼
Tomcat NIO Connector
        │  소켓에서 바이트 수신
        │  HTTP 파싱 → HttpServletRequest 객체 생성
        │  스레드 풀에서 스레드 할당
        │
        ▼
ApplicationFilterChain (Tomcat이 관리)
        │
        ├─ SecurityContextPersistenceFilter
        │       ThreadLocal에 SecurityContext 로딩
        │
        ├─ UsernamePasswordAuthenticationFilter (또는 JWT 필터)
        │       인증 처리
        │
        ├─ CharacterEncodingFilter
        │       request.setCharacterEncoding("UTF-8")
        │
        └─ 마지막 필터 → chain.doFilter() → DispatcherServlet 도달
        │
        ▼
DispatcherServlet.service()
        │  (HttpServlet.service → doPost → processRequest → doService → doDispatch)
        │
        ▼
DispatcherServlet.doDispatch()
```

---

### doDispatch() 내부 전체

```
doDispatch(request, response)
        │
        ▼
① HandlerMapping 탐색
        │
        │  등록된 HandlerMapping 목록 순회
        │  RequestMappingHandlerMapping
        │  → refresh() 시점에 구축된 Map 조회
        │  → "POST /orders" → OrderController.createOrder(OrderRequest)
        │
        │  결과: HandlerExecutionChain
        │  {
        │    handler: OrderController.createOrder 메서드
        │    interceptors: [LoggingInterceptor, AuthInterceptor]
        │  }
        │
        ▼
② HandlerAdapter 선택
        │
        │  HandlerAdapter 목록 순회
        │  → RequestMappingHandlerAdapter.supports(HandlerMethod) = true
        │  → 선택
        │
        ▼
③ Interceptor preHandle() — 순방향
        │
        │  LoggingInterceptor.preHandle()  → true
        │  AuthInterceptor.preHandle()     → true (false면 즉시 중단)
        │
        ▼
④ HandlerAdapter.handle()
        │
        │  RequestMappingHandlerAdapter 내부:
        │  │
        │  ├─ ArgumentResolver 실행
        │  │       @RequestBody OrderRequest
        │  │       → RequestResponseBodyMethodProcessor
        │  │       → HttpInputMessage에서 JSON 읽기
        │  │       → MappingJackson2HttpMessageConverter
        │  │       → JSON → OrderRequest 역직렬화
        │  │
        │  ├─ 컨트롤러 메서드 호출
        │  │       orderController.createOrder(orderRequest)
        │  │       → orderController는 실제로 CGLIB 프록시
        │  │       → 프록시가 @Transactional 감지
        │  │       → TransactionInterceptor 실행
        │  │            PlatformTransactionManager.getTransaction()
        │  │            → HikariCP에서 커넥션 획득
        │  │            → TransactionSynchronizationManager에 바인딩
        │  │            → 원본 createOrder() 실행
        │  │                 orderRepository.save(order)
        │  │                 → 같은 커넥션 재사용 (ThreadLocal)
        │  │            → commit() 또는 rollback()
        │  │
        │  └─ ReturnValueHandler 실행
        │          반환: OrderResponse 객체
        │          → @ResponseBody 감지
        │          → mavContainer.setRequestHandled(true)
        │               (ViewResolver 스킵 플래그)
        │          → MappingJackson2HttpMessageConverter
        │          → OrderResponse → JSON 직렬화
        │          → response.getWriter().write(json)
        │
        ▼
⑤ Interceptor postHandle() — 역방향
        │  AuthInterceptor.postHandle()
        │  LoggingInterceptor.postHandle()
        │
        ▼
⑥ ViewResolver
        │  mavContainer.isRequestHandled() = true
        │  → 스킵 (@ResponseBody 경로)
        │
        ▼
⑦ Interceptor afterCompletion() — 역방향, 예외 무관
        │  AuthInterceptor.afterCompletion()
        │  LoggingInterceptor.afterCompletion()
        │
        ▼
Tomcat → HTTP 응답 전송
        │  response 객체를 바이트로 직렬화
        │  소켓으로 전송
        │  스레드 풀에 스레드 반환
```

## 두 타임라인 합친 전체 지도

```
jar 실행
    │
    ▼
SpringApplication.run()
    │
    ├─ Environment 구성
    │
    ├─ ApplicationContext 생성 (껍데기)
    │
    └─ context.refresh()
            │
            ├─ ① BeanDefinition 수집
            │       메타데이터만, 인스턴스 없음
            │
            ├─ ② BeanFactoryPostProcessor
            │       자동 구성 조건 평가
            │       ${} 플레이스홀더 치환
            │
            ├─ ③ BeanPostProcessor 등록
            │       (실행 아님)
            │
            ├─ ④ onRefresh()
            │       내장 Tomcat 시작
            │       DispatcherServlet 등록
            │       소켓 오픈 (요청 수신 시작)
            │
            └─ ⑤ finishBeanFactoryInitialization()
                    모든 싱글톤 빈 생성
                    AOP 프록시 생성
                    singletonObjects Map 완성
                            │
                            ▼
                    서버 준비 완료
                            │
                            │   (이후 요청마다 반복)
                            ▼
                    HTTP 요청 수신
                            │
                    Tomcat 스레드 할당
                            │
                    Filter Chain
                            │
                    DispatcherServlet.doDispatch()
                            │
                    HandlerMapping 조회 (Map 탐색)
                            │
                    HandlerAdapter 선택
                            │
                    Interceptor preHandle
                            │
                    ArgumentResolver → 역직렬화
                            │
                    Controller 실행
                    (프록시 → AOP → 원본)
                            │
                    ReturnValueHandler → 직렬화
                            │
                    Interceptor postHandle / afterCompletion
                            │
                    HTTP 응답 전송
                    스레드 반환
```

## 자주 헷갈리는 포인트 두 가지

**1. Tomcat이 먼저 뜨고 빈이 나중에 만들어진다**

```
onRefresh()에서 Tomcat 소켓이 열린다.
finishBeanFactoryInitialization()에서 빈이 만들어진다.

즉, 빈 생성 중에 요청이 올 수 있다.
스프링은 refresh() 완료 전까지
DispatcherServlet이 요청을 받아도 처리를 보류하는 방식으로 이를 방어한다.
```

**2. DispatcherServlet이 조회하는 Map은 요청마다 새로 탐색하지 않는다**

```
refresh() → HandlerMapping 초기화
         → @RequestMapping 정보로 Map 구축 (1회)

요청마다:
  Map.get("POST /orders") → HandlerMethod (O(1) 조회)

URL 패턴 매칭이 필요한 경우 (PathVariable 등):
  AntPathMatcher로 패턴 매칭
  결과를 캐시에 저장 → 두 번째 요청부터 캐시 히트
```
