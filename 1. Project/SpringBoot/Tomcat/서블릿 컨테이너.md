#SpringBoot/Tomcat 

서블릿 컨테이너 = **서블릿의 생명주기 전체를 관리하는 런타임 환경**

DispatcherServlet을 담는 통이 맞긴 한데, "담는다"는 것이 단순히 포함한다는 게 아니라 
**생성, 초기화, 실행, 소멸까지 전부 책임진다**는 뜻이다.

```
서블릿 컨테이너 (Tomcat)
│
│  관리 대상:
├── DispatcherServlet (생성 → init() → service() → destroy())
├── Filter Chain 관리
├── Thread Pool 관리
├── TCP 소켓 관리
└── ServletContext 관리 (웹 애플리케이션 전역 저장소)
```

DispatcherServlet은 서블릿 컨테이너가 관리하는 **여러 서블릿 중 하나**다. Spring Boot에서는 DispatcherServlet 하나만 등록하니까 "담는 통"처럼 보이지만, 원래 서블릿 스펙은 하나의 컨테이너 안에 서블릿 여러 개를 등록하는 구조다.

---

### 컨테이너가 서블릿에게 하는 일

```
Tomcat 시작
    │
    ▼
DispatcherServlet 인스턴스 생성 (Tomcat이 함)
    │
    ▼
servlet.init() 호출 (Tomcat이 함)
    │  → Spring ApplicationContext 초기화
    │  → HandlerMapping 스캔 등
    ▼
요청 올 때마다 servlet.service() 호출 (Tomcat이 함)
    │  → doDispatch() 실행
    ▼
Tomcat 종료 시 servlet.destroy() 호출 (Tomcat이 함)
```

DispatcherServlet은 이 호출들을 **받는 쪽**이다. 주도권은 Tomcat(서블릿 컨테이너)에 있다. 이것이 IoC(제어의 역전)가 프레임워크 레벨에서도 적용된 구조다.

---

### 정리

```
서블릿 컨테이너 (Tomcat)
└── DispatcherServlet을 포함한 서블릿들의 생명주기 관리자

"담는 통" → 맞는 방향이지만
"생성/실행/소멸까지 제어하는 런타임 환경" 이 더 정확하다
```

Spring Boot에서는 DispatcherServlet 하나만 쓰니까 1:1처럼 보이지만, 개념적으로 서블릿 컨테이너는 DispatcherServlet보다 상위 레이어다.

---
### 누가 무엇을 만드는가

```
Tomcat이 만드는 것:
- DispatcherServlet 인스턴스
- HttpServletRequest / HttpServletResponse
- Thread Pool

Spring이 만드는 것:
- ApplicationContext (Bean 컨테이너)
- HandlerMapping, HandlerAdapter 등 MVC 컴포넌트들
- @Controller, @Service 등 Bean들
```

Tomcat이 DispatcherServlet을 **new** 해서 생성하고, `init()`을 호출한다. 그 `init()` 안에서 Spring ApplicationContext가 초기화된다.

```
Tomcat.start()
    │
    │  서블릿 스펙: "등록된 서블릿을 생성하고 init() 호출해라"
    ▼
new DispatcherServlet()        ← Tomcat이 생성
    │
    ▼
DispatcherServlet.init()       ← Tomcat이 호출
    │
    ▼
ApplicationContext 초기화      ← 여기서부터 Spring이 주도
    │  @Controller, @Service Bean 생성
    │  HandlerMapping 스캔
    │  MessageConverter 등록
    │  ...
    ▼
초기화 완료 → 요청 받을 준비
```

### "따로 만들어져서 연결된다" — 맞다

더 정확히 표현하면:

```
Tomcat:  "서블릿 하나 실행해줄게. 근데 그 안에서 뭐하는지는 몰라"
Spring:  "내가 init() 안에서 필요한 거 다 세팅할게"

연결 지점 = DispatcherServlet.init()
```

DispatcherServlet이 **Tomcat 세계와 Spring 세계의 연결 다리**다.

```
[Tomcat 세계]                    [Spring 세계]
소켓, 스레드, HTTP 파싱            Bean, DI, MVC 컴포넌트
        │                               │
        └──── DispatcherServlet ─────────┘
               (연결 다리)
        
Tomcat은 DispatcherServlet을 그냥 "서블릿 하나"로 본다
Spring은 DispatcherServlet을 "ApplicationContext의 진입점"으로 본다
```

### Spring Boot가 이걸 어떻게 자동화하는가

원래 Spring MVC만 쓸 때는 `web.xml`에 직접 등록했다.

```xml
<!-- 개발자가 직접 Tomcat에 DispatcherServlet 등록 -->
<servlet>
    <servlet-class>
        org.springframework.web.servlet.DispatcherServlet
    </servlet-class>
    <load-on-startup>1</load-on-startup>
</servlet>
```

Spring Boot는 이걸 `DispatcherServletAutoConfiguration`이 자동으로 처리한다.

```java
// Spring Boot 내부 — 개발자가 안 써도 자동으로 등록됨
@Bean
public DispatcherServlet dispatcherServlet() {
    return new DispatcherServlet();
}

@Bean
public ServletRegistrationBean<DispatcherServlet> dispatcherServletRegistration() {
    // Tomcat에 "이 서블릿을 / 경로에 등록해줘" 라고 알려줌
    return new ServletRegistrationBean<>(dispatcherServlet(), "/");
}
```

### 한 줄 정리

```
Tomcat이 DispatcherServlet을 생성/호출
    → DispatcherServlet.init() 안에서 Spring이 부팅
    → 이후 요청마다 Tomcat이 service() 호출
    → Spring MVC가 처리

"Tomcat이 껍데기를 실행하면, 그 안에서 Spring이 살아난다"
```
