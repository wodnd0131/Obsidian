## B+Tree가 B-Tree에서 뭘 바꿨는가

B-Tree의 문제부터.

```sql
SELECT * FROM users WHERE age BETWEEN 30 AND 70;
```

```
B-Tree에서 30~70 범위 찾기:

        [30 | 70]
       /    |    \
  [10|20] [40|60] [80|90]
  
30 찾음 → 루트로 다시 올라감 → 40 찾음 → 루트로 다시 올라감 → ...
범위 내 값마다 트리를 처음부터 다시 탐색
```

각 노드가 독립적으로 흩어져 있어서 범위 내 다음 값을 찾으려면 매번 루트부터 내려와야 한다.

---

## B+Tree가 바꾼 것 두 가지

**1. 내부 노드는 키만 갖는다 (길 안내 전용)**

```
B-Tree 내부 노드:                B+Tree 내부 노드:
┌─────────────────────┐          ┌─────────────────────┐
│ key: 30                                   │               │ key: 30             │
│ data: (실제 row)                     │          │ (데이터 없음)        │
│ 왼쪽 포인터                              │          │ 왼쪽 포인터         │
│ 오른쪽 포인터                             │          │ 오른쪽 포인터       │
└─────────────────────┘          └─────────────────────┘
```

내부 노드에서 데이터를 뺐으니 같은 16KB 페이지에 키를 더 많이 욱여넣을 수 있다. 트리 높이가 더 낮아진다.

**2. 리프 노드끼리 연결 리스트로 연결된다**

```
        [30 | 70]                ← 내부 노드, 키만 있음
       /    |    \
[10|20] → [30|40|60] → [70|80|90]  ← 리프 노드, 실제 데이터 + 연결
```

리프 노드가 옆 노드의 주소를 알고 있다.

---

## 범위 쿼리가 어떻게 달라지는가

```sql
SELECT * FROM users WHERE age BETWEEN 30 AND 70;
```

```
1. 트리 탐색으로 30이 있는 리프 노드 찾기 (딱 한 번)

        [30 | 70]
       /
  [10|20] → [30|40|60] → [70|80|90]
                ↑
               여기 도착

2. 연결 리스트 따라서 순차 스캔
  [30|40|60] → [70|80|90]
                    ↑
                   70 넘으면 중단
```

트리를 한 번만 탄다. 나머지는 리프 레벨에서 순차 읽기로 끝난다.

---

## 정리

```
B-Tree:
  장점: 운 좋으면 내부 노드에서 바로 데이터 반환
  단점: 범위 쿼리마다 트리 재탐색

B+Tree:
  장점: 범위 쿼리가 리프 순차 스캔으로 처리됨
       내부 노드에 키가 더 많이 들어가서 트리 높이 낮아짐
  단점: 단건 조회도 항상 리프까지 내려가야 함
```

DB는 `BETWEEN`, `ORDER BY`, `LIKE 'abc%'` 같은 범위 쿼리가 필수적이라 B+Tree를 택한 거다.
