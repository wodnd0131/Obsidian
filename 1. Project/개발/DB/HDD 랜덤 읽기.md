HDD: 물리적으로 헤드가 이동해야 함 → 느림 
SSD: 물리적 이동은 없지만 흩어진 주소 찾는 오버헤드 → 그나마 빠름 

공통: 연속된 주소보다 흩어진 주소가 느리다

디스크 조회는 물리적으로 느리다.

## HDD 물리 구조

```
         스핀들(축)
            │
    ┌───────┴───────┐
    │   플래터      │  ← 자성 코팅된 원판, 여기에 데이터 저장
    │  (원판)       │     여러 장이 겹쳐있음
    └───────────────┘
            │
         헤드 ──→ 플래터 표면을 읽는 바늘
         (암)      실제로는 살짝 떠있음 (접촉 안 함)
```

데이터는 플래터 위에 **동심원(트랙)** 형태로 저장된다.

```
플래터 위에서 보면:

    ○○○○○○○○○  ← 트랙 0 (바깥쪽)
   ○○○○○○○○○○○
  ○○○○○○○○○○○○○ ← 트랙 N (안쪽)
```

## 랜덤 읽기가 느린 이유

데이터 A(트랙 0)를 읽다가 데이터 B(트랙 50)를 읽으려면:

```
1단계: Seek (탐색)
   헤드가 트랙 0 → 트랙 50으로 물리적으로 이동
   
   [헤드]━━━━━━━━━━━━━━━━━━━━━━━━→ 이동
   트랙0                        트랙50
   
   시간: 수 ms (기계적 이동)

2단계: Rotational Latency (회전 대기)
   트랙 50에 도착했는데
   원하는 섹터가 지금 헤드 반대편에 있으면
   
        헤드
         ↓
   ──────●──────
   │           │
   │     ×     │  ← 원하는 섹터 (지금 여기)
   │           │
   ──────────── 
   
   원하는 섹터가 헤드 아래로 올 때까지 디스크가 돌아야 함
   최악의 경우 한 바퀴 다 돌아야 함
   7200RPM 기준 최대 8ms 대기

3단계: 그제서야 읽기 시작
```

## 순차 읽기가 빠른 이유

```
트랙 0에서 데이터를 순서대로 읽으면:

   헤드 고정
    ↓
────●────  디스크가 돌면서
   (1)(2)(3)(4)(5)  순서대로 헤드 아래를 지나감
```

헤드가 **안 움직여도 된다.** 디스크가 돌면서 데이터가 알아서 헤드 아래로 지나온다.

## 숫자로 보면

```
HDD 순차 읽기:   100~200 MB/s
HDD 랜덤 읽기:   0.1~2 MB/s    ← 100배 차이
```

B-Tree가 디스크 I/O를 3~4번으로 줄이는 게 중요한 이유가 여기 있다. 랜덤 읽기 자체가 느리니까, 횟수를 최소화하는 게 핵심이다.
