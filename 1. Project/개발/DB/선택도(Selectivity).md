### 정의

```
선택도 = 조건에 매칭되는 행 수 / 전체 행 수
       = 이 조건이 데이터를 얼마나 걸러내는가 (0~1)
```

```
선택도 0.01 = 1%만 통과  → 조건이 매우 선별적 (인덱스 효과 큼)
선택도 0.99 = 99% 통과  → 조건이 거의 안 걸러냄 (인덱스 효과 없음)
```

### 선택도가 카디널리티 추정으로 이어지는 흐름

```
1. 인덱스/히스토그램에서 선택도 계산
   status = 'BANNED' → 선택도 = 0.01

2. 카디널리티 추정
   전체 100만 건 × 0.01 = 10,000건 반환 예상

3. 비용 계산
   랜덤 I/O 10,000번 → 인덱스가 풀스캔보다 저렴

4. 실행 계획 선택
   인덱스 range scan 선택
```

### 선택도가 어디서 계산되는가

```
[인덱스 있는 컬럼]
인덱스의 n_diff (고유값 수) 통계에서 계산

  n_diff = 3 (ACTIVE, INACTIVE, BANNED)
  균등 분포 가정 → 선택도 = 1/3 = 0.33
  → 히스토그램 없으면 모든 값이 33%로 추정됨

[히스토그램 있는 컬럼]
버킷에서 직접 읽음

  BANNED 버킷 = 0.01
  → 선택도 = 0.01 (정확)
```

### 선택도 오차가 실행 계획을 망치는 케이스

```
실제 분포:
  ACTIVE: 95%,  INACTIVE: 4%,  BANNED: 1%

히스토그램 없을 때 추정:
  모든 값 → 선택도 1/3 = 0.33

WHERE status = 'BANNED' 쿼리:
  추정: 33만 건 반환 → 랜덤 I/O 33만 번 → 풀스캔 선택
  실제:  1만 건 반환 → 랜덤 I/O 1만 번  → 인덱스가 훨씬 빨랐는데

WHERE status = 'ACTIVE' 쿼리:
  추정: 33만 건 반환 → 풀스캔 선택
  실제: 95만 건 반환 → 풀스캔이 맞는 선택이긴 했지만 이유가 틀림
```

`BANNED` 케이스가 문제다. 옵티마이저가 인덱스를 써야 하는 쿼리에 풀스캔을 선택했다. **히스토그램이 이 오차를 교정한다.**
