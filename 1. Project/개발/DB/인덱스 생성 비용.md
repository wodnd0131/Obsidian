## 인덱스 비용 — 세 가지 차원

### ① 쓰기 비용 (가장 중요)

테이블에 데이터가 바뀔 때마다 인덱스 B+Tree도 같이 갱신된다.

```sql
INSERT INTO orders (user_id, status, created_at) VALUES (1, 'ACTIVE', NOW());
```

```
인덱스가 3개 있다면:

1. 클러스터드 인덱스(PK) B+Tree 갱신
2. idx_user_id B+Tree에 새 노드 삽입
3. idx_status B+Tree에 새 노드 삽입
4. idx_created_at B+Tree에 새 노드 삽입

INSERT 1번 = B+Tree 갱신 4번
```

노드 삽입 시 **페이지 분할(Page Split)** 이 발생하면 비용이 더 커진다.

```
페이지가 꽉 찬 상태에서 중간에 삽입:

Before:
[10 | 20 | 30 | 40 | 50]  ← 페이지 꽉 참

25 삽입 시도
→ 페이지를 둘로 쪼갬
[10 | 20 | 25]   [30 | 40 | 50]
→ 새 페이지 할당 + 부모 노드 갱신 + 경우에 따라 연쇄 분할
```

AUTO_INCREMENT PK가 UUID보다 유리한 이유가 여기 있다.

```
AUTO_INCREMENT: 항상 오른쪽 끝에 추가 → 페이지 분할 거의 없음
UUID:           랜덤 위치에 삽입 → 페이지 분할 빈번
```

**UPDATE도 인덱스 갱신을 유발한다.**

```sql
UPDATE orders SET status = 'COMPLETED' WHERE id = 1;
```

```
idx_status B+Tree:
  'ACTIVE' 위치에서 해당 노드 삭제
  'COMPLETED' 위치에 새 노드 삽입
→ 삭제 + 삽입 = 2번 B+Tree 조작
```

인덱스 없는 컬럼 UPDATE는 인덱스 갱신 없음. **인덱스 걸린 컬럼을 자주 UPDATE하는 케이스가 특히 비싸다.**

### ② 저장 공간

```
인덱스 = 별도 B+Tree = 별도 디스크 공간

orders 테이블 100GB
  + idx_user_id          10GB
  + idx_status           8GB
  + idx_created_at       12GB
  + idx_covering(created_at, user_id, status)  20GB  ← 커버링은 더 큼
─────────────────────────────────────────────────────
전체                     150GB
```

커버링 인덱스는 리프에 컬럼을 더 저장하므로 일반 인덱스보다 공간을 더 쓴다.

---

### ③ 생성 시점 비용 (운영 중 ALTER)

```sql
-- 운영 중 인덱스 추가
ALTER TABLE orders ADD INDEX idx_new (status, created_at);
```

MySQL 8.0은 Online DDL을 지원해서 테이블 락을 최소화하지만, 내부적으로 하는 일은 무겁다.

```
1. 테이블 전체 풀스캔
2. 스캔하면서 (status, created_at) 값 수집
3. 정렬
4. B+Tree 빌드
5. 작업 중 발생한 DML 변경분 적용
```

**테이블이 클수록 시간이 선형으로 늘어난다.** 수억 건 테이블에서 수십 분~수 시간 소요될 수 있다.

## 그래서 인덱스 설계 판단 기준

```
인덱스를 추가할 때 얻는 것:
  SELECT 속도 향상

인덱스를 추가할 때 잃는 것:
  INSERT / UPDATE / DELETE 속도 저하
  디스크 공간
  버퍼 풀(메모리) 공간 경쟁
```

실무에서 판단하는 기준:

```
읽기 : 쓰기 비율이 높을수록  → 인덱스 추가 유리
쓰기가 압도적으로 많다면     → 인덱스 최소화

자주 바뀌는 컬럼             → 인덱스 비용 큼, 신중하게
거의 안 바뀌는 컬럼          → 인덱스 비용 작음

카디널리티 높은 컬럼(user_id, id)  → 인덱스 효과 큼
카디널리티 낮은 컬럼(status, 성별) → 단독 인덱스 효과 작음
                                     복합 인덱스 앞 컬럼으로는 유효
```

커버링 인덱스는 이 트레이드오프가 더 명확하다.

```
커버링 인덱스:
  얻는 것: 2단계 접근 제거, 랜덤 I/O 0
  잃는 것: 일반 인덱스보다 리프가 크므로 쓰기 비용 더 큼
           공간 더 많이 사용

→ 조회가 압도적으로 많고, 조회 패턴이 고정된 경우에 적합
→ 쓰기가 잦고 조회 패턴이 다양하면 오히려 독
```

## 한 줄 정리

```
인덱스는 SELECT를 위해 INSERT/UPDATE/DELETE를 희생하는 트레이드오프다.
커버링 인덱스는 그 트레이드오프가 더 크다.
읽기가 압도적인 케이스에서만 정당화된다.
```
