## 인덱스 비용 — 세 가지 차원

### ① 쓰기 비용 (가장 중요)

테이블에 데이터가 바뀔 때마다 인덱스 B+Tree도 같이 갱신된다.

```sql
INSERT INTO orders (user_id, status, created_at) VALUES (1, 'ACTIVE', NOW());
```

```
인덱스가 3개 있다면:

1. 클러스터드 인덱스(PK) B+Tree 갱신
2. idx_user_id B+Tree에 새 노드 삽입
3. idx_status B+Tree에 새 노드 삽입
4. idx_created_at B+Tree에 새 노드 삽입

INSERT 1번 = B+Tree 갱신 4번
```

노드 삽입 시 **페이지 분할(Page Split)** 이 발생하면 비용이 더 커진다.

```
페이지가 꽉 찬 상태에서 중간에 삽입:

Before:
[10 | 20 | 30 | 40 | 50]  ← 페이지 꽉 참

25 삽입 시도
→ 페이지를 둘로 쪼갬
[10 | 20 | 25]   [30 | 40 | 50]
→ 새 페이지 할당 + 부모 노드 갱신 + 경우에 따라 연쇄 분할
```

AUTO_INCREMENT PK가 UUID보다 유리한 이유가 여기 있다.

```
AUTO_INCREMENT: 항상 오른쪽 끝에 추가 → 페이지 분할 거의 없음
UUID:           랜덤 위치에 삽입 → 페이지 분할 빈번
```

**UPDATE도 인덱스 갱신을 유발한다.**

```sql
UPDATE orders SET status = 'COMPLETED' WHERE id = 1;
```

```
idx_status B+Tree:
  'ACTIVE' 위치에서 해당 노드 삭제
  'COMPLETED' 위치에 새 노드 삽입
→ 삭제 + 삽입 = 2번 B+Tree 조작
```

인덱스 없는 컬럼 UPDATE는 인덱스 갱신 없음. **인덱스 걸린 컬럼을 자주 UPDATE하는 케이스가 특히 비싸다.**

---

### ② 저장 공간

```
인덱스 = 별도 B+Tree = 별도 디스크 공간

orders 테이블 100GB
  + idx_user_id          10GB
  + idx_status           8GB
  + idx_created_at       12GB
  + idx_covering(created_at, user_id, status)  20GB  ← 커버링은 더 큼
─────────────────────────────────────────────────────
전체                     150GB
```

커버링 인덱스는 리프에 컬럼을 더 저장하므로 일반 인덱스보다 공간을 더 쓴다.

---

### ③ 생성 시점 비용 (운영 중 ALTER)

```sql
-- 운영 중 인덱스 추가
ALTER TABLE orders ADD INDEX idx_new (status, created_at);
```

MySQL 8.0은 Online DDL을 지원해서 테이블 락을 최소화하지만, 내부적으로 하는 일은 무겁다.

```
1. 테이블 전체 풀스캔
2. 스캔하면서 (status, created_at) 값 수집
3. 정렬
4. B+Tree 빌드
5. 작업 중 발생한 DML 변경분 적용
```

**테이블이 클수록 시간이 선형으로 늘어난다.** 수억 건 테이블에서 수십 분~수 시간 소요될 수 있다.

## 그래서 인덱스 설계 판단 기준

```
인덱스를 추가할 때 얻는 것:
  SELECT 속도 향상

인덱스를 추가할 때 잃는 것:
  INSERT / UPDATE / DELETE 속도 저하
  디스크 공간
  버퍼 풀(메모리) 공간 경쟁
```

실무에서 판단하는 기준:

```
읽기 : 쓰기 비율이 높을수록  → 인덱스 추가 유리
쓰기가 압도적으로 많다면     → 인덱스 최소화

자주 바뀌는 컬럼             → 인덱스 비용 큼, 신중하게
거의 안 바뀌는 컬럼          → 인덱스 비용 작음

카디널리티 높은 컬럼(user_id, id)  → 인덱스 효과 큼
카디널리티 낮은 컬럼(status, 성별) → 단독 인덱스 효과 작음
                                     복합 인덱스 앞 컬럼으로는 유효
```

커버링 인덱스는 이 트레이드오프가 더 명확하다.

```
커버링 인덱스:
  얻는 것: 2단계 접근 제거, 랜덤 I/O 0
  잃는 것: 일반 인덱스보다 리프가 크므로 쓰기 비용 더 큼
           공간 더 많이 사용

→ 조회가 압도적으로 많고, 조회 패턴이 고정된 경우에 적합
→ 쓰기가 잦고 조회 패턴이 다양하면 오히려 독
```

## 한 줄 정리

```
인덱스는 SELECT를 위해 INSERT/UPDATE/DELETE를 희생하는 트레이드오프다.
커버링 인덱스는 그 트레이드오프가 더 크다.
읽기가 압도적인 케이스에서만 정당화된다.
```

---

## 측정 도구들

### EXPLAIN ANALYZE — 조회 비용 확인

```sql
-- 인덱스 있을 때
EXPLAIN ANALYZE
SELECT * FROM orders WHERE created_at >= '2024-01-01';

-- 출력:
-- -> Index range scan on orders using idx_created_at
--    (cost=15234 rows=98000)
--    (actual time=0.8..234.5 rows=102341 loops=1)

-- 인덱스 없을 때 (강제로 무시)
EXPLAIN ANALYZE
SELECT * FROM orders IGNORE INDEX (idx_created_at)
WHERE created_at >= '2024-01-01';

-- 출력:
-- -> Table scan on orders
--    (cost=102341 rows=1000000)
--    (actual time=0.5..1823.4 rows=1000000 loops=1)
```

`actual time` 차이가 실제 조회 비용 차이다.

---

### performance_schema — 쓰기 비용 확인

```sql
-- 쓰기 전 측정 시작
UPDATE performance_schema.setup_instruments
SET enabled = 'YES', timed = 'YES'
WHERE name LIKE 'statement/%';

-- INSERT 실행
INSERT INTO orders (user_id, status, created_at, amount)
VALUES (1, 'ACTIVE', NOW(), 10000);

-- 실행 통계 확인
SELECT
    digest_text,
    count_star,
    avg_timer_wait / 1000000000 AS avg_ms,  -- 평균 실행시간(ms)
    sum_rows_affected
FROM performance_schema.events_statements_summary_by_digest
WHERE digest_text LIKE '%INSERT%orders%';
```

인덱스 추가 전후로 같은 INSERT를 측정해서 `avg_ms` 차이를 본다.

---

### information_schema — 인덱스 공간 비용 확인

```sql
SELECT
    table_name,
    index_name,
    ROUND(stat_value * 16 / 1024, 2) AS size_mb  -- 페이지 수 → MB 변환
FROM mysql.innodb_index_stats
WHERE table_name = 'orders'
  AND stat_name = 'size';
```

```
결과 예시:
table_name | index_name      | size_mb
orders     | PRIMARY         | 2048.00   ← 클러스터드 인덱스
orders     | idx_user_id     |  512.00
orders     | idx_status      |  128.00
orders     | idx_created_at  |  768.00
orders     | idx_covering    | 1024.00   ← 커버링은 더 큼
```

## 실제 판단 흐름

```
1. EXPLAIN ANALYZE로 인덱스 있을 때 / 없을 때 actual time 비교
         ↓
2. 해당 쿼리가 얼마나 자주 실행되는지 확인
   (performance_schema.events_statements_summary_by_digest의 count_star)
         ↓
3. 인덱스로 인한 쓰기 비용 증가 측정
   (인덱스 추가 전후 INSERT/UPDATE avg_ms 비교)
         ↓
4. 읽기 이득 × 읽기 빈도  vs  쓰기 손해 × 쓰기 빈도
```

```
읽기 이득:   2000ms → 200ms = 1800ms 단축
읽기 빈도:   초당 100회
읽기 총 이득: 1800ms × 100 = 180,000ms/초

쓰기 손해:   5ms → 8ms = 3ms 증가
쓰기 빈도:   초당 500회
쓰기 총 손해: 3ms × 500 = 1,500ms/초

→ 180,000 >> 1,500 → 인덱스 추가 정당화
```

## 한 줄 정리

```
도구: EXPLAIN ANALYZE(조회 비용) + performance_schema(쓰기 비용) + innodb_index_stats(공간)
판단: (읽기 단축 × 읽기 빈도) vs (쓰기 증가 × 쓰기 빈도)
숫자로 비교해서 이득이 크면 인덱스 추가, 손해가 크면 제거
```
