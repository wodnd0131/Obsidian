# B-Tree와 MySQL 인덱싱

## 1. Problem First

### 인덱스 없는 테이블에서 실제로 무슨 일이 일어나는가

```sql
-- users 테이블: 1,000만 건
SELECT * FROM users WHERE email = 'target@example.com';
```

인덱스가 없다면 MySQL은 **Full Table Scan**을 수행한다.

```
실행 계획:
id | select_type | type | rows     | Extra
1  | SIMPLE      | ALL  | 10000000 | Using where
```

`type = ALL` → 1,000만 행을 **디스크에서 순차적으로 전부 읽는다**.

#### 왜 이게 근본적으로 문제인가

디스크 I/O는 CPU 연산과 비교해서 비용 차이가 극단적이다.

| 연산        | 대략적 지연                 |     |
| --------- | ---------------------- | --- |
| L1 캐시 참조  | ~1 ns                  |     |
| 메모리 참조    | ~100 ns                |     |
| SSD 랜덤 읽기 | ~100,000 ns (0.1 ms)   |     |
| HDD 랜덤 읽기 | ~10,000,000 ns (10 ms) |     |

> _"디스크는 메모리보다 100,000배 느리다"_ — Martin Kleppmann, _Designing Data-Intensive Applications_, Chapter 3

1,000만 행의 데이터가 여러 디스크 페이지에 분산되어 있을 때, 각 페이지를 순차적으로 불러오는 비용이 누적된다. **이것이 인덱스가 없으면 안 되는 근본 이유다** — "느리다"가 아니라, 데이터 규모가 커질수록 비용이 **선형(O(N))으로 증가**하기 때문에 시스템이 한계에 도달하는 건 시간문제다.

## 2. Mechanics

### B-Tree의 정의

B-Tree는 1972년 Rudolf Bayer와 Edward McCreight가 고안한 **자기 균형(self-balancing) 탐색 트리**다.

> "B-trees are the standard index implementation in almost all relational databases" — Martin Kleppmann, _DDIA_, Chapter 3
> 
> (번역) "B-tree는 거의 모든 관계형 데이터베이스에서 표준 인덱스 구현체다"

B-Tree의 핵심 불변식(invariant):

```
1. 모든 리프 노드는 같은 깊이에 있다 (완전 균형)
2. 차수(order) m인 B-Tree에서 각 노드는 최대 m-1개의 키를 가진다
3. 루트를 제외한 모든 노드는 최소 ⌈m/2⌉ - 1개의 키를 가진다
4. 내부 노드의 키 k_i에 대해: k_i보다 작은 값은 왼쪽 자식, 큰 값은 오른쪽 자식
```

### MySQL이 실제로 쓰는 것: B+Tree

MySQL InnoDB는 B-Tree의 변형인 **B+Tree**를 사용한다. 차이가 중요하다.

```
B-Tree:
    [30 | 70]
   /    |    \
 [10|20] [40|60] [80|90]
  ↑각 내부 노드에도 실제 데이터(row pointer)가 있음

B+Tree:
    [30 | 70]          ← 내부 노드: 키만 있음 (라우팅 전용)
   /    |    \
[10|20]-[30|40|60]-[70|80|90]  ← 리프 노드: 실제 데이터 + 연결 리스트
```

**B+Tree가 B-Tree보다 DB에 유리한 이유:**

|특성|B-Tree|B+Tree|
|---|---|---|
|내부 노드|키 + 데이터 포인터|키만 (라우팅 전용)|
|리프 노드|독립|**연결 리스트로 연결**|
|범위 쿼리|트리 재탐색 필요|리프 순차 스캔 가능|
|내부 노드당 키 수|적음 (데이터도 있어서)|더 많음 → **트리 높이 낮아짐**|

### 탐색의 기본 문제

숫자 1~100 중에 내가 생각한 숫자를 맞춰봐.

**방법 1: 순서대로 물어보기**

```
1이야? 2야? 3이야? ... 99야? 100이야?
최악의 경우 100번 질문
```

**방법 2: 절반씩 자르기**

```
50보다 커? → 커
75보다 커? → 작아
62보다 커? → 작아
56보다 커? → 맞아!

4번만에 찾음
```

방법 2가 **이진 탐색**이다. 매번 절반을 버리니까 빠르다.

---

#### 이걸 트리 구조로 표현하면

```
                50
               /  \
             25    75
            / \   / \
          12  37 62  87
```

50 기준으로 작으면 왼쪽, 크면 오른쪽. 찾을 때 위에서 아래로 내려오면서 절반씩 버린다.

이게 **이진 탐색 트리(BST)** 다.

---

#### BST의 문제

노드 하나에 값이 하나뿐이다.

DB는 데이터를 디스크에서 **페이지(16KB) 단위**로 읽어온다. 16KB를 통째로 읽는데 값 하나만 쓰면 낭비다.

```
16KB 읽음 → 값 하나 확인 → 또 16KB 읽음 → 값 하나 확인 → ...
```

---

#### 그래서 나온 게 B-Tree

노드 하나에 값을 **여러 개** 넣는다.

```
BST (노드당 값 1개):
      [50]
      /  \
   [25]  [75]      ← 노드 3개 = 디스크 3번 읽음

B-Tree (노드당 값 여러 개):
   [25 | 50 | 75]  ← 노드 1개 = 디스크 1번 읽음
   /   |   |   \
 ...  ...  ...  ...
```

어차피 16KB 읽을 거, 거기에 키를 최대한 많이 욱여넣자. → 트리 높이가 낮아짐 → 디스크 읽는 횟수가 줄어듦

이게 B-Tree의 전부다.

---

그 다음에 B+Tree는 B-Tree에서 "범위 검색을 더 빠르게" 하려고 개선한 버전이고, MySQL이 실제로 쓰는 게 그거다. 여기까지 되면 본문 다시 읽으면 다 연결될 거야.

### InnoDB의 페이지 구조와 B+Tree의 관계

InnoDB는 데이터를 **페이지(page)** 단위로 읽고 쓴다. 기본값은 **16KB**.

```
디스크 레이아웃:

[ Page 1: Root Node ]
  키: [30 | 70]
  자식 포인터: [Page2, Page3, Page4]

[ Page 2: Internal Node ]        [ Page 3: Internal Node ]
  키: [10 | 20]                    키: [40 | 60]
  자식: [Page5, Page6, Page7]      자식: [Page8, Page9, Page10]

[ Page 5: Leaf ] → [ Page 6: Leaf ] → [ Page 7: Leaf ] → ...
  실제 row 데이터 포함, 이중 연결 리스트로 연결
```

**핵심: 트리 높이 = 필요한 디스크 I/O 횟수**

페이지 하나(16KB)에 키 하나당 약 100바이트라 하면, 약 160개 키가 들어간다. (실제로는 포인터 등을 포함하므로 수백 개 수준)

> OS는 디스크로부터 한 블록, 4KB씩 읽어온다.
> 이때, InnoDB는 4블록, 16KB가 기준이다. (한 페이지를 뜻함)
> 적으면, 원하는 키를 조회못할 수 있고,
> 크면, 조회 대상 이상의 정보도 메모리에 올려야하니 적당선을 찾은것

```
높이 1 (루트만): ~수백 개 레코드 인덱싱
높이 2: 수백 × 수백 = 수만 개
높이 3: 수백 × 수백 × 수백 = 수천만 개  ← 1,000만 건 테이블이 여기
높이 4: 수십억 개
```

**1,000만 건 테이블을 찾는 데 디스크 I/O가 3~4번이면 충분하다.** Full Scan의 수백만 번 I/O와 비교하면 차이는 자명하다.

### 탐색 알고리즘

```
target = 'target@example.com'

1. 루트 페이지 로드 (1회 I/O)
   키: ['m...', 'r...']
   target < 'm...' → 왼쪽 자식

2. 내부 노드 페이지 로드 (1회 I/O)
   키: ['d...', 'h...']
   't...' > 'h...' → 오른쪽 자식

3. 리프 노드 페이지 로드 (1회 I/O)
   키 순차 탐색 → 'target@example.com' 발견
   row data 반환

총 I/O: 3회
시간 복잡도: O(log_m N)  (m = 노드의 최대 자식 수)
```

## 3. 공식 근거

> **MySQL 8.0 Reference Manual — 8.3.1 How MySQL Uses Indexes:** "Most MySQL indexes (PRIMARY KEY, UNIQUE, INDEX, and FULLTEXT) are stored in B-trees. Exceptions: Indexes on spatial data types use R-trees; MEMORY tables also support hash indexes; InnoDB uses inverted lists for FULLTEXT indexes."
> 
> (번역) "대부분의 MySQL 인덱스는 B-tree에 저장된다. 예외: 공간 데이터 타입은 R-tree, MEMORY 테이블은 해시 인덱스, InnoDB FULLTEXT는 역인덱스를 사용한다."
> 
> 출처: https://dev.mysql.com/doc/refman/8.0/en/mysql-indexes.html

> **InnoDB 스토리지 엔진 공식 문서:** "InnoDB uses B-tree indexes... The default page size for InnoDB is 16KB."
> 
> (번역) "InnoDB는 B-tree 인덱스를 사용하며, 기본 페이지 크기는 16KB다."
> 
> 출처: https://dev.mysql.com/doc/refman/8.0/en/innodb-index-types.html

B+Tree가 B-Tree의 변형임에도 MySQL 공식 문서가 "B-tree"로 통칭하는 것은 업계 관행이다. 내부가 B+Tree 구조임은 InnoDB 소스코드(storage/innobase/btr/) 및 아래 문서로 확인된다:

> **MySQL Engineering Blog — "InnoDB: A journey to the core"** 출처: https://dev.mysql.com/blog-archive/innodb-a-journey-to-the-core/

## 4. 이 설계를 이렇게 한 이유

### B+Tree가 해결한 것

- **범위 쿼리 효율**: 리프 노드 연결 리스트 덕분에 `BETWEEN`, `ORDER BY`, `LIKE 'abc%'` 가 리프 레벨에서 순차 스캔으로 처리된다
- **높이 최소화**: 내부 노드에 데이터가 없으므로 한 노드에 더 많은 키를 담을 수 있고, 트리 높이가 낮아져 I/O 횟수가 줄어든다

### B+Tree가 잃은 것 (트레이드오프)

#### 쓰기 비용: 페이지 분할(page split)

삽입 시 리프 노드가 꽉 차면 **분할**이 발생한다.

```
Before (16KB 페이지 꽉 참):
  Leaf: [10, 20, 30, 40, 50]  (만석)

INSERT 35 →

After (페이지 분할):
  Leaf A: [10, 20, 30]
  Leaf B: [35, 40, 50]
  상위 내부 노드: 새 키 35 추가 → 상위도 분할될 수 있음 (cascade)
```

랜덤 INSERT가 많은 워크로드에서 이 cascade split이 반복되면 **write amplification**이 심각해진다.

> _"B-tree writes must overwrite at least one full page on disk. It is sometimes necessary to write several different pages when a page is split."_ — Martin Kleppmann, _DDIA_, Chapter 3
> 
> (번역) "B-tree 쓰기는 최소 하나의 전체 페이지를 덮어써야 한다. 페이지가 분할될 때는 여러 다른 페이지를 써야 할 수도 있다."

이 때문에 InnoDB는 **`AUTO_INCREMENT` PK를 권장**한다. 순차 삽입이면 항상 마지막 리프에만 쓰므로 split이 최소화된다.

UUID를 PK로 쓰면 무작위 위치에 삽입이 발생하여 페이지 분할이 빈번해지고 **인덱스 단편화(fragmentation)**가 심해진다.

#### 인덱스가 오히려 느려지는 상황

```sql
-- 전체 데이터의 30% 이상을 읽어야 하는 경우
SELECT * FROM orders WHERE status = 'COMPLETED';  
-- status 선택도(selectivity)가 낮으면 인덱스 무시하고 Full Scan이 더 빠름
```

MySQL 옵티마이저가 인덱스를 타지 않는 이유: 인덱스를 통한 랜덤 I/O보다 Full Scan의 순차 I/O가 더 효율적인 임계점이 존재하기 때문이다.

## 5. 이어지는 개념

```
B+Tree (현재)
    │
    ├─ 1. 클러스터드 인덱스 vs 세컨더리 인덱스          ← 선행 의존
    │      InnoDB의 PK가 B+Tree 자체인 이유,
    │      세컨더리 인덱스가 PK를 참조하는 구조.
    │      이것 없이는 아래 개념들의 비용 계산이 불가능하다
    │
    ├─ 2. 선택도(Selectivity)와 카디널리티               ← 실무 임팩트
    │      옵티마이저가 인덱스를 탈지 말지 결정하는 기준.
    │      "인덱스 걸었는데 왜 안 타요?" 의 근본 답
    │
    ├─ 3. 복합 인덱스와 Prefix 규칙                      ← 실무 임팩트
    │      (a, b, c) 인덱스에서 (b, c) 조건만으로 왜 안 타는지.
    │      실무에서 가장 많이 만나는 인덱스 설계 실수
    │
    └─ 4. EXPLAIN 실행 계획 읽기                         ← 실무 임팩트
           type / key / rows / Extra 컬럼의 의미.
           위 모든 개념이 여기서 눈에 보이게 된다
```

**1 → 2 → 3 → 4 순서로 진행할 것을 권장한다.** 1을 모르면 세컨더리 인덱스 탐색 비용을 설명할 수 없고, 2를 모르면 3의 복합 인덱스 설계 판단 기준이 없고, 3을 모르면 4의 실행 계획이 해독이 안 된다.
