### 한 줄 정의

> 옵티마이저에게 **"이 컬럼의 값이 어떻게 분포되어 있는지"** 를 알려주는 통계 데이터

```
히스토그램 = 옵티마이저에게 "이 조건 통과하면 많이 남아 vs 적게 남아"를 알려주는 도구
```
### 왜 필요한가

옵티마이저는 기본적으로 **균등 분포를 가정**한다.

```sql
-- users 테이블, status 컬럼, 고유값 3개
-- ACTIVE: 950,000건 (95%)
-- INACTIVE: 40,000건  (4%)
-- BANNED: 10,000건   (1%)
```

히스토그램 없을 때 옵티마이저의 추정:

```
고유값 3개 → 선택도 = 1/3 = 33%
WHERE status = 'ACTIVE'  → 33% 반환 예상
WHERE status = 'BANNED'  → 33% 반환 예상  ← 실제는 1%인데
```

**두 조건이 같은 비용으로 계산된다. 실행 계획이 동일하게 나온다.**

실제로는 `BANNED` 조건은 인덱스가 극도로 효율적이고, `ACTIVE` 조건은 인덱스 쓰는 게 오히려 손해일 수 있는데, 옵티마이저는 이걸 모른다.

### 히스토그램이 있을 때

```
ACTIVE  → 95% 반환 예상 → 풀스캔이 나을 수도 있음 → 풀스캔 선택
BANNED  →  1% 반환 예상 → 인덱스 매우 효율적      → 인덱스 선택
```

**같은 컬럼, 같은 인덱스인데 값에 따라 다른 실행 계획이 나온다.**

## 히스토그램의 내부 구조

MySQL 8.0은 두 가지 타입을 제공한다.

### Singleton 히스토그램

고유값 수가 적을 때. 각 값의 정확한 비율을 저장.

```
status 컬럼:
{
  "ACTIVE":   0.95,
  "INACTIVE": 0.04,
  "BANNED":   0.01
}
```

### Equi-Height 히스토그램

고유값이 많을 때. 전체 범위를 N개 버킷으로 나눠 저장.

```
age 컬럼 (버킷 4개):
버킷 1: 0~25세   → 전체의 15%
버킷 2: 26~35세  → 전체의 40%  ← 이 구간에 데이터 몰림
버킷 3: 36~50세  → 전체의 35%
버킷 4: 51~100세 → 전체의 10%
```

옵티마이저는 `WHERE age BETWEEN 26 AND 35` 가 40%를 반환한다는 걸 알게 된다.

## 실제 명령어

```sql
-- 히스토그램 생성 (ANALYZE TABLE의 확장)
ANALYZE TABLE users
UPDATE HISTOGRAM ON status WITH 100 BUCKETS;

-- 여러 컬럼 동시에
ANALYZE TABLE orders
UPDATE HISTOGRAM ON status, payment_method, region WITH 100 BUCKETS;

-- 히스토그램 삭제
ANALYZE TABLE users
DROP HISTOGRAM ON status;

-- 생성된 히스토그램 확인
SELECT
    column_name,
    histogram->>'$."histogram-type"'  AS type,
    JSON_LENGTH(histogram->>'$."buckets"') AS bucket_count
FROM information_schema.COLUMN_STATISTICS
WHERE table_name = 'users';
```

## 인덱스와 히스토그램의 차이

이게 가장 중요한 구분이다.

|       | 인덱스                       | 히스토그램                        |
| ----- | ------------------------- | ---------------------------- |
| 목적    | **실제 데이터 탐색** 속도 향상       | **옵티마이저 추정** 정확도 향상          |
| 저장 위치 | 테이블과 함께 (B+Tree)          | `information_schema` (메타데이터) |
| 쓰기 비용 | INSERT/UPDATE/DELETE마다 갱신 | `ANALYZE TABLE` 실행 시에만 갱신    |
| 적용 대상 | 인덱스 있는 컬럼만                | **인덱스 없는 컬럼에도 적용 가능**        |
| 크기    | 데이터에 비례 (클 수 있음)          | 고정 크기 (버킷 수만큼)               |

**핵심:** 인덱스는 "빠르게 찾는 도구", 히스토그램은 "옵티마이저에게 분포를 알려주는 지도"다.

둘은 역할이 다르기 때문에 **인덱스 없는 컬럼에 히스토그램만 걸어도 의미가 있다.**

```sql
-- 이 경우 히스토그램이 유용한 전형적 패턴
-- status에 인덱스는 없지만, JOIN 순서 결정에 영향을 줌
SELECT o.id FROM orders o
JOIN users u ON o.user_id = u.id
WHERE u.status = 'BANNED';  -- 1%만 해당

-- 히스토그램 없음: users가 큰 테이블이라 드라이버로 안 씀
-- 히스토그램 있음: BANNED가 1%임을 알고 users를 드라이버로 선택
--                 → 필터 후 작은 집합으로 orders를 조회
```

## 비용 모델 — 어느 수준까지 알면 되나

비용 상수 조정(`mysql.server_cost` 직접 수정)은 DBA 영역이 맞다. 하지만 **비용 모델이 어떻게 작동하는지** 는 알아야 EXPLAIN을 제대로 읽을 수 있다.

개발자가 알아야 할 수준:

```
✅ 알아야 함
- EXPLAIN의 rows × filtered 가 비용 계산의 입력값이라는 것
- rows 추정이 틀리면 실행 계획이 틀린다는 것
- 그 틀림을 고치는 도구가 ANALYZE TABLE과 히스토그램이라는 것

❌ 몰라도 됨 (DBA 영역)
- io_block_read_cost 수치 조정
- SSD 환경에서 engine_cost 재보정
- 비용 상수 변경 후 전체 쿼리 영향 분석
```

## 실무에서 히스토그램을 쓰는 타이밍

```
1. EXPLAIN ANALYZE 실행
2. rows 추정치와 actual rows 차이가 크게 남  ← 여기서 히스토그램 필요 신호
3. 해당 컬럼 값 분포가 편향되어 있는지 확인
4. ANALYZE TABLE ... UPDATE HISTOGRAM 적용
5. 실행 계획 재확인
```

직접 경험한 소프트딜리트 케이스에 대입하면:

```sql
-- deleted_at 컬럼: NULL이 99%, 날짜값이 1%
-- 히스토그램 없으면 옵티마이저가 50/50으로 추정
-- → deleted_at IS NULL 조건의 선택도를 0.99로 알려줄 수 있음

ANALYZE TABLE orders
UPDATE HISTOGRAM ON deleted_at WITH 100 BUCKETS;

-- 이후 옵티마이저는 deleted_at IS NULL이 거의 전체임을 알고
-- 해당 조건만으로는 인덱스 효과 없음을 정확히 판단
```
