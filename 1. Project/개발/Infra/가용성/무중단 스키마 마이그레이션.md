# 무중단 스키마 마이그레이션(Zero-Downtime Schema Migration)

## 문제 상황과 목표 재정의

### 현재 상황

- **운영 중인 MySQL 서버**에 실제 고객 데이터가 저장된 테이블 존재
- **무중단 서비스**가 필요한 환경 (고객이 24시간 서비스 이용 중)
- **스프링부트 애플리케이션**이 해당 테이블을 실시간으로 읽기/쓰기 중

### 목표

1. **컬럼명 변경**: 기존 컬럼의 이름을 새로운 이름으로 변경
2. **새 컬럼 추가**: 추가적인 데이터를 저장할 컬럼 생성
3. **테이블 분리**: 불린값(is_active 등)을 기준으로 하나의 테이블을 두 개로 분리

### 절대 원칙 (데이터와 고객 경험 보호)

- **데이터 손실 금지**: 단 한 건의 데이터도 잃어서는 안 됨
- **서비스 중단 금지**: 고객이 느끼는 어떤 장애도 발생시키면 안 됨
- **롤백 가능**: 언제든 이전 상태로 되돌릴 수 있어야 함

---

## 안전한 데이터 변경 방법론

### 1. 기본 철학: "추가 → 전환 → 제거"

운영 환경에서는 **기존 것을 바로 바꾸지 않는다**. 새로운 것을 추가하고, 점진적으로 전환한 후, 마지막에 기존 것을 제거한다.

### 2. 단계별 안전 전략

### Phase 1: 준비 단계 (데이터 안전성 확보)

```sql
-- 1. 전체 백업 (필수)
mysqldump -u root -p database_name > backup_$(date +%Y%m%d_%H%M%S).sql

-- 2. 바이너리 로그 활성화 확인
SHOW VARIABLES LIKE 'log_bin';

-- 3. 복제 서버 확인 (있다면)
SHOW SLAVE STATUS\\G

```

### Phase 2: 컬럼 추가 (무중단)

```sql
-- 새 컬럼들을 NULL 허용으로 추가 (기존 서비스 영향 없음)
ALTER TABLE users
ADD COLUMN new_column_name VARCHAR(255) NULL COMMENT '기존 old_column_name 대체',
ADD COLUMN new_additional_column VARCHAR(255) NULL COMMENT '새로 추가되는 컬럼';

-- 기존 데이터를 새 컬럼으로 안전하게 복사
UPDATE users
SET new_column_name = old_column_name
WHERE new_column_name IS NULL;

-- 검증
SELECT COUNT(*) as total_rows,
       COUNT(old_column_name) as old_not_null,
       COUNT(new_column_name) as new_not_null
FROM users;

```

### Phase 3: 애플리케이션 이중 쓰기 적용

**핵심**: 기존 방식을 유지하면서 새 방식을 추가

- 쓰기: 기존 컬럼 + 새 컬럼 모두 업데이트
- 읽기: 아직 기존 컬럼 사용 (안전)

### Phase 4: 읽기 전환 (신중하게)

```sql
-- 데이터 일관성 검증 (100% 일치해야 함)
SELECT COUNT(*) as inconsistent_rows
FROM users
WHERE old_column_name != new_column_name
   OR (old_column_name IS NULL AND new_column_name IS NOT NULL)
   OR (old_column_name IS NOT NULL AND new_column_name IS NULL);

```

**결과가 0이 아니면 절대 다음 단계로 진행하지 않는다.**

### Phase 5: 기존 컬럼 제거 (충분한 검증 후)

```sql
-- 최소 1-2주 운영 후 문제없음을 확인한 뒤
ALTER TABLE users DROP COLUMN old_column_name;

```

### Phase 6: 테이블 분리 준비

```sql
-- 분리될 테이블들 생성
CREATE TABLE active_users (
    id BIGINT PRIMARY KEY,
    new_column_name VARCHAR(255),
    new_additional_column VARCHAR(255),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);

CREATE TABLE inactive_users LIKE active_users;

-- 인덱스 복사 (성능 보장)
SHOW CREATE TABLE users;  -- 기존 인덱스 확인
-- 필요한 인덱스들을 새 테이블에 생성

-- 데이터 복사 (부하가 적은 시간에)
INSERT INTO active_users (id, new_column_name, new_additional_column, created_at, updated_at)
SELECT id, new_column_name, new_additional_column, created_at, updated_at
FROM users WHERE is_active = true;

INSERT INTO inactive_users (id, new_column_name, new_additional_column, created_at, updated_at)
SELECT id, new_column_name, new_additional_column, created_at, updated_at
FROM users WHERE is_active = false;

-- 검증
SELECT 'Original' as source, COUNT(*) as active_count, COUNT(*) as inactive_count
FROM users
WHERE is_active = true
UNION ALL
SELECT 'Split' as source,
       (SELECT COUNT(*) FROM active_users) as active_count,
       (SELECT COUNT(*) FROM inactive_users) as inactive_count;

```

### 3. 핵심 안전 장치

### 모니터링 쿼리 (매일 실행)

```sql
-- 데이터 정합성 체크
SELECT
    DATE(NOW()) as check_date,
    (SELECT COUNT(*) FROM users WHERE is_active = true) as original_active,
    (SELECT COUNT(*) FROM active_users) as split_active,
    (SELECT COUNT(*) FROM users WHERE is_active = false) as original_inactive,
    (SELECT COUNT(*) FROM inactive_users) as split_inactive,
    CASE
        WHEN (SELECT COUNT(*) FROM users WHERE is_active = true) = (SELECT COUNT(*) FROM active_users)
         AND (SELECT COUNT(*) FROM users WHERE is_active = false) = (SELECT COUNT(*) FROM inactive_users)
        THEN 'OK'
        ELSE 'ERROR'
    END as status;

```

### 롤백 계획

```sql
-- Phase별 롤백 시나리오

-- Phase 2 롤백: 새 컬럼 제거
ALTER TABLE users
DROP COLUMN new_column_name,
DROP COLUMN new_additional_column;

-- Phase 6 롤백: 분리 테이블 제거
DROP TABLE active_users;
DROP TABLE inactive_users;

```

### 4. 절대 하지 말아야 할 것들

1. **직접 컬럼명 변경**: `ALTER TABLE CHANGE COLUMN` 직접 사용
2. **운영시간 중 대량 데이터 이동**: 서비스 성능에 영향
3. **검증 없는 다음 단계 진행**: 데이터 일관성 미확인 상태로 진행
4. **백업 없는 작업**: 복구 불가능한 상황 발생 위험
5. **한 번에 모든 변경**: 문제 발생 시 원인 파악 곤란

### 5. 성공의 기준

- **데이터 무결성**: 변경 전후 데이터가 100% 일치
- **성능 영향 없음**: 응답시간, 처리량에 변화 없음
- **고객 경험 동일**: 사용자가 변경을 전혀 느끼지 못함
- **롤백 가능**: 각 단계에서 언제든 이전 상태로 복구 가능
