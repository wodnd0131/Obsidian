### 1. Application Layer (Spring)

먼저 애플리케이션 레이어입니다.

- 로그가 발생하는 지점은 **Filter / AOP / ExceptionHandler**입니다.
- 여기서 `BusinessLog.of(...)`, `RequestLog.of(...)`, `ResponseLog.of(...)`, `ErrorLog.of(...)` 같은 DTO 객체를 생성합니다.
- 공통 필드(`request_id`, `client_ip`, `request_uri` 등)는 **MDC**에 추가해두고,
- 로그는 아래처럼 출력합니다:

```java
log.info(objectMapper.writeValueAsString(RequestLog.of(request)));
log.warn(objectMapper.writeValueAsString(BusinessLog.of(...)));
log.error(objectMapper.writeValueAsString(ErrorLog.of(...)));
```

즉, DTO를 `ObjectMapper.writeValueAsString()`으로 직접 직렬화해서 JSON 문자열로 만든 뒤 로그로 남기는 구조였습니다.

---

### 2. Logback Layer

다음은 로깅을 실제로 찍어주는 Logback 레이어입니다.

- **Encoder**: `LogstashEncoder`
- **Appender**: `RollingFileAppender` → `/var/logs/app.log` 파일로 저장

---

### 3. Alloy Layer

로그 수집은 **Alloy**가 담당합니다.

Alloy는 `/var/logs/app.log` 파일을 읽어서 Loki로 전달합니다.

---

### 4. Loki + Grafana Layer

- Loki: 수집된 로그를 저장
- Grafana: 로그 시각화

### 문제점 정리

- **MDC와 message 안 데이터 중복**
    
    request_id, client_ip, request_uri 등이 MDC와 message에 동시에 들어감 → 로그가 더럽게 보임.
    
- **timestamp 불일치**
    
    message 안 timestamp는 애플리케이션(LocalDateTime + Asia/Seoul),
    
    @timestamp는 logback 기본값(UTC) → 서로 달라 혼란 발생.
    
- **message 필드 안 JSON이 문자열로 들어감**
    
    DTO를 JSON으로 만든 의미가 사라짐. 그냥 긴 문자열일 뿐.
    
- **로그 길이 문제**
    
    한 줄이 너무 길어 시각화할 때 가독성 최악.
    
- **검색 불가**
    
    Grafana는 message 내부 JSON을 파싱 못함 → `logType`, `method`, `status`로 필터링 불가능.
    
- **LogstashEncoder 한계**
    
    `%message` 자체만 JSON으로 포함시키는 구조라 커스터마이징 어려움.
    
    결국 Application Layer에서 힘들게 만든 JSON이 `"message": "{...}"`로 들어가버림.
    

---

### 개선 목표

따라서 다음과 같은 목표를 세웠습니다:

1. 로그에서 중복되는 MDC 정보 제거
2. timestamp의 timezone 통일
3. 각 로그의 핵심 정보를 요약(summary)로 제공
4. DTO 필드를 message 문자열이 아닌 JSON key-value로 노출

이 목표를 바탕으로 개선 작업을 하나씩 진행했습니다.

(이후 문서에서는 RequestLog를 대표로 설명합니다. ResponseLog, ErrorLog, BusinessLog에도 동일하게 적용되었습니다.)

# 1. 로그에서 중복되는 MDC 정보는 제거하자

기존 `RequestLog`를 찍는 방식은 다음과 같았습니다.

---

### LogFilter (MDC 넣는 레이어)

`OncePerRequestFilter`를 상속받아 각 요청/응답마다 TraceId를 만들고, `request_uri`, `client_ip` 같은 공통 필드를 MDC에 넣어주고 있었습니다.

```java
@Slf4j
@Component
@RequiredArgsConstructor
public class LogFilter extends OncePerRequestFilter {
    ...
    @Override
    protected void doFilterInternal(
            HttpServletRequest request,
            HttpServletResponse response,
            FilterChain filterChain
    ) throws ServletException, IOException {

        ContentCachingRequestWrapper cacheRequest = new ContentCachingRequestWrapper(request);
        ContentCachingResponseWrapper cacheResponse = new ContentCachingResponseWrapper(response);

        createTraceIds();
        MDC.put("request_uri", request.getRequestURI());
        MDC.put("client_ip", cacheRequest.getRemoteAddr());

        try {
            filterChain.doFilter(cacheRequest, cacheResponse);
            log.info(objectMapper.writeValueAsString(RequestLog.of(cacheRequest)));
        } finally {
            log.info(objectMapper.writeValueAsString(ResponseLog.of(cacheResponse)));
            cacheResponse.copyBodyToResponse();
            MDC.clear();
        }
    }
    ...
}

```

---

### RequestLog 구현

DTO인 `RequestLog`는 아래와 같이 MDC에 들어가는 값들을 필드로 또 들고 있었습니다.

```java
public record RequestLog(
        LogType logType,
        String timestamp,
        String requestId,
        String requestUri,
        String clientIp,
        String method,
        String body
) {
    private static final String NOW_TIME = LocalDateTime.now(ZoneId.of("Asia/Seoul"))
            .format(DateTimeFormatter.ISO_LOCAL_DATE_TIME);

    public static RequestLog of(ContentCachingRequestWrapper request) {
        return new RequestLog(
                LogType.REQUEST,
                NOW_TIME,
                MDC.get("request_id"),
                request.getRequestURI(),
                request.getRemoteAddr(),
                request.getMethod(),
                new String(request.getContentAsByteArray(), StandardCharsets.UTF_8)
        );
    }
}

```

---

### 로그 출력 방식

실제로 로그를 찍을 때는 다음과 같이 `ObjectMapper`로 DTO를 JSON 문자열로 변환해 출력했습니다.

```java
log.info(objectMapper.writeValueAsString(RequestLog.of(cacheRequest)));
```


### LogstashEncoder 특징

📌 **LogstashEncoder**

- 클래스: `net.logstash.logback.encoder.LogstashEncoder`
- 역할: 로그 이벤트 전체를 JSON 한 줄로 직렬화
- 특징:
    - 기본적으로 `@timestamp`, `@version`, `message`, `logger_name`, `thread_name`, `level`, `level_value` 자동 포함
    - MDC는 별도 설정 없이도 자동 포함됨 → `MDC.put("request_id", "...")` 하면 `"request_id": "..."` 필드가 자동으로 찍힘
    - 하지만 **커스텀에 제약**이 많아, 필드 추가/삭제/구조 변경은 어렵습니다.

---

### 문제: MDC 중복

여기서 중복 문제가 생깁니다.

- DTO에도 `requestId`, `requestUri`, `clientIp` 같은 값이 들어있고,
- MDC에도 동일한 값이 들어가서,
- 결국 로그 한 줄에 같은 값이 두 번 찍히게 되었습니다.

즉, 아래처럼 DTO → JSON → `message` 안에 들어간 값과, MDC에서 뽑힌 값이 동시에 존재하는 꼴이었죠.

---

### 해결: MDC는 Logback에게 맡기자

사실 생각해보면 MDC는 “ **각 스레드별 컨텍스트 정보를 저장하고 로깅할 때 해당 정보들을 로그 메시지에 함께 출력할 수 있기 위해**” 쓰는 도구인데, DTO에도 같은 필드를 또 들고 있을 필요는 없습니다.

그래서 **RequestLog에서 MDC 관련 필드를 제거**하고, MDC를 JSON으로 찍는 책임을 Logback 설정(`encoder`)에 위임했습니다.

---

### 변경된 RequestLog

```java
public record RequestLog(
        LogType logType,
        String method,
        String body
) {
    public static RequestLog of(ContentCachingRequestWrapper request) {
        return new RequestLog(
                LogType.REQUEST,
                request.getMethod(),
                new String(request.getContentAsByteArray(), StandardCharsets.UTF_8)
        );
    }
}

```

---

### 효과

- 중복 필드 제거 → 로그 가독성 개선
- DTO는 핵심 데이터(로그 타입, 메서드, 바디)만 책임 → 더 깔끔해짐
- MDC 출력은 Logback 설정에서 `<mdc/>`로 통일

---

### 추가: timestamp 제거

이 과정에서 `LocalDateTime.now(ZoneId.of("Asia/Seoul"))`로 생성하던 `timestamp` 필드도 함께 제거했습니다.

이유는 다음 목표였던 **“timestamp의 timezone을 통일하자”** 작업과 이어지기 때문입니다.

---

# 2. timestamp의 timezone을 통일하자

`LogstashEncoder`는 자동으로 `@timestamp` 필드를 찍어줍니다.

문제는 이 값이 **UTC(세계 표준시)** 기준이라는 점입니다.

한편, DTO 내부에서는 `LocalDateTime.now(ZoneId.of("Asia/Seoul"))`로 한국 시간(KST)을 별도로 찍고 있었죠.

결국 로그 한 줄 안에 **UTC와 KST 두 가지 시간이 섞여서** 나타나는 상황이 벌어졌습니다.

---

### 왜 문제가 되나?

- 운영 로그를 분석하다 보면 **"이 에러가 실제로 몇 시에 발생한 건가?"** 같은 질문이 바로 생깁니다.
- UTC와 KST가 혼재되어 있으면 단순 비교나 타임라인 분석이 꼬여버립니다.
- 특히 Grafana 같은 시각화 툴에서는 타임존을 맞추는 게 매우 중요합니다.

따라서 **timestamp를 한 곳에서, 일관된 방식으로 관리**하는 게 필요했습니다.

---

### LogstashEncoder의 한계

`LogstashEncoder`는 편리하게 `@timestamp`를 자동으로 찍어주지만,

**타임존을 직접 설정하는 옵션이 없습니다.**

즉, 무조건 UTC로 찍히고, 커스터마이징 여지가 거의 없는 구조죠.

이 때문에 DTO에 timestamp 필드를 추가해 한국 시간을 강제로 찍어주고 있었는데,

이건 중복 데이터일 뿐만 아니라 로그 책임도 애매해졌습니다.

---

### 대안: LoggingEventCompositeJsonEncoder

이 문제를 해결하기 위해 선택한 게

`LoggingEventCompositeJsonEncoder`입니다.

📌 **LoggingEventCompositeJsonEncoder**

- 클래스: `net.logstash.logback.encoder.LoggingEventCompositeJsonEncoder`
- 역할: 로그 이벤트를 JSON으로 출력하되, **provider 단위로 원하는 필드를 구성 가능**
- 장점:
    - `<timestamp>` provider를 통해 **필드명(@timestamp)과 타임존(Asia/Seoul) 지정 가능**
    - `<level>`, `<loggerName>`, `<threadName>` 같은 필드도 따로 정의 가능
    - `<mdc/>` provider를 통해 MDC 값 포함
    - `<arguments/>` provider를 통해 `Markers`나 `StructuredArguments`로 전달된 값들을 최상위 JSON으로 풀어서 출력

즉, 단순히 “로그를 JSON으로 찍는다”에서 그치는 게 아니라,

**필드 하나하나를 내가 원하는 방식대로 배치**할 수 있습니다.

---

### 타임존 설정 예시

```xml
<timestamp>
    <fieldName>@timestamp</fieldName>
    <timeZone>Asia/Seoul</timeZone>
</timestamp>
```

이 설정 하나로 이제 모든 로그는 **한국 시간(KST)** 기준으로 찍힙니다.

따라서 DTO 안에서 따로 `timestamp`를 생성할 필요가 없어졌습니다.

---

### 변경된 Logback 설정

```xml
<encoder class="net.logstash.logback.encoder.LoggingEventCompositeJsonEncoder">
    <providers>
        <timestamp>
            <fieldName>@timestamp</fieldName>
            <timeZone>Asia/Seoul</timeZone>
        </timestamp>
        <level>
            <fieldName>severity</fieldName>
        </level>
        <loggerName>
            <fieldName>logger</fieldName>
        </loggerName>
        <threadName>
            <fieldName>thread</fieldName>
        </threadName>
        <logstashMarkers/>
        <arguments/>
        <mdc/>
        <message>
            <fieldName>summary</fieldName>
        </message>
    </providers>
</encoder>

```

---

### 정리

- **이전**: DTO에서 timestamp(KST), Encoder에서 timestamp(UTC) → 시간 정보 불일치
- **이후**: Encoder에서만 timestamp(KST) 관리 → 로그 전역에서 시간 일관성 확보

덕분에 이제 로그 한 줄을 봤을 때, 언제 찍힌 로그인지 혼동할 일이 사라졌습니다.

## 개선 아이디어: Summary + Detail 분리

이 문제를 해결하기 위해, 로그를 **두 층위로 나누는 방식**을 도입했습니다.

1. **Summary**: 사람이 빠르게 훑어볼 수 있는 요약 정보
    - 예: `[REQUEST] POST /rooms`, `[RESPONSE] 200 /rooms`, `[ERROR] HEADER_IS_EMPTY`
2. **Detail**: 그 외 모든 상세 정보(JSON 필드)
    - Grafana에서 필터링/검색/분석할 때 사용

이렇게 되면, 콘솔이나 Grafana에서 로그를 볼 때

먼저 summary만 확인해서 전체 맥락을 이해하고, 필요하다면 detail(JSON)을 펼쳐서 보는 흐름이 가능해집니다.

결과적으로는 다음과 같은 형태로 구성됩니다.

## Log 인터페이스와 summary 메서드 도입

이를 위해 `Log`라는 인터페이스를 만들고, 모든 로그 DTO가 이를 구현하도록 했습니다.

```java
public interface Log {
    Map<String, Object> fields();  // JSON으로 풀릴 필드들
    String summary();              // 요약 문자열
}
```

그다음 `RequestLog` 같은 DTO에서는 다음과 같이 구현했습니다:

```java
public record RequestLog(
        LogType logType,
        String method,
        String uri,
        String body
) implements Log {

    public static RequestLog of(ContentCachingRequestWrapper request, String requestURI) {
        String body = new String(request.getContentAsByteArray(), StandardCharsets.UTF_8);
        return new RequestLog(
                LogType.REQUEST,
                request.getMethod(),
                requestURI,
                body
        );
    }

    @Override
    public Map<String, Object> fields() {
        return Map.of(
                "logType", logType.name(),
                "method", method,
                "uri", uri,
                "body", body
        );
    }

    @Override
    public String summary() {
        return String.format("[%s] %s %s", logType.name(), method, uri);
    }
}
```

---

## 로그 찍는 방식

이제 로그를 남길 때는 다음과 같이 합니다:

```java
RequestLog requestLog = RequestLog.of(cacheRequest, requestURI);

log.info(
    Markers.appendEntries(requestLog.fields()), // JSON detail
    requestLog.summary()                        // 사람이 읽기 좋은 summary
);
```

- 첫 번째 인자: `Markers.appendEntries(requestLog.fields())`
    
    → 필드들을 JSON 최상위로 풀어줌
    
- 두 번째 인자: `requestLog.summary()`
    
    → 사람이 읽기 좋은 요약 메시지
    

---

## Logback 설정에서 summary를 message로 출력

logback 설정에도 약간의 변화가 필요합니다.

```xml
<message>
    <fieldName>summary</fieldName>
</message>
```

이렇게 지정하면, `log.info(..., requestLog.summary())`로 넘긴 summary가

로그 JSON에서 `"summary": "[REQUEST] GET /rooms"` 형태로 출력됩니다.

---

## 개선된 결과

이제 로그는 다음처럼 보입니다:

- Summary만 보면
    
    ```
    [REQUEST] GET /rooms
    ```
    
    처럼 간단히 핵심만 확인할 수 있습니다.
    
- Detail(JSON)은 그대로 최상위 필드에 펼쳐지므로,
    
    Grafana에서는 `logType="REQUEST"` 같은 쿼리로 쉽게 필터링 가능합니다.

# 4. 로그 DTO의 필드를 `message` 문자열 안이 아닌 JSON 최상위 키-값으로 표현하기

이제 마지막 스텝입니다. 그리고 사실 가장 큰 문제였어요.

그동안 아무리 Log DTO를 잘 만들어서 JSON 구조로 찍더라도,

Logback이 그걸 문자열로만 인식해버리면 결과적으로 `"message": "{...}"` 안에 JSON이 또 들어가버리는 꼴이었습니다.

즉, JSON 안에 또 다른 JSON 문자열이 들어가 버리면서 아무 의미가 없어지는 상태였습니다.

---

## 기존 문제

예전 방식으로 찍은 로그를 보면 이렇습니다.

```json
{
  "@timestamp":"2025-09-27T11:20:54.979247616Z",
  "@version":"1",
  "message":"{
    \\"logType\\":\\"REQUEST\\",
    \\"timestamp\\":\\"2025-09-26T17:58:56.716209067\\",
    \\"requestId\\":\\"2678899b\\",
    \\"requestUri\\":\\"/api/v1/pickeats\\",
    \\"clientIp\\":\\"172.18.0.5\\",
    \\"method\\":\\"POST\\",
    \\"body\\":\\"{\\\\n  \\\\\\"name\\\\\\": \\\\\\"점심 맛집 찾기\\\\\\"\\\\n}\\"}"
}

```

겉으로는 JSON처럼 보이지만, 실제로는 `message` 안에 **순수 문자열**로만 들어가 있습니다.

이러면 Grafana에서는 안쪽 JSON을 파싱하지 못하니까 `logType`, `method`, `status` 같은 필드 기준으로 검색이나 필터링이 전혀 불가능했죠.

---

## 해결 방법: LoggingEventCompositeJsonEncoder + Markers API

기존의 `LogstashEncoder`는 구조를 바꾸기가 거의 불가능했습니다.

그래서 더 유연하게 커스터마이징할 수 있는 `LoggingEventCompositeJsonEncoder`로 갈아탔고,

여기에 **Markers API**를 활용해서 DTO의 필드를 JSON 최상위에 그대로 풀어낼 수 있도록 했습니다.

---

### Markers API란?

- **클래스**: `net.logstash.logback.marker.Markers`
- **역할**: SLF4J의 `Marker` 인터페이스를 확장해서, 로그 이벤트에 key-value 데이터를 붙일 수 있게 해줍니다.
- **효과**: Logback/Logstash encoder가 이 key-value를 JSON 최상위 필드로 직렬화해줍니다.

적용 방식은 아주 단순합니다:

```java
log.info(Markers.appendEntries(dto.fields()), dto.summary());
```

- **첫 번째 인자 (Marker)**
    
    → `dto.fields()`로 만든 key-value 맵이 JSON 최상위로 직렬화됩니다.
    
    예: `logType`, `method`, `uri`, `body` 등
    
- **두 번째 인자 (String message)**
    
    → 사람이 읽기 좋은 요약(summary).
    
    예: `[REQUEST] POST /api/v1/pickeats`
    

이렇게 하면 로그는 두 층으로 나뉩니다.

- 머신 필터링/시각화용: JSON 최상위 key-value
- 사람 가독성용: summary

---

## 개선된 결과

이제 로그는 이렇게 찍힙니다.

```json
{
  "@timestamp": "2025-09-27T11:20:54.979247616Z",
  "@version": "1",
  "logType": "REQUEST",
  "timestamp": "2025-09-26T17:58:56.716209067",
  "requestId": "2678899b",
  "requestUri": "/api/v1/pickeats",
  "clientIp": "172.18.0.5",
  "method": "POST",
  "body": "{\\n  \\"name\\": \\"점심 맛집 찾기\\"\\n}",
  "summary": "[REQUEST] POST /api/v1/pickeats"
}

```

이제 더 이상 `message` 안에 JSON 문자열이 들어가는 게 아닙니다.

각 필드가 **최상위 JSON key-value**로 표현되기 때문에,

- Grafana에서 `logType="REQUEST"` 같은 조건으로 바로 필터링 가능
- `method="POST"` 요청만 모아보기 가능
- summary는 여전히 간단하게 `[REQUEST] POST /api/v1/pickeats` 형태로 사람이 빠르게 읽을 수 있음

최종 코드와 설정은 다음과 같습니다.

---

## RequestLog 최종 코드

```java
package com.pickeat.backend.global.log.dto;

import java.nio.charset.StandardCharsets;
import java.util.Map;
import org.springframework.web.util.ContentCachingRequestWrapper;

public record RequestLog(
        LogType logType,
        String method,
        String uri,
        String body
) implements Log {

    public static RequestLog of(ContentCachingRequestWrapper request, String requestURI) {
        String body = new String(request.getContentAsByteArray(), StandardCharsets.UTF_8);
        return new RequestLog(
                LogType.REQUEST,
                request.getMethod(),
                requestURI,
                body
        );
    }

    @Override
    public Map<String, Object> fields() {
        return Map.of(
                "logType", logType.name(),
                "method", method,
                "uri", uri,
                "body", body
        );
    }

    @Override
    public String summary() {
        return String.format("[%s] %s %s", logType.name(), method, uri);
    }
}
```

- `fields()` → 로그에 담길 실제 key-value 데이터들 (JSON 최상위로 풀려서 들어감)
- `summary()` → 사람이 한눈에 알아보기 좋은 텍스트

---

## 로그 찍는 코드

```java
RequestLog requestLog = RequestLog.of(cacheRequest, requestURI);
log.info(
        Markers.appendEntries(requestLog.fields()),
        requestLog.summary()
);
```

- 첫 번째 인자: `Markers.appendEntries(...)` → JSON 최상위로 들어갈 key-value
- 두 번째 인자: `summary()` → 로그 요약 문자열

즉, **머신은 key-value를 보고, 사람은 summary를 보는 구조**가 됐습니다.

---

## logback.xml 최종 설정

```xml
<configuration>
    <!-- 로그 파일이 저장될 루트 경로를 변수로 정의 -->
    <property name="LOGS_ROOT_PATH" value="/var/logs"/>

    <!-- 파일로 로그를 쓰고 롤링(회전)하는 appender 정의 -->
    <appender name="FILE" class="ch.qos.logback.core.rolling.RollingFileAppender">
        <!-- 현재 기록 중인 파일 경로 -->
        <file>${LOGS_ROOT_PATH}/app.log</file>

        <!-- 날짜 + 파일 크기 기준으로 로그를 분할/보관하는 정책 -->
        <rollingPolicy class="ch.qos.logback.core.rolling.SizeAndTimeBasedRollingPolicy">
            <!-- 일자별 디렉터리/파일명 패턴 + 같은 날 여러 파일일 때 인덱스(%i) -->
            <fileNamePattern>${LOGS_ROOT_PATH}/%d{yyyy-MM-dd}.%i.log</fileNamePattern>
            <!-- 한 개 파일 최대 크기 -->
            <maxFileSize>10MB</maxFileSize>
            <!-- 보관할 히스토리(일수 기준). 30일 지나면 정리 -->
            <maxHistory>30</maxHistory>
        </rollingPolicy>

        <!-- JSON 인코더: 필드를 provider로 조합 가능 -->
        <encoder class="net.logstash.logback.encoder.LoggingEventCompositeJsonEncoder">
            <providers>
                <!-- @timestamp 필드 출력 + 타임존 KST 고정 -->
                <timestamp>
                    <fieldName>@timestamp</fieldName>
                    <timeZone>Asia/Seoul</timeZone>
                </timestamp>

                <!-- 로그 레벨을 severity라는 이름으로 출력 -->
                <level>
                    <fieldName>severity</fieldName>
                </level>

                <!-- 로거명을 logger라는 이름으로 출력 (보통 클래스명) -->
                <loggerName>
                    <fieldName>logger</fieldName>
                </loggerName>

                <!-- 스레드명을 thread라는 이름으로 출력 -->
                <threadName>
                    <fieldName>thread</fieldName>
                </threadName>

                <!-- Markers API로 전달한 키-값을 최상위 JSON 필드로 펼침 -->
                <logstashMarkers/>

                <!-- SLF4J 인자(StructuredArguments 포함)를 최상위 JSON으로 직렬화 -->
                <arguments/>

                <!-- MDC에 넣어둔 컨텍스트(예: request_id, client_ip)를 최상위 JSON으로 출력 -->
                <mdc/>

                <!-- 사람이 읽기 좋은 텍스트 요약을 summary라는 이름의 필드로 출력 -->
                <message>
                    <fieldName>summary</fieldName>
                </message>
            </providers>
        </encoder>
    </appender>

    <!-- 루트 로거 레벨: INFO 이상만 출력, 위 FILE appender 사용 -->
    <root level="INFO">
        <appender-ref ref="FILE"/>
    </root>
</configuration>

```

## 끝!

이제 최종적으로 픽잇의 로그는 다음과 같은 구조를 갖습니다:

- **Application Layer**에서는 DTO와 Markers API로 필드 정의 + summary 출력
- **Logback Layer**에서는 `LoggingEventCompositeJsonEncoder`로 JSON 구조를 원하는 대로 컨트롤
- **Alloy + Loki + Grafana Layer**에서는 JSON 필드를 기준으로 필터링/시각화 가능

결과적으로,

1. 중복 제거
2. 타임존 통일
3. 사람이 읽기 좋은 summary 추가
4. DTO 필드 최상위 JSON화

이 네 가지를 모두 달성했습니다.
