
> 가상 스레드는 **"블로킹 코드를 그대로 쓰되, 블로킹의 비용을 OS 스레드에서 힙의 continuation으로 옮긴 것."** I/O 바운드 서비스(챗봇 등)에 정확히 맞고, 주의할 건 pinning · 풀링 안티패턴 · ThreadLocal 메모리 정도다.

## 1. 왜 필요한가 — 플랫폼 스레드의 한계

기존 Java 스레드(이제 **플랫폼 스레드**라 부른다)는 OS 스레드를 1:1로 감싼 래퍼다. 비용이 크다. 스레드 하나당 기본 스택이 약 1MB를 잡고, 생성·컨텍스트 스위칭을 커널이 담당한다. 그래서 한 JVM에서 현실적으로 수천 개가 한계다.

핵심 병목은 **블로킹된 스레드가 OS 스레드를 점유한 채 놓아주지 못한다**는 점이다. 흔히 "작업을 중단(interrupt)하지 못해서"라고 오해하기 쉬운데, 정확히는 _중단의 문제가 아니라 점유의 문제_다. DB나 외부 API 응답을 기다리는 동안 그 스레드는 OS 스레드를 붙들고 아무 일도 안 하면서 묶여 있다.

그 결과:

- **스레드 풀 크기 = 동시 작업 상한**이 된다.
- 이 상한은 **CPU 사용률과 무관**하다. CPU는 놀고 있는데도 스레드가 없어서 요청이 큐에 쌓인다.

챗봇처럼 LLM 응답을 수 초씩 기다리는 I/O 바운드 워크로드에서 이게 특히 치명적이다.

### 세 가지 접근의 비교

|접근|전략|비용|
|---|---|---|
|thread-per-request (전통)|요청당 플랫폼 스레드 하나|스레드 풀이 동시성 상한|
|리액티브 (WebFlux)|블로킹 자체를 없앤다|**코드를 논블로킹(콜백/스트림) 패러다임으로 다시 써야 함**|
|Virtual Threads|코드는 블로킹 스타일 그대로, 블로킹 비용만 없앤다|거의 없음 (단, pinning 등 주의점 존재)|

> **오해 주의**: WebFlux의 전제는 "스레드 튜닝"이 아니라 **코드 작성 방식 자체를 바꾸는 것**이다. 모든 I/O를 논블로킹으로 표현해야 적은 이벤트 루프 스레드로 버틸 수 있다. Virtual Threads는 바로 그 "다시 쓰는 비용"을 없애겠다는 것.

## 2. Virtual Thread의 정체 — M:N 스케줄링

가상 스레드는 OS 스레드가 아니라 **JVM이 관리하는 사용자 공간 스레드**다. 스택을 힙에 저장하고, 생성 비용이 수백 바이트 수준이라 **수백만 개**를 만들 수 있다.

매핑 구조가 핵심이다.

- 기존: 1:1 (가상 = 플랫폼)
- 가상 스레드: **M:N** — 수많은 가상 스레드(M)가 소수의 **캐리어 스레드(carrier thread, N)** 위에서 번갈아 실행된다. 캐리어 스레드는 실제 플랫폼 스레드이고, 기본적으로 CPU 코어 수만큼의 ForkJoinPool로 구성된다.

### mount / unmount / park

```
[대기 큐]            [캐리어 스레드 ≈ 코어 수]            [힙]
 VT #1                  ┌──────────────┐               park된
 VT #2  ── mount ──▶    │  VT 실행 중   │   ── unmount ──▶  VT 스택
 VT #3                  ├──────────────┤   (I/O 블로킹)   (I/O 대기)
  ⋮                     │  VT 실행 중   │
                        ├──────────────┤
                        │  방금 비워짐   │ ◀── 다른 VT 즉시 마운트
                        └──────────────┘
        ▲                                                    │
        └──────────  I/O 완료 → 다시 큐로  ◀──────────────────┘
```

- **mount**: 가상 스레드가 캐리어 위에서 실제로 실행되는 상태.
- **unmount**: 블로킹 I/O를 만나면 JVM이 그 가상 스레드의 **스택 상태를 통째로 힙에 저장(continuation)** 하고 캐리어에서 떼어낸다. 비워진 캐리어는 즉시 다른 가상 스레드를 올린다.

> **중요한 2단계 구분**: 언마운트 = 곧장 실행 큐로 던지는 것이 _아니다_.
> 
> 1. I/O 블로킹으로 언마운트되면 → **힙에 park(보류)** 된다. (아직 실행할 게 없음. 응답 대기 중)
> 2. **I/O가 완료되면** 비로소 "실행 가능(runnable)" 상태가 되어 → **큐로 들어가** 다음 빈 캐리어를 기다린다.

이것이 "블로킹이 공짜에 가까워진다"는 말의 정체다. 가상 스레드를 블로킹해도 **OS 스레드(캐리어)는 블로킹되지 않는다.** 코드는 `response = httpClient.send(...)` 같은 평범한 동기·블로킹 스타일 그대로인데, 런타임이 알아서 캐리어를 양보한다.

### 직관: JVM이 async/await를 자동화한다

리액티브에서 손으로 짜던 "블로킹 지점에서 양보하고 콜백으로 재개"하는 로직을, 컴파일러·런타임이 동기 코드 밑에서 자동으로 수행한다. **외부 의존성 비동기 처리를 체계화·자동화해준 것**이라고 이해하면 정확하다.

## 3. 함정 ① Pinning (고정)

어떤 상황에서는 가상 스레드가 블로킹돼도 **언마운트되지 못하고 캐리어에 고정(pinned)** 된다. 그러면 그 캐리어(= 귀한 플랫폼 스레드) 하나가 통째로 묶여서, 가상 스레드의 장점이 사라진다. 캐리어가 코어 수만큼뿐이라, 핀이 몇 개만 쌓여도 처리량이 무너진다.

핀이 발생하는 두 케이스:

### `synchronized` 안에서 블로킹

- **Java 21~23**: 핀 발생. 핫 패스의 `synchronized`는 `ReentrantLock`으로 바꾸는 게 안전.
- **Java 24+(JEP 491)**: 해결됨. 더 이상 신경 쓸 필요 없음.

### 네이티브 메서드(JNI)나 Foreign Function 안에서 블로킹

**JNI(Java Native Interface)** 와 **Foreign Function & Memory API(FFM)** 는 둘 다 Java가 Java 아닌 네이티브 코드(주로 C/C++)를 호출하는 통로다. JNI는 오래된 방식, FFM은 현대적 대체재. OS 기능을 직접 부르거나 C 라이브러리를 쓸 때 사용한다.

**핀이 걸리는 이유**: 가상 스레드를 언마운트하려면 JVM이 그 스택을 힙에 저장해야 하는데, **네이티브 코드의 스택은 JVM이 관리하는 영역이 아니라서 저장·복원할 수 없다.** 그래서 어쩔 수 없이 캐리어에 고정한다. 이건 Java 24+에서도 여전히 핀이 걸린다.

대부분의 웹 개발에선 직접 마주칠 일이 드물고, 일부 구형 네이티브 JDBC 드라이버 정도가 해당할 수 있다.

### 핀 추적

```bash
-Djdk.tracePinnedThreads=full
```

또는 JFR 이벤트로 확인. 가상 스레드 도입 후엔 핀이 실제로 안 걸리는지 한 번 검증하는 것을 권장.

## 4. 함정 ② ThreadLocal (풀링과는 별개 문제)

`ThreadLocal`은 "스레드마다 독립된 저장 칸"이다(예: 요청별 사용자 정보). 가상 스레드에서도 **기능 자체는 그대로 잘 작동한다.** 다만 새로운 위험이 생겼다 — **메모리**.

플랫폼 스레드는 많아야 수천 개라 스레드마다 객체를 담아도 수천 개분량이다. 그런데 가상 스레드는 **수백만 개**가 될 수 있다. 각 가상 스레드가 무거운 객체를 ThreadLocal에 들고 있으면 그게 수백만 배로 불어나 메모리가 터진다.

> 이건 **스레드 총량을 정하는 문제(= 풀링)가 아니다.** "스레드 수가 폭증했으니 스레드당 저장소를 가볍게 유지하라"는 **메모리 문제**다. ThreadLocal 함정과 풀링 함정은 서로 무관하다.

### "가볍게 유지" — 튜닝의 3가지 축

1. **객체 크기(풋프린트)**: ThreadLocal에 담는 객체 × 가상 스레드 수. 50만 개 × 1KB = 500MB가 그냥 상주한다. 무거운 캐시·버퍼·빌더는 ThreadLocal이 아니라 진짜 공유 객체(싱글톤, 풀)로 빼라.
2. **항목 개수**: 여러 라이브러리가 각자 ThreadLocal을 심는다(트랜잭션 컨텍스트, 로깅 MDC, 보안 컨텍스트 등). 개수 × 스레드 수로 누적된다. "이 요청 경로에서 실제로 몇 개가 살아있나"를 의식할 것.
3. **상속 여부(`InheritableThreadLocal`)**: 자식 스레드 생성 시 부모 값을 _복사_한다. 가상 스레드는 빈번히 생성되므로, 무거운 값이 상속되면 생성마다 복사 비용 + 메모리가 배가된다. 특히 경계할 것.

> **들고 있어도 되는 값**: 작고, 요청 단위로 짧게 살고, 굳이 상속 안 되는 값. **빼야 하는 값**: 크고, 오래 살고, 상속되는 값.

### `InheritableThreadLocal`이란

일반 `ThreadLocal`은 새 스레드를 만들면 값이 자식에게 안 넘어간다(자식은 빈 상태). `InheritableThreadLocal`은 그 예외로, **부모가 자식 스레드를 만들 때 부모의 값을 자식에게 복사**한다. "요청 처리 스레드가 하위 작업 스레드를 띄울 때 사용자 정보를 물려주고 싶다" 같은 상황용이다. 대부분의 개발자가 직접 쓸 일은 드물고, 보통 프레임워크 내부에서 쓰인다. **이 상속 시나리오를 ScopedValue가 복사 없이(공유 참조로) 더 싸고 안전하게 대체한다.**

## 5. ScopedValue — ThreadLocal의 현대적 대체재

ThreadLocal의 구조적 약점 세 가지부터:

1. **가변**: 어디서든 `set()`으로 덮을 수 있어 추적이 어렵다.
2. **무경계**: `set()` 후 `remove()`할 때까지 살아 있다. 스레드 풀 재사용 시 `remove()`를 깜빡하면 다음 요청이 이전 값을 보는 누수 발생.
3. **상속 비용**: 위에서 본 복사 비용.

ScopedValue는 이 셋을 구조로 막는다 — **불변 + 범위(scope) 한정 + 자동 정리.**

```java
public class RequestContext {
    public static final ScopedValue<User> CURRENT_USER = ScopedValue.newInstance();
}

// 값을 "바인딩"하고, 그 범위 안에서 실행
ScopedValue.where(RequestContext.CURRENT_USER, loggedInUser)
    .run(() -> {
        handleRequest();   // 이 안에서 호출되는 모든 메서드가
    });                    // CURRENT_USER를 읽을 수 있음
// run()이 끝나는 순간 바인딩은 사라짐 — remove() 불필요

// 깊은 곳 어디서든 읽기만 가능 (쓰기 불가)
void someDeepMethod() {
    User u = RequestContext.CURRENT_USER.get();
}
```

**멘탈 모델**: ThreadLocal이 "스레드에 붙어서 내가 지울 때까지 떠다니는 값"이라면, ScopedValue는 **"이 코드 블록이 실행되는 동안만 유효한, 호출 스택을 따라 흐르는 값"** 이다. 수명이 스레드가 아니라 `run()`/`call()`의 실행 범위에 묶인다.

### 가상 스레드와 잘 맞는 이유

- **불변** → 여러 가상 스레드가 같은 바인딩을 안전하게 공유. 복사가 아닌 공유 참조라 자식 생성 시 상속 비용이 없다.
- **자동 정리** → 누수가 구조적으로 불가능. `remove()` 깜빡할 여지 자체가 없다.

### 구조적 동시성과의 전파

```java
ScopedValue.where(CURRENT_USER, user).run(() -> {
    try (var scope = StructuredTaskScope.open()) {
        scope.fork(() -> fetchProfile());   // 이 자식들도
        scope.fork(() -> fetchOrders());    // CURRENT_USER를 그대로 봄
        scope.join();
    }
});
```

> **현실적 접근**: ScopedValue는 비교적 최근에야 정식화 단계라, 기존 생태계(Spring Security의 SecurityContext, 로깅 MDC 등)는 여전히 내부적으로 ThreadLocal에 의존하는 경우가 많다. 당장 전부 갈아엎기보다는 **새로 짜는 요청 컨텍스트 전파 코드**에 우선 적용하고, 무거운 값이 ThreadLocal에 잔류·상속되지 않게 관리하는 게 맞다.

## 6. 함정 ③ 풀링하지 말 것

기존엔 스레드 생성이 비싸서 풀로 재사용했다. 가상 스레드는 생성이 거의 공짜라 **재사용할 이유가 없다.** 오히려 고정 크기 풀에 가두면 인위적 동시성 상한이 생겨 장점을 스스로 막는다.

```java
// ❌ 안티패턴: 가상 스레드를 고정 풀로 제한
ExecutorService bad = Executors.newFixedThreadPool(
    200, Thread.ofVirtual().factory());
// → "동시에 200개만"으로 잘려버림. 수십만 동시성이라는 존재 이유를 파괴.

// ✅ 작업마다 새 가상 스레드, 인위적 상한 없음
ExecutorService good = Executors.newVirtualThreadPerTaskExecutor();
```

> "가상 스레드 + `newFixedThreadPool` / `ThreadPoolTaskExecutor` 조합"이 보이면 거의 항상 잘못된 것. (단, 의도적 동시성 제한이 목적이라면 그건 풀이 아니라 세마포어로 — 7장 참고.)

## 7. 함정 ④ 지연시간은 안 줄어든다

가상 스레드는 **처리량(throughput)** 을 위한 기능이지 **지연시간(latency)** 을 위한 게 아니다.

- 요청 하나가 처리되는 **속도 자체는 빨라지지 않는다.**
- 더 많은 요청을 **동시에 버티게** 해줄 뿐이다.

효과가 큰 건 I/O 바운드 워크로드. CPU 바운드 작업(암호화, 이미지 처리, 무거운 연산)에는 거의 무의미하다 — 캐리어가 코어 수만큼뿐이라 결국 CPU에서 직렬화된다.

## 8. Spring Boot 적용 방법

### 기본: 한 줄

```properties
spring.threads.virtual.enabled=true
```

**Java 21+ / Spring Boot 3.2+** 전제. 이걸 켜면 자동 적용 범위가 넓다:

- 톰캣 요청 처리
- `@Async` 메서드
- Spring MVC 비동기 요청
- `@Scheduled` 작업
- Kafka·RabbitMQ·Redis 리스너

대부분의 웹 앱은 이 한 줄로 충분하다.

### 두 레이어로 이해하기

자동 적용이 안 닿는 곳이 있다. 구조를 두 레이어로 나누면 명확하다.

- **레이어 1 — 톰캣(웹 요청 처리)**: `enabled=true`가 자동 처리. 손댈 일 없음. HTTP 요청 하나하나가 가상 스레드 위에서 돈다.
- **레이어 2 — 요청 처리 중 직접 띄우는 작업**: 컨트롤러/서비스에서 `executor.submit(...)`을 호출하거나, `@Async`에 커스텀 실행기를 지정하거나, 직접 `ExecutorService`를 만들어 쓰는 경우. **이 레이어는 자동 설정이 안 닿거나, 명시 지정한 실행기를 그대로 따른다.**

6장의 풀 안티패턴은 정확히 **레이어 2**에서 벌어진다. 톰캣은 멀쩡히 가상 스레드인데, 하위 작업만 고정 풀에 갇히는 식.

커스텀 실행기를 가상 스레드 기반으로 만들려면:

```java
@Bean
public AsyncTaskExecutor applicationTaskExecutor() {
    return new TaskExecutorAdapter(
        Executors.newVirtualThreadPerTaskExecutor());
}
```

## 9. 커스텀 실행기 — 언제, 왜 만드나

커스텀 실행기를 만드는 이유는 **두 종류**로 갈린다. 이 구분이 핵심이다.

|만드는 이유|가상 스레드 환경에서|결론|
|---|---|---|
|① 스레드 생성·재사용 비용 절약|생성이 공짜 → 아낄 게 없음|**불필요해짐**|
|② 동시 실행을 의도적으로 제한|가상 스레드가 해결 못 하는 별개 문제|**여전히 필요**|

### 중요한 오해 두 가지 교정

**"가상 스레드 쓰면 커스텀 실행기 불필요?"** → 절반만 맞다. ①번 목적은 사라지지만 ②번 목적은 남는다.

**"커스텀 실행기 = 플랫폼 스레드?"** → **아니다.** "실행기를 직접 만드는 것"과 "그 실행기가 어떤 스레드를 쓰는가"는 **독립적인 선택**이다. 커스텀 실행기도 가상 스레드로 만들 수 있다. 권장 패턴은 오히려 **가상 스레드 + 세마포어**다.

```java
// 가상 스레드는 그대로(무제한, 가볍게), 동시 진입만 세마포어로 제한
private final Semaphore limiter = new Semaphore(10);

public Result callPartnerApi() throws InterruptedException {
    limiter.acquire();
    try {
        return doBlockingCall();   // 가상 스레드 위에서 실행
    } finally {
        limiter.release();
    }
}
```

> 플랫폼 스레드 고정 풀을 _일부러_ 고르는 경우는 좁다 — CPU 바운드 작업처럼 애초에 가상 스레드가 도움 안 되고 "코어 수만큼만" 돌리는 게 맞을 때 정도.

## 10. "동시 실행 제한"이 필요한 4가지 경우

가상 스레드의 "다 받아라"와 정반대로, 일부러 막아야 하는 상황들.

### ① 외부 시스템이 한도를 강제할 때 (가장 흔함)

외부 API가 "초당 10건", "동시 5건" 같은 rate limit을 건 경우. 제한을 안 하면 `429 Too Many Requests`나 키 차단.

**챗봇 직결 예시 — LLM API 호출**: OpenAI/Anthropic 등은 RPM·TPM 한도가 있다. 사용자 요청은 가상 스레드로 다 받되, **모델로 나가는 출구에만 동시성 제한**을 건다.

```java
private final Semaphore llmLimiter = new Semaphore(8);

public ChatResponse callModel(Prompt prompt) throws InterruptedException {
    llmLimiter.acquire();
    try {
        return chatClient.prompt(prompt).call().chatResponse();
    } finally {
        llmLimiter.release();
    }
}
```

### ② 공유 자원이 한정돼 있을 때 — DB 커넥션

> **흔한 오해 교정**: DB 때문에 **별도 실행기를 두지 않는다.** DB 커넥션 풀(HikariCP 등)이 **이미 세마포어처럼 동작**한다.

풀 크기가 20이면, 커넥션을 요청한 21번째 가상 스레드는 자동으로 **블로킹되어 대기**한다(가상 스레드라 이 대기는 공짜 — 언마운트됨). 별도 세마포어/실행기가 불필요하다.

가상 스레드 도입 시 DB에서 할 일은 딱 하나 — **커넥션 풀 크기·타임아웃 점검**.

```properties
spring.datasource.hikari.maximum-pool-size=30
spring.datasource.hikari.connection-timeout=10000
```

→ ①번(LLM API)은 제한 장치가 없어 세마포어를 직접 얹지만, ②번(DB)은 풀이 이미 그 역할이라 **설정값만 맞추면 끝.**

### ③ 하류 시스템 보호 (격벽/bulkhead)

여러 외부 의존성 중 하나가 느려지거나 죽으면, 제한이 없을 경우 그쪽에 가상 스레드가 무한정 쌓여 공유 자원을 다 빨아들이고 **멀쩡한 다른 기능까지 같이 죽는다.** 의존성별 동시 호출 한도를 따로 둬서, A가 죽어도 A 한도만 차고 B·C는 영향 없게 격리한다. (배의 격벽이 한 칸 침수돼도 안 가라앉는 원리)

### ④ 작업 자체가 무겁거나 위험할 때

CPU 많이 쓰는 작업(이미지 변환, 대량 계산)을 무제한으로 띄우면 서로 CPU를 뺏어 전체가 느려진다. 메모리 많이 먹는 작업은 OOM 위험. "동시에 코어 수만큼만" 같은 상한이 오히려 안정적. (가상 스레드가 CPU 바운드엔 무의미한 것과 같은 맥락)

### 관통하는 원리

네 경우 모두 **제한의 이유가 "스레드"가 아니라 스레드 너머의 무언가**에 있다 — 외부 API 한도, DB 커넥션 수, 하류 시스템 체력, CPU·메모리. 가상 스레드는 "스레드가 비싸다"는 문제만 없앴을 뿐, 이 진짜 한계들은 그대로다.

## 11. 핵심 원리 — 두 개의 다이얼

가상 스레드 시대의 가장 중요한 사고 전환:

> **"무엇으로 실행하느냐(가상/플랫폼)"** 와 **"동시에 몇 개를 허용하느냐(제한)"** 는 **분리된 두 개의 다이얼**이다.

- **옛날**: 스레드 풀 크기 하나로 두 다이얼이 붙어 있었다 — 스레드도 아끼고(자원) 동시성도 막고(제한).
- **지금**: 둘을 따로 돌린다 — **스레드는 가볍게 무제한으로 두고, 진짜 막아야 할 자원에만 세마포어/풀로 한도를 건다.** 관심사가 깔끔히 분리된다.

또한 "실행기를 나누면 완전히 별개 작업인가?" → 실행 컨텍스트는 격리되지만 **공유 자원(메모리, DB 커넥션 풀, CPU)까지 분리되지는 않는다.** 같은 JVM에서 돌기 때문.

## 12. 왜 아직 주류 전환이 더딘가

가상 스레드는 기술적으로 분명히 옳은 방향이고 신규/I/O 바운드 프로젝트엔 이미 합리적이다. 그런데 기존 시스템의 전면 전환은 신중하게 진행 중이다. 이유:

1. **pinning이 조용히 장점을 무력화한다.** 오래된 라이브러리·드라이버가 `synchronized`나 네이티브 호출로 블로킹하면 핀이 걸려 효과가 안 난다. 더 나쁜 건 "별로 안 빨라지네" 하고 원인을 모른 채 넘어가기 쉽다는 점. Java 24에서 synchronized 핀은 해결됐지만 많은 프로덕션이 아직 Java 21 LTS.
2. **생태계가 여전히 ThreadLocal 기반.** Spring Security, MDC, 다수 APM·트레이싱이 ThreadLocal로 컨텍스트를 전파한다. 작동은 하지만 가상 스레드 친화적 재설계 전까진 미묘한 비용·호환성 이슈가 남는다.
3. **보수성.** WebFlux와 동기 thread-per-request는 수년~수십 년 검증됐다. 가상 스레드는 2023년 9월(Java 21) 정식화라 신참이고, 멀쩡히 도는 시스템을 바꿀 동기가 약하다.

> 즉 "스위치 하나로 공짜 성능"이 아니라, 위 세 가지를 점검해야 효과를 보는 기능이다. **레거시 부채가 없는 신규 프로젝트가 가장 유리한 케이스.**
