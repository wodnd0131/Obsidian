자바는 main을 기반으로 클래스를 객체화하여 진행합니다. 
자바가 어떻게 실행되는지 공부하시면 좋을 것같아요.

## Java의 실행 구조: main을 기반으로 클래스를 객체화

### 핵심 개념부터

다른 언어(Python, C 등)는 파일을 위에서 아래로 그냥 실행하거나, 함수를 바로 정의해서 씁니다.

Java는 **모든 것이 클래스 안에** 있어야 합니다. 코드 한 줄도 클래스 밖에 존재할 수 없어요.

---

### main 메서드 — 진입점

```java
public class Main {
    public static void main(String[] args) {
        // 여기서부터 시작
    }
}
```

JVM(Java 가상머신)이 프로그램을 실행할 때 **`main` 메서드를 찾아서 시작**합니다. `public static void main(String[] args)` — 이 시그니처는 약속이라 바꿀 수 없어요.

> Python의 `if __name__ == "__main__":` 과 같은 역할이지만, Java에서는 강제입니다.

---

### "클래스를 객체화" — 이게 핵심

클래스는 **설계도**, 객체는 **실제로 만들어진 것**입니다.

```java
// 설계도 (클래스)
public class Dog {
    String name;

    void bark() {
        System.out.println(name + ": 멍멍!");
    }
}

// main에서 객체화 (인스턴스 생성)
public class Main {
    public static void main(String[] args) {
        Dog myDog = new Dog();   // 설계도로 실제 객체를 생성
        myDog.name = "콩이";
        myDog.bark();             // 콩이: 멍멍!
    }
}
```

`new Dog()` 가 **객체화(인스턴스화)** 입니다. 설계도를 보고 실물을 찍어내는 것.

---

### 왜 "main을 기반으로" 라고 하냐면

프로그램의 흐름이 항상 **main에서 시작**되고, main에서 필요한 객체들을 만들어서 동작시키는 구조이기 때문입니다.

```
JVM 실행
  └─► main() 호출
        └─► new SomeClass() → 객체 생성
              └─► 객체의 메서드 호출 → 실제 로직 수행
```

---

### 다른 언어와 비교하면

| |Python|Java|
|---|---|---|
|코드 위치|파일 어디든|반드시 클래스 안|
|진입점|`if __name__ == "__main__"` (선택)|`main()` (필수)|
|실행 단위|함수/스크립트|객체|
|자유도|높음|낮음 (대신 구조화)|

---

### 한 줄 요약

> Java는 **"main에서 시작해서, 필요한 클래스를 객체로 만들고, 그 객체들이 서로 협력하며 동작"** 하는 언어입니다.

처음엔 왜 이렇게 복잡하게 하나 싶지만, 규모가 커질수록 이 구조 덕분에 관리가 훨씬 쉬워집니다. Spring Boot도 결국 이 구조의 연장선이에요.

# 자바의 실행 과정
## Java 실행 과정

### 전체 흐름 한눈에

```
.java 파일 (소스코드)
    │
    │  javac (컴파일러)
    ▼
.class 파일 (바이트코드)
    │
    │  JVM (Java Virtual Machine)
    ▼
실제 실행 (기계어)
```

---

### 1단계 — 컴파일 (`javac`)

```bash
javac Main.java  →  Main.class 생성
```

Java 소스코드를 **바이트코드**로 변환합니다.

바이트코드는 기계어가 아닙니다. CPU가 바로 읽을 수 없고, **JVM만 읽을 수 있는 중간 언어**예요.

> C/C++은 컴파일하면 바로 기계어(exe)가 됩니다. Java는 여기서 한 단계 더 있어요.

---

### 2단계 — JVM이 실행

```bash
java Main  →  JVM이 Main.class를 읽어서 실행
```

JVM이 `.class` 파일을 받아서 내부적으로 여러 단계를 거칩니다.

```
.class (바이트코드)
    │
    ▼
① Class Loader       — 필요한 클래스 파일을 메모리에 로드
    │
    ▼
② Bytecode Verifier  — 바이트코드 유효성 검사 (보안)
    │
    ▼
③ JIT Compiler       — 바이트코드 → 기계어 변환 (실시간)
    │
    ▼
실제 CPU 실행
```

---

### JVM 메모리 구조 (간단히)

실행되면 JVM이 메모리를 이렇게 나눠서 관리합니다.

```
┌─────────────────────────────────┐
│           Method Area           │  클래스 정보, static 변수
├─────────────────────────────────┤
│               Heap              │  new로 생성된 객체들
├─────────────────────────────────┤
│         Stack (스레드별)         │  메서드 호출, 지역변수
├─────────────────────────────────┤
│          PC Register            │  현재 실행 중인 명령어 위치
└─────────────────────────────────┘
```

- `new Dog()` 하면 → **Heap**에 올라감
- 메서드 안 지역변수 → **Stack**에 올라감
- `static` 변수 → **Method Area**에 올라감

---

### 핵심 — "Write Once, Run Anywhere"

Java가 이런 구조를 택한 이유가 여기 있습니다.

```
          [Windows]
.class ──► JVM(Windows용) ──► 실행
          [Mac]
.class ──► JVM(Mac용)     ──► 실행
          [Linux]
.class ──► JVM(Linux용)   ──► 실행
```

**같은 `.class` 파일**이 어떤 OS에서든 동작합니다. OS별로 다른 건 JVM이고, 개발자는 신경 쓸 필요 없어요.

> C는 OS마다 다시 컴파일해야 합니다. Java는 `.class` 하나로 어디서든 실행.

---

### JIT 컴파일러 — 성능 얘기

초기 Java가 느리다는 오명이 있었는데, JIT(Just-In-Time) 덕분에 많이 해소됐습니다.

```
인터프리터 방식 (초기)
  바이트코드를 한 줄씩 해석 → 느림

JIT 방식 (현재)
  자주 실행되는 코드를 감지 → 미리 기계어로 캐싱 → 빠름
```

**자주 쓰는 코드일수록 점점 빨라지는** 구조입니다.

---

### 한 줄 요약

> **소스코드 → (javac) → 바이트코드 → (JVM) → 실행** JVM이 중간에서 OS 차이를 흡수해주기 때문에, 개발자는 플랫폼 걱정 없이 코드만 짜면 됩니다.

Spring Boot도 결국 JVM 위에서 돌아가는 것이고, 컨테이너(Docker)로 배포할 때도 내부적으로는 이 구조 그대로입니다.
