## 핵심 원칙 하나

> **구현을 보지 않고 이름만으로 의도를 알 수 있어야 한다**

이게 전부입니다. 나머지 규칙은 전부 이것의 파생입니다.

---

## 규칙 1 — 동사 + 명사

```java
// 동사만
process()
handle()
execute()
// → 뭘 하는지 모름

// 동사 + 명사
processPayment()
handleAuthFailure()
executeQuery()
// → 의도가 보임
```

---

## 규칙 2 — and가 들어가면 메서드를 쪼개야 한다는 신호

```java
// 나쁜 예
validateAndSaveUser()
fetchAndCacheProduct()

// 좋은 예
validate(user)
save(user)

fetchProduct(id)
cache(product)
```

and는 두 가지 책임이 섞였다는 신호입니다.

---

## 규칙 3 — 반환값이 있는지 없는지를 이름에서 유추 가능해야 한다

```java
// 반환값 없음 — 행위 자체가 목적
send()
save()
delete()
notify()

// 반환값 있음 — 가져오거나 만들거나 확인
getUser()
findById()
calculateTax()
isValid()
hasPermission()
```

`get`, `find`, `calculate`, `is`, `has`, `check` 같은 접두사가 "이 메서드는 뭔가를 반환한다"는 신호입니다.

---

## 규칙 4 — get vs find 구분

실무에서 자주 혼용되는데 의미가 다릅니다.

```java
// get — 항상 결과가 있다고 가정. 없으면 예외
User getUser(Long id)          // 없으면 throw Exception

// find — 없을 수도 있음
Optional<User> findUser(Long id)   // 없으면 Optional.empty()
```

Spring Data JPA가 이 컨벤션을 그대로 씁니다. `getById`는 없으면 예외, `findById`는 Optional 반환.

---

## 규칙 5 — 부정형 메서드 이름은 피한다

```java
// 나쁜 예 — 부정형
isNotValid()
hasNoPermission()

// 좋은 예 — 긍정형으로 뒤집기
isValid()
hasPermission()
```

부정형은 조건문에서 이중 부정이 됩니다.

```java
if (!isNotValid()) { }    // 읽기 어려움
if (isValid()) { }        // 명확
```

---

## 규칙 6 — 맥락이 있으면 생략한다

```java
// 클래스가 UserService인데
class UserService {
    User getUserByUserService(Long userId)  // 과잉
    User getUser(Long id)                   // 충분
}
```

클래스 이름이 이미 맥락을 제공합니다. 메서드에 반복할 필요 없습니다.

---

## 판단 기준 — 이름이 좋은지 테스트하는 법

코드 리뷰할 때 이 질문을 던집니다.

```
1. 이 메서드가 하는 일을 한 문장으로 설명할 수 있는가?
2. 그 한 문장이 메서드 이름과 일치하는가?
3. 구현을 보지 않고 이름만으로 반환값을 예측할 수 있는가?
```

세 가지 다 yes면 좋은 이름입니다.

## 실무에서 자주 보이는 나쁜 패턴

```java
// 1. 너무 추상적
process()
handle()
manage()
deal()

// 2. 구현 방식이 이름에 노출됨
getUserFromDatabase()     // DB 바꾸면 이름도 바꿔야 함
getUserFromCache()
→ getUser()로 충분, 어디서 가져오는지는 내부 관심사

// 3. 숫자/순서가 이름에 들어감
processStep1()
processStep2()
→ 각 단계의 의미로 이름 짓기
validateOrder()
calculateTotal()

// 4. 동의어 혼용
fetch / get / retrieve / load / read
→ 팀 내에서 하나로 통일
```

## 한 줄 정리

> 좋은 메서드 이름은 **주석이 필요 없는 이름**입니다. 이름을 보고 주석을 달고 싶어진다면, 이름이 나쁜 겁니다.
