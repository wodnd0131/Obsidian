# N+1 문제와 LAZY 로딩

---

## 먼저 LAZY 로딩이 뭔지

엔티티 조회 시 **연관된 엔티티를 즉시 가져오지 않고, 실제로 접근할 때 가져오는 것.**

```java
@Entity
public class Order {
    @Id private Long id;
    
    @ManyToOne(fetch = FetchType.LAZY)  // LAZY
    private Member member;
}
```

```java
Order order = em.find(Order.class, 1L);
// 이 시점: SELECT * FROM order WHERE id=1 만 실행
// member는 아직 안 가져옴 — 프록시 객체로 대체

String name = order.getMember().getName();
// 이 시점에 SELECT * FROM member WHERE id=? 실행
// 실제로 접근하는 순간 DB에 쿼리
```

반대는 `EAGER` — 조회 시 연관 엔티티를 즉시 JOIN해서 가져온다. JPA 기본값은 `@ManyToOne`이 EAGER, `@OneToMany`가 LAZY다. 실무에서는 **모두 LAZY로 설정하는 게 관례**다. EAGER는 예측 못한 쿼리가 나가는 원인이 된다.

---

## N+1 문제

LAZY 로딩이 루프를 만나면 터진다.

```java
// 주문 목록 조회
List<Order> orders = orderRepository.findAll();
// SELECT * FROM order → 주문 10건 조회 (쿼리 1번)

for (Order order : orders) {
    System.out.println(order.getMember().getName());
    // 여기서 member가 LAZY → 주문마다 SELECT * FROM member WHERE id=?
    // 10번 실행
}

// 총 쿼리: 1 + 10 = 11번
// 주문이 100건이면 1 + 100 = 101번
```

```
실행된 SQL:
SELECT * FROM orders;                        -- 1번
SELECT * FROM member WHERE id = 1;           -- order 1
SELECT * FROM member WHERE id = 2;           -- order 2
SELECT * FROM member WHERE id = 3;           -- order 3
...                                          -- N번
```

N+1이라는 이름이 여기서 나온다. **1번 조회했더니 N번 추가 쿼리가 발생.**

---

## 해결 방안

### 1. Fetch Join — 가장 기본

```java
// JPQL
@Query("SELECT o FROM Order o JOIN FETCH o.member")
List<Order> findAllWithMember();
```

```sql
-- 실행되는 SQL (쿼리 1번으로 끝)
SELECT o.*, m.*
FROM orders o
INNER JOIN member m ON o.member_id = m.id
```

한 번에 다 가져오니 추가 쿼리 없다.

**한계:** `@OneToMany` 관계에서 컬렉션 Fetch Join + 페이징을 같이 쓰면 문제가 생긴다.

```java
// Order 1개에 OrderItem 여러 개인 경우
@Query("SELECT o FROM Order o JOIN FETCH o.orderItems")
Page<Order> findAll(Pageable pageable); // ← 위험
```

```
ORDER 1 — ITEM A
ORDER 1 — ITEM B   ← JOIN 결과가 row 중복
ORDER 2 — ITEM C
```

JPA가 페이징을 메모리에서 처리해버린다(HibernateWarning: `HHH90003004`). 데이터 전체를 메모리에 올린 후 잘라내는 것이라 OOM 위험이 있다.

---

### 2. @BatchSize — 컬렉션 페이징 문제 해결

```java
@Entity
public class Order {
    @Id private Long id;
    
    @BatchSize(size = 100)   // 또는 글로벌 설정
    @OneToMany(mappedBy = "order", fetch = FetchType.LAZY)
    private List<OrderItem> orderItems;
}
```

```
기존 N+1:
SELECT * FROM order_item WHERE order_id = 1;
SELECT * FROM order_item WHERE order_id = 2;
SELECT * FROM order_item WHERE order_id = 3;
...N번

@BatchSize 적용:
SELECT * FROM order_item WHERE order_id IN (1, 2, 3, ..., 100);
-- IN절로 한 번에 묶어서 조회
```

N번이던 쿼리가 `N / BatchSize`번으로 줄어든다. 주문 1000건이면 10번.

글로벌 설정으로 전체 적용 가능하다.

```yaml
# application.yml
spring:
  jpa:
    properties:
      hibernate:
        default_batch_fetch_size: 100
```

---

### 3. QueryDSL + DTO Projection — 조회 전용일 때

엔티티가 아닌 DTO로 바로 뽑으면 연관관계 로딩 자체가 없다.

```java
List<OrderSummaryDto> result = queryFactory
    .select(new QOrderSummaryDto(
        order.id,
        order.orderDate,
        member.name      // JOIN해서 바로 DTO로
    ))
    .from(order)
    .join(order.member, member)
    .fetch();
// 쿼리 1번, N+1 없음, 필요한 컬럼만
```

화면 출력용 조회 API에서 가장 깔끔한 방법이다. 엔티티를 굳이 영속성 컨텍스트에 올릴 필요가 없을 때 쓴다.

---

## 정리

|상황|해결책|
|---|---|
|`@ManyToOne` 단건 조회|Fetch Join|
|`@OneToMany` + 페이징|`@BatchSize` (글로벌 설정 권장)|
|조회 전용 API|QueryDSL DTO Projection|
|컬렉션 Fetch Join + 페이징|**하지 말 것** — BatchSize로 대체|

실무에서는 글로벌 `default_batch_fetch_size: 100` 설정을 기본으로 깔고, 성능이 중요한 조회는 QueryDSL DTO로 내려가는 방식을 가장 많이 쓴다.

# ORM — Object Relational Mapping

---

## M은 Mapping

**객체(Object)** 와 **관계형 DB(Relational)** 사이의 **매핑(Mapping)**

둘의 패러다임이 근본적으로 다르다는 게 출발점이다.

```
Java 객체 세계          관계형 DB 세계
─────────────────       ─────────────────
클래스                  테이블
필드                    컬럼
객체 참조               외래키(FK)
상속                    없음 (흉내만 가능)
그래프 탐색             JOIN
```

ORM은 이 간극을 자동으로 메워주는 기술이다.

```java
// 객체 세계 — 참조로 탐색
order.getMember().getAddress().getCity();

// DB 세계 — JOIN으로 탐색
SELECT a.city
FROM orders o
JOIN member m ON o.member_id = m.id
JOIN address a ON m.address_id = a.id
WHERE o.id = ?
```

ORM이 없으면 위 객체 탐색 코드가 아래 SQL로 변환되는 걸 개발자가 직접 해야 한다.

---

## 실무에서 ORM의 장점을 포기하는 이야기

말한 "간접참조"가 정확히 어떤 패턴인지부터 짚겠다.

### ORM이 원래 주는 것 — 객체 그래프 탐색

```java
// ORM의 이상향
Order order = orderRepository.findById(1L);
String city = order.getMember().getAddress().getCity();
// 객체 탐색만으로 연관 데이터 접근
```

### 실무에서 포기하는 방식 — ID로만 참조

```java
// 실무 코드
@Entity
public class Order {
    @Id private Long id;
    
    private Long memberId;  // 객체 참조 대신 ID만 보관
    // @ManyToOne private Member member; ← 이걸 안 씀
}

// 필요하면 직접 조회
Member member = memberRepository.findById(order.getMemberId());
```

---

## 왜 이렇게 하냐

### 이유 1. 경계를 명확히 — 도메인 주도 설계(DDD)의 Aggregate

```
Order Aggregate       Member Aggregate
─────────────         ───────────────
Order                 Member
OrderItem             Address
                      Point
```

`Order`에서 `Member`로 객체 참조가 뚫려 있으면 `Order` 수정하다가 `Member`까지 딸려 변경될 수 있다. 영속성 컨텍스트가 두 객체를 모두 추적하고 있기 때문이다.

```java
@Transactional
public void updateOrder(Long orderId) {
    Order order = orderRepository.findById(orderId);
    order.getMember().setName("변경됨"); // 의도치 않게 Member도 dirty checking
    // → Member UPDATE 쿼리가 나감
}
```

ID로만 참조하면 이 경계가 물리적으로 막힌다.

### 이유 2. MSA 환경 — 서비스가 DB를 공유 안 한다

```
주문 서비스 DB        회원 서비스 DB
─────────────         ──────────────
orders 테이블         members 테이블
```

DB가 분리되면 JPA 객체 참조 자체가 불가능하다. `memberId`만 들고 있다가 회원 서비스 API를 호출하는 수밖에 없다.

```java
// MSA에서의 조회
Order order = orderRepository.findById(orderId);
MemberDto member = memberClient.findById(order.getMemberId()); // HTTP 호출
```

### 이유 3. 성능 — 객체 그래프가 깊을수록 조회가 무거워진다

```java
// 이 한 줄이 내부적으로 어디까지 타고 들어갈지 예측이 안 됨
order.getMember().getTeam().getCompany().getCeo().getAddress();
```

객체 참조가 뚫려 있으면 개발자가 실수로 연쇄 조회를 유발할 수 있다. ID 참조면 그 호출 자체가 명시적이라 인지가 된다.

---

## 그럼 ORM을 왜 쓰나 — 남는 장점

객체 참조를 포기해도 ORM이 주는 게 여전히 많다.

|포기하는 것|여전히 얻는 것|
|---|---|
|객체 그래프 탐색|단일 Aggregate 내 매핑 자동화|
|연관관계 자동 JOIN|변경 감지(dirty checking)|
|투명한 LAZY 로딩|1차 캐시|
||트랜잭션 쓰기 지연|
||스키마 자동화|

`Order` 안의 `OrderItem` 목록처럼 **같은 Aggregate 내부**에서는 객체 참조를 그대로 쓴다. 포기하는 건 **Aggregate 경계를 넘는 참조**다.

---

## 한 줄 요약

> ORM의 객체 그래프 탐색은 강력하지만, 경계가 없으면 통제가 안 된다. 실무에서 ID 참조로 전환하는 건 ORM을 포기하는 게 아니라, **ORM을 쓰되 Aggregate 경계 안에서만 쓰겠다**는 설계 결정이다.
