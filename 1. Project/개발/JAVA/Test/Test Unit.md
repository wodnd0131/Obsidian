#Test/Test_Unit #Status/정리
나는 메세지 전달의 관점에서, TDD를 진행하며, 해당 도메인이 요구사항을 수행하기위해 입출력을 낼수있는가를 위주로 보는것같아. 요구사항 대비 입출력만 확인하는거지

그렇기에, 비즈니스 로직이 없는 getter/setter나 Equals 같은 경우는 테스트를 안해. 그런데, 비즈니스 로직이 있다고 가정해도, 단순한 계산로직같은건 어떻게 봐야할까? 같은거야
## 단위 테스트 작성 기준 — 다양한 관점

### 먼저 네 관점 정리

> "도메인 신뢰성을 위해 내부 로직은 모두 테스트한다"

이건 **테스트를 명세(specification)로 보는 관점**이에요. 로직이 단순해도 "이 객체는 이렇게 동작해야 한다"는 계약을 코드로 남기는 거죠. 나중에 누군가 코드를 바꿀 때 안전망이 돼요.

---

### 다양한 관점들

**1. 켄트 벡 — 자신감이 기준**

> "테스트가 두려움을 지루함으로 바꿔줄 때까지 작성하라"

테스트를 작성하는 기준이 로직의 복잡도가 아니라 **"이게 틀렸을 때 내가 불안한가"** 예요.

```java
// 이것도 테스트할 가치가 있을 수 있음
// "혹시 1이 아닌 값을 반환하면?" 이라는 불안이 있다면
public int getMinLottoNumber() {
    return 1;
}
```

**2. 마틴 파울러 — 테스트할 가치가 있는 것만**

> "완벽한 테스트보다 좋은 테스트를 작성하라"

단순한 getter/setter는 테스트하지 않아도 된다는 입장이에요. 프레임워크나 언어가 보장하는 영역까지 테스트하는 건 낭비라고 봐요.

```java
// 마틴 파울러라면 이건 안 써도 된다고 할 것
@Test
void getter_테스트() {
    Lotto lotto = new Lotto(List.of(1,2,3,4,5,6));
    assertThat(lotto.getNumbers()).isEqualTo(List.of(1,2,3,4,5,6));
}
```

**3. 블라디미르 코리코프 — 비즈니스 가치가 기준**

오브젝트와 함께 국내에서 많이 읽히는 **단위 테스트** 책의 저자예요.

```
테스트 비용 vs 테스트 가치

비용 — 작성 시간, 유지보수 시간
가치 — 버그 발견, 회귀 방지, 문서화
```

단순 로직은 **버그가 생길 가능성이 낮아서 가치 대비 비용이 높다**고 봐요. 복잡한 도메인 로직에 집중하라는 입장이에요.

**4. TDD 관점 — 테스트가 설계를 이끈다**

```java
// TDD로 짜면 단순한 로직도 자연스럽게 테스트가 생김
// 테스트 작성 여부를 고민할 필요가 없음
// 코드보다 테스트가 먼저이기 때문에
@Test
void 로또_번호는_1에서_45_사이여야_한다() { ... }
// → 이 테스트를 통과시키는 코드를 작성
```

TDD를 철저히 따르면 네 관점과 자연스럽게 일치해요.


---

### 테스트 작성 여부 판단 기준 비교

```
켄트 벡       — 불안하면 써라
마틴 파울러   — 가치 있는 것만 써라
코리코프      — 비용 대비 가치를 따져라
네 관점       — 도메인이면 다 써라 (신뢰성 우선)
```

---

### 실무에서의 현실적 양상

팀마다 달라요. 보통 이런 기준을 많이 써요.

```
✅ 무조건 테스트
- 도메인 핵심 로직 (당첨 계산, 수익률)
- 경계값이 있는 로직 (1~45, 1000원 단위)
- 버그가 난 적 있는 곳
- 여러 사람이 건드리는 곳

🤔 팀 기준에 따라
- 단순 생성자 검증
- 단순 getter

❌ 보통 안 씀
- 프레임워크 동작 검증
- 자바 언어 자체 동작 검증
```

## 네 관점 정리부터

먼저 네가 말한 걸 명확히 표현하면 이거예요.

```
객체에게 메시지를 보낸다
        ↓
요구사항에 맞는 결과가 나오는가?
        ↓
YES/NO로 검증
```

이건 코리코프가 말한 **"블랙박스 테스트"** 와 정확히 일치해요. 내부 구현이 아니라 입출력 계약을 검증하는 거죠.

---

### 단순 계산 로직의 딜레마

네가 고민하는 케이스를 구체화하면 이래요.

```java
// 비즈니스 로직은 맞는데 단순한 계산
public double calculateProfitRate(int totalPrize, int purchaseAmount) {
    return (double) totalPrize / purchaseAmount * 100;
}
```

여기서 핵심 질문은 두 가지예요.

**질문 1 — 이게 요구사항에서 명시된 행위인가?**

```
로또 요구사항: "수익률을 계산한다"
        ↓
YES → 요구사항에서 명시된 행위
        ↓
입출력 검증 대상이 맞음
```

**질문 2 — 이게 틀렸을 때 다른 테스트가 잡아주는가?**

```java
// calculateProfitRate를 쓰는 상위 테스트가 있다면?
@Test
void 전체_수익률_계산() {
    // 여기서 이미 잡힘
}
// → 별도 테스트 불필요할 수 있음

// 독립적으로 쓰인다면?
// → 별도 테스트 필요
```

---

### 단순 로직 테스트의 실질적 가치

단순해 보여도 테스트가 가치있는 경우가 있어요.

**경계값이 있을 때**

```java
// 단순해 보이지만
public double calculateProfitRate(int totalPrize, int purchaseAmount) {
    return (double) totalPrize / purchaseAmount * 100;
}

// 이런 케이스들이 숨어있음
calculateProfitRate(0, 1000)     // 수익 없음 → 0.0
calculateProfitRate(1000, 1000)  // 본전 → 100.0
calculateProfitRate(5000, 1000)  // 수익 → 500.0
// 단순 계산이지만 경계값 자체가 요구사항
```

**형변환 버그가 숨어있을 때**

```java
// 이거 틀린 거 바로 보여요?
public double calculateProfitRate(int totalPrize, int purchaseAmount) {
    return totalPrize / purchaseAmount * 100; // 정수 나눗셈 버그
}

// 테스트가 없으면 잡기 어려움
@Test
void 수익률_계산() {
    assertThat(calculateProfitRate(5000, 3000))
        .isCloseTo(166.6, offset(0.1)); // 여기서 잡힘
}
```

---

### 네 관점에서의 실용적 기준

```
이 로직이 요구사항에서 명시된 행위인가?
        ↓
YES → 단순해도 테스트 대상

다른 테스트가 이미 이걸 커버하는가?
        ↓
YES → 중복 테스트, 생략 가능
NO  → 독립 테스트 필요

경계값이나 타입 변환이 있는가?
        ↓
YES → 단순해 보여도 반드시 테스트
```

---

### 결론적으로

네 관점인 **"메시지 입출력 기반 + 요구사항 중심"** 은 사실 매우 탄탄한 기준이에요.

단순 계산 로직도 결국 이 질문 하나로 귀결돼요.

```
"이 메시지의 입출력이 요구사항을 표현하는가?"
        ↓
YES → 단순해도 테스트
NO  → 구현 세부사항, 생략 가능
```

getter/setter를 테스트 안 하는 이유도 같은 맥락이에요. 요구사항이 "번호를 꺼내라"가 아니라 "당첨 여부를 판단하라" 이기 때문이죠.
