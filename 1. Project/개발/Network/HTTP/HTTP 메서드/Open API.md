# Open API 시스템 개발/운영

채용공고나 역량 정의서에서 나오는 표현이라 CS 개념 학습 지침보다는 **도메인 맥락 해석** 질문이네요. 지침 형식 대신 실무 관점에서 풀겠습니다.

---

## 이 표현이 의미하는 것

"Open API 시스템"은 **외부 개발자/파트너/고객사에게 자사 서비스 기능을 API로 공개하는 플랫폼**을 말합니다.

카카오 로그인 API, 네이버 지도 API, 토스페이먼츠 결제 API 같은 것들이 전형적인 예시입니다.

---

## 내부 서비스 API vs Open API — 뭐가 다른가

|구분|내부 서비스 API|Open API|
|---|---|---|
|호출자|자사 팀|불특정 외부 개발자|
|인증|사내 SSO, 서비스 계정|API Key, OAuth 2.0, JWT|
|버전 관리|배포 맞춰 일괄 변경 가능|**하위 호환 필수**, 폐기 정책 필요|
|트래픽 제어|대략 예측 가능|예측 불가, 폭발적 증가 가능|
|문서화|팀 내 공유로 충분|Developer Portal, SDK, 샘플 코드|
|장애 영향|내부 서비스만|외부 고객사 서비스까지 연쇄|
|SLA|느슨|계약 수준의 가용성 보장|

## "개발/운영 경험"에서 실제로 보는 역량

채용 담당자가 이 표현으로 검증하려는 건 보통 아래입니다.

**1. API Gateway / 인증 레이어 설계 경험**

- API Key 발급/만료/재발급 생명주기
- OAuth 2.0 Client Credentials / Authorization Code Flow 구현
- Spring Security + JWT 필터 체인 직접 구성

**2. Rate Limiting / Quota 관리**

- 호출자별 초당/일별 요청 제한
- Redis 기반 슬라이딩 윈도우 카운터 같은 구현 경험
- 429 Too Many Requests 처리 정책

**3. 버전 관리와 하위 호환**

- `/v1/`, `/v2/` URI 버전 전략 또는 헤더 버전 전략
- Deprecated 필드를 언제 어떻게 제거하는가
- Breaking change 없이 스펙을 확장하는 설계 감각

**4. 외부 노출 수준의 문서화/계약**

- OpenAPI Spec (Swagger) 작성 경험
- 에러 코드 체계 설계 (RFC 7807 Problem Details 등)

**5. 운영 관측성**

- 외부 호출자별 트래픽 모니터링
- SLA 위반 감지, 알림 체계
- AWS API Gateway + CloudWatch, 또는 자체 구축 경험

---

## 한 줄 요약

> **"우리 서비스를 외부에 API로 팔아본 적 있냐"** — 설계부터 인증, 트래픽 제어, 버전 관리, 운영까지 전 사이클 경험을 보는 것

내부 CRUD API 만든 경험과는 **책임 범위와 설계 난이도**가 다르다는 걸 전제한 표현입니다.
