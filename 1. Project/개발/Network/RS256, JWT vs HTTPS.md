# JWT 비대칭키 vs HTTPS 비대칭키 — 목적과 동작의 차이

---

## 핵심 차이 한 줄 요약

```
HTTPS 비대칭키: "나(서버)가 진짜임을 증명" — 신원 인증
JWT 비대칭키:  "내가 만든 토큰임을 증명" — 데이터 무결성 + 발급자 인증
```

같은 RSA 알고리즘을 쓰더라도 **무엇을 보호하려는가**가 완전히 다르다.

---

## 1. HTTPS의 비대칭키 — TLS Handshake

**목적: 클라이언트가 "이 서버가 진짜 google.com인가"를 검증**

```
클라이언트                          서버 (google.com)
    │                                    │
    │──── ClientHello (TLS 버전, 암호 스위트) ──▶│
    │                                    │
    │◀─── ServerHello + 인증서(공개키 포함) ────│
    │                                    │
    │  [인증서 검증]                       │
    │  CA 서명 확인                        │
    │  도메인 일치 확인                    │
    │  유효기간 확인                       │
    │                                    │
    │──── PreMasterSecret                 │
    │     (서버 공개키로 암호화) ──────────▶│
    │                                    │  서버 개인키로 복호화
    │◀═══════ 이후 대칭키(세션키)로 암호화 통신 ══▶│
```

**비대칭키의 역할이 여기서 끝난다.** 실제 데이터 전송은 이 과정에서 교환한 **대칭키(세션키)** 로 한다. 
RSA 연산이 너무 느리기 때문이다.

**핵심: 신뢰의 근거가 CA(Certificate Authority)다.**

```
브라우저/OS에 내장된 Root CA 목록
    ↓
Root CA가 서명한 Intermediate CA
    ↓
Intermediate CA가 서명한 서버 인증서
    ↓
"이 공개키는 진짜 google.com 것이다"를 CA 체인이 보증
```

```
누가 개인키를 갖고 있는가: 서버 (google.com)
누가 공개키를 검증하는가: CA 체인 (브라우저가 신뢰하는 제3자)
비대칭키 쓰임:           서버 신원 증명 + 세션키 교환
```

> 📎 **근거:** RFC 8446 — _"The Transport Layer Security (TLS) Protocol Version 1.3"_, §4.4 Authentication Messages

## 2. 소셜 로그인 JWT의 비대칭키 — RS256

**목적: 리소스 서버가 "이 토큰을 Google이 만들었는가"를 검증**

```
사용자          내 서버          Google Auth 서버
  │                │                               │
  │─ 로그인 요청 ──▶│                   │
  │                │── 인증 요청 ───────▶│
  │                │                                        │ [개인키로 JWT 서명]
  │                │◀── JWT 발급 ────────│
  │                │                                          │
  │                │ [Google 공개키로 서명 검증]
  │                │ (HTTPS로 공개키 받아옴)
  │                │                                        │
  │◀── 로그인 완료 ─│                   │
```

**JWT 서명 검증 과정:**

```
JWT 구조:
eyJhbGciOiJSUzI1NiJ9          ← Header  (Base64)
.eyJzdWIiOiIxMjM0In0          ← Payload (Base64)
.SflKxwRJSMeKKF2QT4fwpMeJf36P ← Signature

서명 생성 (Google 서버):
SHA256(header.payload) → digest
RSA_Sign(Google_개인키, digest) → Signature



서명 검증 (내 서버):
RSA_Verify(Google_공개키, Signature) == SHA256(header.payload) ?
→ true: Google이 만든 토큰
→ false: 위조된 토큰
```

**핵심: 신뢰의 근거가 Google의 공개키 자체다.**

```
누가 개인키를 갖고 있는가: Google Auth 서버
누가 공개키를 갖고 있는가: 누구나 (공개)
비대칭키 쓰임:           토큰 발급자 인증 + 데이터 무결성
```

Google은 공개키를 JWKS(JSON Web Key Set) 엔드포인트로 공개한다:

```
GET https://www.googleapis.com/oauth2/v3/certs

{
  "keys": [{
    "kty": "RSA",
    "kid": "abc123",   ← Key ID (JWT 헤더의 kid와 매칭)
    "n": "...",        ← RSA 공수 (모듈러스)
    "e": "AQAB"        ← 공개 지수
  }]
}
```

```java
// Spring Security에서 검증하는 흐름
@Bean
public JwtDecoder jwtDecoder() {
    return JwtDecoders.fromIssuerLocation("https://accounts.google.com");
    // 내부적으로:
    // 1. /.well-known/openid-configuration 조회
    // 2. jwks_uri 획득
    // 3. 공개키 목록 다운로드 + 캐싱
    // 4. JWT 수신 시 kid로 키 매칭 → 서명 검증
}
```

> 📎 **근거:** RFC 7517 — _"JSON Web Key (JWK)"_, OpenID Connect Core 1.0 §10.1 — _"Signing"_

## 3. 두 방식의 구조적 차이

```
┌─────────────────┬──────────────────────┬──────────────────────┐
│                 │ HTTPS TLS            │ JWT RS256            │
├─────────────────┼──────────────────────┼──────────────────────┤
│ 보호 대상        │ 전송 채널             │ 토큰 데이터           │
│ 개인키 소유자    │ 서버 (google.com)     │ 발급자 (Google Auth) │
│ 공개키 신뢰 근거 │ CA 체인 (제3자)       │ JWKS 엔드포인트 직접  │
│ 비대칭키 수명    │ Handshake 동안만      │ 키 교체 전까지        │
│ 이후 암호화      │ 대칭키로 교체         │ 없음 (서명만, 암호화X)│
│ 목적            │ 채널 기밀성 + 서버인증 │ 발급자 인증 + 무결성  │
└─────────────────┴──────────────────────┴──────────────────────┘
```

**JWT는 암호화가 아니다 — 가장 흔한 오해:**

```java
String payload = new String(Base64.decode(jwt.split("\\.")[1]));
// Base64는 인코딩이지 암호화가 아님
// Payload 내용은 누구나 볼 수 있다

// HTTPS가 전송 구간을 암호화하기 때문에
// JWT를 HTTPS 위에서 쓰면 전송 중 노출은 막힘
// 하지만 JWT 자체는 평문
```

**둘은 계층이 다르다:**

```
[ JWT (토큰 레이어) — 발급자 인증, 무결성 ]
         위에서 동작
[ HTTPS (전송 레이어) — 채널 암호화, 서버 인증 ]
```

JWT를 HTTPS 없이 쓰면 토큰이 네트워크에서 그대로 노출된다. HTTPS를 JWT 없이 쓰면 채널은 안전하지만 토큰 위조를 막을 수 없다. **둘은 대체재가 아니라 역할이 다른 보완재다.**

## 4. 소셜 로그인 전체 흐름에서 두 비대칭키가 동시에 쓰이는 위치

```
사용자 브라우저
    │
    │ HTTPS (TLS 비대칭키 — 채널 암호화)
    │ ══════════════════════════════════════════════
    ▼
내 서버 (Spring)
    │
    │ HTTPS (TLS 비대칭키 — 채널 암호화)
    │ ══════════════════════════════════════════════
    ▼
Google Auth 서버
    │
    │ JWT 발급 (RS256 비대칭키 — 토큰 서명)
    │ ──────────────────────────────────────────────
    ▼
내 서버: JWT 서명 검증 (Google 공개키로)
    │
    │ 자체 JWT 발급 (내 서버의 개인키로 서명)
    │ ──────────────────────────────────────────────
    ▼
사용자 브라우저: 내 서버 JWT 저장
```

Google의 JWT는 "Google이 이 사용자를 인증했다"는 증명. 내 서버의 JWT는 "내 서버가 이 사용자에게 접근 권한을 부여했다"는 증명. **두 JWT 모두 HTTPS 위에서 전송된다 — 계층이 다르다.**

## 이어지는 개념

| 순서  | 개념                         | 이유                                                                  |
| --- | -------------------------- | ------------------------------------------------------------------- |
| 1   | **OAuth 2.0 / OIDC 전체 흐름** | 소셜 로그인에서 JWT가 어떤 역할인지(ID Token vs Access Token) — RS256이 왜 필요한지의 맥락 |
| 2   | **TLS 1.3 Handshake 상세**   | HTTPS 비대칭키가 세션키로 교체되는 정확한 과정 — 왜 대칭키로 전환하는지 성능 근거까지                 |
| 3   | **JWT 키 로테이션 전략**          | JWKS의 `kid` 필드가 왜 있는지 — 개인키가 노출됐을 때 어떻게 대응하는가                       |

---



```
[클라이언트가 검증할 때]
받은 인증서에서 서명(XYZ)을 꺼냄
CA_공개키로 복호화(XYZ) = ABC123  ← 원래 해시값 복원

인증서 내용을 직접 Hash = ABC123  ← 직접 계산

둘이 같으면 → CA가 서명한 게 맞다
둘이 다르면 → 위조됨
```

---

## 핵심 구조

```
서명 = 개인키로 암호화한 해시값
검증 = 공개키로 복호화한 해시값 == 직접 계산한 해시값
```

내용을 숨기는 게 아니라 **내용이 변조되지 않았고, CA가 만든 게 맞는지** 를 확인하는 것이다.

---

## 정리

```
암호화 목적:  공개키 암호화 → 개인키 복호화  (내용을 숨김)
서명 목적:    개인키 암호화 → 공개키 복호화  (만든 사람을 증명)
                                    ↑
                              해독이 맞긴 한데
                              꺼낸 값을 "읽는" 게 아니라
                              "해시 비교"에 쓰는 것
```

그래서 인증서 내용(공개키, 도메인 등)은 누구나 볼 수 있고, CA 공개키로 하는 건 **그 내용이 CA가 보증한 것과 동일한지 확인** 하는 것이다.
