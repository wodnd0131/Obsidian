# 자유 정리
운영체제는 자원할당자이자 제어자
운영체제는 커널, 미들웨어 프레임워크(모바일), 시스템 프로그램(커널에 속하지 않음)
공유 메모리 사이의 연결 파이프인 `시스템 버스` (통신 경로)가 존재하여, 각 장치의 컨트롤러에 연결됨( 컨트롤러 연결 대상은 둘둘 이상일 수 있음)
장치 컨트롤러는 일부 로컬 버퍼 저장소,특수 목적 레지스터 집합 유지,
제어하는 주변 장치와 로컬 버퍼 저장소 간 데이터 전송 관리

OS에는 장치 컨트롤러마다 장치 드라이버가 존재
장치 드라이버는 OS에게 제공되는 장치의 인터페이스

CPU와 장치 컨트롤러는 병렬로, 메모리 사이클를 두고 경쟁
메모리 컨트롤러는 메모리에 대한 액세스를 동기화한다.

인터럽트 : CPU의 조치가 필요한 이벤트를 경고함

) 입출력 작업 수행 예시
입출력 작업을 위해 장치 드라이버는 장치 컨트롤러의 레지스터 값 적재
컨트롤러는 [[장치 레지스터|레지스터]] 내용 검사하여 수행할 작업 결정
컨트롤러는 장치에서 로컬 버퍼로 데이터를 전송
데이터 전송이 완료되면 장치컨트롤러는 드라이버에게 작업 완료를 알림
읽기 작업의 경우, 데이터와 데이터를 읽던 포인터를 반환하여 OS에게 제어 넘김
컨트롤러가 장치 드라이버에게 작업 완료를 알리는 것은 **인터럽트**가 담당

### 인터럽트
하드웨어는 시스템 버스를 통해 CPU에게 인터럽트를 보낼 수 있다.
인터럽트는 OS와 하드웨어 상호작용 방식의 핵심

CPU가 인터럽트되면, 하던 작업을 중단하고 고정된 위치로 실행을 옮긴다.
고정된 위치는 일반적으로 인터럽트를 위한 서비스 루틴이 위치한 시작 주소를 가지고있다.
그리고 인터럽트 서비스 루틴이 실행된다.
서비스 루틴이 완료되면 CPU는 인터럽트 되었던 연산을 재개한다. (이러한 연산의 시간 일정 =타임 라인)

인터렙트 서비스 루틴의 두 방식 
- 방식 A (직관적 방법): 인터럽트가 오면 일단 범용 루틴이 먼저 호출되어 "어떤 장치에서 온 인터럽트인지" 조사한 다음, 그 결과에 따라 고유 핸들러를 호출. → 2단계 경로
- 방식 B (인터럽트 벡터 테이블): 장치마다 고유 번호가 있으니, 그 번호로 테이블을 바로 색인해서 해당 핸들러 주소로 곧장 점프. → 1단계 경로

인터럽트 구조는 또한 인터럽트된 모든정보를 저장해야 인터럽트를 처리한 후 이 정보를 복원할 수 있다.
인터럽트 루틴이 처리기의 상태를 변경할 필요가 있다면, 인터럽트 루틴은 반드시 명시적으로 형재의 상태를 저장해야하며, 복귀 주소를 프로그램 카운터에 적재하고, 인터럽트에 의해 중단되었던 연산이 인터럽트가 발생되지 않았던 것처럼 다시 시작된다.

인터럽트가 호출자(원래 실행 중이던 프로세스) 입장에서 "아무 일도 없었던 것처럼" 보여야 한다는 요구사항

- 인터럽트 서비스 루틴(ISR): 인터럽트 발생 시 CPU가 실행하는 코드. "핸들러"와 동의어로 쓰임
- 범용 루틴(generic routine): 어떤 장치의 인터럽트인지 조사한 뒤 고유 핸들러를 호출하는 중간 단계 (방식 A)
- 고유 핸들러(device-specific handler): 실제 그 장치를 위한 처리 코드. 두 방식 모두에서 최종적으로 실행됨
- 인터럽트 벡터(Interrupt Vector): 핸들러 주소들을 모아둔 포인터 배열. 보통 하위 메모리에 저장됨
- 장치 번호로 색인: 인터럽트 벡터에서 어느 핸들러로 갈지 결정하는 키 값 
  — 범용 루틴의 "조사" 단계를 생략시켜주는 장치
- 상태 저장/복원(save/restore): 
  ISR이 레지스터 등을 변경해야 할 경우, 진입 시 저장하고 종료 시 복원해야 함 
  => 인터럽트가 호출자(원래 실행 중이던 프로세스) 입장에서 "아무 일도 없었던 것처럼" 보여야 함
  = ISR에 진입하여 변경되는 레지스터가 인터럽트된 CPU의 연산을 방해하지 않도록 이전 상태 저장 및 복원함
  => 미니 컨텍스트 스위칭
- 프로그램 카운터(PC) 복귀: 인터럽트 처리 종료 후 원래 실행 흐름으로 투명하게 복귀하기 위한 핵심 장치
- 타임라인(timeline) 관점: "정상 실행 → 인터럽트 발생 → ISR 실행 → 원래 실행 재개"라는 시간 순서로 전체 그림을 보는 게 이해에 도움
- 인터럽트 요청 라인 = 명령어 실행 완료마다 CPU가 이 선을 감지, 컨트롤러가 인터럽트 요청을 보내는 곳

**인터럽트 매커니즘**
인터럽트는 비동기 이벤트 처리를 위해 사용
	raise → catch → dispatch → clear
	장치 컨트롤러가 인터럽트 요청 라인에 신호를 보내 인터럽트를 발생(raise)
	CPU가 인터럽트를 포착(catch)
	인터럽트 핸들러(=ISR)로 디스패치(dispatch / 제어를 넘김)
	핸들러가 장치를 서비스하고 인터럽트를 지운다(clear)

정교한 인터럽트 처리가 요구됨 **-> CPU 및 인터럽트 컨트롤러 하드웨어에 의해 제공**
- 중요한 처리 중 인터럽트 처리 연기가 가능해야한다. → **벡터 방식 인터럽트** + **인터럽트 체인**이 그 해결책
- ISR을 효율적으로 디스패치할 방법 → **인터럽트 우선순위 레벨**(마스크 불가능/마스킹 가능 라인)이 그 해결책
- 인터럽트의 우선순위를 구별할 다단계 인터럽트 
  → **마스킹 가능 인터럽트 라인**으로 구현, 
  마스킹 가능 라인을 CPU가 일시적으로 꺼두면, 그 인터럽트는 처리가 "연기"되는 셈이거든요.


백터 방식 인터럽트 기법의 목적은 모든 인터럽트 소스의 검색을 줄이기 위함 (속도 개선)
하지만, 벡터의 주소 수보다 많은 장치가 있다
인터렙트 체인을 통해 해결
-> 인터럽트 벡터의 각 원소가 핸들러 리스트의 헤드를 가리킨다 
(큰 구조의 인터럽트 테이블 오버헤드와 하나의 인터럽트 핸들러로 디스패치하는 비효율성의 절충안)

인터럽트 기법은 인터럽트 우선순위 레벨을 구현한다.
마스크 불가능 인터럽트 라인 - 복구할 수 없는 메모리 오류 등의 이벤트를 위해 예약됨
마스킹 가능 인터럽트 라인 
- CPU가 어떤 중요한 명령어 시퀀스(예: 자료구조를 일관성 있게 갱신하는 도중 등)를 실행하기 전에, 이 마스킹 가능 라인을 의도적으로 꺼서(mask off) 인터럽트가 들어오지 못하게 막을 수 있다는 거예요. 그 작업이 끝나면 다시 켜서(unmask) 정상적으로 인터럽트를 받습니다.
마스킹 정보를 벡터 테이블로 관리 -> 마스킹 하지 않고도  우선순위 비교

>지금까지 다룬 건 모두 하드웨어(장치 컨트롤러)가 발생시키는 인터럽트였습니다. 
>반면 0으로 나누기나 잘못된 메모리 접근, 시스템 콜 요청처럼 소프트웨어(실행 중인 명령어 자체)가 일으키는 인터럽트를 트랩(trap) 또는 예외(exception)라고 따로 구분합니다.
>처리되는 메커니즘(상태저장-디스패치-처리-복원)은 동일하지만, "누가 발생시켰는가"가 다르다

### 저장 장치 구조
부트스트랩 프로그램 - 전원 켤때 OS를 적재 = bios
RAM은 휘바성이라 부트스트랩 유지 용도론 사용 x

컴퓨터는 전기적으로 소거 가능한 프로그램 가능 읽기 전용 메모리(EEPROM) 
기타 형태의 펌웨어(쓰기 작업이 자주 발생하지 않고 비 휘발성인 저장장치)
```
ROM(마스크 ROM) — 제조 시점에 내용이 고정되어 출고 후엔 전혀 변경 불가  
→ PROM — 출고 후 한 번은 사용자가 직접 굽듯이(burn) 기록 가능, 그 이후엔 불가  
→ EPROM — 자외선(UV)을 쪼여서 지우고 재기록 가능  
→ EEPROM — 전기적으로(자외선 없이) 지우고 재기록 가능, 다만 느리고 자주 못 함  
→ 플래시 메모리 — EEPROM 계열을 블록 단위 삭제로 더 빠르게 개량한 버전 (USB, SSD 등에 사용)
```
(Rom에 포함되는 개념)EEPROM은 변경할 수는 있지만 자주 변경할 수 없고, 속도가 느리므로 주로 정적 프로그램과 데이터를 다룬다. ex 장치의 일련번호, 하드웨어 정보

펌웨어 단계(BIOS/UEFI) — 하드웨어 초기 점검(POST), 그다음 단계 프로그램을 어디서 찾을지 결정  
부트로더 단계(GRUB, Windows Boot Manager 등) — 실제 OS 커널 파일을 디스크에서 찾아 메모리에 적재  
커널 실행 단계 — 적재된 커널에게 제어가 완전히 넘어가면서 OS 본격 시작

모든 형태의 메모리는 바이트의 배열로 제공, 각 바이트는 자신의 주소를 가지고 있다. (바이트 단위 주소지정)

| 주소(Address)     | 0x0000   | 0x0001   | 0x0002   | 0x0003   | 0x0004   | ... |
| --------------- | -------- | -------- | -------- | -------- | -------- | --- |
| 값(1byte = 8bit) | 01001010 | 11100001 | 00000000 | 10101100 | 00011111 | ... |

실제로 CPU가 한 번에 가져오는 단위는 1바이트가 아니라 워드(Word) 단위인 경우가 많습니다. 예를 들어 64비트 CPU라면 워드 크기가 8바이트(64비트)인데, 이때는 위 표를 8칸씩 묶어서 한 번에 가져오는 식으로 동작

정리하면 바이트는 데이터 1칸의 크기 단위, 주소는 그 칸의 위치를 나타내는 정수값이고, 그 정수값 자체를 표현하는 데 필요한 비트 수(32비트, 64비트 등)가 시스템의 주소 체계를 결정한다고 보시면 됩니다.

|구분|무엇을 나타내는가|64비트 시스템 기준|
|---|---|---|
|1바이트|메모리 한 칸의 크기|8비트, 고정|
|주소 값의 크기|주소 자체를 표현하는 비트 수|보통 8바이트(64비트)|
|워드|CPU가 한 번에 처리하는 데이터 묶음|보통 8바이트 = 연속된 8칸|
