
레지스터는 제어 명령 저장을, 로컬 버퍼는 실 데이터의 저장을 담당

# 레지스터
여기서 말하는 레지스터는 CPU 레지스터가 아니라, **장치 컨트롤러(I/O 컨트롤러) 내부에 있는 소형 저장 공간**을 말합니다. 디스크 컨트롤러, 키보드 컨트롤러처럼 각 하드웨어 장치를 관장하는 컨트롤러 칩 안에 박혀있는 작은 레지스터들이에요.

보통 장치 컨트롤러는 몇 개의 레지스터를 갖고 있는데, 역할이 대략 이렇게 나뉩니다.

- 데이터 레지스터(data-in/data-out): 실제로 주고받을 데이터가 잠깐 머무는 공간
- 명령 레지스터(command register): "읽어라", "써라" 같은 명령어를 적어넣는 공간
- 상태 레지스터(status register): 컨트롤러가 지금 작업 중인지, 끝났는지, 에러가 났는지를 비트로 표시하는 공간

본문에서 "장치 드라이버가 레지스터 값을 적재한다"는 건, 드라이버(소프트웨어)가 이 명령/데이터 레지스터에 원하는 작업 정보를 써넣는다는 뜻이고, "컨트롤러가 레지스터 내용을 검사한다"는 건 컨트롤러(하드웨어)가 그 레지스터 값을 읽어서 "아, 읽기 작업을 하라는 거구나" 하고 해석한다는 뜻이에요. 작업이 끝나면 상태 레지스터 값을 바꿔두거나, 인터럽트를 걸어서 CPU/드라이버에게 "다 했다"고 알려주는 거죠.

비유하자면 CPU 레지스터는 CPU가 연산할 때 쓰는 임시 작업대이고, 장치 컨트롤러 레지스터는 CPU와 그 장치 사이의 "우편함" 같은 거예요. CPU(혹은 드라이버)가 우편함에 명령을 넣어두면, 컨트롤러가 그걸 꺼내서 실제 하드웨어 동작(디스크 헤드 이동, 데이터 전송 등)을 수행하는 구조입니다.

참고로 CPU가 이 레지스터에 접근하는 방식은 두 가지가 있는데, 메모리 주소 공간 일부를 레지스터에 매핑해서 일반 메모리처럼 읽고 쓰는 메모리 매핑 I/O(memory-mapped I/O) 방식과, 별도의 I/O 주소 공간과 전용 명령어를 쓰는 포트 매핑 I/O(port-mapped I/O) 방식이 있어요. 1장/12장에서 이 부분이 더 자세히 나올 가능성이 높으니, 거기서 한 번 더 짚고 넘어가면 좋습니다.

#  그럼 로컬 버퍼는?
여기서 로컬 버퍼는 장치 컨트롤러 안에 내장된 임시 저장 공간을 말합니다. 레지스터보다는 좀 더 큰 단위의 데이터(보통 한 섹터나 한 블록 단위)를 잠깐 담아두는 작은 메모리 칩이라고 보면 됩니다.

왜 이게 필요한지 생각해보면, 디스크 헤드가 플래터에서 데이터를 읽어내는 속도와, 그 데이터를 시스템 버스를 통해 메인 메모리로 옮기는 속도가 서로 다르기 때문이에요. 디스크가 비트 단위로 한 글자씩 천천히 읽어내는 동안 CPU나 버스가 매번 기다릴 수는 없으니, 컨트롤러가 일단 자기 내부의 로컬 버퍼에 데이터를 다 모아뒀다가, 한 블록이 통째로 채워지면 그제서야 버스를 통해 메인 메모리로 한 번에 쏟아내는 방식을 씁니다.

본문 흐름으로 다시 보면 이렇습니다.

1. 디스크 장치에서 로컬 버퍼로 데이터가 채워짐 (장치 → 컨트롤러 내부)
2. 로컬 버퍼가 다 차면, 컨트롤러가 드라이버에게 "전송 완료" 인터럽트를 보냄
3. 그 다음 단계(보통 DMA)로 로컬 버퍼의 내용이 메인 메모리로 옮겨짐

즉 로컬 버퍼는 "장치 ↔ 컨트롤러" 사이의 속도 차이를 흡수하는 중간 정거장이고, 메인 메모리는 그 다음 단계인 "컨트롤러 ↔ 시스템" 사이에서 최종적으로 데이터가 도착하는 곳이라고 구분하면 헷갈리지 않을 거예요.

---
SSE 비유는 정확합니다. 컨트롤러가 능동적으로 "끝났다"는 신호를 보내는 거니까, 클라이언트가 계속 물어보는 폴링이 아니라 서버가 푸시해주는 SSE/웹소켓 방식과 같은 발상이에요.

다만 "컨트롤러가 알리는 것 = CPU 작업"이냐는 질문은 한 단계 나눠서 봐야 합니다.

정리하면 신호 자체는 하드웨어 이벤트, 그 신호를 받은 다음의 처리(드라이버 코드 실행)는 CPU 작업이라고 나눠서 이해하시면 됩니다.

두 번째 질문, 드라이버의 위치는 정정이 필요해요. 장치 드라이버는 보통 **커널에 포함됩니다.** 커널 모드에서 동작하고, 리눅스라면 .ko 파일처럼 커널 모듈로 컴파일되어 커널 주소공간에 로드돼요. 메모해두신 "시스템 프로그램(커널에 속하지 않음)" 카테고리는 컴파일러, 셸, 텍스트 에디터처럼 사용자 모드에서 동작하면서 OS와 함께 배포는 되지만 커널 바이너리/주소공간에는 속하지 않는 프로그램들을 말하는 거고, 드라이버는 그쪽이 아니라 커널 쪽입니다.

다만 예외적으로 마이크로커널(microkernel) 구조에서는 드라이버를 의도적으로 사용자 영역으로 빼기도 하는데, 이건 1장/2장에서 본 "모놀리식 vs 마이크로커널" 구조 차이와 연결되는 부분이니 그쪽 키워드 정리할 때 다시 짚어보면 좋을 것 같아요.
