## 1. Tomcat Worker 풀 (HTTP 요청 처리용)

```yaml
server:
  tomcat:
    threads:
      max: 200                # Worker 최대 개수 (기본 200)
      min-spare: 10           # 최소 유지 개수 (기본 10)
    max-connections: 8192     # 동시 수용 가능 연결 수 (기본 8192)
    accept-count: 100         # accept queue 크기 (기본 100)
    connection-timeout: 20000 # 연결 후 첫 바이트까지 대기 (ms, 기본 20000)
```

각 항목의 의미를 이전 토픽들과 연결해서 보면:

|설정|의미|어디서 작동?|
|---|---|---|
|`threads.max`|Worker 스레드 최대 개수|`http-nio-8080-exec-*` 풀 상한|
|`threads.min-spare`|항상 유지할 최소 Worker|갑작스러운 트래픽 대비|
|`max-connections`|Poller에 등록 가능한 소켓 최대|이걸 넘으면 accept 안 함|
|`accept-count`|OS accept queue 크기 (listen backlog)|이것마저 넘치면 연결 거부|
|`connection-timeout`|연결 후 데이터 안 오면 끊음|Slow HTTP 공격 방어|

### 큐가 막히는 순서

```
1. 클라이언트가 연결 시도
   ↓
2. OS accept queue (accept-count=100)에 들어감
   ↓
3. Acceptor가 accept() → max-connections=8192 체크
   ↓ (여기 넘으면 더 이상 받지 않음)
4. Worker 풀(threads.max=200)에서 처리
   ↓ (여기 다 차면 큐에서 대기)
5. 처리 후 응답
```

200개 다 점유되면 들어온 요청은 **Worker 풀 내부 큐에서 대기**하고, 그것마저 가득 차면 **accept-count 큐에서 대기**, 그것도 넘치면 **OS가 연결 거부**한다.

## 2. @Async 풀 (Spring 백그라운드 작업)

```yaml
spring:
  task:
    execution:
      pool:
        core-size: 8           # 기본 유지 스레드 (기본 8)
        max-size: 2147483647   # 최대 스레드 (기본 Integer.MAX_VALUE)
        queue-capacity: 2147483647  # 대기 큐 크기 (기본 Integer.MAX_VALUE)
        keep-alive: 60s        # idle 스레드 유지 시간
      thread-name-prefix: task-  # 스레드 이름 (기본 "task-")
      shutdown:
        await-termination: true
        await-termination-period: 30s
```

### 주의할 함정: 기본값이 사실상 무한대

`max-size`와 `queue-capacity`가 `Integer.MAX_VALUE`(약 21억)인 게 핵심이다. 이게 의미하는 바:

```
@Async 작업 폭주 시나리오 (기본 설정)
  ↓
core-size=8개 가득 참
  ↓
queue-capacity=21억 → 큐에 무한정 쌓임
  ↓
max-size까지 가지도 못함. OOM 발생까지 큐만 쌓임
  ↓
JVM 죽음
```

**실무에서는 명시적으로 줄여야 한다.**

```yaml
spring:
  task:
    execution:
      pool:
        core-size: 8
        max-size: 16
        queue-capacity: 100   # 큐가 가득 차면 RejectedExecutionException
```

## 3. @Scheduled 풀 (Spring 스케줄링)

```yaml
spring:
  task:
    scheduling:
      pool:
        size: 1               # 스케줄러 스레드 수 (기본 1)
      thread-name-prefix: scheduling-
      shutdown:
        await-termination: true
        await-termination-period: 30s
```

### 함정: 기본값 1개

스케줄러가 1개면 **`@Scheduled` 메서드가 여러 개일 때 직렬 실행된다.**

```java
@Scheduled(cron = "0 0 * * * *")  // 매시 정각
public void taskA() { /* 10분 걸림 */ }

@Scheduled(cron = "0 0 * * * *")  // 매시 정각 (같은 시각)
public void taskB() { /* 빨리 끝나야 함 */ }
```

`pool.size=1`이면 taskA가 끝날 때까지 taskB는 대기. 동시에 돌아야 하는 스케줄이 여럿이면 `size`를 늘려야 한다.

## 4. HikariCP (DB 커넥션 풀)

엄밀히 말하면 스레드 풀은 아니지만, **풀이라는 점에서 같이 묶어 보는 게 실무적**이다.

```yaml
spring:
  datasource:
    hikari:
      maximum-pool-size: 10         # 최대 커넥션 수 (기본 10)
      minimum-idle: 10              # 최소 유지 커넥션 (기본 maximum-pool-size와 동일)
      connection-timeout: 30000     # 커넥션 얻기까지 대기 (ms, 기본 30000)
      idle-timeout: 600000          # idle 커넥션 유지 시간 (기본 10분)
      max-lifetime: 1800000         # 커넥션 최대 수명 (기본 30분)
      leak-detection-threshold: 0   # 누수 감지 (0=비활성, 권장 60000)
```

### Worker 풀과의 관계 — 실무 핵심

**`server.tomcat.threads.max`(200)와 `spring.datasource.hikari.maximum-pool-size`(10)의 관계를 이해해야 한다.**

```
Worker 200개 vs DB Connection 10개

극단 시나리오:
  HTTP 요청 200개가 동시에 도착
  → 200개 Worker가 다 점유
  → 각 Worker가 @Transactional 진입 = DB Connection 필요
  → 10개만 받음. 나머지 190개는 connection-timeout(30초) 대기
  → 30초 후 SQLException 발생
```

**핵심 원칙**: Worker 수가 DB Connection 수보다 훨씬 크면 **DB가 병목**이 된다.

그래서 일반적 권장:

- Worker는 트래픽 기준으로 크게
- DB Connection은 **DB 서버가 감당 가능한 수준**으로 (DB는 비싸다)
- 부족분은 큐잉으로 흡수하되 timeout 모니터링

HikariCP 저자의 추천 공식이 있다.

```
connections = ((core_count * 2) + effective_spindle_count)
```

CPU 8코어 SSD 시스템이면 약 `8*2 + 1 = 17`개. 무조건 크게 잡지 않는다.

## 5. Servlet 비동기 (Callable/DeferredResult) — 별개로 설정 가능

이건 Spring Boot가 별개 설정을 지원하지 않아서, **기본적으로 위 `spring.task.execution` 풀과 공유한다.** 분리하려면 Java 설정이 필요하다.

```java
@Configuration
public class AsyncMvcConfig implements WebMvcConfigurer {
    @Override
    public void configureAsyncSupport(AsyncSupportConfigurer configurer) {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(20);
        executor.setThreadNamePrefix("mvc-async-");
        executor.initialize();
        configurer.setTaskExecutor(executor);
        configurer.setDefaultTimeout(30_000);
    }
}
```

## 6. 공식 근거

**Spring Boot 공식 — Application Properties**

> `server.tomcat.threads.max` — Maximum amount of worker threads. Doesn't have an effect if virtual threads are enabled. (Worker 스레드 최대 개수. 가상 스레드 활성화 시 효과 없음.)
> 
> `spring.task.execution.pool.max-size` — Maximum allowed number of threads. If tasks are filling up the queue, the pool can expand up to that size to accommodate the load. (스레드 최대 수. 큐가 차면 이 크기까지 늘어남.)
> 
> `spring.task.scheduling.pool.size` — Maximum allowed number of threads. (최대 스레드 수.)

→ 출처: [Spring Boot Application Properties — Server / Task properties](https://docs.spring.io/spring-boot/appendix/application-properties/index.html)

**HikariCP 공식 — Pool Sizing**

> "About Pool Sizing ... A formula which has held up pretty well across a lot of benchmarks for years is that for maximizing throughput the number of active connections should be close to ((core_count * 2) + effective_spindle_count)."
> 
> (수년간 많은 벤치마크에서 잘 들어맞은 공식: 처리량 최대화를 위한 활성 커넥션 수는 ((코어 수 * 2) + 디스크 스핀들 수)에 가까워야 한다.)

→ 출처: [HikariCP GitHub Wiki — About Pool Sizing](https://github.com/brettwooldridge/HikariCP/wiki/About-Pool-Sizing)

## 7. 실무 통합 예시

production 환경 yml 한 벌:

```yaml
server:
  tomcat:
    threads:
      max: 200
      min-spare: 20
    max-connections: 8192
    accept-count: 100
    connection-timeout: 5s

spring:
  task:
    execution:
      pool:
        core-size: 8
        max-size: 16
        queue-capacity: 100
      thread-name-prefix: async-
    scheduling:
      pool:
        size: 4
      thread-name-prefix: scheduled-

  datasource:
    hikari:
      maximum-pool-size: 20
      minimum-idle: 20
      connection-timeout: 3000
      max-lifetime: 1800000
      leak-detection-threshold: 60000
```

이 설정이 시사하는 바:

- Worker는 200개로 트래픽 흡수
- DB Connection은 20개로 보수적 (DB 서버 보호)
- @Async는 16개 + 큐 100개 → 폭주해도 무한 적재 막음
- 스케줄은 4개로 동시 실행 보장

**각 풀이 yml에서 별개 키로 설정된다**는 게 핵심이다. 하나의 설정이 모든 풀에 영향을 주지 않는다. 이게 풀이 별개라는 가장 직관적인 증거이기도 하다.
