### 서블릿이란

서블릿은 **Java에서 HTTP 요청을 처리하는 표준 인터페이스**다.

```java
// 서블릿 스펙의 핵심 인터페이스
public interface Servlet {
    void init(ServletConfig config);      // 초기화
    void service(ServletRequest req,
                 ServletResponse res);    // 요청 처리
    void destroy();                       // 종료
}
```

Tomcat 입장에서 서블릿은 그냥 **"이 인터페이스를 구현한 객체"** 다. 
안에서 뭘 하는지 모른다. 요청 오면 `service()` 호출하는 게 전부다.

서블릿 스펙은 Sun Microsystems(현 Oracle)가 정의한 **Java EE 표준(JSR-369)** 이다. 
Tomcat은 이 스펙의 구현체다.

---

### 서블릿이 DispatcherServlet만 있는 건 아니다

원래 서블릿 스펙 시절엔 URL마다 서블릿을 등록했다.

```
/users   → UserServlet
/orders  → OrderServlet
/items   → ItemServlet
```

Spring MVC를 쓰면 이 구조가 바뀐다.

```
/*  (모든 URL) → DispatcherServlet (하나만)
```

DispatcherServlet이 모든 요청을 받아서 내부에서 Controller로 분배한다. 그래서 **Spring Boot 애플리케이션에서는 사실상 DispatcherServlet 하나만 등록된다.**

하지만 기술적으로는 서블릿을 여러 개 등록할 수 있다.

```java
// Spring Boot에서 서블릿 직접 추가 등록도 가능
@Bean
public ServletRegistrationBean<MyServlet> myServlet() {
    return new ServletRegistrationBean<>(new MyServlet(), "/legacy/*");
}
// /legacy/* 경로는 MyServlet이 처리
// 나머지는 DispatcherServlet이 처리
```

레거시 시스템 연동이나 특수한 경우에 쓰인다.

---

### 서블릿의 위치를 계층으로 보면

```
[HTTP 요청]
    │
    ▼
Tomcat (서블릿 컨테이너)
    │  소켓, 스레드, HTTP 파싱
    │  "서블릿 스펙대로 service() 호출해줄게"
    ▼
서블릿 (Servlet 인터페이스 구현체)
    ├── DispatcherServlet  ← Spring MVC가 구현한 서블릿
    ├── MyServlet          ← 직접 만든 서블릿 (가능)
    └── ...

DispatcherServlet 내부에서
    → HandlerMapping
    → HandlerAdapter
    → Controller
    → ...
```

### 핵심 관계 정리

```
서블릿:            HTTP 요청 처리를 위한 Java 표준 인터페이스
서블릿 컨테이너:   서블릿의 생명주기를 관리하는 런타임 (Tomcat)
DispatcherServlet: Spring MVC가 서블릿 인터페이스를 구현한 구현체

"Tomcat이 서블릿 컨테이너"
= Tomcat이 서블릿 인터페이스를 구현한 객체들을 관리한다
= 그 중 하나가 DispatcherServlet일 뿐
```

Spring MVC 이전엔 서블릿이 여러 개였다. Spring MVC가 **"서블릿은 나(DispatcherServlet) 하나만 두고, 그 안에서 내가 다 분배할게"** 라는 설계를 택한 것이다. 서블릿 컨테이너가 DispatcherServlet 전용 컨테이너처럼 보이는 건 이 때문이다.
