## 1. 기획 정리

### 1.1 해결하려는 문제

기술력은 갖췄지만 기획 경험이 없는 주니어 팀은 "무엇을 만들지"조차 정하지 못해 프로젝트가 시작 단계에서 멈춘다. 이들이 현재 하는 행동은 대체로 세 가지로 나뉜다. 아는 기술을 연습하는 토이 프로젝트에 그치거나, 아이디어 회의를 하다 흐지부지되거나, 기획자를 구하러 팀빌딩 플랫폼을 전전한다.

### 1.2 타겟

**1차 타겟** — 스펙을 쌓으려는 주니어 개발·디자인 팀 (1~2년차, 3~5인)

이들을 1차 타겟으로 삼는 이유는 명확하다. 기획 방법론을 따로 배운 적이 없고, 기획자를 영입하기엔 애매한 규모이며, 포트폴리오라는 결과물에 대한 동기가 강하다.

**2차 타겟** — 창업 전 아이디어를 검증하려는 예비 창업팀

**데모 타겟** — 개발 연합동아리 회원. 이들이 가장 먼저 서비스를 접하는 집단이며, 동시에 1차 유저이기도 하다. 따라서 "동아리 회원이 실제로 쓸 것 같은가"가 아이디어 후보를 거르는 가장 강력한 기준이 된다.

### 1.3 경쟁 대비 차별점

|구분|기존 서비스|이 서비스|
|---|---|---|
|ValidatorAI, IdeaProof 등|아이디어 있음 → 검증|**아이디어 없음 → 발굴 → 검증**|
|ChatGPT|무엇을 물어야 할지 모름|선택지·질문을 먼저 제시|
|비사이드, 렛플|팀원 매칭|팀이 생긴 이후 단계|
|Notion 템플릿|무엇을 채울지 모름|토론하며 자연스럽게 채워짐|

해외에는 StartupMate, IdeaProof, ValidatorAI 같은 아이디어 검증 서비스가 있으나 모두 창업자·사업가 대상이며 "아이디어가 이미 있는" 단계를 전제한다. 국내 서비스는 대부분 팀빌딩·팀원 매칭에 집중되어 있다. **아이디어 발굴 단계부터, 주니어 개발팀을 대상으로, 한국어로 다루는 서비스는 현재 없다.**

핵심 차별점을 한 줄로 요약하면: **"무엇을 만들지 모르는 팀"을 위한 첫 번째 서비스.**

### 1.4 핵심 아이디어 — 키워드 도출 엔진

이 서비스의 진짜 엔진은 "좋은 키워드를 어떻게 던져주느냐"이다. AI에게 막연히 "아이디어 줘"라고 하면 맥락이 없어 쓸모없는 답이 나온다. 키워드는 맥락을 좁혀주는 트리거 역할을 한다.

```
너무 넓음     "아이디어 뭐 있어?"
                    ↓
키워드 투입   "실시간 / 취준 / 불편함"
                    ↓
구체화        "취준생이 코테 준비를 혼자 하면 지속이 안 됨"
                    ↓
아이디어      "코딩테스트 같이 푸는 앱"
```

좋은 키워드는 외부에서 주입하는 것이 아니라 **팀에게서 캐낸다.** 그 출발점은 두 가지다.

1. **기술 관심사 (이력서 요소)** — 실시간성, AI 연동, 지도 기반, 결제 등 팀원이 "써보고 싶은" 기술
2. **도메인 관심사** — 운동, 취준, 금융, 여행 등 팀원이 관심 있는 분야

이 둘의 교차점에서 아이디어 후보가 나온다.

```
실시간 × 취준    → 코딩테스트 같이 푸는 앱
AI × 이력서      → 포트폴리오 피드백 도구
지도 × 동아리    → 해커톤 팀원 근처 매칭
알림 × 취준      → 채용공고 맞춤 알리미
```

### 1.5 두 가지 필터

아이디어 후보는 반드시 두 필터를 통과해야 한다.

- **필터 ① 구현 가능성** — 1~2년차 팀이 실제로 만들 수 있는 범위. AI 기능이라도 단순 API 호출 수준, 실시간이라면 WebSocket 정도. 라이브 스트리밍처럼 무거운 건 제외.
- **필터 ② 데모 가능성** — 개발 연합동아리 회원이 데모에서 직접 체험하고 공감할 수 있는 것.

### 1.6 중복 아이디어 처리

비슷한 연합동아리에서 이미 나온 주제(예: 독서 루틴 앱)는 걸러야 한다. 다만 목적은 "제외"가 아니라 **"차별점 유도"**다. 이미 있는 주제라면 "이미 N개 있으니, 그것들이 못한 틈새를 파보라"는 방향으로 안내한다.

DB 전략은 단계적으로 간다.

- **초기** — 디스콰이엇·깃허브·데모데이 후기 등에서 주요 동아리 프로젝트 100~200개 수동 큐레이션
- **운영** — 유저 제보로 확장 ("우리 동아리에 이런 거 있어요")
- **나중** — 패턴이 파악되면 크롤링 자동화

이렇게 쌓인 "주니어 팀 사이드 프로젝트 아카이브"는 다른 서비스가 따라오기 어려운 데이터 자산이 된다.

### 1.7 진행 방식 — 디스코드 봇 + 웹

서비스는 음성 토론을 AI가 옆에서 정리해주는 방식이다. **AI가 주도하지 않고, 팀이 주도하며 AI는 서기 역할**을 한다. 인터뷰처럼 무겁지 않고 잡담처럼 가볍게 진행되는 것이 핵심 톤이다.

진입점은 웹, 토론은 디스코드, 결과 확인은 다시 웹이다.

```
[웹]      기술 관심사 선택
            ↓
          도메인 관심사 선택
            ↓
          팀원 역할 입력
            ↓
          봇 초대 링크 발급
═══════════════════════════════  (디스코드로 전환)
[디스코드] 음성채널에서 팀 토론
            ↓
          봇이 텍스트 채널에 실시간 기록
          (화자별 발언 요약 + 키워드 추출,
           팀이 채팅으로 키워드 추가·삭제)
            ↓
          /기획완료 입력
═══════════════════════════════  (웹으로 전환)
[웹]      결과물 확인
          (타겟 정의 · 경쟁 서비스 맵 ·
           MVP 기능 리스트 · 구현 가능성 판정)
```

웹이 두 번 등장하는 것이 포인트다. 처음엔 **세팅 도구**, 마지막엔 **결과물 뷰어**. 디스코드는 중간 토론 구간만 담당한다.

### 1.8 플랫폼 결정 근거

- **웹 우선** — 팀이 함께 접속해 실시간으로 진행하는 작업(화이트보드 성격)은 화면이 넓어야 하고 텍스트 밀도가 높아 모바일에 부적합하다.
- **디스코드 봇으로 한정** — 팀이 이미 쓰는 환경에 녹아들어 진입장벽이 0에 가깝다. 별도 음성 통화방을 직접 구현할 필요도 없다. (슬랙 확장은 추후 과제)

## 2. 기술 스택 정리

### 2.1 전체 구조

봇 서버(Node.js)가 중앙 허브 역할을 하며 디스코드, 외부 API, DB, 웹을 모두 연결한다.

```
[웹 프론트]          [봇 서버 (Node.js)]         [외부 API]
 React        ──REST──▶ discord.js                OpenAI Whisper
 WebSocket수신 ◀─WS──   @discordjs/voice  ──────▶  OpenAI GPT-4o
 결과물 렌더링          Whisper API 호출
                       GPT API 처리              [DB / 저장소]
[디스코드]             WebSocket 서버    ──────▶  PostgreSQL
 음성채널(WebRTC)──▶   REST API                   동아리 프로젝트 DB
 텍스트채널(출력)◀──
```

### 2.2 컴포넌트별 기술

|영역|기술|역할|
|---|---|---|
|웹 프론트|React|세팅 페이지, 결과물 페이지|
|웹 프론트|WebSocket 클라이언트|실시간 키워드 수신·표시|
|봇 서버|discord.js|봇 기본 동작, 슬래시 커맨드|
|봇 서버|@discordjs/voice|화자별 음성 스트림 수신|
|봇 서버|Whisper API 연동|음성 → 텍스트 (STT)|
|봇 서버|GPT API 연동|키워드 추출, 요약, 결과물 생성|
|봇 서버|WebSocket 서버|웹으로 실시간 전달|
|봇 서버|REST API|웹 세팅 데이터 수신|
|외부 API|OpenAI Whisper|음성 인식|
|외부 API|OpenAI GPT-4o|키워드·요약·판정|
|DB|PostgreSQL|세팅 데이터, 결과물 저장|
|DB|동아리 프로젝트 DB|중복 필터용 데이터|

### 2.3 음성 처리 기술 검증 결과

`@discordjs/voice`의 `VoiceReceiver`는 유저 ID별로 개별 음성 스트림을 생성할 수 있어 **화자 분리가 가능하다.**

```js
receiver.subscribe(userId, { end: EndBehaviorType.AfterSilence })
```

|항목|가능 여부|
|---|---|
|화자별 음성 분리 수신|가능|
|실시간 스트림 처리|가능|
|자동 silence 보정|직접 구현 필요|
|다중 유저 동기화|직접 구현 필요|

discord.js는 유저가 말을 멈춘 구간에 silence를 자동으로 채워주지 않고, 다중 유저 스트림의 타이밍 동기화도 수동 처리가 필요하다. 다만 MVP 단계에서 silence 동기화 이슈는 키워드 추출 정확도에 큰 영향이 없어 초기엔 무시 가능하다.

Whisper API는 실시간 스트리밍이 아니라 **3~5초 청크 단위** 처리가 현실적이다. 완전한 실시간은 비용과 복잡도가 크다.

### 2.4 팀 역할 분리

|역할|담당 영역|
|---|---|
|백엔드 개발자|봇 서버 전체, REST API, DB, 외부 API 연동|
|앱 개발자|React 웹 (세팅 페이지 + 결과물 페이지 + WebSocket 수신)|
|디자이너|웹 UI 전반, 결과물 레이아웃|

### 2.5 구현 난이도 핵심 포인트

가장 까다로운 두 군데를 먼저 프로토타입으로 검증하면 나머지는 붙이기 수월하다.

1. **음성 청크 타이밍 처리** — Whisper 호출 간격을 어떻게 끊을 것인가
2. **WebSocket 실시간 반영** — 봇이 뽑은 키워드를 웹에 지연 없이 표시

## 3. 검증해야 할 가설 (위험 요소)

서비스를 만들기 전에 확인이 필요한 가정들이다.

1. **진입 시점** — 팀이 "아이디어 없이" 뭉치는 경우가 실제로 얼마나 되는가? (아이디어 먼저인가, 팀 먼저인가)
2. **행동 수용성** — "팀이 음성 토론을 AI가 정리해준다"는 방식을 실제로 쓸 의향이 있는가?
3. **결과물 신뢰도** — AI가 뽑아준 MVP 리스트를 팀이 신뢰하고 따를까?
4. **디스코드 사용률** — 타겟 주니어 팀의 디스코드 사용 비율이 충분한가?

이 가설들은 코드 없이도 검증 가능하다. 랜딩페이지·설문·타겟 유저 인터뷰로 먼저 확인하는 것이 안전하다.

## 4. MVP 우선순위

가장 먼저 만들어야 할 단 하나의 핵심 기능:

> **디스코드 음성채널의 발언을 받아 키워드를 추출해 텍스트 채널에 출력하는 것**

이 한 가지가 작동하면 서비스의 핵심 가치가 증명된다. 웹 세팅 페이지, 결과물 생성, 중복 필터, 슬랙 확장 등은 모두 그 다음 순서다.
