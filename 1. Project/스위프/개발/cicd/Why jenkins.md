## CI/CD란 무엇인가

CI/CD는 Continuous Integration(지속적 통합)과 Continuous Delivery/Deployment(지속적 배포)를 합친 개념입니다. 핵심 아이디어는 코드 변경사항이 저장소에 올라오는 순간을 트리거로 삼아, 사람이 수동으로 개입하지 않아도 빌드와 테스트, 배포까지 일련의 검증 과정이 자동으로 이어지게 만드는 것입니다. CI는 "이 변경사항이 기존 코드와 충돌 없이 합쳐지고, 정상적으로 컴파일되고, 기존 테스트를 깨뜨리지 않는가"를 매번 자동으로 확인하는 단계를 말하고, CD는 그 검증을 통과한 결과물을 실제 서버에 자동으로 올리는 단계를 말합니다. 두 단계를 합쳐 운영함으로써, 사람이 수작업으로 빌드하고 옮기는 과정에서 발생하는 실수와 지연을 구조적으로 줄이는 것이 CI/CD의 본질적인 목적입니다.

## CI/CD에서 가능한 작업

실제로 GitHub Actions로 구성해본 파이프라인을 기준으로 보면, CI/CD가 다루는 작업의 범위를 크게 두 갈래로 나눌 수 있습니다. 하나는 PR이 올라왔을 때 수행하는 검증 작업으로, 표준 환경에서의 컴파일 확인, `./gradlew test`를 통한 자동화 테스트 실행, 그 결과를 PR에 체크 형태로 게시하는 작업이 여기에 속합니다. 다른 하나는 특정 브랜치에 push가 일어났을 때 수행하는 배포 작업으로, JAR 빌드, Docker 이미지 빌드 및 레지스트리 push, 그리고 운영 서버에 접속해 새 버전을 무중단으로 교체하는 블루그린 배포까지가 여기에 포함됩니다. 정리하면 CI/CD는 검증(테스트, 정적 분석, 빌드 성공 여부 확인)과 인도(이미지 빌드, 배포, 환경 설정 반영)라는 두 축의 작업을 코드 변경 이벤트에 묶어 자동화하는 것이라고 볼 수 있습니다.

## 젠킨스의 역할

젠킨스는 이러한 CI/CD 파이프라인을 직접 운영하는 자동화 서버입니다. GitHub Actions와 비교했을 때 가장 큰 구조적 차이는, GitHub Actions는 웹훅 수신과 워크플로우 스케줄링(Controller 역할)을 GitHub라는 SaaS가 대신 처리해주고 실행 환경(Agent 역할)만 hosted runner 또는 self-hosted runner로 선택하는 구조인 반면, 젠킨스는 이 두 역할을 본인이 직접 구축한 인프라 위에서 운영해야 한다는 점입니다. 젠킨스 Controller는 UI를 서빙하고, Job을 큐잉하고 스케줄링하며, Credential과 플러그인을 관리하고, 어떤 빌드를 어떤 Agent에게 보낼지를 결정하는 역할을 담당합니다. 실제로 코드를 체크아웃하고 테스트를 돌리고 배포 스크립트를 실행하는 행위는 Controller가 아니라 별도로 등록된 Agent가 수행합니다.

## 면접 우대 요건이 되는 이유

채용 공고에서 "젠킨스 사용 경험"을 요구하는 것은 단순히 CI/CD 개념을 아는지를 묻는 것이 아닙니다. 실제로는 이미 회사 내부에 구축되어 있는 Jenkins 인프라에 합류해서, Jenkinsfile 작성뿐 아니라 Controller-Agent 운영, Credential 관리, 플러그인 충돌 같은 인프라 운영 이슈까지 다룰 수 있는 사람인지를 묻는 질문에 가깝습니다. GitHub Actions가 등장한 이후로도 다수의 회사가 이미 수년간 구축해온 Jenkins 인프라를 그대로 유지하고 있고, 보안 정책상 외부 클라우드 러너 접근이 막힌 망분리 환경에서는 자체 호스팅이 강제되는 Jenkins가 유일한 선택지가 되는 경우도 많습니다. 이러한 배경 때문에 Jenkins 경험은 곧 "복잡한 운영 인프라를 다뤄본 경험"으로 해석되는 경향이 있습니다.

다만 단순히 빈도나 보안 정책 때문만이 아니라, 기능적으로도 GitHub Actions가 완전히 대체하지 못하는 지점이 분명히 존재합니다. 그 핵심을 세 가지로 좁혀볼 수 있습니다.

**Controller-Agent 분리**

젠킨스는 빌드를 요청받는 주체(Controller)와 빌드를 실제로 실행하는 주체(Agent)가 처음부터 구조적으로 분리되어 있습니다. Controller의 Executor 수를 0으로 설정해두면 Controller에는 아무런 빌드 부하가 걸리지 않고, Agent를 라벨로 구분해서 특정 Job이 특정 장비에서만 실행되도록 명시적으로 라우팅할 수 있습니다. 예를 들어 PR 검증용 빌드는 범용 Agent에서, 운영 서버 배포는 `prod` 라벨이 붙은 Agent에서 실행되도록 Jenkinsfile에서 직접 지정할 수 있습니다. 이는 GitHub Actions의 `runs-on: [self-hosted, dev]`와 발상은 비슷하지만, Jenkins는 이 Agent들을 Node 단위로 미리 등록하고 UI에서 실시간으로 연결 상태를 확인할 수 있다는 점에서 더 명시적이고 추적 가능한 구조를 제공합니다.

**Groovy의 동적 파이프라인 제어**

Jenkinsfile은 Groovy라는 실제 프로그래밍 언어로 작성되기 때문에, 파이프라인의 구조 자체를 빌드 실행 도중에 동적으로 만들어낼 수 있습니다. 예를 들어 모노레포 환경에서 `git diff` 결과로 변경된 모듈 목록을 런타임에 뽑아낸 다음, 그 목록을 순회하며 변경된 모듈 수만큼 stage를 즉석에서 생성하는 식의 로직이 가능합니다. GitHub Actions의 YAML은 워크플로우 시작 전에 구조가 고정되는 선언적 방식이라, 이런 동적 처리를 하려면 한 Job에서 변경 목록을 outputs로 내보내고 다음 Job의 matrix 입력으로 다시 주입하는 우회가 필요합니다. Groovy는 일반적인 반복문과 함수 호출만으로 이를 구현할 수 있어, 복잡한 분기나 동적 구조가 필요한 파이프라인일수록 체감 차이가 커집니다.

**코드 이벤트와 무관한 범용 작업 실행 포털**

Jenkins Job은 git push나 PR 같은 코드 이벤트에 묶여 있을 필요가 없습니다. `parameters` 블록을 사용하면 Jenkins가 자동으로 웹 입력 폼을 만들어주기 때문에, 사람이 브라우저에서 드롭다운으로 환경을 선택하고 버튼을 눌러 DB 백업, 서버 재시작, 특정 버전으로의 롤백 같은 운영 작업을 직접 실행할 수 있습니다. 이 Job들은 어떤 커밋과도 연결되어 있지 않으며, SSH 키나 서버 구조를 모르는 팀원도 웹 화면만으로 운영 작업을 수행할 수 있게 해줍니다. GitHub Actions의 `workflow_dispatch`도 수동 트리거와 입력값을 지원하지만 결국 특정 레포의 워크플로우 파일에 종속되는 반면, Jenkins는 하나의 Controller 안에 여러 레포와 여러 서버를 아우르는 Job들을 동시에 두고 운영 콘솔처럼 활용할 수 있다는 점에서 차이가 있습니다.
