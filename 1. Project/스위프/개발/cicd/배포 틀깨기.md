아니요, 포트 기반 blue-green + 단일 호스트 + 쉘 스크립트로 직접 컨테이너와 nginx를 조작하는 이 구조 자체는, 제약이 없는 환경이라면 최선이라고 보기는 어렵습니다. 지금까지 얘기한 개선들(compose 분리, 라우팅 분리, docker ps 기반 상태 판단)은 "이 구조 안에서 더 깨끗하게"였고, 구조 자체를 더 좋게 바꿀 여지는 분명히 있습니다. 몇 가지 축으로 짚어볼게요.

가장 큰 차이는 "포트를 바꿔치기하는 방식의 blue-green"이 사실 진짜 의미의 blue-green이 아니라 그 축소판이라는 점입니다. 원래 blue-green의 핵심 아이디어는 완전히 분리된 두 환경(보통 서버/인스턴스 단위)을 두고 로드밸런서가 트래픽을 어느 쪽으로 보낼지 스위치하는 겁니다. 지금처럼 같은 호스트에서 포트만 8081/8082로 나누는 건, 호스트 자체가 단일 장애점(SPOF)으로 남아 있다는 뜻입니다. 호스트가 죽으면 blue든 green이든 둘 다 같이 죽습니다. 진짜 이상적인 형태는 별도의 인스턴스(혹은 오토스케일링 그룹) 두 세트를 두고, ALB/ELB 같은 로드밸런서의 타겟 그룹을 전환하는 식입니다. 이러면 nginx 설정 파일을 직접 rewrite하고 reload하는 행위 자체가 사라지고, "타겟 그룹 A → B로 트래픽 전환"이라는 더 원자적이고 검증된 메커니즘으로 대체됩니다.

두 번째로, 지금은 배포 로직이 전부 imperative shell script입니다. "이렇게 해라, 저렇게 해라"를 순서대로 명령하는 방식이죠. 더 이상적인 형태는 declarative + reconciliation 모델, 흔히 GitOps라고 부르는 방식입니다. "지금 상태가 어떻게 되어야 하는지"를 git에 선언해두고(예: "green이 활성 상태여야 한다"), Argo CD나 Flux 같은 컨트롤러가 그 선언된 상태와 실제 상태를 계속 비교하면서 알아서 맞춰줍니다. 이 모델의 장점은 deploy.sh처럼 "한 번 실행하고 끝"이 아니라, 누가 수동으로 컨테이너를 건드려도 컨트롤러가 다시 원하는 상태로 되돌려놓는 self-healing이 자동으로 따라온다는 점입니다. 이건 사실 지난번에 얘기했던 "active-profile 파일과 실제 상태가 어긋날 수 있다"는 문제를 구조적으로 원천 차단하는 방식이기도 합니다.

세 번째는 헬스체크와 전환의 이분법적 구조입니다. 지금은 "헬스체크 통과 → 100% 전환"으로 한 번에 다 넘기는데, 더 정교한 형태는 canary 방식으로 트래픽의 일부(5%, 10%)만 먼저 신규 버전으로 보내보고, 에러율이나 레이턴시 같은 실제 운영 지표를 일정 시간 관찰한 뒤 점진적으로 비중을 늘려가는 겁니다. `/actuator/health`가 200을 반환한다고 해서 그 버전이 실제 트래픽 부하에서도 괜찮다는 보장은 없으니까요. 이런 progressive delivery는 보통 서비스 메시(Istio 등)나 쿠버네티스의 Argo Rollouts 같은 도구가 담당합니다.

네 번째로 롤백 시점의 문제입니다. 지금 구조는 헬스체크가 전환 전에만 있고, 전환 이후에 문제가 생기면(예: 헬스체크는 통과했는데 특정 API에서만 500이 터진다거나) 사람이 알아서 다시 deploy.sh를 거꾸로 돌려야 합니다. 더 이상적인 형태는 전환 후에도 일정 기간 모니터링 지표(에러율, p99 레이턴시 등)를 자동으로 관찰하면서, 기준을 넘으면 자동으로 직전 버전으로 되돌리는 automated rollback입니다.

다섯 번째, 이건 좀 더 철학적인 차이인데, deploy(배포)와 release(릴리즈)를 분리하는 개념도 있습니다. 새 버전을 서버에 올리는 행위와, 그 버전을 실제 사용자에게 보여주는 행위를 나누는 거죠. feature flag를 쓰면 새 코드는 이미 프로덕션에 떠 있지만 특정 사용자/비율에게만 노출시키고, 문제가 생기면 flag만 끄면 되니 컨테이너를 내리고 올리는 무거운 작업 없이 즉시 되돌릴 수 있습니다.

이걸 전부 종합하면, 쿠버네티스나 ECS 같은 오케스트레이터를 쓰는 환경에서는 사실 이 모든 게 거의 기본으로 제공됩니다. rolling update, readiness/liveness probe, 타겟 그룹 전환, 심지어 Argo Rollouts 같은 도구로 canary까지 선언적으로 설정 가능합니다. 즉 "이상적인 형태"라는 건 결국 직접 쉘 스크립트로 인프라를 손으로 조작하는 모델에서, 오케스트레이터/컨트롤러에게 "원하는 상태"만 알려주고 나머지는 위임하는 모델로 옮겨가는 방향입니다.

다만 이걸 지금 단계에서 다 가져와야 한다는 뜻은 전혀 아닙니다. 팀 규모, 트래픽, 운영 인력을 고려하면 지금의 단일 EC2 + nginx + shell script 구조는 충분히 합리적인 절충이고, 오히려 k8s 같은 걸 지금 무리하게 들였다가는 복잡도가 감당이 안 될 수도 있습니다. 다만 "이 구조가 절대적으로 이상적이다"가 아니라 "지금 제약 하에서의 합리적 근사치다"라는 인식을 갖고 계시면, 나중에 트래픽이 늘거나 다중 인스턴스로 갈 시점에 어느 방향으로 단계적으로 옮겨가야 할지 미리 그림이 잡히실 거예요.

# ALB 배포
ALB 비용부터 정확히 보고 갈게요.ALB 비용부터 말씀드리면, 기본 구조는 시간당 고정요금 + 트래픽 기반 가변요금(LCU)으로 나뉩니다. 고정요금은 리전마다 약간 다르지만 대략 시간당 $0.0225~0.0252 수준이고, 한 달(730시간) 기준으로 환산하면 시간당 약 0.0252달러 × 730시간 = 약 18.40달러 정도가 ALB 하나를 그냥 켜두는 것만으로 발생하는 기본 비용입니다. 여기에 LCU 비용이 더 붙는데, LCU당 시간당 0.008달러이고 신규 연결, 활성 연결, 처리 바이트, 규칙 평가 중 가장 높은 값을 기준으로 과금합니다. 지금 swyp 트래픽 규모(거의 빈코드 테스트 단계)라면 LCU 사용량은 거의 1 미만일 가능성이 높아서, 1 LCU 기준이면 한 달에 약 5.84달러 정도가 추가됩니다. 합치면 대략 월 22~25달러선이 ALB 자체 비용이라고 보시면 됩니다. 참고로 신규 AWS 계정은 매달 ALB와 CLB 통합 750시간, ALB LCU 15개를 무료 제공하니 계정이 새것이면 첫해는 거의 무료로 쓸 수도 있습니다.

다만 이게 끝이 아니라는 점이 중요합니다. ALB 자체는 월 20달러 안팎이지만, "진짜" blue-green을 하려면 별도 인스턴스 두 대가 동시에 떠 있어야 하는데, 그 EC2 인스턴스 비용이 사실 ALB보다 더 큰 변수입니다. 지금 1.9GiB짜리 인스턴스 하나 쓰시는 비용을 그대로 두 배로 늘리는 셈이니, ALB 도입 자체보다 "인스턴스를 두 대로 늘리는 결정"이 비용의 본질이라고 보시는 게 맞습니다. (배포 윈도우에만 잠깐 두 대를 띄우는 Auto Scaling 전략을 쓰면 이 비용을 줄일 수는 있지만, 그러면 구조가 더 복잡해집니다.)

# Self-healing
self-healing/GitOps 부분은 솔직히 말씀드리면 도커 리태깅과는 다른 층위의 개념이고, 동시에 큰 그림을 더 보시면 "사실 이건 거의 쿠버네티스에 종속된 개념"이라는 걸 아셔야 할 것 같습니다.

먼저 push와 pull이라는 두 가지 배포 모델 차이를 짚어볼게요. 지금 구조(Jenkins가 트리거 → deploy.sh 실행)는 push 모델입니다. "이벤트가 발생하면(코드 push, 머지 등) 외부에서 타겟 서버로 명령을 밀어넣는다"는 방식이죠. GitOps는 반대로 pull 모델입니다. 컨트롤러(Argo CD나 Flux 같은)가 별도 프로세스로 계속 돌면서, git 저장소에 적힌 "원하는 상태"(예: "이 이미지의 이 버전이 떠 있어야 한다")를 주기적으로 읽어오고, 실제 클러스터 상태와 비교해서 다르면 스스로 맞춥니다. 이게 reconciliation loop입니다. 핵심은 "누군가 트리거해야 동작"이 아니라 "항상 감시하면서 스스로 교정"한다는 점이고, 그래서 누가 수동으로 컨테이너를 건드려도 다음 reconcile 주기에 원래 상태로 되돌아갑니다.

그런데 여기서 솔직하게 말씀드려야 할 부분이, Argo CD와 Flux는 둘 다 쿠버네티스 컨트롤러입니다. 이게 동작하려면 쿠버네티스의 API 서버(선언적 리소스를 받아주고, 변경을 감시할 수 있는 인터페이스)가 전제되어 있어야 합니다. k8s 없이 순수 docker 환경에서 "진짜" GitOps를 하려면 이 API 서버 역할을 대신할 무언가를 직접 만들어야 하는데, 현실적으로는 cron이나 systemd timer로 몇 분마다 "git의 원하는 상태 ↔ 현재 docker ps 상태"를 비교해서 다르면 deploy.sh를 실행시키는 식의 자체 제작 스크립트가 됩니다. 이게 불가능한 건 아니지만, 사실상 미니 k8s 컨트롤러를 직접 짜는 것과 비슷한 수준의 작업이라, "GitOps"라는 이름이 주는 만큼의 성숙한 도구 생태계(충돌 감지, 동기화 상태 시각화, 권한 관리 등) 없이 그 개념의 일부만 흉내내는 정도가 됩니다. 그래서 이 항목은 사실상 "k8s를 도입하지 않는 한 본격적으로는 못 가져온다"는 게 정직한 결론입니다. k8s를 안 가져가시기로 한 결정과 직접 충돌하는 지점이라, 지금 단계에서는 참고 개념으로만 알아두시고 실제 적용은 보류하는 게 맞을 것 같습니다.

# canary
**네 번째, k8s까지는 절대 안 가신다는 전제 이해했습니다. 그 안에서 canary 비슷한 걸 흉내내고 싶으면, 실제로 쓸 수 있는 가장 가벼운 도구는 nginx 자체의 weight 기반 라우팅입니다. nginx의 upstream 블록에서 서버마다 weight를 줄 수 있는데(예: `server blue:8080 weight=9; server green:8080 weight=1;`), 이러면 신버전한테 트래픽의 10%만 보내고 90%는 구버전이 그대로 받게 할 수 있습니다. 그 상태로 몇 분~몇십 분 정도 지켜보면서, 이미 구축해두신 Prometheus + Grafana로 에러율이나 p99 레이턴시를 관찰하고, 괜찮으면 weight를 점진적으로 올려서 100%까지 가는 식입니다. 이게 서비스 메시 없이도 흉내낼 수 있는 가장 현실적인 canary입니다.**

k6는 이거랑은 조금 다른 목적입니다. k6/JMeter는 보통 "배포 전에" 신버전이 부하를 견디는지 미리 테스트하는 용도(load test)고, canary는 "배포 중/직후에" 실제 사용자 트래픽 일부로 검증하는 용도(production validation)라서 상호보완적이지 대체 관계는 아닙니다. 둘 다 쓰시면 좋지만, 지표 관찰이라는 본질은 이미 갖고 계신 Grafana 대시보드로 충분히 커버됩니다.

다섯 번째, 롤백 부분에서 "이게 도커 이미지 리태깅이네"라고 하셨는데, 정확히는 약간 다른 두 가지가 묶여 있는 개념입니다. 하나는 이미지 태깅 전략이고, 다른 하나는 자동 복구 트리거입니다.

지금 deploy.sh는 `IMAGE="app5swyp6team/swyp-spring:latest"`로 항상 latest를 pull합니다. 문제는 이 방식으로는 "직전에 배포했던 정확한 버전"을 다시 가져올 방법이 없다는 점입니다. latest는 항상 가장 최근 빌드를 가리키니, 새 이미지가 또 push되면 "되돌리고 싶은 그 버전"이 무엇이었는지 식별할 수단이 사라집니다. 그래서 첫 단계로는 이미지 태그를 git commit SHA나 빌드 번호로 고정하는 게 좋습니다(예: `app5swyp6team/swyp-spring:a1b2c3d`). 이러면 "직전 배포 = OLD_CONTAINER가 쓰던 태그"라는 게 명확해지고, 롤백은 단순히 그 태그로 다시 deploy.sh를 거꾸로 도는 일이 됩니다. 이게 "리태깅"에 해당하는 부분입니다.

automated rollback은 그 위에 한 단계 더 얹는 겁니다. 사람이 모니터링하다가 이상하면 수동으로 롤백 명령을 치는 게 아니라, 전환 후 N분 동안 에러율/레이턴시 같은 지표를 스크립트가 자동으로 체크하고, 기준치를 넘으면 사람 개입 없이 곧바로 직전 태그로 deploy.sh를 재실행하는 겁니다. 이건 별도의 모니터링 스크립트(Grafana API나 Prometheus 쿼리를 직접 curl로 찔러서 체크하는 정도의 가벼운 형태로도 충분히 구현 가능합니다) + 위의 버전 태깅 전략이 합쳐져야 완성됩니다. k8s나 GitOps 없이도 이 정도는 충분히 손으로 구현할 수 있는 영역이라, 다른 항목들보다 현실적으로 가져오기 쉬운 개선이라고 봅니다.

# k8s

마지막으로 k8s(쿠버네티스)가 뭔지 짧게 짚어드리면, 여러 서버(노드)를 하나의 클러스터로 묶어서 컨테이너 배치, 스케일링, 장애 복구, 트래픽 라우팅을 사람이 일일이 스크립트로 안 짜고 "원하는 상태"만 선언하면 자동으로 맞춰주는 컨테이너 오케스트레이션 플랫폼입니다. 지금까지 얘기한 "이상적인 형태"들(타겟 그룹 전환, reconciliation, canary, 자동 롤백)이 전부 k8s 생태계 도구(Argo CD, Argo Rollouts, Istio)에서 기본 제공되거나 표준 패턴으로 자리잡혀 있어서 계속 비교 대상으로 나온 거고, 그래서 자꾸 k8s 얘기가 따라붙는 겁니다. 다만 지금 1.9GiB 단일 인스턴스, 트래픽도 적은 단계에서 k8s를 들이면 클러스터 자체를 운영하는 오버헤드(컨트롤 플레인, 노드 관리, 학습 곡선)가 지금 풀고자 하는 문제보다 훨씬 커지니, 안 가시는 게 맞는 판단이라고 봅니다.

# Self-healing 도입해볼까??
락만 빼면 나머지가 쉬워 보이는 건 어느 정도 맞는데, 몇 가지 짚어드릴 부분이 있습니다.

먼저 락을 빼는 것에 대해서인데, 이건 "어렵은 부분을 제거했다"가 아니라 "리스크를 감수하기로 한 결정"에 더 가깝습니다. 폴링하는 reconciler를 따로 두는 순간, deploy.sh를 실행할 수 있는 주체가 두 개(Jenkins 트리거 + 주기적 reconciler)가 되는 거라서, 둘이 동시에 같은 컨테이너를 건드릴 가능성 자체는 사라지지 않습니다. 다만 배포 빈도가 낮으면 충돌할 확률이 작으니 "그냥 받아들이자"는 선택을 하신 거고, 그건 충분히 합리적인 트레이드오프입니다. 다만 완전히 안 만들기보다는, 아주 가벼운 형태의 락(예: `/tmp/deploy.lock` 같은 플래그 파일을 만들고, reconciler가 시작 전에 그 파일 존재 여부만 확인하는 정도)은 몇 줄이면 되는 일이라 굳이 빼실 필요는 없을 것 같습니다. 진짜 분산 락(etcd, Redis 기반)까지 갈 필요가 없다는 거지, 파일 하나 체크하는 수준은 들이는 비용 대비 얻는 안전망이 꽤 큽니다.

상태 비교 쪽은 사실 셋 중에 제일 만만치 않은 부분입니다. 겉보기엔 "git에 적힌 태그랑 docker ps로 본 태그를 비교하면 되는 거 아냐?" 싶지만, blue-green 특성상 단순 일치/불일치로 안 끝나는 중간 상태들이 꽤 있습니다. 예를 들면 컨테이너는 떠 있는데 헬스체크가 아직 안 끝난 경우(방금 막 띄운 직후), 컨테이너가 죽었다가 `--restart unless-stopped`로 재시작되는 중인 경우, nginx 설정은 blue를 가리키는데 active-profile 파일은 green이라고 적혀 있는 경우(지난번에 짚었던 그 불일치) 같은 것들이요. 이런 케이스를 "동기화됨/드리프트됨/진행중" 중 어디로 판정할지 하나씩 정의해야 하는데, 이게 막상 만들다 보면 처음 생각보다 분기가 늘어나는 부분입니다. 다만 못 만들 정도로 어려운 건 아니고, "동기화됐는지 판단하는 기준은 nginx가 가리키는 컨테이너가 살아있고 헬스체크가 통과한 상태"처럼 기준을 단순하게 하나로 못 박아두면 충분히 관리 가능한 범위입니다.

재시도/백오프는 말씀하신 대로 정말 단순합니다. 실패하면 일정 시간 띄우고 다시 시도, 몇 번 연속 실패하면 알림만 보내고 멈추는 정도면 충분하고 코드량도 적습니다.

관찰 가능성도, 이미 MDC 기반 구조화 로깅이랑 Alloy + Prometheus + Grafana 스택을 갖고 계셔서 사실 이 부분은 거의 "이미 가진 인프라를 재활용"하는 수준입니다. reconciler가 동기화 시도/성공/실패를 로그로 찍기만 하면 기존 대시보드에 바로 얹을 수 있으니, 처음부터 새로 만드는 다른 회사들보다 훨씬 유리한 출발점입니다.

그런데 한 가지 꼭 정정하고 싶은 부분이 있는데, 이 reconciler를 만든다고 해서 "롤백 시점의 문제"가 자동으로 같이 풀리는 건 아닙니다. 이 둘은 사실 서로 다른 문제입니다.

self-healing/drift 감지는 "선언된 상태(git)와 실제 상태(docker)가 다르면 고친다"는 논리입니다. 그런데 막 배포가 끝난 직후 상황을 생각해보면, git에는 이미 새 버전(예: abc124)이 적혀 있고, 실제로도 abc124가 떠서 헬스체크도 통과했습니다. 즉 선언된 상태와 실제 상태가 완벽히 일치하는 상태입니다. 그런데 그 abc124가 특정 API에서만 500을 뱉고 있다면, reconciler 입장에서는 "다른 게 하나도 없는데?"라는 결론을 내립니다. 드리프트가 아니라 "선언한 그대로 잘 배포됐는데, 그 배포된 것 자체가 사실은 나쁜 버전"인 상황이라서, drift 비교 로직으로는 절대 못 잡습니다.

이 문제를 풀려면 완전히 별개의 감시 로직이 필요합니다. 배포 직후 N분 동안 Prometheus에서 에러율/레이턴시를 조회해서 기준치를 넘으면, git의 manifest를 "직전 태그로 되돌려라"로 다시 커밋(또는 직접 deploy.sh 역방향 실행)하는 watcher입니다. 이건 reconciler가 공짜로 따라오는 부수효과가 아니라, 따로 만들어야 하는 두 번째 컴포넌트라는 점은 분명히 해두고 싶습니다. 다만 둘 다 결국 "deploy.sh를 실행하는 능력"을 공유하니, reconciler를 먼저 만들어두면 그 실행 경로를 재사용해서 두 번째 watcher를 더 쉽게 얹을 수 있다는 이점은 있습니다.

종합하면, 도입해볼 만하냐는 질문에는 네, 라고 답하고 싶습니다. 이미 갖고 계신 관찰 인프라 덕분에 시작 비용이 낮고, self-healing 하나만 해도 (drift 감지 + 가벼운 플래그 락 + 재시도) 충분히 독립적으로 가치 있는 결과물입니다. 다만 이게 "오늘 오후에 끝낼 작업"은 아니고, 상태 비교 로직의 엣지케이스를 정리하는 데 생각보다 시간이 들어가는 사이드 프로젝트급 작업이라는 점은 미리 가늠해두시는 게 좋을 것 같습니다. 순서로는 self-healing 쪽(drift 감지 + 가벼운 락)을 먼저 작게 완성해서 독립된 가치를 얻고, 그다음 여유가 되면 별도 컴포넌트로 자동 롤백 watcher를 얹는 식으로 단계를 나누시는 걸 추천합니다. 지금 OS 공부랑 레주메, Swiipe 기능 개발도 같이 돌리고 계신 걸 감안하면, 이걸 메인 트랙으로 당장 끌고 오기보다는 여유 있을 때 짬짬이 만지는 사이드 트랙으로 두시는 정도가 현실적일 것 같습니다.
