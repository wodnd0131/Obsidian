
# 로깅 시스템의 의의
응답 및 예외를 중앙 집중식으로 캡처할 때 깨끗하고 체계적인 방식으로 로깅을 구현하는 것


## Issue Number

feat/centralize-logging

## As-Is

- 로그가 없어 HTTP 요청/응답, 비즈니스 액션, 예외 발생 여부를 추적할 수 없었음
- 예외 처리가 산재되어 있고 일관된 응답 형태가 없었음
- `ErrorCode` enum에 예외 상태와 메시지를 모아 관리하는 구조로, 새 예외를 추가할 때마다 enum을 수정해야 했음
- `LoginException`이 별도 존재했으나 `GlobalExceptionHandler`에 핸들러가 없어 500으로 처리됨

## To-Be

**로깅 시스템 구축**
- `LogFilter`: 모든 HTTP 요청/응답을 자동 로깅, MDC로 `request_id` 추적
- `@BusinessLogging`: AOP 기반 선언적 비즈니스 액션 로깅 (userId, action 기록)
- 로그 타입 분리: `RequestLog`, `ResponseLog`, `BusinessLog`, `ErrorLog`, `InfoLog`
- `logstash-logback-encoder` 도입, 운영 환경에서 JSON 출력으로 ELK 연동 가능한 구조
- `logback-spring.xml` 추가: 로컬(텍스트) / 운영(JSON) 환경별 분리

**예외 처리 정비**
- `BusinessException`에서 `HttpStatus` + `message` 직접 보유
- `LoginException` 제거, `BusinessException`으로 통일, 도메인별 Exception 정의를 통일
- `GlobalExceptionHandler`에 JWT 예외, Spring MVC 바인딩 예외 핸들러 추가
- 예외 타입별 로그 레벨 분기: 4xx → INFO, 401/403 → WARN, 5xx → ERROR

## 1. 예외 클래스 구조  
  
```  
RuntimeException  
├── BusinessException          # 일반 비즈니스 로직 예외  
├── LoginException             # 로그인 관련 예외  
├── ExternalApiException       # 외부 API 응답 오류  
└── ExternalApiConnectionException  # 외부 API 연결 오류  
```  
  
### BusinessException  
  
비즈니스 로직에서 발생하는 예외. `HttpStatus`와 `message`를 직접 지정한다.  
  
```  
throw new BusinessException(HttpStatus.UNAUTHORIZED, "refresh Token 값 불일치");  
throw new BusinessException(HttpStatus.NOT_FOUND, "유저를 찾을 수 없습니다.");  
throw new BusinessException(HttpStatus.CONFLICT, "이미 존재하는 닉네임입니다.");  
```  
  
### ExternalApiException  
  
외부 API(카카오, 네이버 등)로부터 오류 응답을 받은 경우.  
  
```  
throw new ExternalApiException("잘못된 액세스 토큰","KAKAO",HttpStatus.UNAUTHORIZED);  
```  
  
### ExternalApiConnectionException  
  
외부 API에 아예 연결 자체가 실패한 경우 (네트워크 오류, 타임아웃 등). 항상 503으로 응답한다.  
  
```  
throw new ExternalApiConnectionException("카카오 서버 연결 실패","KAKAO");  
```

## 2. 예외 핸들링 흐름  
  
모든 예외는 `GlobalExceptionHandler`에서 처리되며 `ProblemDetail` 형태로 응답한다.  
  
### 예외별 처리 규칙  
  
| 예외                                       | HTTP 상태          | 로그 레벨 |  
|------------------------------------------|------------------|-------|  
| `BusinessException` (401, 403)           | 예외의 status 그대로   | WARN  |  
| `BusinessException` (그 외)                | 예외의 status 그대로   | INFO  |  
| `ExternalApiException` (5xx)             | 외부 API 응답 status | ERROR |  
| `ExternalApiException` (4xx)             | 외부 API 응답 status | WARN  |  
| `ExternalApiConnectionException`         | 503              | ERROR |  
| `MethodArgumentNotValidException`        | 400              | INFO  |  
| `MethodArgumentTypeMismatchException`    | 400              | INFO  |  
| `HttpMessageNotReadableException`        | 400              | INFO  |  
| `HttpRequestMethodNotSupportedException` | 400              | INFO  |  
| `HttpMediaTypeNotSupportedException`     | 400              | INFO  |  
| `MultipartException`                     | 400              | INFO  |  
| `Exception` (그 외 모든 예외)                  | 500              | ERROR |  
  
### 응답 형태 (ProblemDetail)  
  
```json  
{  
  "status": 401,  "title": "UNAUTHORIZED",  "detail": "refresh Token 값 불일치"  
}  
```  
  
Validation 오류의 경우 `fieldErrors` 추가:  
  
```json  
{  
  "status": 400,  "title": "BAD_REQUEST",  "detail": "입력 데이터 검증에 실패했습니다.",  
  "fieldErrors": {    "email": "이메일 형식이 올바르지 않습니다.",  
    "password": "비밀번호는 8자 이상이어야 합니다."  
  }}  
```

## 3. 로그 시스템 구조  
  
### 로그 타입  
  
| LogType          | 설명         | 생성 주체                         |  
|------------------|------------|-------------------------------|  
| `REQUEST`        | HTTP 요청 정보 | `LogFilter` (자동)              |  
| `RESPONSE`       | HTTP 응답 정보 | `LogFilter` (자동)              |  
| `BUSINESS`       | 비즈니스 액션    | `BusinessLogAspect` (자동)      |  
| `CLIENT_ERROR`   | 4xx 오류     | `GlobalExceptionHandler` (자동) |  
| `SERVER_ERROR`   | 5xx 오류     | `GlobalExceptionHandler` (자동) |  
| `EXTERNAL_ERROR` | 외부 API 오류  | `GlobalExceptionHandler` (자동) |  
| `INFO`           | 정보성 로그     | 개발자가 직접 필요할 경우 작성(ex. 서킷브레이커) |  
  
### 로그 출력 형태 (JSON)  
  
모든 로그는 `Markers.appendEntries()`를 통해 JSON 직렬화되어 ELK 스택으로 전송 가능한 구조로 출력된다.  
  
```json  
{  
  "logType": "REQUEST",  "method": "POST",  "uri": "/api/login",  "body": "{\"email\":\"user@example.com\"}",  "request_id": "a1b2c3d4",  "request_uri": "/api/login",  "client_ip": "192.168.1.1"}  
```  
  
`request_id`, `request_uri`, `client_ip`는 `LogFilter`가 MDC에 세팅하므로 모든 로그에 자동으로 포함된다.

## 4. 로그 사용법  
  
### HTTP 요청/응답 — 자동  
  
별도 작업 없이 모든 요청/응답이 자동 기록된다.    
단, 아래 URI는 제외된다.  
  
- `/actuator/**`  
- `/swagger-ui/**`  
- `/v3/api-docs/**`  
  
```  
{ "logType": "REQUEST", "method": "DELETE", "uri": "/api/users/me" }  
{ "logType": "RESPONSE", "uri": "/api/users/me", "status": 200 }  
```  
  
---  
  
### 비즈니스 액션 — `@BusinessLogging`  
  
`LogFilter`는 HTTP 입출력(어떤 URL에 요청이 왔는지)을 기록하지만, **누가 했는지는 알 수 없다.**    
`@BusinessLogging`은 userId를 직접 기록해 **사람 기준의 행위 추적**을 가능하게 한다.  
  
단순 조회·검색처럼 부작용 없는 행위는 HTTP 로그로 충분하며, 아래와 같이 **나중에 근거가 필요한 행위**에만 사용한다.  
  
- 돈/자산이 오가는 행위 (결제, 쿠폰 사용)  
- 계정/권한이 변경되는 행위 (탈퇴, 비밀번호 변경)  
- 과금이 발생하는 외부 API 호출 (SMS, 이메일 인증)  
- 관리자 행위 (강제 로그아웃, 계정 정지)  
  
```  
@BusinessLogging("delete account")  
public void deleteUser(Long userId) { ... }  
  
@BusinessLogging("change role")  
public void changeUserRole(Long userId, UserRole role) { ... }  
```  
  
> **주의:** 파라미터 중 첫 번째 `Long` 타입을 userId로 자동 추출한다. userId는 항상 첫 번째 파라미터로 위치시킨다.  
  
출력되는 로그:  
  
```  
{  
  "logType": "BUSINESS",  "userId": 123,  "action": "delete account",  "request_id": "a1b2c3d4",  "request_uri": "/api/users/me",  "client_ip": "192.168.1.1"}  
```  
  
`request_id`, `request_uri`, `client_ip`는 `LogFilter`가 MDC에 세팅한 값으로, Logback이 자동으로 포함시킨다.    
따라서 `request_id`로 묶으면 한 요청 안의 흐름을 순서대로 추적할 수 있다.  
  
```  
request_id=a1b2c3d4  
  → [REQUEST]  DELETE /api/users/me  → [BUSINESS] User 123 executed delete account  → [RESPONSE] /api/users/me 200```  
  
> **비동기 주의:** `@Async` 또는 별도 스레드로 넘어가는 경우 MDC는 스레드 로컬이라 `request_id`가 전파되지 않는다. 이 경우 MDC 값을 직접 복사해서 넘겨야 한다.  
  
---  
  
### 정보성 로그 — `InfoLog`  
  
스케줄러, 배치 등 HTTP 요청과 무관한 로직에서 수동으로 기록한다.  
  
```  
// 속성 없이  
InfoLog infoLog = InfoLog.of(LogType.INFO, "SCHEDULER", "배치 작업 완료");  
log.info(Markers.appendEntries(infoLog.fields()),infoLog.summary());  
  
// 추가 속성과 함께  
InfoLog infoLog = InfoLog.of(LogType.INFO, "SCHEDULER", "배치 작업 완료",  
        Map.of("duration", 120, "processedCount", 3000));  
log.info(Markers.appendEntries(infoLog.fields()),infoLog.summary());  
```  
  
---  
  
### 예외 로그 — 자동  
  
`GlobalExceptionHandler`에서 자동으로 처리되므로 별도 작업 불필요.    
직접 에러 로그가 필요한 경우 (ex. catch 블록 내부):  
  
```  
// 서버 오류  
ErrorLog errorLog = ErrorLog.createServerErrorLog(500, e, "CUSTOM_CODE");  
log.error(Markers.appendEntries(errorLog.fields()),errorLog.summary());  
  
// 외부 API 오류  
ErrorLog errorLog = ErrorLog.createExternalErrorLog(502, e, "KAKAO");  
log.error(Markers.appendEntries(errorLog.fields()),errorLog.summary());  
```

## 5. 필터 실행 순서  
  
```  
HTTP 요청  
    │    ▼LogFilter          ← @Component, Servlet 체인 (모든 요청 로깅, MDC 세팅)  
    │    ▼Spring Security 체인  
    │    ▼JwtAuthenticationFilter  ← SecurityConfig에서 수동 등록  
    │    ▼UsernamePasswordAuthenticationFilter  
    │    ▼Controller / ExceptionHandler  
    │    ▼LogFilter (finally)  ← RequestLog, ResponseLog 기록, MDC 초기화  
    │    ▼HTTP 응답  
```  
  
`LogFilter`는 Security 체인 바깥에 위치하므로 인증 실패 응답도 빠짐없이 로깅된다.
