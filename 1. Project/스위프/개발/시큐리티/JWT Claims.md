## PII (Personally Identifiable Information)

**개인 식별 정보** — 특정 개인을 식별할 수 있는 데이터.

이메일, 이름, 전화번호, 주민번호 등이 해당됨.

JWT는 Base64로 인코딩만 되어있어서 **암호화가 아님** → 누구나 디코딩 가능. 그래서 payload에 PII를 넣으면 탈취 시 그대로 노출됨. `id(숫자)`는 그 자체로 개인정보가 아니라서 안전.

## JWT 클레임 구성 분석

### 설정된 클레임

| 클레임     | 코드                       | 설명                   |
| ------- | ------------------------ | -------------------- |
| `sub`   | `.subject(accountId)`    | 토큰 주체 식별자 (누구의 토큰인가) |
| `roles` | `.claim("roles", roles)` | 커스텀 클레임 — 권한 정보      |
| `jti`   | `.id(jti)`               | JWT ID — 토큰 고유 식별자   |
| `iat`   | `.issuedAt(now)`         | 발급 시각                |
| `exp`   | `.expiration(expiry)`    | 만료 시각                |

### jti란?

JWT Token ID. 이 토큰 자체의 고유 ID (보통 UUID).

**왜 필요한가:**

- **토큰 재사용 공격 방지** — 같은 토큰이 두 번 쓰이는 걸 서버에서 감지 가능
- **로그아웃 구현** — `jti`를 블랙리스트(Redis 등)에 저장해서 강제 무효화
- **감사 로그** — 어떤 토큰이 어떤 요청을 했는지 추적 가능

```
jti 없으면 → 로그아웃해도 exp 전까지 토큰 계속 유효
jti 있으면 → 블랙리스트에 jti 등록하면 즉시 무효화 가능
```

### 설정되지 않은 표준 클레임

| 클레임                | 의미                             | 없을 때 영향                 |
| ------------------ | ------------------------------ | ----------------------- |
| `iss` (issuer)     | 토큰 발급자 식별 (예: `"swyp-server"`) | 다중 서비스 환경에서 출처 검증 불가    |
| `aud` (audience)   | 토큰 수신 대상 (예: `"swyp-client"`)  | 다른 서비스에서 이 토큰을 수락할 수 있음 |
| `nbf` (not before) | 이 시각 이전엔 유효하지 않음               | 없어도 보통 문제없음             |

**현재 규모에서 실질적으로 중요한 건 `iss`** — 나중에 토큰 검증 시 발급자 확인 용도로 넣어두면 좋음.

```java
.issuer("swyp")
```

`aud`는 MSA나 멀티 클라이언트 구조 아니면 당장 필요없음.
