public CustomClaims generateToken(String type, String accountId, String[] role) {
    // 1. 처음부터 Instant로 현재 시간 가져오기
    Instant now = Instant.now(); 
    
    // 2. Duration을 이용해 직관적으로 만료 시간 계산하기
    long expMillis = type.equals("ACCESS") ? ACCESS_EXP : REFRESH_EXP;
    Instant expiry = now.plusMillis(expMillis);

    // 3. Jwts.builder()에 Date 대신 Instant를 그대로 주입 (최신 jjwt 라이브러리 지원)
    String generatedToken = Jwts.builder()
                                    .subject(accountId)
                                    .claim("roles", role)
                                    .issuedAt(Date.from(now))       // 만약 구버전 jjwt라면 Date.from() 사용
                                    .expiration(Date.from(expiry)) // 최신 버전이면 .issuedAt(now) 처럼 Instant 바로 가능
                                    .signWith(key)
                                    .compact();

    // 4. 변환 과정(.toInstant()) 없이 깔끔하게 그대로 반환
    return new CustomClaims(accountId, generatedToken, now, expiry);
}

## 토큰 타입 혼용 문제 — 두 방향

### 방향 1: access token → `/auth/refresh`

```
클라이언트가 실수(또는 공격자)로 access token 을 refresh 엔드포인트에 보냄

validateToken()       → 서명·만료 통과 (유효한 토큰이니까)
validateTokenUUID()   → claims.getJti() 와 DB jti 비교
```

**jti 를 access token 에 안 만들면:**

```
claims.getJti() = null
DB 의 jti     = "550e8400-e29b-..."

null ≠ UUID → BusinessException → 차단 ✅
```

---

### 방향 2: refresh token → 일반 API 헤더

```
Authorization: Bearer <refresh_token>
```

```
JwtAuthenticationFilter
  → resolveToken()    : Bearer 떼고 토큰 추출
  → validateToken()   : 서명·만료만 확인
  → getAuthentication(): SecurityContext 에 저장
  → 컨트롤러 진입 허용 ❌
```

`JwtAuthenticationFilter` 는 jti 를 아예 보지 않아요.  
refresh token 도 서명은 같은 키로 만들어졌고 만료도 안 됐으면 **그냥 통과해요.**

---

### 정리

||access → /auth/refresh|refresh → 일반 API|
|---|---|---|
|jti 미생성 방식|✅ 차단|❌ 통과|
|타입 클레임 방식|✅ 차단|✅ 차단|

jti 미생성은 한쪽만 막아요.  
refresh token 이 탈취돼서 일반 API에 쓰이는 걸 막으려면 타입 클레임이 필요하고,  
그걸 감수할 수 있다면 jti 미생성으로도 충분한 선택이에요.
