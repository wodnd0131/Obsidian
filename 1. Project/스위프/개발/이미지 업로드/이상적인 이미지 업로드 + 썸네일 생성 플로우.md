# 이상적인 이미지 업로드 + 썸네일 생성 플로우

```
[1] 클라이언트 (브라우저/앱)
    원본 이미지를 Canvas API / 네이티브 압축 라이브러리로 1차 축소
    (예: 장변 4000px 이상이면 4000px로 축소, JPEG 품질 85% 압축)
        ↓
[2] S3 원본 버킷 업로드 (Presigned URL 통한 PutObject)
    경로: s3://bucket/original/{image_id}.jpg
    버킷 정책: 원본은 무손실 보존, 라이프사이클 정책으로 장기보관/Glacier 전환 가능
        ↓ (S3 PutObject 이벤트 자동 발생)
        
[3] 메타데이터 기반 라우팅 Lambda (경량, 헤더만 읽음)
    S3 객체의 Content-Length(파일 크기)와 이미지 헤더(가로/세로 해상도)만 확인
    - libvips의 메타데이터 전용 읽기 기능으로 전체 다운로드 없이 헤더만 파싱
    - 기준치 이하 (예: 10MB 미만, 4000px 미만) → 기본 처리 Lambda(SQS A)로 라우팅
    - 기준치 이상 → 고사양 처리 Lambda(SQS B)로 직접 라우팅
        ↓
        ├──────────────────────┐
        ↓                      ↓
[4-A] 기본 처리 Lambda      [4-B] 고사양 처리 Lambda
    메모리: 512MB~1GB           메모리: 3GB~10GB
    (S3 → Lambda 구간은 둘 다 스트리밍 방식으로 객체를 읽음:
     libvips/sharp가 S3 Range Request로 필요한 바이트 범위만 순차 요청,
     전체 파일을 메모리에 미리 적재하지 않음)
        ↓                      ↓
[5] 중간 해상도 정규화 (선택적 계층)
    원본 → 디스플레이 상한 해상도(예: 2000px)로 다운스케일한 "중간본" 생성 및 별도 저장
    경로: s3://bucket/normalized/{image_id}.jpg
    (이후 모든 썸네일은 원본이 아닌 이 중간본 기준으로 생성 → 입력 편차 고정)
        ↓
[6] 썸네일 생성 (여러 사이즈 동시 생성 가능: thumb_small, thumb_medium...)
    경로: s3://bucket/thumbnail/{size}/{image_id}.jpg
        ↓
   ┌─── 성공 ───────────────────┐         ┌─── 실패 ───────────────────┐
   ↓                            │         ↓                            │
[7] 썸네일 S3 저장 완료                                        │   [8] SQS DLQ #1 적재
   DB에 썸네일 URL/상태 업데이트                    │       (maxReceiveCount: 3회 설정,
                                 │        visibility timeout으로 일시적
                                 │        에러는 자체 재시도되도록 함)
                                 │            ↓
                                 │       [9] 재시도 Lambda (실패 사유 분류)
                                 │           - OOM/리소스 부족 → 고사양으로 재시도(최대 1회)
                                 │           - 파일 손상/형식 오류 → 재시도 없이 바로 사용자 알림 큐로
                                 │           - 원인 불명 → SQS DLQ #2로
                                 │            ↓
                                 │       ┌─── 재시도 성공 ──→ [7]로 합류
                                 │       │
                                 │       └─── 재시도 실패 ──┐
                                 │                          ↓
                                 │                  [10] SQS DLQ #2 (최종 실패 큐)
                                 │                  메시지에 실패 사유, 이미지 메타데이터 포함
                                 └──────────────────────────┘
                                                              ↓
[11] CloudWatch (Lambda 실행 메트릭, DLQ #2 메시지 수 메트릭 수집)
        ↓
[12] Grafana (대시보드 + 임계치 기반 알림 규칙)
        ↓ (DLQ #2 메시지 수 > 임계치, 또는 일정 시간 내 발생 시)
[13] Slack 알림 (실패 사유/이미지ID/링크 포함하여 사람이 바로 확인 가능하게)


[병행 트랙] 프론트엔드 Fallback
    썸네일 조회 시 404/실패 응답 → 일정 시간(3초) 대기 후 원본 또는 중간본 표시
    (사용자에게는 무한 로딩 대신 항상 무언가가 보이도록 보장)
```

## 단계별 인프라 요소 요약

| 단계     | 인프라 컴포넌트                     | 역할                                       |
| ------ | ---------------------------- | ---------------------------------------- |
| 클라이언트  | 브라우저/앱 (Canvas API 등)        | 업로드 전 1차 축소, 네트워크 비용 절감                  |
| 업로드    | S3 (Presigned URL)           | 원본 무손실 보관, 클라이언트가 직접 업로드                 |
| 라우팅    | 경량 Lambda                    | 메타데이터만 읽어 적합한 처리 경로로 분기                  |
| 처리     | Lambda (사양 분리: A/B)          | 스트리밍 라이브러리(libvips/sharp)로 메모리 효율 처리     |
| 정규화    | S3 (normalized 경로)           | 이후 모든 처리의 입력 편차를 고정                      |
| 실패 처리  | SQS (DLQ #1, #2)             | maxReceiveCount로 일시적 오류 자동 흡수, 영구 실패만 분리 |
| 재시도 로직 | Lambda (사유 분류)               | 자동 복구 가능한 것과 사람이 봐야 하는 것을 구분             |
| 모니터링   | CloudWatch → Grafana → Slack | 진짜 예외 케이스만 사람에게 알림                       |
| UX 보완  | 프론트엔드 fallback               | 백엔드 장애가 사용자 경험으로 직결되지 않게 차단              |

## 기존 구조 대비 핵심 차이

기존 구조가 **"실패 → 재시도 → 알림"** 중심의 사후 대응이었다면, 이 플로우는

1. **사전 예방** (메타데이터 라우팅, 스트리밍 처리, 중간본 정규화)으로 1차 실패율 자체를 낮추고
2. **재시도 로직 고도화** (사유 분류, 자동 재시도 횟수 조정)로 사람이 봐야 하는 케이스를 줄이고
3. **기존 DLQ + 모니터링 + 알림 구조는 그대로 안전망으로 유지**

하는, "예방이 주력, 복구는 안전망"인 구조로 전환된 형태입니다.

## 썸네일이라는 용도가 만드는 제약 조건

**1. 즉시성(latency)이 핵심 가치** 썸네일은 본질적으로 "목록을 빠르게 훑어보기 위한" 용도예요. 갤러리나 피드에서 수십 장을 동시에 보여줘야 하니, 사용자는 거의 즉각적인 로딩을 기대합니다. 원본 이미지 자체는 좀 늦게 떠도 괜찮지만(상세 화면에서 한 장만 보니까), 썸네일이 안 뜨면 "리스트 전체가 깨진 것처럼" 보여서 체감 장애가 훨씬 크게 느껴져요. 그래서 실패 시 fallback(3초 후 원본 표시)이라는 보완책이 의미가 있는 거고요.

**2. 비동기 생성이라 "생성 실패"라는 상태가 항상 존재** 원본 업로드는 동기적으로 끝나지만, 썸네일은 이 다이어그램처럼 S3 이벤트 → Lambda 트리거 구조로 **항상 비동기로 나중에 생성**됩니다. 즉 "원본은 있는데 썸네일은 아직(혹은 영원히) 없는" 중간 상태가 구조적으로 존재할 수밖에 없어요. 이게 동기 처리라면 애초에 "생성 중 사용자가 기다리는" 형태가 되니 이런 DLQ 복구 설계 자체가 필요 없었을 거예요.

**3. 입력값(원본 이미지)을 서비스가 통제할 수 없음** 썸네일 생성은 사용자가 업로드하는 원본 이미지의 크기/해상도에 그대로 영향을 받아요. 일반 텍스트나 정형 데이터 처리와 달리, 이미지 파일은 사용자마다 천차만별(스마트폰 사진은 작고, 작품 스캔본은 수십~수백 MB일 수 있음)이라 **워커(Lambda)의 리소스를 사전에 고정하기 어렵다**는 문제가 생겨요. 이 글에서 "고화질 작품 사진"이 OOM을 유발한 것도 이 지점이죠.

**4. 비용과 품질의 트레이드오프가 명확히 드러나는 작업** 썸네일은 "어차피 작게 리사이즈할 거니까" 굳이 고사양 리소스를 상시 할당하는 게 낭비예요. 반면 실패하는 소수 케이스(고화질)만 고사양으로 처리하면 효율적이죠. 이런 비용 최적화 여지가 뚜렷하게 보이는 게 이미지 리사이즈/썸네일 작업의 특징이에요. 단순 CRUD API라면 이런 식의 "사양을 나눠서 처리"하는 구조가 잘 안 나오거든요.

## 그래서 일반화하면

이 패턴이 필요한 건 **"썸네일"이라서**라기보다, 다음 조건을 동시에 만족하는 작업이기 때문이에요:

- 비동기/이벤트 기반으로 처리됨
- 입력 데이터의 편차가 커서 리소스 요구량을 예측하기 어려움
- 실패해도 전체 서비스가 죽지는 않지만 사용자 경험에 누적된 악영향을 줌 (무한 로딩, 반복 업로드)
- 처리 비용이 트래픽 규모에 민감해서 일괄 고사양화가 비효율적

이 조건들은 썸네일 생성뿐 아니라, 예를 들어 **동영상 트랜스코딩, PDF 미리보기 생성, OCR, 음성 파일 변환** 같은 작업에서도 똑같이 적용돼요. 

즉 "이미지 썸네일"은 이 패턴이 등장하는 가장 흔한 예시일 뿐, 핵심은 "비동기 + 입력 편차 큼 + 실패해도 즉시 전체 장애는 아님 + 비용 민감"이라는 조건의 조합이라고 보면 됩니다.

---
네, "사용자 행동 제한"이 아니라 **기술로 해결하는 방법**들을 정리해볼게요. 핵심은 "원본을 그대로 서버에 던지지 않고, 서버에 도달하기 전에 이미 다루기 쉬운 형태로 만든다"는 방향이에요.

## 1. 클라이언트 사이드 사전 리사이징 (제한이 아니라 자동 변환)

사용자한테 "용량 제한이니 줄여서 올리세요"라고 요구하는 게 아니라, **브라우저/앱이 알아서 줄여서 업로드**하는 방식이에요.

- 웹: `Canvas` API나 `createImageBitmap()`으로 업로드 전에 클라이언트에서 리사이즈/압축 후 전송
- 모바일 앱: 네이티브 이미지 압축 라이브러리(iOS `UIImage` 리사이즈, Android `Bitmap.compress`)로 업로드 전 처리

이렇게 하면 사용자는 "제한당했다"고 느끼지 않아요. 그냥 평소처럼 올렸는데 내부적으로 적정 크기로 변환되는 거니까요. 다만 원본 보존이 필요한 서비스(작품 사진처럼 디테일이 중요한 경우)라면 "원본은 별도 경로로, 화면 표시용은 자동 축소본으로" 이원화하는 게 나아요.

## 2. Presigned URL + 멀티파트 업로드 후 서버사이드에서 단계적 처리

S3 업로드 자체는 그대로 받되, 업로드 직후 **원본은 그대로 보관하고 화면 노출용 변환은 별도 파이프라인**으로 분리하는 거예요. 이게 사실 지금 구조와 비슷한데, 차이는 "원본=원본 보관용, 디스플레이용=무조건 한 단계 다운스케일된 중간본을 거쳐서 생성"하는 계층을 추가하는 거죠. 즉 원본 → 중간 해상도(예: 2000px 이하) 정규화 → 거기서 썸네일 생성. 이러면 썸네일 Lambda는 항상 일정 크기 입력만 받게 되어 OOM 가능성 자체가 사라져요.

## 3. 스트리밍 방식 이미지 처리 (메모리에 전체를 안 올림)

지금 OOM이 나는 근본 이유는 Lambda가 **이미지 전체를 메모리에 올려서** 처리하기 때문일 가능성이 커요(Pillow, Sharp 같은 라이브러리를 기본 방식으로 쓰면 그렇게 동작). `libvips` 기반 처리(Node의 `sharp`도 내부적으로 libvips 사용)는 스트리밍 방식이라 동일한 이미지를 처리해도 메모리 사용량이 훨씬 적어요. 즉 도구 자체를 메모리 효율적인 라이브러리로 바꾸는 것도 "근본 원인" 해결책 중 하나예요. 이건 라우팅이나 클라이언트 변경 없이도 적용 가능한, 가장 손쉬운 기술적 개선일 수 있어요.

## 4. 업로드 시점에 변환을 강제하는 아키텍처 (Origin 자체를 변환 게이트로)

S3에 직접 원본을 받는 대신, 업로드 게이트웨이(예: CloudFront + Lambda@Edge, 혹은 API Gateway 뒤의 업로드 핸들러)에서 **업로드가 완료되는 시점에 무조건 한 번 정규화 처리**를 거치게 만드는 구조예요. 사용자/클라이언트가 무엇을 보내든 시스템 입구에서 일관된 크기로 normalize 하니, 이후 모든 다운스트림(썸네일 생성 등)은 예측 가능한 입력만 다루게 됩니다.

## 정리하면

| 방법                      | 핵심 아이디어                 | 장점                       |
| ----------------------- | ----------------------- | ------------------------ |
| 클라이언트 사전 리사이즈           | 업로드 전 브라우저/앱에서 자동 축소    | 네트워크 비용도 같이 줄음           |
| 원본→중간본→썸네일 계층화          | 항상 일정 크기 입력만 썸네일 단계에 도달 | 서버 파이프라인이 예측 가능해짐        |
| 메모리 효율 라이브러리(libvips 등) | 처리 방식 자체를 가볍게           | 코드 변경만으로 적용 가능, 가장 빠른 개선 |
| 업로드 게이트웨이에서 정규화 강제      | 입구에서 한 번 무조건 통일         | 다운스트림 전체가 안정화            |

이 중에서 **가장 빠르고 리스크 적은 건 4번째 줄(libvips 같은 스트리밍 처리 라이브러리로 교체)**이에요. 코드 레벨 변경만으로 메모리 사용량을 크게 줄일 수 있어서, 지금 운영 중인 구조를 갈아엎지 않고도 OOM 발생 빈도 자체를 낮출 수 있거든요. 그다음 단계로 "원본/중간본 분리"를 도입하면 구조적으로 거의 OOM이 안 나게 만들 수 있어요.

---
좋은 의문이에요. "이미지를 부분부분 본다면서 어떻게 전체를 줄이는 변환을 하지?"라는 거죠. 핵심은 **이미지 파일 포맷 자체가 처음부터 "부분적으로 읽을 수 있게" 설계되어 있다**는 점이에요.

## 이미지가 저장되는 방식부터 보면

JPEG, PNG 같은 포맷은 픽셀을 위에서 아래로(또는 타일 단위로) **순차적으로** 인코딩해서 저장해요. 즉 파일을 처음부터 끝까지 한 번에 안 읽어도, "지금까지 읽은 부분까지는 디코딩 가능"한 구조예요.

```
파일 구조 예시 (단순화):
[헤더: 가로세로 크기, 포맷 정보] → [1번째 스캔라인] → [2번째 스캔라인] → ... → [마지막 스캔라인]
```

## 리사이즈가 실제로 일어나는 방식

리사이즈는 "이미지를 다 펴놓고 한 번에 줄이는" 연산이 아니라, **입력 스캔라인 몇 줄을 모아서 출력 스캔라인 1줄을 계산**하는 식으로 처리할 수 있어요.

예를 들어 세로를 1/4로 줄인다고 하면:

```
입력 스캔라인 1~4줄 읽음 → 평균/보간 계산 → 출력 스캔라인 1줄 생성 → 출력 스트림에 바로 씀 → 입력 1~4줄은 메모리에서 버림
입력 스캔라인 5~8줄 읽음 → 출력 스캔라인 2줄 생성 → 씀 → 버림
... (반복)
```

즉 **항상 "지금 계산에 필요한 일부분"만 메모리에 있고, 계산이 끝나면 바로 버려져요.** 전체 이미지를 한 번에 메모리에 올릴 필요가 없는 거죠.

## libvips는 이걸 더 정교하게 함 (데맨드 기반 파이프라인)

libvips는 "이 픽셀이 필요해질 때"가 되어서야 그 부분을 디코딩하는 **지연 평가(lazy evaluation) 방식**으로 동작해요.

```
요청: "원본 이미지를 가로세로 1/4로 리사이즈해줘"

libvips 내부 동작:
1. 출력 이미지의 픽셀을 채우기 위해 필요한 영역을 계산
2. 그 영역에 해당하는 입력 파일 부분만 디코딩
3. 계산 → 출력
4. 다음 영역으로 이동, 반복
```

비유하면:

- **일반 방식**: 큰 그림을 전부 인화해서 책상에 펼쳐놓고, 색연필로 줄여 그리기 (그림 전체가 책상에 있어야 함)
- **스트리밍 방식**: 그림을 작은 창문(타일)으로만 보면서, "지금 보이는 부분만큼만" 줄여 그리고, 다 그리면 그 부분 그림은 치우고 다음 창문으로 이동

## 그럼 왜 이게 가능한가? (수학적으로)

리사이즈(다운샘플링)는 본질적으로 **국소적인 연산**이에요. 출력 이미지의 특정 픽셀 1개를 계산하려면 입력 이미지의 "그 근처 일부 영역"만 알면 되지, 이미지 전체를 알 필요가 없어요. (블러나 일부 고급 필터처럼 넓은 범위를 참조하는 연산도 있지만, 그래도 "전체"는 아니고 일정 반경 정도예요.)

그래서 **순차적으로 들어오는 데이터를 부분부분 처리해도 정확한 리사이즈 결과를 만들 수 있는 것**이고, 이게 가능하기 때문에 스트리밍/타일 기반 처리가 메모리 효율적으로 동작할 수 있는 거예요.

---

정리하면, "나뉘어서 온다"는 게 "처리가 불완전하다"는 뜻이 아니라, **"어차피 부분부분만 봐도 정확하게 계산 가능한 연산이라서, 굳이 전체를 한꺼번에 들고 있을 필요가 없다"**는 의미예요. 그래서 메모리 사용량은 입력 이미지 전체 크기가 아니라, "한 번에 처리하는 작은 윈도우 크기" 정도로 고정될 수 있는 거죠.
