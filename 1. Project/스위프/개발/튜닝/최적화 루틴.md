
# 기초 개념
**메모리부터 보면**, 핵심은 "힙이 실제로 얼마나 차 있는가"와 "그게 GC 이후에도 안 줄어드는 진짜 live set인가"를 구분하는 겁니다. docker stats에 찍히는 숫자는 JVM이 미리 committed 해놓은 영역까지 포함하기 때문에 실제 사용량과는 다릅니다. 그래서 보셔야 할 건 세 가지인데,

먼저 GC 로그입니다. `-Xlog:gc*:file=/var/log/gc.log:time,uptime,level,tags` 정도로 켜두면 Full GC(또는 Old GC) 직후의 used 사이즈가 찍히는데, 이게 곧 "GC가 다 청소하고도 남는 진짜 살아있는 객체량"입니다. 이 값이 힙 사이즈를 정하는 진짜 하한선이에요. Xmx를 이 값보다 약간 여유 있게(보통 1.3~1.5배) 잡는 게 기본 원칙입니다.

다음은 idle 상태 기준선입니다. 트래픽 없을 때 jcmd로 `jcmd <pid> GC.heap_info`를 찍어보면 애플리케이션 자체(Spring 컨텍스트, 캐시, 커넥션 풀 등)가 깔고 시작하는 메모리가 보입니다. 이게 최소값이고, 이 위로 트래픽에 따라 얼마나 늘어나는지를 봐야 피크를 알 수 있습니다.

그래서 세 번째가 부하 테스트입니다. 이미 k6/JMeter 쓰고 계시니까, 예상 동시 사용자 수 정도로 ramp 테스트를 돌리면서 Actuator/Micrometer가 노출하는 `jvm_memory_used_bytes{area="heap"}`를 Grafana에서 실시간으로 보는 게 제일 정확합니다. 그 테스트에서 찍힌 피크치가 진짜 기준이 되는 거고, Xmx는 그 피크치 + 여유분으로 정하면 됩니다. 추측이 아니라 관찰값으로 역산하는 거죠.

비힙(메타스페이스, 코드캐시, 스레드 스택)도 감으로 잡지 말고 `-XX:NativeMemoryTracking=summary`를 켠 다음 `jcmd <pid> VM.native_memory summary`로 보면 영역별 실측치가 나옵니다. 코멘트의 MetaspaceSize=64M, CodeCacheSize=48m 같은 값도 이 출력 보고 실제 쓰는 양보다 살짝 위로 잡는 게 맞고, 무작정 줄이면 오히려 메타스페이스 부족으로 GC가 더 자주 도는 역효과가 날 수 있습니다.

**스레드 풀 쪽은 기준이 조금 다릅니다.** Tomcat max thread는 메모리 문제가 아니라 동시성 문제로 접근해야 해요. Little's Law로 보면 필요한 동시 처리량 ≈ 초당 요청 수 × 평균 응답시간입니다. 예를 들어 평균 응답시간이 100ms고 피크 시 초당 20건이 들어온다면 필요한 동시 스레드는 대략 2개 수준이고, 거기에 스파이크 여유를 더하는 식이죠. 50이라는 숫자가 합리적인지는 이 계산 없이는 사실 알 수 없습니다.

여기서도 추측보다 실측이 우선인데, Actuator가 노출하는 `tomcat_threads_busy_threads`와 `tomcat_threads_current_threads`를 부하 테스트 중에 Grafana로 같이 보시면 됩니다. 목표 트래픽으로 테스트 돌렸을 때 busy 스레드가 50 근처까지 안 가고 5~10 정도에서 머문다면 50은 과한 설정이고, 반대로 풀이 거의 다 차서 큐잉이 생긴다면 늘려야 하는 거고요.

그리고 한 가지 더 중요한 게, 스레드 풀 크기는 HikariCP 풀 크기와 묶어서 봐야 합니다. 지금 코드처럼 DB I/O가 블로킹이면, 스레드가 100개 있어도 HikariCP 커넥션이 5개면 그 이상 동시에 처리 못 하고 나머지는 그냥 대기 상태로 스레드 스택 메모리만 먹습니다. 그러니까 Tomcat max thread를 정할 때는 "DB 커넥션 풀 사이즈 대비 얼마나 여유를 둘 것인가"를 같이 고려해야 의미가 있어요. 실제 동시 DB 호출이 5개를 넘기 어렵다면 스레드를 50까지 둘 이유가 크지 않은 거죠.

정리하면, 순서는 (1) Actuator/Micrometer로 jvm.memory, tomcat.threads, hikaricp 메트릭 노출 확인 → (2) GC 로그 + NMT로 idle 기준선 측정 → (3) k6/JMeter로 예상 피크 트래픽 시뮬레이션하면서 Grafana로 실시간 관찰 → (4) 관찰된 피크치 + 여유분으로 Xmx, max thread, Hikari pool 값을 역산 → (5) 적용 후 재테스트로 GC pause나 thread 큐잉이 안 생기는지 검증, 이렇게 데이터 기반으로 한 바퀴 돌리는 게 맞는 접근입니다. 이미 구축해두신 Alloy/Prometheus/Grafana 스택이 있으니 추가 인프라 없이 바로 측정 들어가실 수 있을 거예요.

# JVM 메모리/스레드 튜닝 — 데이터 기반 적용 가이드

목표: 추측값(Xmx 320m, thread max 50 등)이 아니라 실측 데이터로 컨테이너 메모리/스레드 설정값을 도출한다.

---

## 0단계. 사전 점검 — 메트릭 노출 확인

- [ ] `micrometer-registry-prometheus` 의존성이 build.gradle에 있는지 확인
- [ ] application.yml에 아래 설정 추가

```yaml
management:
  endpoints:
    web:
      exposure:
        include: health,prometheus,metrics
  metrics:
    export:
      prometheus:
        enabled: true
```

- [ ] `curl http://localhost:8080/actuator/prometheus | grep -E "jvm_memory|tomcat_threads|hikaricp"` 로 세 종류 메트릭이 다 찍히는지 확인

이게 안 나오면 2~3단계 측정 자체가 불가능하므로 가장 먼저 확정해야 함.

---

## 1단계. Idle 기준선 측정 (트래픽 없는 상태)

- [ ] GC 로그 활성화

```
-Xlog:gc*:file=/var/log/gc.log:time,uptime,level,tags
```

- [ ] NMT 활성화

```
-XX:NativeMemoryTracking=summary
```

- [ ] 앱 기동 후 5~10분 idle 유지한 뒤 측정

```bash
jcmd <pid> GC.heap_info
jcmd <pid> VM.native_memory summary
```

- [ ] 기록할 값
    - idle heap used / committed
    - idle metaspace / code cache used
    - idle thread count

이 값이 "아무것도 안 해도 깔고 가는" 최소 바닥값.

---

## 2단계. 부하 테스트로 피크 관찰

- [ ] k6/JMeter로 목표 트래픽 기준 ramp 테스트 설계 (예: 5분간 0→목표 RPS, 10분 유지)
- [ ] Grafana에 아래 PromQL 패널 추가

|패널|PromQL|
|---|---|
|Heap 사용량|`jvm_memory_used_bytes{area="heap"}`|
|GC pause|`jvm_gc_pause_seconds_sum` / `_count`|
|Tomcat 사용 스레드|`tomcat_threads_busy_threads` vs `tomcat_threads_current_threads`|
|Hikari 활성/대기|`hikaricp_connections_active` / `hikaricp_connections_pending`|

- [ ] 테스트 중 기록할 값
    - heap peak
    - Full(Old) GC 빈도/pause 시간
    - busy thread peak
    - hikari active peak / pending이 0보다 큰 구간 있었는지

---

## 3단계. 수치 역산

|항목|산출 방식|
|---|---|
|Xmx|(idle 직후 GC 안정화 heap used, 또는 부하테스트 heap peak 중 더 큰 값) × 1.3~1.5|
|Xms|Xmx의 50~70%|
|Metaspace/CodeCache|NMT 실측치 + 20~30% 여유|
|Tomcat max threads|(목표 RPS × 평균 응답시간(초)) × 1.5~2 (Little's Law 기반 spike 여유)|
|Hikari pool size|부하테스트 중 관찰된 active connection peak + 1~2|
|컨테이너 `-m`|Xmx + Metaspace max + CodeCache + (스레드수 × 1MB) + 50~100MB 여유|

추측치가 아니라 위 표의 입력값은 모두 1~2단계에서 관찰한 실측 숫자여야 함.

---

## 4단계. 적용

- [ ] `deploy.sh`에 `-m`, `--memory-swap` 반영
- [ ] Dockerfile `JAVA_OPTS` 갱신 (Xms/Xmx, Metaspace, CodeCache)
- [ ] `application.yml`의 tomcat.threads, hikari.maximum-pool-size 반영
- [ ] 재배포

---

## 5단계. 재검증

동일한 부하 테스트를 다시 돌려서 확인:

- [ ] heap 사용량이 새 Xmx 대비 70~80% 이하로 유지되는가
- [ ] Full GC 빈도가 이전보다 늘지 않았는가
- [ ] busy thread가 새 max thread 대비 여유 있는가
- [ ] hikari pending이 0에 가까운가

문제가 있으면 3단계 산출값을 조정하고 4~5단계 반복. 트래픽 패턴이 크게 바뀔 때마다(신규 기능 배포, 사용자 급증 등) 이 사이클을 다시 도는 게 안전함.
