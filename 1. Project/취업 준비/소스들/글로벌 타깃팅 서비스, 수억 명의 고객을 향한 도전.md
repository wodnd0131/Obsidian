# 글로벌 타깃팅 서비스, 수억 명의 고객을 향한 도전
우아한형제들 서버 개발자 박영제님이 발표한 **"글로벌 타깃팅 서비스, 수억 명의 고객을 향한 도전"** 동영상의 전체 내용을 이전과 같이 상세히 정리해 드립니다.

### 1. 배경: 우아한 버즈(Buzz)와 타깃팅의 중요성

우아한형제들의 '버즈(Buzz)'는 배달의민족 유저 데이터를 기반으로 하는 타깃팅 시스템입니다. 타깃팅 없이 모든 유저에게 동일한 쿠폰이나 메시지를 보내면 스팸으로 여겨질 수 있지만, **적절한 타깃팅을 통해 꼭 필요한 유저에게 맞춤형 혜택을 제공하면 높은 비즈니스 성과로 전환**될 수 있습니다. 연령, 성별, 최근 주문 내역, 보유 쿠폰 등 수백 개의 유저 속성을 조합하여 타깃이 되는 대상 그룹인 '세그먼트(Segment)'를 생성합니다.

### 2. 세그먼트 생성을 위한 핵심 기술: 오픈서치(OpenSearch)

복잡한 조건의 세그먼트 고객 수 분포를 단 1초 만에 로드하기 위해 '오픈서치' 데이터베이스를 활용합니다.

- **도입 이유:** 빠른 검색과 집계가 가능하며, 스키마 리스(Schema-less) JSON 형태로 다양한 타입의 속성 데이터를 유연하게 저장할 수 있습니다.
- **강점:** AND, EXCLUDE 등의 복잡한 조건을 묶어주는 강력한 쿼리 DSL을 지원하며, 내부 샤드 구성을 통해 고객 데이터가 수백 개의 속성을 가져도 수평적 확장이 용이합니다.

### 3. 글로벌 서비스 '세그멘텀(Segmentum)'으로의 확장과 한계 극복

버즈 시스템을 글로벌 타깃팅 서비스인 '세그멘텀'으로 확장하면서 총 5개 리전, 70개 국가의 약 1억 5천만 명의 고객을 대상으로 서비스를 제공하게 되었습니다. 특히 글로벌 진출과 함께 유저가 앱을 켰을 때 **실시간으로 맞춤형 광고를 노출하는 '풀 퍼스널라이제이션(Full Personalization)' 기능을 새롭게 구축**해야 했습니다. 이를 위해 **엄격한 API 응답 속도(95분위 60ms 이하), 거대한 데이터 스케일, 수만 TPS의 처리량, 그리고 70개국 곱하기로 불어날 수 있는 인프라 비용 효율화**라는 네 가지 요구 사항을 만족시켜야 했습니다.

### 4. 아키텍처 결정 및 데이터베이스 비용 최적화

글로벌 서비스의 레이턴시와 비용 문제를 해결하기 위해 다음과 같은 아키텍처 및 DB 최적화를 진행했습니다.

- **리전 단위 배포:** 중앙 집중화(속도 저하/장애 전파)나 70개국 개별 배포(막대한 비용/관리 불가) 대신, 한국 포함 총 4개의 리전 단위로 배포하여 레이턴시와 운영 부담을 최소화했습니다.
- **다이나모DB(DynamoDB) 선택:** 레디스(비용 문제), 카산드라(운영 난이도) 대신 무한 확장이 가능하고 일관된 성능을 내는 다이나모DB를 최종 저장소로 선택했습니다.
- **쓰기 비용 절감 (세그먼트 CDC):** 다이나모DB의 비싼 쓰기 비용을 줄이기 위해, 매일 전체 데이터를 삭제/삽입하는 대신 전날과 비교하여 **추가되거나 삭제된 약 3~4%의 변경분(Delta)만 업데이트**하는 스냅샷 방식을 도입했습니다.
- **스토리지 비용 절감 (TTL 활용):** 캠페인이 끝난 세그먼트 데이터를 명시적으로 삭제하면 추가 과금이 발생하므로, **다이나모DB의 TTL(Time To Live) 기능을 이용해 캠페인 종료 시 무료로 자동 삭제**되도록 설계했습니다.

### 5. 서버 응답 속도(Latency) 및 처리량 극대화 전략

풀 퍼스널라이제이션 과정에서 응답 속도를 한계까지 끌어올리기 위해 서버 단에서 여러 기법을 적용했습니다.

- **로컬 캐시 도입:** 세그먼트의 유효성을 검사하는 PostgreSQL DB는 블로킹 I/O 기반이라 성능 병목이 발생할 수 있으므로, **앞단에 '카페인 캐시(Caffeine Cache)'를 두어 15초마다 데이터를 폴링**하게 함으로써 다이나모DB 호출을 1회로 줄였습니다.
- **커넥션 재사용 및 페일패스트(Fail-fast):** TLS 핸드쉐이크 시간을 아끼기 위해 커넥션 풀을 유지하고, 다이나모DB 응답이 지연될 경우 **차라리 타임아웃을 짧게(예: 50ms) 설정해 빨리 실패 처리한 후 재시도**하는 페일패스트 전략을 썼습니다.
- **리퀘스트 헤징(Request Hedging):** 요청을 연달아 두 개 보내고 먼저 온 응답을 채택하여 테일 레이턴시(Tail Latency)를 최적화했습니다.
- **놀라운 성과:** 이러한 노력 덕분에 글로벌 세그멘텀과 사내 버즈 서비스 모두 **응답 속도 10~20ms를 달성했으며, 초당 15,000~20,000 이상의 막대한 트래픽을 안정적으로 처리**하게 되었습니다.

### 6. 전면적인 코드와 인프라의 개편

- **하이버네이트(Hibernate) 필터 적용:** 국가(글로벌 엔티티 ID)를 구분하기 위해 모든 소스 코드를 수정하는 대신, **AOP와 `@Filter` 어노테이션을 활용해 세션 단위에서 쿼리에 국가 조건절이 동적으로 자동 삽입**되도록 하여 기존 코드의 변경을 최소화했습니다.
- **인프라 통합의 어려움 극복:** 사내 시스템(버즈)과 딜리버리히어로 센트럴 시스템(세그멘텀) 간의 소스 저장소(GitLab vs GitHub), 인증(Google vs Okta), 배포(EC2 vs EKS) 방식이 모두 달라, 이를 인터페이스로 추상화하여 구현하는 방식으로 해결했습니다.

### 7. 글로벌 론칭 과정에서 얻은 교훈 (후기)

- **기술 부채 청산의 중요성:** 확장성을 고려하지 않았던 기존 코드를 글로벌과 통합 운영하기 위해 3개월 이상 빚을 갚는 고통이 있었으나, 이 과정에서 **미래를 대비하는 아키텍처 설계 역량**을 기를 수 있었습니다.
- **넓어진 기술 스택과 운영 부담:** 개발자들이 넓어진 기술 스택(인프라, 네트워크, 테라폼 등)을 직접 다뤄야 했고, 독일과의 7시간 시차 탓에 낮과 밤을 나눠 개발해야 하는 어려움이 있었습니다.
- **원팀(One Team) 문화:** 비록 물리적 거리와 언어/문화의 장벽은 있었지만, 도입하는 기술의 타당성을 꼼꼼히 검토하고 서로 신뢰하고 존중하며 협업하는 자산을 얻을 수 있었다고 평가하며 발표를 마쳤습니다.
