## 실행 결과로 생기는 것들

### 1. 아카이브 테이블 6개

```
archive_users
archive_room
archive_room_user
archive_template
archive_template_wish
archive_wish
```

각 테이블의 컬럼 구조는 동일해요.

```sql
id             -- 아카이브 테이블 자체 PK
original_id    -- 원본 테이블의 id
archived_data  -- 원본 row 전체를 JSON으로 직렬화한 값
created_at     -- 원본 row의 created_at
updated_at     -- 원본 row의 updated_at
source_table   -- 어느 테이블에서 왔는지 (예: 'users')
archived_at    -- 아카이빙된 시각
flyway_version -- 아카이빙 당시 DB 스키마 버전
data_hash      -- 원본 row 전체의 SHA-256 해시
```

---

### 2. 프로시저 1개 — `archive_table`

호출하면 아래 순서로 동작해요.

```
1. flyway_schema_history에서 현재 스키마 버전 조회
2. 대상 테이블의 컬럼 목록을 INFORMATION_SCHEMA에서 동적으로 읽음
3. deleted = 1인 행을 JSON으로 직렬화해서 아카이브 테이블에 INSERT
4. 원본 테이블에서 해당 행 DELETE
5. 실패하면 전체 ROLLBACK
```

---

### 3. 이벤트 스케줄러 1개 — `archive_all_daily`

매일 **KST 새벽 2시**에 자동 실행돼요. (`STARTS '2025-10-17 17:00:00'` → UTC 기준이라 KST +9 = 02:00)

실행 순서는 아래와 같아요.

```
template_wish → wish_v2 → room_user → room → users → template
```

순서가 의미 있는 게, 참조 관계상 자식 테이블을 먼저 아카이빙하고 부모를 나중에 처리하는 구조예요.

---

### 한 사이클의 전체 흐름 요약

```
[새벽 2시]
  ↓
스케줄러 실행
  ↓
각 테이블에서 deleted = 1인 행 추출
  ↓
archive_* 테이블에 JSON + 해시 + 버전 정보와 함께 INSERT
  ↓
원본 테이블에서 DELETE
  ↓
원본 테이블은 active 데이터만 남음 → 조회 성능 회복
```

# 소프트 딜리트 데이터 아카이빙 방법 비교

## 비교 대상 3가지

| |같은 DB 아카이브 테이블|별도 DB|S3|
|---|---|---|---|
|**현재 설계**|✅|||
|**저장 위치**|같은 MySQL 인스턴스|별도 RDS 인스턴스|오브젝트 스토리지|
|**조회 방법**|SQL 그대로|SQL 그대로|Athena / 앱 레이어|
|**비용**|추가 없음|RDS 인스턴스 비용|매우 저렴|

---

## 각 방법의 실제 차이

### 같은 DB 아카이브 테이블 (현재 설계)

**실제로 얻는 것**

- 아카이브 데이터 조회할 때 SQL 그대로 사용 가능
- 추가 인프라 없음, 추가 비용 없음
- 트랜잭션으로 원본 DELETE와 아카이브 INSERT가 원자적으로 처리됨

**실제로 잃는 것**

```
원본 테이블 부하 → 아카이브 테이블도 같이 영향받음

예: RDS 인스턴스 디스크 IOPS 한계에 도달하면
    원본 테이블 조회 + 아카이브 INSERT 가 같이 느려짐
```

그리고 아카이브 테이블이 커질수록 같은 DB 용량을 잡아먹어요. 원본 테이블 성능을 개선하려고 아카이빙했는데, 같은 인스턴스 안에서 디스크는 계속 늘어나는 구조예요.

---

### 별도 DB

**실제로 얻는 것**

- 원본 DB 부하와 완전히 분리
- 아카이브용 DB는 스펙을 낮게 잡아도 됨 (읽기 거의 없음)
- SQL 조회 가능

**실제로 잃는 것**

```
트랜잭션 범위가 DB를 넘을 수 없음

원본 DB DELETE ──┐
                  ├── 두 작업이 원자적으로 묶이지 않음
별도 DB INSERT ──┘

→ 원본 DELETE 성공, 별도 DB INSERT 실패하면 데이터 유실
→ 이걸 막으려면 애플리케이션 레이어에서 별도 처리 필요
```

그리고 RDS 인스턴스를 하나 더 띄우는 비용이 발생해요. 아카이브 데이터를 거의 조회 안 한다면 SQL 기능에 비해 비용이 과한 구조예요.

---

### S3

**실제로 얻는 것**

- 비용이 가장 저렴 (GB당 $0.023, RDS 대비 1/10 수준)
- 용량 제한 없음
- 장기 보관에 최적

**실제로 잃는 것**

```
SQL로 직접 조회 불가능

아카이브 데이터 조회가 필요한 경우:
  - Athena 쿼리 (비용 발생, 쿼리 속도 느림)
  - 앱 레이어에서 S3 → 파싱 → 처리 로직 직접 구현

그리고 트랜잭션 없음:
  원본 DB DELETE와 S3 PUT이 원자적으로 묶이지 않음
  → 실패 시 롤백 불가, 재시도 로직 직접 구현 필요
```

---

## 현재 설계가 맞는 상황 vs 바꿔야 할 상황

**현재 설계(같은 DB)가 맞는 상황**
- 아카이브 데이터를 가끔이라도 SQL로 조회해야 할 때
- 인프라 복잡도를 늘리고 싶지 않을 때
- 아카이브 데이터 규모가 수억 건 수준이 아닐 때

**S3로 바꿔야 할 상황**
- 아카이브 데이터가 폭발적으로 쌓여서 같은 DB 디스크 비용이 부담될 때
- 아카이브 조회가 거의 없고 법적 보관 목적일 때

**별도 DB가 맞는 상황**
- 아카이브 데이터를 SQL로 자주 조회해야 하는데, 원본 DB 부하 분리가 반드시 필요할 때
- 사실 이 케이스는 별도 DB보다 읽기 전용 레플리카가 더 현실적인 선택이에요
