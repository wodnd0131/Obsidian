
## Tag 캐싱 아키텍처 분석

### 전체 구조 요약

```
TIL 생성
  └─> TilTagGenerateListener (비동기 이벤트)
        └─> TilTagService.createTagsDirect()   ← AI 호출로 Tag/SubTag 생성 → DB 저장
              실패 시 → TagCreationOutboxService → 재시도

TagScheduledJob (30분마다, matilda.cache.tag.enabled=true 일 때만)
  └─> TagRelationService.renewCoreTagsRelation()  ← DB에서 TagRelation 재계산
  └─> TilTagService.getRecentWroteTags/SubTags()  ← 최근 500일치 Tag/SubTag 조회
  └─> new TilTagRelations(...)                    ← 도메인 객체 조립
  └─> RecentTilTagsCache.updateRecentTagRelations() ← Redis에 저장

GET /api/tags/recent
  └─> RecentTilTagsCache.getRecentTagRelations()  ← Redis에서 조회
```

---

### 1. Redis 캐시 설정 ([RedisConfig.java](https://claude.ai/epitaxy/src/main/java/com/tilguys/matilda/common/config/RedisConfig.java))

| 항목      | 값                                           |
| ------- | ------------------------------------------- |
| 캐시 이름   | `tilTags`                                   |
| TTL     | **30분**                                     |
| 키 직렬화   | `StringRedisSerializer`                     |
| 값 직렬화   | `GenericJackson2JsonRedisSerializer` (JSON) |
| null 캐싱 | 비활성화 (`disableCachingNullValues`)           |

---

### 2. RecentTilTagsCache ([cache/RecentTilTagsCache.java](https://claude.ai/epitaxy/src/main/java/com/tilguys/matilda/tag/cache/RecentTilTagsCache.java))

Redis 캐시를 직접 감싸는 컴포넌트로 두 가지 레이어를 가집니다:

```
Redis (tilTags::recent:til:relations)   ← 1차 저장소
  실패 시 ↓
this.recentTagRelations (인스턴스 필드) ← 2차 폴백 (JVM 인메모리)
```

- **캐시 키**: `"recent:til:relations"` (고정)
- **Failover 로직**: Redis에서 캐시 조회 실패(예외 발생 또는 null) 시 인스턴스 변수 `recentTagRelations`로 폴백
- **주의점**: `updateRecentTagRelations()`가 최초 한 번 호출되기 전엔 폴백도 `null`이라 NPE 위험

---

### 3. TagScheduledJob ([schedule/TagScheduledJob.java](https://claude.ai/epitaxy/src/main/java/com/tilguys/matilda/tag/schedule/TagScheduledJob.java))

**캐시 갱신 주기**: `0 */30 * * * *` → **30분마다**

**조건부 활성화**: `@ConditionalOnProperty(name = "matilda.cache.tag.enabled", havingValue = "true")` → 이 프로퍼티가 없으면 배치 자체가 빈으로 등록되지 않음

**갱신 흐름**:

1. `renewCoreTagsRelation()` → `tag_relation` 테이블 전체 삭제 후 최근 500일치 TIL 기반으로 재계산 (배치: 1,000건 단위)
2. 최근 500일(`TAG_GET_START_DAY = 500`)의 Tag, SubTag 조회 + 삭제된 TIL 필터링
3. `TilTagRelations` 도메인 객체 생성
4. Redis에 `put`

---

### 4. TilTagRelations ([domain/TilTagRelations.java](https://claude.ai/epitaxy/src/main/java/com/tilguys/matilda/tag/domain/TilTagRelations.java))

캐시에 직렬화되어 저장되는 핵심 도메인 객체로 세 가지 맵을 보유합니다:

|필드|타입|의미|
|---|---|---|
|`keywordTagMap`|`Map<String, List<String>>`|코어태그 → 서브태그(키워드) 목록|
|`tagTilIdMap`|`Map<String, List<Long>>`|코어태그 → 해당 태그를 가진 TIL ID 목록|
|`tagRelationMap`|`Map<String, List<String>>`|코어태그 → 연관 태그 문자열 목록|

초기 용량을 `Math.max(16, tags.size() / 2)`로 잡아 HashMap 리사이징을 최소화합니다.

---

### 5. 태그 생성 흐름 (캐시 비갱신)

TIL 생성 → 태그 DB 저장은 캐시를 **직접 건드리지 않습니다**. 스케줄러가 30분마다 일괄 재계산하는 **배치 갱신** 방식입니다.

```
TilTagGenerateListener (AFTER_COMMIT, Async)
  ├─ createTagsDirect() 성공 → 완료
  └─ 실패 → Outbox 저장 → processEventAsync(100ms 후)
        실패 → 30초마다 PENDING 재처리 / 1분마다 FAILED 재시도
        (재시도 간격: 5분 * retryCount, 매일 새벽 2시 7일 이상 이벤트 정리)
```

---

### 잠재적 문제점

1. **캐시 콜드 스타트**: 서버 시작 직후 30분 배치 전까지 `recentTagRelations`가 `null`이므로 `/api/tags/recent` 호출 시 500 오류 발생 가능
2. **Redis TTL = 배치 주기**: TTL(30분) = 배치 주기(30분)인데, 배치가 늦어지면 TTL 만료 후 Redis가 비어 폴백(null)으로 내려갈 수 있음
3. **단일 캐시 키**: `"recent:til:relations"` 고정 키 하나로만 저장되므로 사용자별/카테고리별 분리 없이 전체 공유
4. **`renewCoreTagsRelation()` deleteAllInBatch**: TagRelation 테이블 전체 삭제 후 재삽입하는 방식으로, 배치 실행 중간에 `getRecentRelationTagMap()` 조회가 들어오면 빈 결과를 볼 수 있음 (쓰기-읽기 간 레이스)

---


## 개발자가 고려한 설계 요소들

### 1. 왜 "스케줄러 배치 갱신"을 택했는가

태그 조회는 **전체 유저가 공유**하는 데이터다. TIL이 하나 생길 때마다 캐시를 갱신하면 초당 수십 번의 캐시 재계산이 일어날 수 있다.

```
이벤트 기반 갱신 (나쁜 이유)
  TIL 생성 → AI 호출(수초) → Tag 저장 → 캐시 재계산(DB 대량 조회)
  → TIL이 동시에 100개 생성되면? → 100번 재계산

배치 갱신 (선택한 이유)
  30분에 딱 한 번, 아무리 TIL이 많이 생겨도 재계산은 1번
```

**면접 포인트**: "실시간성보다 처리 비용을 우선했다. 태그 추천은 30분 오래된 데이터여도 사용자가 체감 못 한다. 반면 DB 대량 조회를 매 TIL 생성마다 하면 DB에 심각한 부하가 생긴다."

---

### 2. `TilTagRelations`를 별도 도메인 객체로 분리한 이유 (Read Model 패턴)

세 개의 Map을 미리 조립해서 캐시에 넣는다. API가 `keywordTagMap`, `tagRelationMap`, `tagTilIdMap`을 각각 따로 Redis에서 조회하는 게 아니라 **하나의 객체**를 한 번에 꺼낸다.

```
나쁜 설계 (Redis 키 3개 따로 관리)
  GET recent:keyword:map   → 역직렬화
  GET recent:relation:map  → 역직렬화
  GET recent:til:id:map    → 역직렬화

선택한 설계 (Redis 키 1개)
  GET recent:til:relations → TilTagRelations 하나로 역직렬화
```

**면접 포인트**: "CQRS의 Read Model 개념과 유사하다. 쓰기(Tag/SubTag 저장)와 읽기(태그 추천 응답)의 모델을 분리하여 읽기 경로를 최적화했다."

---

### 3. `@ConditionalOnProperty`로 환경별 배치 분리

```java
@ConditionalOnProperty(name = "matilda.cache.tag.enabled", havingValue = "true", matchIfMissing = false)
```

`matchIfMissing = false`이므로 **프로퍼티가 없으면 빈 자체가 등록 안 된다**.

**의도**: 로컬/테스트 환경에서는 AI 호출도 없고 배치 스케줄러도 안 돌게 한다. 테스트 파일에서 `@TestPropertySource(properties = {"matilda.cache.tag.enabled=true"})`로 명시적으로 켜는 걸 보면 의도적인 설계다.

**면접 포인트**: "Feature Toggle 개념이다. 인프라가 없는 개발 환경에서 의도치 않은 외부 호출을 차단하는 방어적 설계다."

---

### 4. 이중 폴백 (Redis → 인메모리)

```java
public TilTagRelations getRecentTagRelations() {
    try {
        TilTagRelations cached = cache.get(...);
        if (cached == null) throw new MatildaException(...);
        return cached;
    } catch (Exception e) {
        log.error(...);
        return recentTagRelations;  // JVM 인메모리 폴백
    }
}
```

Redis가 다운되거나 네트워크 단절 시에도 **직전에 메모리에 올려둔 값**으로 서비스를 유지한다.

**면접 포인트**: "Circuit Breaker 없이 수동으로 구현한 Failover 패턴이다. Redis 의존성을 소프트하게 처리하여 가용성을 높였다."

---

### 5. `JOIN FETCH`로 N+1 사전 차단

```java
// TagRepository
@Query("SELECT t FROM Tag t JOIN FETCH t.til WHERE t.createdAt >= :start")

// SubTagRepository  
@Query("SELECT st FROM SubTag st JOIN FETCH st.tag t JOIN FETCH t.til WHERE st.createdAt >= :start")
```

배치에서 수천 개의 Tag/SubTag를 가져올 때 TIL 정보가 Lazy Loading으로 N+1이 터지는 걸 미리 막았다.

**면접 포인트**: "배치 컨텍스트에서는 영속성 컨텍스트 생명주기가 길지 않아 LazyInitializationException도 발생할 수 있다. JOIN FETCH는 성능과 안전성 두 가지를 동시에 해결한다."

---

### 6. `TagRelation`을 별도 테이블로 관리하는 이유

같은 TIL에 묶인 태그들을 `(tag, otherTag)` 쌍으로 미리 펼쳐서 저장한다. 예를 들어 TIL에 태그 A, B, C가 붙으면 `(A,B), (A,C), (B,A), (B,C), (C,A), (C,B)` 6개를 저장한다.

```
런타임에 연관 태그 계산하면?
  "A와 연관된 태그" → Tag 전체를 같은 TIL 기준으로 join → 복잡한 쿼리, 느림

TagRelation 테이블이 있으면?
  SELECT otherTag FROM TagRelation WHERE tag = 'A' → 단순 조회
```

**면접 포인트**: "Materialized View 개념의 애플리케이션 레벨 구현이다. 쓰기 비용(30분 배치 재계산)을 지불하고 읽기 비용을 0에 가깝게 만들었다."

---

### 7. Transactional Outbox 패턴 (태그 생성)

```
TIL 생성 트랜잭션 커밋
  └─> AFTER_COMMIT 이벤트 → AI 직접 호출 시도
        성공 → 완료
        실패 → Outbox 테이블에 PENDING 저장
                → 30초마다 재처리, 1분마다 실패 재시도
                → 재시도 간격: 5분 * retryCount (지수 백오프)
                → 매일 새벽 2시 7일치 이벤트 정리
```

**면접 포인트**: "AI API는 외부 의존성이라 실패할 수 있다. Outbox 패턴으로 'TIL은 생성됐지만 태그가 없는' 상태를 일시적으로 허용하고 Eventually Consistent하게 보정한다."

### 문제 2: TTL과 배치 주기의 위험한 동기화

TTL 30분 = 배치 주기 30분인데, 배치가 **1초라도 늦으면** Redis에서 데이터가 만료된 후 빈 상태로 요청이 들어온다. 그 사이 인메모리 폴백이 버티지만 멀티 인스턴스 환경(아래 참고)에서는 각 인스턴스마다 폴백 값이 다를 수 있다.

**해결 방향**: TTL을 배치 주기보다 여유있게 설정 (`Duration.ofMinutes(60)` 등) 하여 겹치는 구간을 확보.

---

### 문제 3: 멀티 인스턴스에서 인메모리 폴백 불일치

```java
@Component
public class RecentTilTagsCache {
    private TilTagRelations recentTagRelations;  // 인스턴스 변수
```

인스턴스 A는 마지막 배치 결과를 갖고 있지만, 새로 뜬 인스턴스 B는 배치가 한 번도 안 돌았으면 `null`이다. 로드밸런서가 B로 요청을 보내면 NPE.

**해결 방향**: Redis를 단일 진실 공급원(Single Source of Truth)으로 두고, 서버 시작 시 Redis에서 읽어 인메모리를 채우는 초기화 로직 추가.

### 문제 4: `renewCoreTagsRelation()`의 공백 구간

### 문제 6: `GenericJackson2JsonRedisSerializer`의 타입 정보 문제

```java
.serializeValuesWith(...new GenericJackson2JsonRedisSerializer())
```

이 직렬화기는 JSON에 `@class` 필드를 포함시킨다. `TilTagRelations` 클래스 경로가 변경되면 Redis에 남은 이전 데이터를 역직렬화하지 못해 예외가 발생한다.

**면접 포인트**: "클래스 리팩터링(패키지 이동, 클래스명 변경)이 운영 중 배포 시 캐시 역직렬화 오류로 이어질 수 있다. 배포 전 Redis flush 또는 버전 키 도입이 필요하다."

### 문제 7: 단일 캐시 키의 확장성 한계
