
## Tag 캐싱 아키텍처 분석

### 전체 구조 요약

```
TIL 생성
  └─> TilTagGenerateListener (비동기 이벤트)
        └─> TilTagService.createTagsDirect()   ← AI 호출로 Tag/SubTag 생성 → DB 저장
              실패 시 → TagCreationOutboxService → 재시도

TagScheduledJob (30분마다, matilda.cache.tag.enabled=true 일 때만)
  └─> TagRelationService.renewCoreTagsRelation()  ← DB에서 TagRelation 재계산
  └─> TilTagService.getRecentWroteTags/SubTags()  ← 최근 500일치 Tag/SubTag 조회
  └─> new TilTagRelations(...)                    ← 도메인 객체 조립
  └─> RecentTilTagsCache.updateRecentTagRelations() ← Redis에 저장

GET /api/tags/recent
  └─> RecentTilTagsCache.getRecentTagRelations()  ← Redis에서 조회
```

---

### 1. Redis 캐시 설정 ([RedisConfig.java](https://claude.ai/epitaxy/src/main/java/com/tilguys/matilda/common/config/RedisConfig.java))

| 항목      | 값                                           |
| ------- | ------------------------------------------- |
| 캐시 이름   | `tilTags`                                   |
| TTL     | **30분**                                     |
| 키 직렬화   | `StringRedisSerializer`                     |
| 값 직렬화   | `GenericJackson2JsonRedisSerializer` (JSON) |
| null 캐싱 | 비활성화 (`disableCachingNullValues`)           |

---

### 2. RecentTilTagsCache ([cache/RecentTilTagsCache.java](https://claude.ai/epitaxy/src/main/java/com/tilguys/matilda/tag/cache/RecentTilTagsCache.java))

Redis 캐시를 직접 감싸는 컴포넌트로 두 가지 레이어를 가집니다:

```
Redis (tilTags::recent:til:relations)   ← 1차 저장소
  실패 시 ↓
this.recentTagRelations (인스턴스 필드) ← 2차 폴백 (JVM 인메모리)
```

- **캐시 키**: `"recent:til:relations"` (고정)
- **Failover 로직**: Redis에서 캐시 조회 실패(예외 발생 또는 null) 시 인스턴스 변수 `recentTagRelations`로 폴백
- **주의점**: `updateRecentTagRelations()`가 최초 한 번 호출되기 전엔 폴백도 `null`이라 NPE 위험

---

### 3. TagScheduledJob ([schedule/TagScheduledJob.java](https://claude.ai/epitaxy/src/main/java/com/tilguys/matilda/tag/schedule/TagScheduledJob.java))

**캐시 갱신 주기**: `0 */30 * * * *` → **30분마다**

**조건부 활성화**: `@ConditionalOnProperty(name = "matilda.cache.tag.enabled", havingValue = "true")` → 이 프로퍼티가 없으면 배치 자체가 빈으로 등록되지 않음

**갱신 흐름**:

1. `renewCoreTagsRelation()` → `tag_relation` 테이블 전체 삭제 후 최근 500일치 TIL 기반으로 재계산 (배치: 1,000건 단위)
2. 최근 500일(`TAG_GET_START_DAY = 500`)의 Tag, SubTag 조회 + 삭제된 TIL 필터링
3. `TilTagRelations` 도메인 객체 생성
4. Redis에 `put`

---

### 4. TilTagRelations ([domain/TilTagRelations.java](https://claude.ai/epitaxy/src/main/java/com/tilguys/matilda/tag/domain/TilTagRelations.java))

캐시에 직렬화되어 저장되는 핵심 도메인 객체로 세 가지 맵을 보유합니다:

|필드|타입|의미|
|---|---|---|
|`keywordTagMap`|`Map<String, List<String>>`|코어태그 → 서브태그(키워드) 목록|
|`tagTilIdMap`|`Map<String, List<Long>>`|코어태그 → 해당 태그를 가진 TIL ID 목록|
|`tagRelationMap`|`Map<String, List<String>>`|코어태그 → 연관 태그 문자열 목록|

초기 용량을 `Math.max(16, tags.size() / 2)`로 잡아 HashMap 리사이징을 최소화합니다.

---

### 5. 태그 생성 흐름 (캐시 비갱신)

TIL 생성 → 태그 DB 저장은 캐시를 **직접 건드리지 않습니다**. 스케줄러가 30분마다 일괄 재계산하는 **배치 갱신** 방식입니다.

```
TilTagGenerateListener (AFTER_COMMIT, Async)
  ├─ createTagsDirect() 성공 → 완료
  └─ 실패 → Outbox 저장 → processEventAsync(100ms 후)
        실패 → 30초마다 PENDING 재처리 / 1분마다 FAILED 재시도
        (재시도 간격: 5분 * retryCount, 매일 새벽 2시 7일 이상 이벤트 정리)
```

---

### 잠재적 문제점

1. **캐시 콜드 스타트**: 서버 시작 직후 30분 배치 전까지 `recentTagRelations`가 `null`이므로 `/api/tags/recent` 호출 시 500 오류 발생 가능
2. **Redis TTL = 배치 주기**: TTL(30분) = 배치 주기(30분)인데, 배치가 늦어지면 TTL 만료 후 Redis가 비어 폴백(null)으로 내려갈 수 있음
3. **단일 캐시 키**: `"recent:til:relations"` 고정 키 하나로만 저장되므로 사용자별/카테고리별 분리 없이 전체 공유
4. **`renewCoreTagsRelation()` deleteAllInBatch**: TagRelation 테이블 전체 삭제 후 재삽입하는 방식으로, 배치 실행 중간에 `getRecentRelationTagMap()` 조회가 들어오면 빈 결과를 볼 수 있음 (쓰기-읽기 간 레이스)
