```
import java.util.*;

class Solution {
    public String solution(String[] participant, String[] completion) {
        Map<String,Integer> map = new HashMap<>();
        
        for(String p : participant){
            map.put(p,map.getOrDefault(p,0)+1);
        }
        for(String c : completion){
            map.put(c,map.get(c)-1);
        }
        for(String p : participant){
            if(map.get(p)!=0){
                return p;
            }
        }
        return "";
    }
}

```
시간 복잡도 O(n).



vs. 다른 풀이
```
Arrays.sort(participant);
Arrays.sort(completion);
for (int i = 0; i < completion.length; i++) {
    if (!participant[i].equals(completion[i])) return participant[i];
}
return participant[participant.length - 1];
```

## 왜 해시는 O(1)이 가능한가

배열에서 특정 값을 찾으려면 원래는 처음부터 끝까지 훑어야 해요 (O(n)). 근데 해시는 다른 전략을 씁니다.

**핵심 아이디어**: "찾는다"는 동작을 "계산한다"로 바꿔버리는 거예요.

```
"apple" → 해시함수 → 12345 → 12345 % 배열크기 → 인덱스 7
```

`HashMap`은 내부에 배열을 갖고 있고, key를 해시함수에 넣으면 그 key가 저장될 **인덱스가 바로 계산**돼요. 그래서 찾을 때도 순회 안 하고 같은 계산을 한 번 더 해서 바로 그 인덱스로 점프합니다. 배열 인덱스 접근은 O(1)이니까, 결과적으로 조회도 O(1)이 되는 거예요.

 데이터**대가는 뭐냐면** — 이 내부 배열을 미리 넉넉하게 만들어둬야 해요.가 늘어나면 배열도 키워야 하고(리해싱), 저장 안 된 빈 슬롯도 많아요. 그래서 "추가 메모리를 쓴다"는 게 이 얘기예요. 데이터 n개를 저장하는데 실제로는 그보다 훨씬 큰 배열 공간을 확보해두는 거죠.

## 왜 정렬은 O(n log n)인가

정렬은 계산으로 위치를 알아내는 게 아니라, **비교해서** 순서를 맞춰나가는 방식이에요 (병합정렬 기준으로 생각하면):

- 전체를 반으로 쪼갠다 → 또 반으로 → ... (이 깊이가 log n번)
- 각 깊이마다 합치면서 비교하는 데 n번 걸림
- 그래서 n번 × log n번 = O(n log n)

비교 기반 정렬은 "값을 직접 계산해서 위치를 알아내는" 게 아니라 값들끼리 **관계**를 따져가며 순서를 잡는 거라서, 이론적으로 비교 기반 정렬은 O(n log n)보다 빠를 수 없다는 게 증명돼 있어요 (결정트리 하한선). 대신 정렬은 원본 배열 안에서(또는 최소한의 추가 공간으로) 자리를 바꾸는 방식이라 **메모리를 거의 안 씁니다** (in-place 정렬이면 O(1) 추가 공간).

## 한 줄로 정리하면

| |해시|정렬|
|---|---|---|
|방식|계산으로 위치를 바로 알아냄|값끼리 비교해서 순서를 맞춤|
|시간|O(1) 평균|O(n log n)|
|공간|추가로 큼 (해시테이블)|거의 안 씀|
|트레이드오프 이유|"직접 점프"의 대가로 미리 넓은 공간 확보|"비교"만으로 정렬하는 대신 공간은 아낌|

그래서 실무에서도 "메모리는 넉넉한데 속도가 중요하다" → 해시, "메모리가 빠듯하거나 정렬된 상태 자체가 다른 데도 필요하다" → 정렬, 이런 식으로 선택하는 거예요.

이 감각 챙겼으면 전화번호 목록 넘어가볼까요? 이것도 해시 vs 정렬 선택이 갈리는 문제예요.
