## 규칙 레이어 — 3단계 강제 시스템

하네스의 핵심은 **"부탁이 아닌 차단"** 이야. 규칙을 3단계로 나눠서 각 단계가 실패하면 다음으로 못 넘어가게 막는 구조야.

```
┌─────────────────────────────────────────────────────┐
│  LAYER 1: Local (Pre-commit)     — 커밋 전 차단      │
│  LAYER 2: Build-time             — 빌드 전 차단      │  
│  LAYER 3: CI/CD                  — 머지 전 차단      │
└─────────────────────────────────────────────────────┘
```

### Layer 1 — Pre-commit Hook (로컬 즉각 차단)

**목적:** 나쁜 패턴이 리포지토리에 들어오기 전 차단. 에러 메시지에 수정 지침을 직접 담는다.

```
.husky/pre-commit
  │
  ├── [Frontend] ESLint (커스텀 룰 포함)
  │     └── 실패 시: "components/는 hooks/에 직접 의존 불가 → services/ 경유할 것"
  │
  ├── [Frontend] Prettier
  │
  ├── [Backend] Checkstyle
  │     └── 실패 시: "Controller는 Service 레이어만 호출 가능. Repository 직접 호출 금지"
  │
  ├── [Shared] 아키텍처 경계 린터 (custom script)
  │     └── 레이어 의존성 방향 위반 감지
  │
  └── [Docs] AGENTS.md 100줄 초과 검사
```

**커스텀 린터 핵심 — 에러 메시지가 곧 수정 지침:**

```javascript
// scripts/lint/check-arch.js
const RULES = [
  {
    pattern: /import.*from.*['"]\.\.\/repository/,
    layer: 'service',
    message: `
      [ARCH VIOLATION] Service → Repository 직접 import 감지
      ✗ 현재: import UserRepository from '../repository/UserRepository'
      ✓ 수정: Repository는 Spring DI를 통해 주입받을 것
      참고: docs/BACKEND.md#layer-rules
    `
  }
]
```

이게 원문의 핵심 포인트야. 일반 린터는 "위반"이라고만 하는데, 여기선 **에러 메시지 자체가 AI(혹은 개발자)에게 다음 행동을 알려줌.**

---

### Layer 2 — Build-time 강제 (빌드 전 차단)

**목적:** 컴파일/번들 단계에서 구조적 불변 조건 검증.

```
Frontend Build (Vite/Webpack)
  │
  ├── TypeScript strict mode
  │     └── any 사용 금지 (에러 메시지: "docs/FRONTEND.md#type-rules 참고")
  │
  ├── 순환 의존성 검사 (madge)
  │     └── 감지 시 빌드 실패 + 의존성 그래프 출력
  │
  └── Bundle size 제한
        └── 초과 시: "어떤 모듈이 얼마나 기여했는지" 리포트 출력

Backend Build (Gradle/Maven)
  │
  ├── ArchUnit 테스트 (구조적 아키텍처 강제)
  │     └── Controller→Service→Repository 방향만 허용
  │
  ├── Checkstyle (Google Java Style 기반 커스텀)
  │
  └── SpotBugs (정적 분석)
```

**ArchUnit 예시 — 아키텍처 경계를 테스트로 강제:**

```java
// 이게 원문의 "기계적 강제 적용"의 실체
@AnalyzeClasses(packages = "com.pickeat")
class ArchitectureTest {

    @ArchTest
    static final ArchRule layerRule = layeredArchitecture()
        .layer("Controller").definedBy("..controller..")
        .layer("Service").definedBy("..service..")
        .layer("Repository").definedBy("..repository..")
        .whereLayer("Controller").mayOnlyBeAccessedByLayers("Service")
        // 위반 시 메시지:
        // "Controller는 Service를 통해서만 접근 가능
        //  docs/BACKEND.md#architecture 참고"
        ;
}
```

### Layer 3 — CI/CD 파이프라인 (머지 전 최종 차단)

**목적:** 로컬을 통과해도 CI에서 한 번 더 잡고, 실패 정보를 에이전트가 읽을 수 있는 형태로 출력.

```
PR Open
  │
  ├── [Gate 1] Lint & Format 검사        ← Layer 1 재검증
  │
  ├── [Gate 2] Build 검사                ← Layer 2 재검증  
  │
  ├── [Gate 3] TDD 기반 테스트 실행
  │     ├── Unit Test (커버리지 80% 미만 시 실패)
  │     ├── Integration Test
  │     └── 실패 시: 테스트명 + 실패 원인 + 관련 파일 경로 출력
  │
  ├── [Gate 4] 문서 freshness 검사
  │     └── 코드 변경 있는데 관련 docs 변경 없으면 경고
  │
  └── [Gate 5] 아키텍처 드리프트 검사
        └── 새로운 의존성 패턴 감지 시 리포트

Merge → main
  │
  └── [Background] Garbage Collection 에이전트 트리거
        └── 패턴 스윕 → 이상 감지 시 PR 자동 생성
```

## TDD 구조 — 레이어별 테스트 작성 요령

원문 철학: **"테스트는 에이전트에게 눈을 준다"** — 실패 메시지가 다음 행동의 지침이 되어야 함.

```
테스트 피라미드
          /\
         /  \    E2E (소수, 핵심 user journey만)
        /────\
       /      \  Integration (레이어 경계 검증)
      /────────\
     /          \ Unit (비즈니스 로직 집중)
    /────────────\
```

### Backend TDD (Spring Boot)

```
테스트 작성 순서 (Red → Green → Refactor)

1. Unit Test — 비즈니스 로직
   └── given / when / then 구조 강제 (Checkstyle 룰로)
   └── Mock은 Service 레이어까지만 허용

2. Integration Test — 레이어 경계
   └── @SpringBootTest 없이 Slice Test 우선
       (@WebMvcTest, @DataJpaTest)
   └── 실제 DB는 Testcontainers 사용

3. Architecture Test — 구조 불변 조건
   └── ArchUnit으로 레이어 규칙 검증
   └── 이건 TDD라기보다 "회귀 방지 울타리"
```

**실패 메시지 설계 원칙:**

```java
@Test
@DisplayName("음식점 검색 시 비활성 음식점은 제외되어야 한다")
void searchRestaurants_excludesInactive() {
    // given: 테스트 의도가 명확하게
    // when: 단일 행동
    // then: 실패 시 "무엇이 기대됐고 무엇이 나왔는지" 출력
    assertThat(result)
        .as("비활성 음식점이 검색 결과에 포함됨 → RestaurantService#search 로직 확인")
        .doesNotContain(inactiveRestaurant);
}
```

### Frontend TDD (React)

```
1. Unit Test — 컴포넌트/훅 로직
   └── React Testing Library (구현 아닌 동작 테스트)
   └── "사용자가 보는 것" 기준으로 쿼리

2. Integration Test — 페이지 단위
   └── MSW로 API 모킹 (실제 네트워크 없이)

3. 작성 금지 패턴 (린터로 강제):
   └── enzyme, snapshot test → 구현 의존성 높음
   └── getByTestId 남용 → 접근성 속성 우선
```

## 실패 처리 플로우 — "실패를 규칙으로 전환"

원문의 핵심 루프야. 실패가 발생하면 그냥 고치는 게 아니라 **재발 방지 구조로 인코딩**하는 게 목표.

```
실패 발생
    │
    ▼
실패 원인 분류
    ├── 린트/포맷 문제     → Pre-commit 룰 추가
    ├── 아키텍처 위반      → ArchUnit/커스텀 린터 룰 추가  
    ├── 테스트 없는 버그   → 해당 케이스 테스트 추가 (TDD)
    ├── 문서-코드 불일치   → doc-gardening 에이전트 트리거
    └── 반복 패턴          → Garbage Collection 스윕 대상 등록
    │
    ▼
docs/ 또는 scripts/lint/ 에 인코딩
    │
    ▼
AGENTS.md 업데이트 (필요 시 목차만)
```

**"성공은 조용히, 실패는 시끄럽게"** 원칙:

```yaml
# .github/workflows/ci.yml
- name: Run Tests
  run: ./gradlew test
  # 실패 시 출력:
  # ✗ 실패한 테스트: RestaurantServiceTest#searchRestaurants_excludesInactive
  # ✗ 관련 파일: src/main/java/.../RestaurantService.java:47
  # ✗ 참고 문서: docs/BACKEND.md#service-layer-rules
  # → AI/개발자가 다음 행동을 즉시 알 수 있도록
```

## 요약 — 단계별 구축 순서

이걸 한 번에 다 만들 필요 없어. 원문도 실패하면서 점진적으로 쌓았다고 했으니까.

```
Phase 1 (기반): 패키지 구조 + AGENTS.md(목차) + docs/ 뼈대
Phase 2 (울타리): Pre-commit Hook + 커스텀 린터 (에러 메시지 포함)
Phase 3 (검증): TDD 구조 + ArchUnit + CI Gate
Phase 4 (자동화): Garbage Collection + doc-gardening 에이전트
```
