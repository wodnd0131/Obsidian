## 1. 기획 배경
기존 회의 기록 도구는 대체로 "받아쓰기"를 보조한다. 발화를 텍스트로 옮기고, 끝난 뒤 요약을 만들어준다. 하지만 실제로 부족한 것은 텍스트가 아니라 구조다. 회의가 끝난 뒤 참가자에게 남는 것은 긴 트랜스크립트 하나뿐이고, 그 안에서 "지금 무슨 안건이 논의 중이었는지", "각 안건이 어디서 시작돼서 어떻게 흘렀는지"는 다시 처음부터 읽어야 알 수 있다.

이 서비스는 그 지점을 정면으로 다룬다. 회의가 끝나기를 기다리지 않고, 대화가 오가는 그 순간에 내용을 안건 단위로 구조화해서 눈에 보이게 만든다.
## 2. 서비스 한 줄 정의

음성 채널의 대화를 실시간으로 텍스트화하고, 그 내용을 안건별 토픽 노드로 구조화해 마인드맵처럼 시각화하며, 노드를 클릭하면 해당 발언 지점으로 즉시 이동하고 필요하면 그 자리에서 GitHub 이슈로 전환할 수 있는 실시간 회의 구조화 서비스.

## 3. 해결하려는 문제

### 3.1 사용자의 실제 문제

사용자는 회의를 기록하지 못하는 것이 아니라, 회의 내용을 구조화하지 못한다.

- 회의 중 서기 역할을 맡으면 대화를 따라가면서 동시에 정리하기 어렵다.
- 회의가 길어지면 초반에 나온 안건이 뒤로 갈수록 흐릿해진다.
- 회의가 끝난 뒤 트랜스크립트를 다시 읽어야 무슨 얘기가 있었는지 파악된다.
- 논의된 안건이 실제 할 일(이슈, 태스크)로 이어지지 않고 회의록 안에 갇힌다.
- Discord로 진행하는 스터디, 사이드 프로젝트, 회의는 이런 기록/정리 도구의 사각지대에 놓여 있다.
	-  이러한 회의 환경에서 방향성이 통일되지 않거나, 익숙하지 않은 사용자에게 도움이 될 것을 기대한다.

## 4. 핵심 사용자

### 4.1 1차 타깃

- 사이드 프로젝트를 진행하는 개발팀
- 스터디 그룹
- Discord로 소통하는 커뮤니티 운영진
- 비동기 근무가 잦은 소규모 원격 팀

### 4.2 주요 사용자 유형

#### 서기 부담형

회의 때마다 누군가 한 명이 서기를 맡아야 하고, 그 사람은 대화에 온전히 집중하지 못한다. 이 서비스는 서기 역할 자체를 시스템으로 대체한다.

#### 사후 정리 회피형

회의 후에 회의록을 정리해야 한다는 걸 알지만 항상 미룬다. 회의가 끝나는 순간 이미 정리된 결과물이 나와 있으면 이 부담이 사라진다.

#### 실행 연결형

회의에서 나온 이야기가 실제 작업(이슈, 태스크)으로 이어지길 바라지만, 지금은 회의록과 이슈 트래커가 분리되어 있어 매번 수동으로 옮겨 적어야 한다. 이 서비스의 이슈 생성 기능이 이 유형을 직접 겨냥한다.

## 5. 제품의 핵심 매력

이 서비스의 매력은 "회의를 잘 기록해준다"는 조용한 약속에 있지 않다. 회의가 진행되는 동안 화면 위에서 대화가 살아있는 구조로 자라나는 것을 보는 경험, 그리고 그 구조에서 바로 실행으로 넘어갈 수 있다는 즉시성에 있다.

### 5.1 회의가 끝나기 전에 이미 정리되어 있다

기존 도구는 회의가 끝난 뒤에야 요약을 제공한다. 이 서비스는 대화가 오가는 15~30초 단위로 토픽 노드가 갱신되기 때문에, 회의 중간에 참가자가 "지금까지 뭐가 논의됐지"를 바로 확인할 수 있다. 늦게 들어온 참가자도 트랜스크립트를 처음부터 읽을 필요 없이 노드 구조만 보면 맥락을 따라잡는다.

### 5.2 노드 클릭 한 번으로 그 순간의 대화로 돌아간다

각 토픽 노드는 그 안건이 처음 언급된 채팅 로그 위치에 연결되어 있다. 노드를 클릭하면 채팅 로그가 해당 지점으로 스크롤되고 색상으로 하이라이트된다. "그 얘기 정확히 뭐였지"라는 질문에 검색 없이 바로 답이 된다.

### 5.3 정리에서 끝나지 않고 실행으로 이어진다

토픽 노드에서 바로 GitHub 이슈를 생성할 수 있다. 회의에서 나온 안건이 회의록 안에 머무르지 않고, 안건의 배경(발언 인용, 요약된 논의 내용)이 그대로 담긴 이슈로 전환된다. 이 지점이 기존 회의 기록 도구가 채우지 못한 "기록 이후" 구간을 메운다.

## 6. 서비스 작동 방식

### 6.1 기본 플로우

1. 사용자가 Discord 음성 채널에 봇을 초대한다.
2. 봇이 음성을 STT로 변환해 웹페이지로 실시간 전송한다.
3. 웹 화면의 채팅 로그에 발화가 순서대로 쌓인다.
4. 일정 주기(15~30초)마다 누적된 텍스트가 LLM으로 전달된다.
5. LLM이 현재 토픽 노드 상태를 갱신한다 (신규 노드 생성, 기존 노드 업데이트, 노드 병합).
6. 갱신된 노드가 마인드맵 형태로 실시간 렌더링된다.
7. 참가자가 노드를 클릭하면 해당 안건이 시작된 채팅 로그 위치로 이동하고 하이라이트된다.
8. 참가자가 특정 노드를 선택해 GitHub 이슈 생성을 요청하면, 안건 제목과 논의 내용이 담긴 이슈가 생성된다.
9. 회의가 끝나면 전체 토픽 맵과 채팅 로그가 최종 결과물로 남는다.

### 6.2 토픽 노드 등록 항목

- 노드 ID
- 안건 제목
- 안건 목차(하위 논의 포인트)
- 시작 지점(origin_message_id)
- 색상(채팅 로그 하이라이트와 매칭)
- 마지막 갱신 시각
- 연결된 GitHub 이슈 URL(생성된 경우)

### 6.3 처리 파이프라인

```
[Discord 음성]
      ↓ STT
[Discord 봇(Node.js)] --WebSocket--> [웹 프론트]
                                          ↓
                              [채팅 로그 렌더링] (메시지별 ID 부여)
                                          ↓
                          (15~30초 주기) 텍스트 델타 누적
                                          ↓
              [현재 노드 상태 + 델타] → LLM → 노드 업데이트
                    (노드에 origin_message_id, color 포함)
                                          ↓
                          [마인드맵 시각화] (실시간 노드/엣지)
                                          ↓
                    클릭 시 → 채팅 로그 스크롤 + 색상 하이라이트
                                          ↓
                        [이슈 작성 버튼] → GitHub API → 이슈 생성
```

## 7. 노드 구조화 설계

노드 구조화 로직은 이 서비스의 핵심 상품이다. 너무 잦은 갱신은 화면이 어지럽고, 너무 드문 갱신은 실시간성이 사라진다. 따라서 안정적인 상태 유지와 빠른 반응성 사이에서 균형을 잡아야 한다.

### 7.1 설계 원칙

#### 상태는 누적되어야 한다

매 주기마다 전체 트랜스크립트를 다시 넣지 않고, "현재까지 정리된 노드 상태 + 신규 텍스트 델타"만 LLM에 전달한다. 이는 토큰 비용과 응답 속도를 동시에 개선한다.

#### 노드 ID는 안정적으로 유지되어야 한다

같은 안건이 계속될 때 노드가 매번 새로 생성되면 화면이 깜빡이고 사용자가 흐름을 놓친다. 프롬프트에 기존 ID를 유지하도록 명시해야 한다.

#### 앵커(anchor)는 필수 필드다

모든 노드는 origin_message_id를 가져야 한다. 이 값이 없으면 "노드 클릭 시 채팅 로그로 이동"이라는 핵심 기능이 성립하지 않는다.

#### 병합과 분리는 완벽하지 않아도 된다

두 안건이 비슷해 보여 병합되어야 하는데 안 되거나, 반대로 갈라져야 하는데 안 갈라지는 경우가 생길 수 있다. 이는 이후 반복 개선의 대상이지, 초기 단계에서 완벽하게 풀어야 할 문제는 아니다.

### 7.2 노드 갱신 강도 단계

#### 1단계: 신규 노드 생성

새로운 주제가 감지되면 새 노드를 만든다. 가장 기본적인 동작이다.

#### 2단계: 기존 노드 갱신

이미 있는 안건에 대한 추가 발언이 감지되면 해당 노드의 outline에 포인트를 추가한다.

#### 3단계: 노드 병합

두 개의 노드가 사실은 같은 안건에 대한 이야기였다고 판단되면 하나로 합친다.

#### 4단계(스트레치): 임베딩 기반 1차 필터

노드 수가 많아지는 경우, 새 발화의 임베딩과 기존 노드 centroid의 유사도를 먼저 비교해 임계값 이상이면 LLM 호출 없이 즉시 귀속시킨다. 애매한 경우에만 LLM을 호출해 판단 비용을 줄인다.

### 7.3 노드 상태 스키마 예시

```json
{
  "current_topic": "예산안 논의",
  "agenda_items": [
    {
      "id": "a1",
      "title": "예산 배분",
      "outline": ["팀장 승인 필요", "다음 주까지 초안"],
      "origin_message_id": "msg_0042",
      "color": "#378ADD",
      "github_issue_url": null
    }
  ]
}
```

## 8. 정확성과 신뢰성 확보

이 서비스의 신뢰도는 "노드가 실제 대화 내용을 정확히 반영하는가"와 "클릭했을 때 정말 맞는 위치로 이동하는가"에 달려 있다.

### 8.1 검증 방식

#### STT 정확도 관리

발음, 잡음, 화자 중첩에 따라 STT 정확도가 떨어질 수 있다. 데모나 초기 사용 환경에서는 조용한 곳, 명확한 발화를 전제로 하며, 향후 화자 분리 기능으로 보완할 수 있다.

#### 앵커 정합성 확인

노드가 가리키는 origin_message_id가 실제로 그 안건이 처음 언급된 지점인지 LLM 응답 시점에 검증한다. 근거 메시지를 함께 반환하도록 요구하면 잘못된 앵커링을 줄일 수 있다.

#### 사람의 최종 확인

자동 생성된 이슈나 노드 요약은 초안으로 취급하고, 사용자가 이슈 생성 버튼을 누르기 전 내용을 확인하고 수정할 수 있는 단계를 둔다.

### 8.2 좋은 노드 vs 나쁜 노드

나쁜 노드:

- "이야기함"
- "논의"
- "회의 내용"

좋은 노드:

- "예산 배분 – 팀장 승인 필요, 다음 주까지 초안"
- "일정 조율 – QA 일정과 겹치는 이슈 확인 필요"

노드 제목과 목차가 구체적일수록 클릭 후 채팅 로그를 다시 읽지 않아도 맥락이 파악된다.

## 9. 주요 기능

### 9.1 실시간 채팅 로그

Discord 음성 채널의 발화가 순서대로 웹 화면에 쌓인다. 화자, 시각, 발화 내용이 함께 표시된다.

### 9.2 실시간 토픽 마인드맵

누적된 대화가 15~30초 주기로 안건 노드로 구조화되어 시각화된다. 새 안건은 새 노드로, 이어지는 발언은 기존 노드 갱신으로 반영된다.

### 9.3 노드-채팅 연동 하이라이트

노드를 클릭하면 채팅 로그가 해당 발언 위치로 스크롤되고, 노드와 동일한 색상으로 배경이 하이라이트된다.

### 9.4 GitHub 이슈 생성

선택한 노드의 제목과 목차를 기반으로 GitHub 이슈 초안을 자동 구성하고, 사용자 확인 후 실제 리포지토리에 이슈를 생성한다.

### 9.5 회의 종료 후 최종 보드

회의가 끝나면 완성된 토픽 맵을 이미지나 문서로 내보낼 수 있다.

## 10. 기술 스택

| 영역           | 기술                                  |
| ------------ | ----------------------------------- |
| Discord 봇    | `@discordjs/voice` (Node.js)        |
| STT          | STT API 또는 `faster-whisper` (로컬)    |
| 상태 갱신 로직     | LLM API (Structured output/JSON 강제) |
| 실시간 통신       | WebSocket (Socket.io / ws)          |
| 프론트 시각화      | React Flow 기반 마인드맵                  |
| (스트레치) 안건 매칭 | 임베딩 API + 경량 벡터 저장소                 |

## 11. MVP 설계

처음부터 임베딩 기반 매칭, 다국어 지원, 팀 권한 관리를 모두 완성할 필요는 없다. MVP의 목적은 "실시간으로 구조화된 시각화가 실제로 회의 내용 파악에 도움이 되는가"를 검증하는 것이다.

### 11.1 MVP 핵심 가설

참가자는 회의 종료를 기다리지 않고도 실시간 토픽 노드를 통해 논의 내용을 더 빠르게 파악하고, 회의에서 나온 안건을 더 쉽게 실행(이슈화)으로 옮긴다.

### 11.2 MVP 기능

- Discord 음성 → STT → 웹 채팅 로그 전송
- 채팅 로그 실시간 렌더링 (메시지 ID 부여)
- 주기적 LLM 호출을 통한 토픽 노드 갱신
- 마인드맵 시각화
- 노드 클릭 → 채팅 로그 이동 및 하이라이트
- GitHub 이슈 생성 (PAT 기반)

### 11.3 MVP에서 제외해도 되는 것

- 임베딩 기반 안건 매칭 1차 필터
- 화자 분리 및 화자별 색상 구분
- 다국어 STT 지원
- 팀/권한 관리 시스템
- 세션 간 검색(RAG 기반 과거 회의 조회)
- Notion, Slack 등 GitHub 외 연동

## 12. 사용자 경험 시나리오

### 12.1 사이드 프로젝트 팀 회의 시나리오

세 명이 Discord 음성 채널에서 기능 기획 회의를 한다. 봇이 채널에 들어와 있고, 웹 화면에는 채팅 로그와 빈 토픽 맵이 떠 있다.

- 회의 시작: "오늘 안건은 예산이랑 일정이에요."
- 30초 후: "예산 배분" 노드가 생성된다.
- 대화가 이어지며 "일정 조율" 노드가 추가된다.
- 한 명이 예산 노드를 클릭해 팀장 승인 관련 발언을 다시 확인한다.
- 회의 막바지에 "GitHub 이슈 연동" 안건이 나오고, 담당자가 해당 노드에서 바로 이슈를 생성한다.
- 회의가 끝난 시점에 세 개의 노드와 각각의 GitHub 이슈 링크가 남아 있다.

### 12.2 해커톤 팀 스크럼 시나리오

해커톤 기간 중 매일 짧은 스크럼을 Discord로 진행하는 팀이 있다. 회의는 10분 내외로 짧지만 매일 진행되어 누적되면 놓치는 내용이 생긴다. 이 서비스를 사용하면 각 스크럼마다 생성된 토픽 노드가 그날의 결정사항으로 남고, 필요한 것은 바로 이슈로 전환되어 팀 보드에 쌓인다.

## 13. 제품 톤과 사용자 경험 원칙

이 서비스는 "회의를 대신 기록해주는 도구"라는 인상보다 "회의 중에 함께 정리해주는 동료"에 가까운 경험을 지향한다.

### 13.1 원칙

- 방해가 되지 않아야 한다 — 화면은 계속 갱신되지만 대화의 흐름을 끊지 않는다.
- 즉시 이해 가능해야 한다 — 노드 제목만 봐도 무슨 안건인지 알 수 있어야 한다.
- 클릭 한 번으로 원본에 닿아야 한다 — 노드에서 채팅 로그로, 채팅 로그에서 이슈로 이어지는 경로가 항상 짧아야 한다.

### 13.2 화면 문구 예시

- "새로운 안건이 감지되었습니다."
- "이 안건, 이슈로 만들까요?"
- "발언 위치로 이동합니다."
- "회의가 종료되었습니다. 정리된 보드를 확인하세요."

## 14. 확산 관점

이 서비스는 화상회의 도구가 아니라 Discord 커뮤니티, 스터디, 사이드 프로젝트 팀을 타겟으로 하기 때문에 확산 경로도 이들 커뮤니티 안에서 자연스럽게 만들어진다.

### 14.1 확산 포인트

- "Discord 스터디용 실시간 회의 정리 봇"
- "회의 끝나기 전에 이미 정리되는 회의록"
- "회의 안건이 바로 GitHub 이슈가 되는 도구"

### 14.2 공유 요소

- 완성된 토픽 맵 이미지 공유
- 회의 요약 카드 공유
- 생성된 이슈 링크를 팀 채널에 자동 게시

## 15. 법적·윤리적 고려사항

이 서비스는 음성 대화를 기록하고 텍스트화하는 도구이므로 참가자 동의와 데이터 처리 원칙이 명확해야 한다.

1. 봇이 채널에 들어오는 순간 녹음 및 기록이 시작됨을 참가자에게 고지해야 한다.
2. 기록된 트랜스크립트와 생성된 노드는 회의 참가자 외에는 접근할 수 없어야 한다.
3. 회의 데이터를 모델 학습에 사용하지 않는다는 원칙을 명확히 한다.
4. 사용자가 요청하면 기록을 삭제할 수 있어야 한다.

## 16. 핵심 지표

### 16.1 제품 지표

- 세션당 생성된 노드 수

- 노드 → 이슈 전환율
- 재사용률 (동일 팀의 반복 사용)

### 16.2 효과 지표

- 회의 종료 직후 참가자의 내용 파악 소요 시간 (기존 트랜스크립트 재확인 대비)
- 노드에서 생성된 이슈의 실제 처리(완료)율
- 회의 안건이 실행으로 이어지는 비율 변화

## 17. 경쟁 서비스와 차별점

### 17.1 기존 회의 인텔리전스 도구 (Otter, Fireflies, Fathom 등)

이들은 Zoom, Meet, Teams 중심이며 선형 트랜스크립트와 사후 요약에 초점을 둔다. 이 서비스는 Discord 음성 채널을 대상으로 하며, 회의 중 실시간 구조화 시각화에 초점을 둔다.

### 17.2 Notion AI 회의 노트

Notion 생태계 안에서만 의미가 있고 독립적 가치가 낮다. 이 서비스는 특정 워크스페이스에 종속되지 않고, 대신 GitHub라는 실행 도구와 직접 연결된다.

### 17.3 일반 회의록 정리 도구

일반 도구는 회의가 끝난 뒤 요약을 만든다. 이 서비스는 회의 도중 실시간으로 구조가 만들어지고, 그 구조에서 바로 실행(이슈 생성)으로 이어진다는 점에서 다르다.

## 18. 확장 방향

### 18.1 화자 분리 및 색상 구분

발화자별로 채팅 로그 색상을 구분해 누가 어떤 안건을 제기했는지 추적한다.
discord 서버로부터 사용자 정보를 매핑가능한지 확인해보자.

### 18.2 세션 간 검색

여러 회의를 넘나드는 질의응답("저번 회의 때 이거 뭐라고 했지?")은 벡터 검색(RAG)이 적합한 영역이다. 다만 이는 세션 하나 내 실시간 구조화와는 별개 문제이므로 MVP 이후 확장 기능으로 분리한다.

### 18.3 다른 실행 도구 연동

GitHub 외에 Notion, Jira, Slack으로도 안건을 전환할 수 있도록 확장한다.

### 18.4 팀/권한 관리

여러 팀이 사용할 경우 채널별 접근 권한과 데이터 분리가 필요해진다.

## 19. 가장 중요한 제품 원칙

이 서비스는 회의를 완벽하게 받아쓰는 도구가 아니다. 회의 중 대화를 구조로 바꾸고, 그 구조에서 바로 다음 행동으로 이어지게 만드는 도구다.

핵심 원칙은 다음과 같다.

1. 실시간성이 핵심이다 — 회의가 끝난 뒤가 아니라 진행 중에 구조화되어야 한다.
2. 노드는 항상 원본 발언에 닿아 있어야 한다 — 앵커 없는 노드는 신뢰할 수 없다.
3. 정리에서 끝나지 않고 실행으로 이어져야 한다 — 이슈 생성 기능이 이 원칙의 핵심이다.
4. 화면은 대화를 방해하지 않아야 한다 — 시각화는 보조 수단이지 회의의 중심이 아니다.
5. 완벽한 클러스터링보다 데모 가능한 안정성이 우선이다 — 초기 단계에서는 일반화보다 시나리오 내 정확성을 우선한다.

## 20. 결론

이 서비스의 본질은 회의를 기록하는 것이 아니라 회의를 구조로 바꾸는 것이다.

기존 도구들은 회의가 끝나기를 기다렸다가 요약을 건넨다. 이 서비스는 대화가 오가는 그 순간에 안건을 노드로 세우고, 노드를 누르면 그 발언이 있었던 자리로 데려가고, 그 자리에서 바로 다음 할 일을 만든다. Discord라는 아무도 다루지 않은 공간에서, 회의 기록이 끝나는 지점이 아니라 실행이 시작되는 지점을 만드는 것이 이 서비스의 목표다.

잘 설계된다면 이 서비스는 "회의를 받아 적는 앱"이 아니라 "회의가 곧 작업이 되는 앱"이 될 수 있다.
