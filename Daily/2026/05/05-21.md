### 서블릿 컨테이너
서블릿 = HTTP 처리에 대한 자바 표준 인터페이스
서블릿 컨테이너 = 서블릿의 생명주기 관리자 => 톰캣
서블릿 컨텍스트에서 URL별 복수의 서블릿을 관리하였지만, 디스패처 서블릿ㅡ 프론트컨트롤러를 둠으로서
복수의 서블릿이 필요하지 않게됨.

스프링부트
톰캣{코요테 - 카탈리나 } 디스패처 서블릿 - 애플리케이션 콘텍스트(IoC 컨테이너)
#### 코요테
코요테는 http 처리 모듈
엔드포인트, 프로세서, 어뎁터로 구성
Nio는 넌블록킹 I O // NIO 엔드포인트는 Accepter thread, Puller thread, Worker Thread로구성
스프링 자체적으로도 Thread가 존재함. 이 경우 비동기, 스케쥴, 히카리cp등이 영역에 포함됨.
thread는 결국 동일 인테페이스의 구현체로 os, jvm 입장에선 동일하나, 역할이 다르고, 튜닝 방식도 다름.

#### 카탈리나
코요테로부터 전달받은 httpserviceRequest를 처리
어느 애플리케이션, 어느 서블릿이 처리할지 결정한다.
컨테이너 개층 구조 - 엔진(카탈리나) > 호스트(로컬호스트) > 컨텍스트(기본주소) > 래퍼(디스패처서블)

컨텍스트 - 클래스로더, 필터 등 mvc 개념 자체가 들어갔다?


- **Tomcat Host/Context를 선택하면**: 진정한 격리를 잃는다. JVM-level 장애에 모든 앱이 함께 죽는다. 또한 Spring Boot의 fat-jar 모델, 자동 설정, 단순한 main() 부팅을 잃는다. 외부 Tomcat에 WAR로 배포해야 하니까.
- **Nginx + 멀티 프로세스를 선택하면**: 메모리/CPU 자원을 더 쓴다. JVM 하나 띄우는 비용(warm-up 시간, JIT 캐시, 힙 오버헤드)이 N배. 작은 서비스를 여러 개 운영하면 모든 JVM마다 똑같은 Spring Context, 똑같은 라이브러리가 메모리에 N번 로드된다. 또한 프로세스 간 통신이 네트워크를 타게 되어 in-process 호출(메서드 호출 ns 단위) 대비 ms 단위로 느려진다.
- **모놀리식을 선택하면**: 격리를 잃는다. 한 모듈의 메모리 누수가 전체를 죽인다. 한 모듈만 배포 못 한다. 보안 등급이 다른 모듈이 같은 권한으로 실행된다. 하지만 운영은 가장 단순하다.
---
- **Filter**: 인증/인가, 요청/응답 압축, 인코딩, CORS, 로깅(원시 데이터), CSRF 방어 — Spring과 무관하게 "모든 요청"에 적용해야 하는 것
- **Interceptor**: 핸들러 메서드와 연관된 처리. 컨트롤러 어노테이션 검사, 핸들러 실행 시간 측정, ThreadLocal 정리 — Handler 정보가 필요한 것
[ Spring MVC ] = DispatcherServlet + 그 내부 컴포넌트들
                 (HandlerMapping, HandlerAdapter, Interceptor,
                  ArgumentResolver, ExceptionResolver, ...)
                 + @Controller, @RequestMapping 같은 어노테이션 모델

[ ApplicationContext ] = Spring MVC를 포함하는 더 큰 그릇.
                         웹과 무관한 Bean들도 여기 산다.

[ Filter ] = Servlet 스펙. DispatcherServlet 밖에서 동작.
             ApplicationContext의 Bean으로 만들 수도 있고
             순수 Servlet 컴포넌트로 만들 수도 있다.

[ Interceptor ] = Spring MVC의 일부. DispatcherServlet 안에서 동작.
                  반드시 ApplicationContext의 Bean.
